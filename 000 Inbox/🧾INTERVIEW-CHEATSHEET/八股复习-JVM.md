## 内存模型

### JVM内存模型介绍一下？

![image-20250321222404809](/Users/czl/Library/Application Support/typora-user-images/image-20250321222404809.png)

- **程序计数器（PC Register）**
  - 记录当前线程正在执行的字节码指令地址，属于线程私有。
  - 若执行的是 `Native` 方法，则 PC 为空 (`null`)。
  - 该区域不会发生 `OutOfMemoryError`。
- **Java 虚拟机栈（VM Stack）**
  - 每个线程私有，存储方法执行时的栈帧，包括局部变量、操作数栈、动态链接等信息。
  - 方法调用时创建栈帧，执行完成后销毁。
  - 可能抛出 `StackOverflowError`（栈空间不足）或 `OutOfMemoryError`（栈扩展失败）。
- **本地方法栈（Native Method Stack）**
  - 主要用于 `Native` 方法的调用，与 Java 虚拟机栈类似。
  - HotSpot 虚拟机中，本地方法栈与虚拟机栈合并。
  - 可能抛出 `StackOverflowError` 或 `OutOfMemoryError`。
- **Java 堆（Heap）**
  - JVM 内存中最大的一块区域，所有对象实例都存储在这里。
  - 启动时创建，线程共享。
  - 分代存储：新生代（Eden、Survivor From、Survivor To）、老年代。
  - 堆空间不足时抛出 `OutOfMemoryError`。
- **方法区（元空间 MetaSpace）**
  - JDK6,方法区在永久代（PermGen），和堆是分开的
  - JDK7， 逐步移除永久代
    - 大部分内容仍在**永久代**
      - 运行时常量池（Runtime Constant Pool）
      - 类的元数据（类名、字段、方法、访问权限等）
      - 静态变量（Static Variables）
      - JIT 编译后的代码
    - 部分内容被移动到**堆**
      - 字符串常量池（String Constant Pool）
      - 静态变量中的字符串（如果是字符串字面量）
      - Interned String（调用 `String.intern()` 存入的字符串）
  - JDK 1.8 之后，`方法区` 被 `元空间` 取代，使用**本地内存**存储类信息、常量、静态变量等。
  - 可选择不进行垃圾回收，空间不足时抛出 `OutOfMemoryError`。
- **运行时常量池（Runtime Constant Pool）**
  - 始终位于方法区内部，存储编译期生成的常量和符号引用。
  - JDK6， 运行时常量池 -> 方法区 -> **永久代**
  - JDK7，字符串常量池（String Constant Pool）从**永久代**移到**堆**， 运行时常量池还在永久代
  - 运行时可能动态扩展，若空间不足，会抛出 `OutOfMemoryError`。
- **直接内存**
  - JVM 之外的内存，主要用于 `NIO` 进行高速数据传输，提高 I/O 性能。
  - 受本地内存管理限制，可能抛出 `OutOfMemoryError`。

### 栈和堆的区别？

* **用途**：
  * 栈用于存储方法调用相关的信息，如局部变量、方法参数、返回地址和临时数据。每次方法调用时都会创建一个栈帧，方法执行结束后，栈帧会自动销毁。
  * 堆用于存储对象实例和数组，使用 new 关键字创建的对象会分配在堆上。

* 生命周期：
  * 栈中的数据生命周期与方法执行周期一致，方法返回后，栈帧被销毁，局部变量也随之消失。
  * 堆中的对象生命周期不确定，由垃圾回收机制管理，只有当对象没有任何引用时，才会被 gc 回收。

* 存取速度：
  * 栈的访问速度较快，采用后进先出LIFO的方式进行操作，存取高效。
  * 堆的访问速度相对较慢，因为对象分配和垃圾回收的成本较高，可能会影响性能。

* 存储空间：
  * 栈的空间较小且固定，由操作系统管理，每个线程都有自己的栈。
  * 堆的空间较大，可动态扩展，由 jvm 进行管理，堆内存过大可能导致内存溢出。

* 可见性：
  * 栈是线程私有的，每个线程都有自己的栈空间，彼此隔离，数据不可共享。
  * 堆是线程共享的，所有线程都可以访问堆中的对象。

* 异常情况：
  * 栈可能会发生 stackoverflowerror，通常由于递归调用过深或局部变量过大导致。
  * 堆可能会发生 outofmemoryerror，通常是由于创建过多对象或垃圾回收未能及时回收导致。

### 栈中存的到底是指针还是对象？

**JVM 内存模型**

- **栈（Stack）**：存方法调用上下文，局部变量
- **堆（Heap）**：存对象实例和数组

**栈存储的内容**

- **基本数据类型的值**（int、double 等）
- **对象的引用，不是对象本身**

**示例**

```java
MyObject obj = new MyObject();
```

- `obj` 存在 **栈** 上，指向 **堆** 中的对象
- 引用是固定大小的（如 64 位系统 8 字节）

**结论**

- **栈存引用，堆存对象**

### 堆分为哪几部分？

<img src="/Users/czl/Library/Application Support/typora-user-images/image-20250321230219251.png" alt="image-20250321230219251" style="zoom:50%;" />

1. **新生代**（Young Generation）

   - Eden 区：大部分新对象存放，Eden 满了触发 Minor GC

   - Survivor 区（S0、S1）
     - Minor GC 时，存活对象在 S0/S1 之间复制交换
       - Eden + 当前 Survivor 存活对象 → 复制到另一个 Survivor
       - 达到一定次数**（**MaxTenuringThreshold**）**后**，**晋升老年代

2. 老年代（Old Generation）

   - 存放生命周期长的对象

   - 对象经过多次 Minor GC 仍存活，则晋升

   - 触发 Major bGC（Full GC），进行标记-清除或标记-整理，耗时长

3. **元空间**（Metaspace，JDK 8+）

   - 存储类的元数据（类名、字段、方法等**）**

   - 使用本地内存**（**Native Memory），不在堆中

   - 避免永久代 OOM 问题

4. **大对象区**（Large Object Space）

   - 存放大数组、大对象

   - 大对象直接分配到老年代，避免 Survivor 复制消耗

   - G1 GC 设有 Humongous Objects 专区（存放超大对象）

### 程序计数器的作用，为什么是私有的？

* **程序计数器的作用**
  - 记录当前线程正在执行的字节码指令地址，指向下一条即将执行的指令
  - 线程执行时，程序计数器会自动更新，确保代码按顺序执行
  - 如果执行本地方法，程序计数器的值为null
* **为什么程序计数器是私有的**
  - 线程切换时，每个线程都需要独立记录自己的执行位置，防止混乱
  - 每个线程都有自己的程序计数器，不会相互影响，保证指令正确执行
  - 由于程序计数器仅用于记录指令位置，它是 JVM 唯一不会发生 GC 的区域

### 方法区中的方法的执行过程？

* **解析方法调用**
  - JVM 根据方法的符号引用解析出实际方法地址，若方法未解析过，则先执行解析

* **栈帧创建**

  - 调用方法前，JVM 在当前线程的虚拟机栈中为该方法创建新的栈帧

  - 栈帧存储局部变量表、操作数栈、动态链接、方法出口等信息
    - 动态链接：记录**方法调用的符号引用**，在运行时解析为实际内存地址
    - 方法出口：记录**方法执行完后**的返回地址，确保方法执行完后能回到正确位置继续执行

* **执行方法**
  - 执行方法字节码指令，涉及局部变量的读写、操作数栈操作、跳转控制、对象创建、方法调用等

* **返回处理**
  - 方法执行完成后，返回结果给调用者，并清理当前栈帧，恢复调用者执行环境



### 方法区中还有哪些东西？

**类信息**

- 存储类的结构信息，如类名、访问修饰符、父类、实现的接口等

**常量池**

- 存储类文件和接口中的常量，包括字面量、符号引用
- 运行时生成 **运行时常量池**，用于存储动态计算生成的常量

**静态变量**

- 存储 `static` 变量，这些变量在类加载时初始化

**方法字节码**

- 存储方法对应的字节码指令，即 JIT 编译后的代码

**符号引用**

- 存储类和方法的符号引用，用于解析方法调用、字段访问

**运行时常量池**

- 类加载时，从常量池中复制部分数据到方法区，供运行时使用

**常量池缓存**

- 用于提升加载效率，将常量缓存起来便于访问

### String s = new String（“abc”）执行过程中分别对应哪些内存区域？

* 如果`"abc"`原本不存在，会在堆上创建两个String对象：

  - 一个是字面量`"abc"`对应的对象，其引用会被存入常量池

  - 另一个是通过`new String()`创建的对象，其引用赋值给变量`s`

* 如果`"abc"`原本已存在，则只会在堆上创建一个新的String对象(通过`new String()`)，常量池中已有的引用会被重用

### 引用类型有哪些？

* **强引用（Strong Reference）**
  - 最常见的引用类型，例如 `A a = new A()`
  - 只要存在强引用，垃圾回收器永远不会回收该对象
* **软引用（Soft Reference）**
  - 使用 `SoftReference` 表示，用于描述非必须但有用的对象
  - 当内存不足时，垃圾回收器会尝试回收软引用指向的对象
* **弱引用（Weak Reference）**
  - 使用 `WeakReference` 表示，引用强度比软引用更弱
  - 只要发生垃圾回收，不管内存是否紧张，弱引用指向的对象都会被回收
* **虚引用（Phantom Reference）**
  - 使用 `PhantomReference` 表示，是最弱的一种引用
  - 必须和 `ReferenceQueue` 搭配使用
  - 不能通过虚引用获取对象，主要用于在对象被回收后收到通知，常用于堆外内存的管理

###  内存泄漏和内存溢出的理解？

 **内存泄漏的理解**

- 指程序运行中不再使用的对象仍然被引用，导致垃圾回收器无法回收，内存占用逐渐增多
- Java 虽有自动垃圾回收机制，但如果对象一直被引用，就不会被清理，最终可能引发内存问题

**常见内存泄漏原因**

- 静态集合：静态变量持有集合（如 HashMap、ArrayList）未清理
- 事件监听：未取消监听，导致监听器引用对象不释放
- 线程：线程未结束但持有对象引用，导致对象无法回收

**内存溢出的理解**

- 指 JVM 在申请内存时找不到足够的空间，导致程序抛出 OutOfMemoryError
- 通常发生在堆内存不足以容纳新对象时

**常见内存溢出原因**

- 大量对象创建：频繁创建对象，超过堆大小限制
- 持久引用：缓存或集合长时间持有对象，阻碍回收
- 递归调用：递归层数过深，耗尽栈空间

### 有具体的内存泄漏的例子么请举例及解决方案?

* **静态变量导致内存泄漏**
  - 问题：使用 static 修饰的集合类存储大量对象，生命周期随应用程序结束，不会自动释放
  - 解决方案：减少 static 变量使用，方法执行后清空集合，避免单例模式滥用缓存
* **ThreadLocal 使用不当导致内存泄漏**
  - 问题：ThreadLocal 的 key 是弱引用，key 被回收但 value 仍残留，无法被 GC 回收
  - 解决方案：使用完 ThreadLocal 后调用 remove 方法，避免 set(null) 方式清理，放在 finally 块中确保释放

* **资源未关闭导致内存泄漏**

  - 问题：数据库连接、文件流、网络连接等未正确关闭，导致 GC 无法清理资源

  - 解决方案：使用 try-with-resources 语法自动关闭资源，或在 finally 块中手动释放

### 有具体的内存溢出的例子么请举例及解决方案?

* **大量对象创建**：短时间内创建过多对象，超出 JVM 堆大小限制
  - 解决方案：减少对象创建，优化数据结构，调整 JVM 堆参数
* **递归过深导致栈溢出**：递归调用无终止条件，导致栈空间耗尽
  - 解决方案：确保递归有终止条件，适当调整栈大小，使用循环代替递归
* **大对象占用内存**：加载大文件、大数组等超出堆大小限制
  - 解决方案：使用流式处理，增大堆内存或使用更高效的数据结构

## 类初始化和类加载

### 创建对象的过程？

* **类加载检查**
  - 虚拟机遇到 `new` 指令时，检查这个指令的参数是否能在常量池中定位到一个类的符号引用，并且检查这个符号引用代表的类是否已经被加载，解析和初始化过
  - 若类未加载，则执行类加载过程
* **分配内存**
  - 根据类的内存大小，在堆中分配对象所需的内存
  - 内存分配后，确保对象头等结构完整
* **初始化为零值**
  - 将分配的内存空间初始化为零（不包括对象头）
  - 保证 Java 代码使用对象前，成员变量已有默认值
* **必要设置（对象头）**
  - 设定对象的元数据，如类信息、对象哈希码、GC 分代状态等
  - 若对象启用了锁机制，也会进行锁标志位初始化
* **执行构造方法 init**
  - 调用类的构造方法，对对象进行显式初始化
  - 执行 `clinit` 进行静态成员初始化
  - `new` 指令完成后返回对象引用，正式交付使用

### 对象的生命周期？

* **创建**
  - 通过 `new` 关键字在堆中分配内存
  - 调用构造方法初始化对象

* **使用**

  - 通过引用访问对象的属性和方法

  - 对象在程序运行过程中不断被使用

* **销毁**

  - 当对象不再被引用时，垃圾回收器回收其占用的内存

  - 释放内存空间，完成对象的销毁过程

### 类加载器有哪些？

* **启动类加载器（Bootstrap ClassLoader）**
  - 负责加载 Java 核心类（如 `rt.jar`）
  - 由 C++ 实现，属于 JVM 的一部分
  - 无法被 Java 代码直接引用
* **扩展类加载器（Extension ClassLoader）**
  - 继承 `ClassLoader`，用 Java 语言实现
  - 负责加载 `jre/lib/ext` 或 `Java.ext.dirs` 目录下的类
  - 由 **启动类加载器** 作为父加载器
* **系统类加载器 / 应用程序类加载器（System / Application ClassLoader）**
  - 负责加载 `ClassPath` 路径下的类
  - Java 应用默认使用的类加载器
  - 其父加载器是 **扩展类加载器**
* **自定义类加载器（Custom ClassLoader）**
  - 由开发者自定义，按需求加载类
  - 可从网络、数据库、加密文件等加载类
  - 主要用于动态扩展 Java 应用的功能
* **类加载器的双亲委派机制**
  - 类加载器不会自己加载类，而是先委派给父类加载器
  - 只有当父类加载器找不到类时，子加载器才会尝试加载
  - 确保 Java 核心类不会被错误覆盖，提高安全性

### 双亲委派模型的作用？

* 保证类的**唯一性**：所有类加载请求会先传递给启动类加载器，避免重复加载，防止用户自定义类覆盖 Java 核心类。
* 保证**安全性**：只有可信路径下的类才会被加载，防止恶意代码伪装成核心类，提高系统安全性。
* 支持**隔离**和层次划分：不同类加载器负责不同任务，如加载用户代码或扩展类，层级分明，有利于维护和扩展。
* **简化加载流程**：通过委派机制，大多数类由父类加载器加载，减少处理次数，提高加载效率。

### 讲一下类加载过程？

* 加载（Loading）
   通过类的全限定名（包名+类名）获取 `.class` 文件对应的字节流，转化为方法区的结构，生成 `java.lang.Class` 对象表示该类。
* 连接（Linking）
   包括以下三个子阶段：
  - 验证（Verification）：
     保证字节码格式正确、语义合理，保障虚拟机运行安全。分为文件格式、元数据、字节码、符号引用四种验证。
  - 准备（Preparation）：
     为静态变量分配内存并设置默认值（如 int 为 0）。`final static` 常量不会在此阶段赋值。
  - 解析（Resolution）：
     将常量池中的符号引用（如类名、字段名、方法名）解析为直接引用（内存地址或句柄）。
* 初始化（Initialization）
   执行 `<clinit>` 方法，对静态变量赋初值，运行静态代码块（由编译器收集并合并生成）。
* 使用（Using）
   类被主动使用或通过反射调用、实例化等方式参与运行。
* 卸载（Unloading）
   类满足以下条件才可卸载：
  - 所有实例被回收
  - 加载该类的类加载器被回收
  - 无引用指向该类的 `Class` 对象

## 垃圾回收

### 什么是java中的垃圾回收？如何触发垃圾回收？

* **垃圾回收（GC）** 是 Java 虚拟机自动管理内存的机制，用于释放不再被引用的对象所占内存，减少内存泄漏和管理错误的风险。
* **触发方式主要包括以下几种：**内手参与
  - 内存不足时：
     当 JVM 检测到堆内存不足、无法为新对象分配内存时，会自动触发垃圾回收。
  - 手动请求：
     虽然 GC 是自动的，但开发者可以使用 `System.gc()` 或 `Runtime.getRuntime().gc()` 主动建议 JVM 进行回收（仅为建议，无法保证立即执行）。
  - JVM 参数：
     启动 Java 应用时可通过参数控制 GC 行为，如 `-Xmx`（最大堆大小）、`-Xms`（初始堆大小）等，间接影响垃圾回收的触发时机。
  - 对象数量或内存使用达到阈值：
     垃圾收集器内部有策略监控对象创建和内存使用情况，超过某一阈值后会触发 GC。

### 判断垃圾的方法有哪些？

1. 引用计数法（Reference Counting）

   * 原理：每个对象维护一个引用计数，有引用时加 1，引用失效时减 1；当计数为 0 时，说明该对象不再被引用，可回收。
   * 缺点：无法处理循环引用的问题，例如两个对象互相引用但无外部引用，计数器不为 0，无法回收。

2. 可达性分析法（Reachability Analysis）

   * Java 虚拟机主要采用该方法。

   *  原理：以 GC Roots 作为起点，向下搜索能连接到的对象，构成引用链。如果某个对象无法通过任何 GC Roots 链接到达，则判定为不可达，可回收。

   *  GC Roots 包括以下对象：

     * 虚拟机栈中引用的对象（如局部变量表中的对象）

     - 方法区中类的静态属性引用的对象

     - 方法区中常量引用的对象

     - 本地方法栈中 JNI 引用的对象

     - 活跃线程引用的对象等

### 垃圾回收算法是什么，是为了解决了什么问题？

* 垃圾回收算法是 JVM 提供的一种自动内存管理机制，其核心目的是解决对象生命周期结束后，内存无法自动释放的问题。它主要解决以下几个方面：
  - **避免内存泄漏**
     程序中有些对象虽然不再使用，但仍被引用或没有及时清理，导致占用内存不释放，长时间积累就会造成内存泄漏。
  - **避免内存溢出**
     当程序不断创建新对象，且旧对象没有被及时回收，可能会导致内存不足，从而抛出 OutOfMemoryError 异常。
  - **提高开发效率**
     程序员无需手动释放内存，减少出错概率，降低内存管理的复杂性，专注于业务逻辑开发。
  - **提高程序稳定性和安全性**
     自动内存回收机制可以避免非法访问已释放内存等底层错误，使程序运行更加稳定和安全。
  - **提供更高级的内存模型**
     通过分代回收、区域化管理等手段，优化内存使用效率，减少停顿时间，提高性能。

### 垃圾回收算法有哪些？

* **标记-清除算法**
  - 原理：分为“**标记**”和“**清除**”两个阶段，先标记出所有需要回收的对象，再统一清除。
  - 优点：实现简单。
  - 缺点：
    - 标记和清除过程效率较低，需要遍历对象两次
    - 清除后可能产生大量内存碎片，影响大对象分配，可能导致频繁 GC
* **复制算法**
  - 原理：将内存分为两块，每次只使用一块。当这一块满了，将存活对象复制到另一块，再清空原区域。
  - 优点：不会产生内存碎片，复制过程简单快速。
  - 缺点：每次只能使用一半内存，内存利用率低。
* **标记-整理算法**
  - 原理：先标记所有存活对象，然后将它们移动到内存的一端，最后清除掉边界以外的内存。
  - 优点：解决内存碎片问题。
  - 缺点：对象移动带来额外开销，GC 停顿时间长于复制算法。
* **分代回收算法**
  - 原理：将堆划分为新生代和老年代，不同代使用不同回收策略。
  - 新生代：对象生命周期短，采用复制算法，通常将 Eden 区与 Survivor 区之间复制。
  - 老年代：对象存活时间长，采用标记-清除或标记-整理算法。
  - 晋升机制：
    - 对象在新生代经历多次 GC 后仍存活，其“年龄”每次加 1
    - 当年龄超过阈值（默认是 15，可通过 `-XX:MaxTenuringThreshold` 设置），对象会晋升到老年代

### 垃圾回收器有哪些？

* **Serial 收集器（单线程）**
  - Serial：新生代收集器，使用**复制**算法，适用于单核 CPU，Client 模式
  - Serial Old：老年代收集器，使用**标记-整理**算法，是 Serial 的老年代版本
* **ParNew 收集器（多线程）**
  - ParNew：新生代收集器，使用**复制**算法，是 Serial 的多线程版本
    - 支持与 CMS 配合使用，适用于多核 CPU 的 Server 模式
* **Parallel 系列收集器（吞吐量优先）**
  - Parallel Scavenge：新生代收集器，使用**复制**算法，注重吞吐量和 CPU 利用率
    - 适用于后台任务多、交互要求低的场景（如日志系统、定时处理系统）
  - Parallel Old：老年代收集器，使用**标记-整理**算法，是 Parallel Scavenge 的老年代版本
* **CMS（Concurrent Mark Sweep）收集器**
  - 类型：老年代收集器，使用**标记-清除**算法，支持并发回收
  - 特点：低停顿时间，适用于对响应速度要求高、交互频繁的 B/S 系统
  - 缺点：容易产生内存碎片，已在 **JDK 9 被标记为废弃**
* **G1（Garbage-First）收集器**
  - 类型：面向服务端应用的新一代收集器，覆盖新生代和老年代，统一管理整个 Java 堆
  - 算法：**混合使用复制算法和标记-整理算法**，避免内存碎片
  - 特点：
    - 并发 + 并行结合
    - 可以精确控制最大停顿时间（通过 `-XX:MaxGCPauseMillis` 设置）
    - JDK 7 引入，JDK 9 起成为默认收集器

### G1垃圾回收流程

* **基本概念**
  - G1 是面向服务端的垃圾回收器
  - 将堆划分为多个大小相等的 Region（如 1MB ~ 32MB）
  - 新生代、老年代和大对象都在这些 Region 中分布
  - 回收以 Region 为单位进行，而非传统整代回收
* **回收类型**
  - Young GC（年轻代回收）
    - 触发时机：Eden 区满时
    - 回收范围：Eden + From 区中的对象
    - 使用复制算法
    - 流程：
      - **初始标记**（STW）：标记 GC Roots 可达对象
      - 复制存活对象到 To 区或老年代空 Region
      - 清空 Eden 和 From 区
  - Mixed GC（混合回收）
    - 触发时机：老年代占用超过阈值（如 InitiatingHeapOccupancyPercent）
    - 回收范围：年轻代 + 一部分老年代
    - 使用标记-整理算法
    - 流程：
      - **初始标记**（STW）：标记 GC Roots 可达对象
      - 并发标记：与应用线程并发执行，追踪可达性
      - **再标记**（STW）：标记并发期间新创建或变动引用的对象
      - **清理阶段**（STW）：标记无存活对象的 Region，准备回收
      - **拷贝（整理）阶段**（STW）：将存活对象搬迁到新 Region，释放碎片空间
* 停顿控制机制
  - G1 支持设定最大停顿时间（如 -XX:MaxGCPauseMillis=200）
  - 收集器根据历史数据预测每个 Region 的回收耗时
  - 按权重优先选择回收收益高、回收成本低的 Region
* Region 类型
  - Eden：新对象分配区
  - Survivor：幸存对象缓冲区
  - Old：晋升对象所在区域
  - Humongous：大对象区域，占用多个连续 Region
  - Free：空闲可用的区域
* 总结
  - Young GC 使用复制算法
  - Mixed GC 使用标记-整理算法
  - 整个堆按 Region 管理，目标是减少停顿、提高吞吐

	### CMS垃圾回收流程？

* 使用范围
  - 主要用于老年代的垃圾回收
  - 新生代通常配合 ParNew 收集器使用
* 回收目标
  - 尽可能减少停顿时间（低延迟），适合对响应时间敏感的应用
* 工作模式
  - 基于**标记-清除算法**
  - 引入多阶段并发机制，减少 Stop-The-World（STW） 时间
* 回收流程（共 6 个阶段）：
  - **初始标记（Initial Mark）**
    - STW
    - 从 GC Roots 出发，标记直接可达的对象
    - 时间很短
  - **并发标记（Concurrent Mark）**
    - 与应用线程并发执行
    - 遍历整个对象图，找出所有可达对象
    - 执行时间较长，不会暂停应用
  - **预清理（Concurrent Preclean）**
    - 阶段位置
      - 处于 CMS 回收流程中“并发标记”之后、“重新标记”之前
    - 执行方式
      - 与用户线程并发执行
      - 不会引发 Stop-The-World（STW）停顿
    - 主要目的
      - 在不暂停应用线程的前提下，**提前处理并发标记期间发生的引用变动**
      - 尽量减少“重新标记”阶段的 STW 停顿时间
    - 依赖机制：写屏障和卡表
      - **写屏障（Write Barrier）**：JVM 在对象引用变更操作时，自动记录引用变动信息
      - **卡表（Card Table）**：JVM 用来追踪“哪些内存区域的引用被修改过”，每个卡片对应一小段堆内存（如 512 字节）
      - 当引用发生修改时，写屏障会将对应卡片标记为“脏卡”（Dirty Card）
    - 预清理做的事情
      - GC 线程扫描卡表中所有被标记为脏的区域
      - 重新检查这些区域的对象引用
      - 对于新增引用对象，**进行补充标记**
      - 对于引用变动的情况，记录变动信息供重新标记阶段进一步处理
    - 效果和意义
      - 提前处理一部分变动引用，降低后续重新标记的工作量
      - 提升标记准确性
      - 有助于缩短重新标记的 STW 停顿时间
    - 可选项
      - 可通过参数开启或关闭：`-XX:+CMSPrecleaningEnabled`（默认开启）
  - **重新标记（Remark）**
    - STW
    - 修正并发标记期间发生引用变动的对象状态
    - 这是 CMS 中最重要的一次 STW，时间相对较长
  - **并发清除（Concurrent Sweep）**
    - 与应用线程并发执行
    - 清除不可达对象，释放内存
    - 会产生浮动垃圾（此时的应用线程产生的）
  - **重置（Concurrent Reset）**
    - 重置内部状态，为下一次 GC 做准备
    - 与应用线程并发执行
* 特点总结：
  - 多阶段并发，STW 时间短
  - 采用标记-清除算法，**不会移动对象**，易产生内存碎片
  - 会产生浮动垃圾，需要预留内存
  - 若清除阶段内存不足，可能触发 **Concurrent Mode Failure**，退化为 Serial Old 收集器

### minorGC、majorGC、fullGC的区别，什么场景触发full GC？

* minor GC（也称 young GC）
  - 回收范围：新生代（Eden 区和一个 Survivor 区）
  - 触发条件：Eden 区满时触发
  - 算法：复制算法
  - 特点：频繁发生，回收速度快，停顿时间短，对程序影响小
* major GC（也称 old GC）
  - 回收范围：老年代，不一定包含新生代和元空间
  - 触发条件：老年代空间不足，或者年轻代对象晋升速度过快
  - 算法：通常使用标记-整理或标记-清除
  - 特点：回收频率低，但停顿时间较长；在部分收集器中实际等价于 full GC
* full GC
  - 回收范围：整个堆，包括新生代、老年代、元空间（方法区、类元信息）
  - 特点：最重型的一次 GC，停顿时间最长，对应用影响最大
  - 注意：在 Serial、Parallel 等收集器中，major GC 往往就是 full GC

* 触发 full GC 的常见场景：
  - 显式调用 `System.gc()` 或 `Runtime.getRuntime().gc()`（JVM 有权忽略）
  - 老年代空间不足，且无法通过普通 major GC 回收足够空间
  - 永久代（JDK 8 前）或元空间（JDK 8 后）空间不足
  - 新生代 GC 后对象晋升失败，但老年代也没有足够空间容纳
  - CMS 回收失败或出现 concurrent mode failure
  - G1 收集器在 mixed GC 无法满足内存需求时自动触发 full GC
  - 类加载过多导致元空间膨胀
  - 强引用大量堆对象，且无法通过普通 GC 释放

### 垃圾回收器 CMS 和 G1的区别？

- 使用范围不同
  - CMS 只负责老年代，需要搭配 ParNew 或 Serial 用于新生代回收
  - G1 统一管理新生代与老年代，不依赖其他收集器
- 停顿时间模型不同
  - CMS 追求最小化停顿，尽可能并发执行 GC
  - G1 支持通过 `-XX:MaxGCPauseMillis` 等参数控制停顿时间，具有可预测性
- 内存碎片处理不同
  - CMS 使用标记-清除算法，回收后不整理空间，易产生碎片
  - G1 使用标记-整理算法，回收时会移动对象，避免碎片问题
- 回收过程不同
  - CMS 回收过程分为初始标记、并发标记、重新标记、并发清除
  - G1 将堆划分为多个 Region，以 Young GC 和 Mixed GC 两种方式回收，采用按 Region 增量式回收方式
- 是否会产生浮动垃圾
  - CMS 在“并发清除”阶段仍允许用户线程运行，此时创建的新对象可能变为“浮动垃圾”，只能等下次 GC 再清除
    - 因此 CMS 必须预留足够内存以容纳这些浮动垃圾
    - 若预留不足，将触发 `Concurrent Mode Failure`，退化为单线程 Serial Old 收集器，效率下降
  - G1 不存在浮动垃圾的问题，其并发阶段不会立即清除新对象
    - 用户线程新创建的对象，会在下一次 GC 中统一处理
    - 若内存预留不足同样可能出现 Full GC，但不会退化为串行 GC

### G1回收器的特色是什么？

* 引入“分区（Region）”概念
  - 将整个 Java 堆划分为多个大小相等的 Region，而不再严格区分新生代与老年代
  - 弱化传统的代（young/old）概念，实现更加灵活的内存管理
* 增量回收、均衡资源利用
  - 在每个 GC 周期中，根据 Region 的垃圾比例、回收成本等信息选择“回收收益最大”的部分区域进行回收
  - 提高内存回收的性价比，避免集中式清理造成大停顿

### GC只会对堆进行GC吗？

* 堆（heap）
  - 用于存储对象实例
  - 大部分 gc 操作都发生在堆上
  - gc 会清除堆中不再被引用的对象，释放内存空间
* 方法区（method area）或元空间（metaspace）
  - 用于存储类信息、静态变量、常量池等
  - gc 会清除其中不再使用的类信息、无效常量等
  - 类信息的回收通常发生在类卸载（class unloading）时，例如动态加载类、热更新系统等
