## Socket
### 网络视角
* Socket 是**应用层**与**TCP/IP 协议族**通信的中间抽象层，一个网络连接是由一堆 Socket 决定的, Socket 就是这 5 个要素在操作系统中的**编程接口**。
$$\{ \text{源IP}, \text{源端口}, \text{目的IP}, \text{目的端口}, \text{协议(TCP/UDP)} \}$$
#### 生命周期TCP
*  服务器端 (Server) - 被动等待
	1. **`socket()`**：创建 Socket
	2. **`bind()`**：绑定 IP 和 端口，比如 6379
	3. **`listen()`**：进入监听状态
	4. **`accept()`**：返回一个新的 Socket 专门用于和这个客户端说话
*  客户端 (Client) - 主动发起
	1. **`socket()`**：创建 Socket
	2. **`connect()`**：发起 TCP 三次握手
	3. **`write()`**：发送请求

服务器返回新的 Socket 处理和客户端的连接，即被 Linux 当成 FD 处理。
### OS视角
* Socket本质上是一个文件，可以用FD[[05-死锁与资源管理#FD的分类]]表示
* 因此`read(), write(), close()`完全通用语 Socket
## Unix5种IO模型
- 阻塞式 I/O 
- 非阻塞式 I/O
- I/O 复用 (select 和 poll)
- 信号驱动式 I/O (SIGIO)
- 异步 I/O (POSIX 的 aio_ 系列函数)
输入操作包含**两个阶段**：
1. 等待数据准备好（网络中到达）
2. 从内核向进程复制数据（数据从内核缓冲区复制到应用进程缓冲区）
### 阻塞式 I/O Blocking I/O
* 当进程发起一个 IO 操作时，如果数据尚未准备好，进程会被系统挂起（进入睡眠状态），直到操作完成或发生错误
* 在输入操作两个阶段都会被阻塞，即
	* 在数据等待阶段，进程调用系统调用等待数据，数据到达之前，进程都是被挂起（blocked）的
	* 在复制阶段， 进程在内核态将数据从缓冲区拷贝到应用缓冲区，进程在内核态进行操作，用户进程仍挂起
### 非阻塞式 I/O
* **特点**：通知内核，当 IO 需要将进程挂起（Sleep）才能完成时，不要挂起而是返回一个错误。进程持续轮询（polling）内核以查看某个操作是否完成
* 两个阶段
	* 应用进程保持运行（running）状态在一个循环中反复查看数据是否就绪
	* 一旦数据就绪， 进程在内核态将数据从内核缓冲区拷贝到进程缓冲区，进程会被短暂阻塞 （Blocked）
### IO复用模型
* IO 复用即在单个线程中同时监听多个 FD 的方法
#### select
* 数据结构
	* 使用 BitMap 来表示FD 集合，即`fd_set`，每一个位代表一个FD，默认最大 1024
* 流程
	* 用户进程创建 fd 集合，将需要监控的 FD 设置为 1
	* 调用 select系统调用，将 set 从用户空间拷贝到内核空间
	* 内核遍历这些 set 位图，调用每个 FD 对应驱动程序的 poll 处理函数，检查该 fd 是否就绪（如缓冲区是否有数据）
		* 若有就绪的 FD，内核在位图中标记该位，并返回
		* 若没有，如果设置了超时时间，进程会进入睡眠；若有数据中断或超时发生，进程被唤醒，内核再次遍历检查
	* 内核将修改后的位图，从内核空间拷贝回用户空间
	* select 返回 后，用户进程并不知道哪个 FD 准备好了，必须再次遍历整个 fd_set来判断具体的 FD 是否可以进行 IO
* 缺点
	* FD 数量限制
	* fd_set要在用户态和内核态之间来回拷贝
	* 内核遍历 FD 是线性的时间线复杂度为O(n)
	* 每次调用后位图会被内核修改，用户进程在下次调用前要重置fd_set


#### epoll
* 数据结构
	* 红黑树
		* 存储所有需要监控的 FD，通过`epoll_ctl()`注册的 FD 会被插到这颗树中
	* 就绪链表 
		* 一个双向链表，存储已经就绪的 FD
* 流程-三个核心函数
	- `epoll_create()` 创建 epoll 实例，初始化红黑树和就绪链表。
	- `epoll_ctl()` 向 epoll 注册/修改/删除事件。
		- 注册 FD 时，内核会为该 FD对应的驱动程序注册一个回调函数
	- `epoll_wait()` 进程调用此函数进入阻塞状态，等待事件发生
		- 当某个 FD 发生 IO 时，硬件终端处理程序会调用之前注册的回调函数，该函数直接将该 FD 挂到就绪链表。
		- `epoll_wait()`只检查就绪链表是否为空，如果不为空，直接将链表中的数据拷贝到用户空间提供的数组中。
- 两种触发模式
	- LT 水平触发
		- 只要缓冲区存在数据，内核就会不断通知
	- ET 边缘触发
		- 只有缓冲区状态发生变化，内核才会通知

## 网络编程事件处理模型
### Reactor模型（同步 IO）
* 流程
	- **注册 (Register)**：应用程序将 Socket 及其感兴趣的事件（如 `Accept`、`Read`）注册到 Reactor。
	- **监听 (Monitor)**：Reactor 运行事件循环，调用 `select/epoll` 阻塞等待。
	- **就绪 (Ready)**：当有数据到达内核缓冲区，内核通知 Reactor 该 Socket 已就绪（可读）。
	- **分发 (Dispatch)**：Reactor 停止阻塞，将就绪事件分发给对应的 Handler。
	- **同步执行 (Handle)**：Handler 亲自调用 `read` 系统调用，将数据从内核缓冲区拷贝到用户缓冲区，然后执行业务逻辑。
* 变体
	* 单 Reactor 单线程如Redis：所有监听分发读写业务逻辑都在一个线程完成
	* 单 Reactor 多线程：Reactor 负责监听和读写，业务逻辑交给 Worker 线程池处理
	- 主从 Reactor 多线程：MainReactor 仅负责处理连接请求，SubReactor 负责处理已建立连接的读写，业务交给线程池（如 Netty）。
### Proactor模型（异步 IO）
基于事件完成通知的异步 IO 模型
* 流程
	- **发起 (Initiate)**：应用程序调用异步读操作（如 `aio_read`），并向内核提供用户缓冲区地址和回调函数，应用程序不等待结果，继续处理其他事务。
	- **内核处理 (Kernel Execution)**：内核等待数据并将其拷贝到应用程序提供的用户缓冲区，无需应用线程参与。
	- **通知 (Notify)**：拷贝完成后，内核产生一个“完成事件”，将其放入完成队列。
	- **回调 (Callback)**：**Proactor**（事件循环）从队列中获取完成事件，并触发 **Completion Handler**。
	- **直接消费**：Handler 直接处理缓冲区里的数据，不再需要执行任何 I/O 读取操作。
- Proactor 最根本的就是将数据拷贝从 APP 放到了内核上，使得该操作不再是同步的，使得 CPU 可以几乎全量用于业务逻辑。
