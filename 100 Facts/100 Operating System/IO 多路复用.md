## 思想
IO多路复用是一种高效处理多个IO时间的机制，允许一个（或少量）线程同时监听多个IO[[文件描述符FD]]，一旦有IO就需，就被唤醒去处理，避免为每个链接都创建一个线程或进程。

## 流程
- 应用程序通过一个多路复用接口（如 `select`、`poll`、`epoll`）将一批文件描述符（socket）注册到内核；
- 内核监视这些描述符；
	- PS 内核的关键数据结构
		- FD table
		- File Object：即 fd 指向的内核文件对象
		- 内核 IO 队列：当每一个 FD 状态发生变化，内核会将该事件放入该队列
- 当其中任意一个就绪（可读/可写）时，内核通知应用；
- 应用再去对这个就绪的描述符执行 `read` 或 `write`。

## 机制
 1. `select`（最早期）
	- 使用一个 **fd_set 位图** 存储关注的文件描述符；
	- 内核轮询，返回就绪的；
	- 缺点：
	    - 内核限制了 fd_set的大小位 1024，越界代价自己承担
	    - 每次调用都要拷贝整个 fd 集合，性能差
 2. `poll`
	- 使用链表存储 fd，没有数量上限；
	- 每次都遍历整个列表；
	- 依然是 **O(n)** 的线性扫描
 3. `epoll`（Linux 下主流）
	- 使用事件驱动机制，注册后不必每次传递 fd；
	- 内核维护就绪队列；
	- 只有真正就绪的 fd 才会返回；
	- **时间复杂度 O(1)**，性能远优于 `select`/`poll`。
	- 使用时显示梨花一个 epoll 然后存入 fd，fd 实际放在内核的红黑树中。
