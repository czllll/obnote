位于 `java.util.concurrent.locks` 包下
## 内部结构
内部有两个重要的内部类，`NonfairSync`和`FairSync`都继承自`Sync`(`Sync`继承自[[深入理解AQS]])
## 实现原理
### 获取锁 (lock / acquire) 的过程
当我们调用 `reentrantLock.lock()` 方法时，实际上是调用了内部 `sync` 对象的 `acquire(1)` 方法。这个过程可以分为两大步：**尝试获取锁** 和 **获取失败后的处理**。
#### 尝试获取锁 (`tryAcquire`)
这是 `ReentrantLock` 实现的核心逻辑，公平锁和非公平锁在此处的实现不同。
- **非公平锁 (`NonfairSync`)**
    1. 直接尝试 CAS 修改 `state`：当一个线程调用 `lock()` 时，不管等待队列中是否有其他线程在排队，它都会直接尝试通过 CAS 将 `state` 从 0 修改为 1。
    2. 如果成功：表示它“插队”成功了，立即获取到锁，并将锁的持有者设置为当前线程。
    3. 如果失败：说明锁已经被其他线程持有。这时会检查持有锁的线程是否是当前线程。
    4. 检查是否重入：如果是当前线程持有锁，就将 `state` 的值加 1，完成一次**重入**。
    5. 获取锁彻底失败：如果 `state` 不为 0 且持有者不是当前线程，则 `tryAcquire` 失败。
- **公平锁 (`FairSync`)**
    1. 检查等待队列：与非公平锁不同，公平锁在尝试获取锁之前，会先检查等待队列的头部是否有其他线程正在等待（通过 `hasQueuedPredecessors()` 方法）。
    2. 如果队列为空或自己是队首：才会去尝试通过 CAS 修改 `state`。后续的重入逻辑与非公平锁相同。
    3. 如果队列中有其他线程：则直接返回失败，放弃获取锁。这保证了排在前面的线程能优先获取到锁，体现了“公平”。
**小结**：非公平性主要体现在新来的线程有机会在等待队列中的线程被唤醒之前抢占锁。这虽然可能导致“饥饿”，但在高并发下能减少线程上下文切换，所以吞吐量通常更高。
#### 获取失败后的处理 (`acquireQueued`)
如果 `tryAcquire` 失败了，AQS 框架会接管后续流程：
1. 创建节点并入队：将当前线程封装成一个 `Node` 对象，并通过 CAS 操作安全地将其添加到等待队列的尾部。
2. 自旋与阻塞：线程进入一个“自旋”循环。在每次循环中，它会检查自己是否是队列的头部节点的下一个节点（即实际上的队首，因为头节点是虚节点）。如果是，它会再次尝试调用 `tryAcquire` 获取锁。
3. 挂起线程 (`park`)：如果再次尝试失败，或者它发现自己不是队首，线程就会调用 `LockSupport.park(this)` 将自己挂起，等待被前一个节点唤醒（前提是当前线程 CAS 将前面节点的的waitStatus改为 SIGNAL 即-1）。这样就避免了 CPU 的空转，提高了效率。
### 释放锁 (unlock / release) 的过程
当我们调用 `reentrantLock.unlock()` 方法时，实际上是调用了内部 `sync` 对象的 `release(1)` 方法。这个过程相对简单：
1. 调用 `tryRelease`：这个方法会检查当前线程是否是锁的持有者。如果不是，会抛出 `IllegalMonitorStateException`。
2. 状态减一：如果是持有者，就将 `state` 的值减 1。这对应了锁的重入概念，只有 `state` 减到 0，锁才被真正释放。
3. 判断是否完全释放：
    - 如果 `state` 减 1 后仍然大于 0，说明只是完成了一次重入的释放，锁还未完全释放，`unlock` 方法直接返回。
    - 如果 `state` 减 1 后等于 0，说明锁已经被完全释放。此时会将锁的持有者设置为 `null`。
4. 唤醒后继节点：在锁被完全释放后，会查找等待队列中头节点的下一个节点（即等待时间最长的那个节点），并调用 `LockSupport.unpark()` 唤醒该节点的线程，让它*有机会*重新尝试获取锁。（有机会指的是可能会输给刚刚使用lock()方法的线程）
