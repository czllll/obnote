# 本地事务
* 概念
	* 仅操作单一事务资源的、不需要全局事务管理器进行协调的事务
* 场景
	* 适用于单个服务使用单个数据源的场景
* 依赖
	* 直接依赖于**数据源**本身提供的事务能力来工作的，事务的开启、终止、提交、回滚、嵌套、设置隔离级别，乃至与应用代码贴近的事务传播方式，全部都要依赖底层数据源的支持才能工作
## 实现原子性和持久性
* 是什么？
	* 原子性保证了事务的多个操作要么都生效要么都不生效，不会存在中间状态
	* 持久性保证了一旦事务生效，就不会再因为任何原因而导致其修改的内容被撤销或丢失
* 问题？
	* 写入存在中间状态以及崩溃无法避免，有下列两个情形
		* 未提交事务，写入后崩溃：程序还没修改完所有数据，但是已经有写入db了，此时崩溃，也就是说*操作并未完成*；db应该*有能力*回滚修改的数据；保证原子性
		* 已提交事务，写入前崩溃：程序已经修改完所有数据，但是db还未完全写入，此时崩溃，也就是说*操作其实已经完成*了；db应该*有能力*将来不及落盘的数据重新写入；保证持久性
	* 写入中间状态与崩溃都是无法避免的
* 解决？
	* 只能在崩溃后采取恢复的补救措施，叫崩溃恢复（Crash Recovery）
	* “Commit Logging”（提交日志）
		* **步骤**
			* 不直接修改db数据，而是先以*日志*方式记录到磁盘
			* 日志记录全部都安全落盘，数据库在日志中看到代表事务成功提交的“提交记录”（Commit Record）
			* 根据日志上的信息对真正的数据进行修改，修改完成后，再在日志中加入一条“结束记录”（End Record）表示事务已完成持久化
		* 为什么能保证原子性和持久性？
			* 日志一旦成功写入 Commit Record，那整个事务就是成功的，即使实际修改db崩溃了，也能重新继续修改，保证持久性；
			* 若日志没有成功写入 Commit Record就崩溃，那么说明事务失败，系统重启会看到一部分没有commit record的日志，标记为回滚即可。保证原子性
		* **缺陷**？
			* 所有对数据的真实修改都必须发生在事务提交以后，即日志写入了 Commit Record 之后，对性能不友好，所以引出了*Write-Ahead Logging*
	* Write-Ahead Logging（提前写入日志）
		* 何时写入变动数据按照事务提交的时间点有两种情况
			* FORCE/NO-FORCE
				* 事务提交后，是否要同时完成变动数据的写入
			* STEAL/NO-STEAL
				* 事务提交前，是否允许变动数据提前写入
		* Commit Logging 允许 NO-FORCE，但不允许 STEAL；Write-Ahead Logging 允许 NO-FORCE，也允许 STEA
		* WAL的使用STEAL但是不会错误解决方法：增加了Undo Log当变动数据写入磁盘前，必须先记录UndoLog，注明修改了哪个位置的数据从什么修改成什么，以便在崩溃或者事务回滚对提前写入的数据变动进行删除；之前记录用于崩溃恢复的重演日志即为Redo Log
		* **步骤**
			* 分析：确定哪些事务在崩溃前未完成、哪些页面被修改但未刷盘（dirty），从而确定恢复工作需要关注哪些事务和数据页
			* 重做：**重复崩溃前已成功完成的操作**，恢复数据库至崩溃瞬间的状态
			* 回滚阶段（Undo Phase）：撤销未完成事务的影响，恢复到一致状态。
## 实现隔离性
* 没有并发就是串行就是天然同步的。
* 串行化：对事务所有读、写的数据全都加上读锁、写锁和范围锁即可做到`可串行化`
* 可重复读：对事务所涉及的数据加读锁和写锁，且一直持有至事务结束，但不再加范围锁
* 读已提交：事务涉及的数据加的写锁会一直持续到事务结束，但加的读锁在查询操作完成后就马上会释放
* 读未提交：对事务涉及的数据只加写锁，会一直持续到事务结束，但完全不加读锁

* MVCC
	* 思路：数据库的任何修改都不会直接覆盖之前的数据，而是产生一个新版副本与老版本共存，以此达到读取时可以完全不加锁的目的

# 全局事务
* 一种在分布式环境中仍追求强一致性的事务处理方案，不适用于微服务系统，几乎仅应用于单服务多数据源
* XA架构
	* 定义了全局的事务管理器和局部的资源管理器
	* **过程**：（2PC两阶段提交）
		* 准备阶段
			* 协调者询问事务的所有参与者是否准备好提交，参与者如果已经准备好提交则回复 Prepared并锁定相关资源，否则回复 Non-Prepared
			* 即不写commit record的在redo log中记录全部事务提交操作所要做的内容
		* 提交阶段
			* 协调者如果在上一阶段收到所有事务参与者回复的 Prepared 消息，则先自己在本地持久化事务状态为 Commit，在此操作完成后向所有参与者发送 Commit 指令，所有参与者立即执行提交操作；否则，任意一个参与者回复了 Non-Prepared 消息，或任意一个参与者超时未回复，协调者将自己的事务状态持久化为 Abort 之后，向所有参与者发送 Abort 指令，参与者立即执行回滚操作
	* 2PC的问题
		* 单点问题：协调者挂了所有参与者必须等待
		* 性能问题：所有参与这要经历两次远程服务调用，三次数据持久化（准备阶段写redo log，协调者做状态持久化，提交阶段在日志写入commit record）
		* 一致性风险：
			* 宕机：“[FLP 不可能原理](https://en.wikipedia.org/wiki/Consensus_\(computer_science\)#Solvability_results_for_some_agreement_problems)”，证明了如果宕机最后不能恢复，那就不存在任何一种分布式协议可以正确地达成一致性结果；
			* 网络：网络不稳定导致协调者在明确收到参与者的事务是可以提交的以后协调者会先持久化事务状态
	* 3PC
		* 阶段
		* CanCommit、PreCommit，把提交阶段改称为 DoCommit 阶段，多了一个询问阶段，相当于分解了一下准备阶段的操作，因为2PC的准备阶段负载太高了， 如果有一个参与者没法完成，那么相当于所有参与者做了一次白用功。

# 分布式事务
* 在分布式环境下的事务处理机制
## CAP 和 ACID
* CAP：Consistency、Availability、Partition Tolerance Theorem
* 一个分布式的系统中，涉及共享数据问题时，以下三个特性最多只能同时满足其中两个
* 一致性（Consistency）：代表数据在任何时刻、任何分布式节点中所看到的都是符合预期的。
* 可用性（Availability）：代表系统不间断地提供服务的能力
	* 可靠性（Reliability）
	* 可维护性（Serviceability）
* 分区容忍性（Partition Tolerance）：代表分布式环境中部分节点因网络原因而彼此失联后，即与其他节点形成“网络分区”时，系统仍能正确地提供服务的能力
* 放弃
	* CA，放弃P，即假设节点之间的通信永远可靠；
	* CP，放弃A，即假设一旦网络发现分区，节点之间的信息同步时间可以无限制延长
	* AP，放弃C，通常做法，因为P是网络属性，A是建设目的
## 柔性事务
* 相较于使用ACID的事务， 我们把其他常见的分布式事务叫柔性事务
### 可靠事件队列
* 编程时：一个事务内的事件按照出错概率大小安排操作顺序
* 定义：保证消息/事件在传递过程中不会丢失，不会重复或顺序错乱的事件队列机制，常用于分布式系统中的异步通信和最终一致性保障
* 方法：MQ
### TCC事务
* Try-Confirm-Cancel
* 可靠消息队列无法保证隔离性
* **阶段**
	* - **Try**：尝试执行阶段，完成所有业务可执行性的检查（保障一致性），并且预留好全部需用到的业务资源（保障隔离性）。
	- **Confirm**：确认执行阶段，不进行任何业务检查，直接使用 Try 阶段准备的资源来完成业务处理。Confirm 阶段可能会重复执行，因此本阶段所执行的操作需要具备幂等性。
	- **Cancel**：取消执行阶段，释放 Try 阶段预留的业务资源。Cancel 阶段可能会重复执行，也需要满足幂等性。
### SAGA事务
* TCC侵入性太强了
* SAGA 把一个**长事务**拆成多个**本地短事务**，每个短事务有对应的**补偿事务**：
	- **正向事务（Tx）**：正常的业务操作。
	- **补偿事务（Cx）**：与 Tx 相反的操作，用于回滚该步的业务结果。
SAGA 的一致性保证是**通过执行补偿事务来回退已完成的步骤**，从而实现最终一致性。
