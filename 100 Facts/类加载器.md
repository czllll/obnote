* **作用**
	* 就是将 字节码文件 加载到 JVM 内存中
* JVM有自己的类加载过程，默认是使用[[双亲委派]]模型来加载一个类
* 顺序
```java
	// JDK8
	BootstrapClassLoader
	    ↑
	ExtClassLoader (扩展类加载器)
	    ↑
	AppClassLoader (应用类加载器)
	    ↑
	自定义ClassLoader

```
```java
	// JDK9 +
	BootstrapClassLoader   （加载 java.base 模块等核心模块）
	        ↑
	PlatformClassLoader    （加载标准模块，如 java.sql, java.xml）
	        ↑
	AppClassLoader         （加载应用模块 / 类路径下的类）
	        ↑
	自定义 ClassLoader

```
## 类加载器
* **BootstrapClassLoader**
	* 最顶层的类加载器，是 JVM 内部的类加载器，负责加载Java 核心类库
		* JDK8：（JDK 内部的 `rt.jar` 里的一部分：`java.lang.*`, `java.util.*`, `sun.*` 等）
			* PS：`rt.jar` = **Runtime library JAR**，是 java8 之前最重要的JAR 包
		* JDK9+: **`java.base` 模块**（它包含 `java.lang`, `java.util`, `java.io`, `java.math` 等，所有应用都必须依赖）
* **ExtClassLoader**
	* 扩展类加载器，现已废弃
		* 为什么废弃？
			* 有隐式全局覆盖的问题，jdk9 要求所有依赖必须要显式写在 **classpath/modulepath**，而不是丢在 ext 里就能生效
	* 用来加载 `$JAVA_HOME/jre/lib/ext/` 目录下的 jar 包
* **PlatformClassLoader**
	* 加载平台模块（比如 `java.sql`, `java.naming` 等），即那些不在 `java.base` 里，但仍然属于 JDK 的模块。
* **AppClassLoader**
	* jdk8 加载 classpath 的代码
		* `java -cp target/classes com.example.Main`
	* jdk9 加载module path 的代码
		* `java --module-path out -m com.greeting/com.greeting.Main`

### [[Java命令]] mvn 命令的关系
* mvn 封装了javac 和 java 命令，帮我们拼 classpath/modulepath
	* `mvn compile` → 调用 `javac -cp ...`
	* `mvn exec:java` → 调用 `java -cp ... Main`
	* `mvn package` → 调用 `jar cf ...`
* ide 的绿三角
	* 编译源码，javac
	* 拼 cp，把编译后的 `target/classes`，再加上所有依赖 jar（来自 Maven/Gradle 的 `~/.m2/repository` 或 `~/.gradle/caches`），拼成一个很长的 `-cp`。
	* 执行 java 命令
