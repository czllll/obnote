1. **解析 URL 生成 HTTP 请求**：
    - 浏览器首先解析 URL，得到域名和路径信息。
    - 浏览器使用 HTTP 协议生成请求报文，并将其发送到服务器。
2. **[[DNS]] 域名解析得到目的 IP**：
    - 如果域名未缓存，浏览器会向 DNS 服务器请求解析。
    - DNS 服务器逐级查询并返回 IP 地址，浏览器使用该 IP 发送 HTTP 请求。
3. **TCP 三次握手建立可靠连接**：
	- 浏览器通过 TCP 协议与目标服务器建立连接。TCP 三次握手完成后，数据才能开始可靠传输。
4. **生成 IP 报文实现远程定位**：
	* TCP 将数据传递给 IP 层，IP 层会生成 IP 报文，加入源和目的 IP 地址，确保数据能够在网络中定位到正确的目标。
5. **添加 MAC 头部实现以太网传输**
	* 数据到达链路层，生成 MAC 帧，用于在本地网络传输。ARP 协议用于获取目标设备的 MAC 地址。
	- 通过 ARP（地址解析协议）将 IP 地址映射为 MAC 地址，实现数据链路层的传输。
6. **数据从网卡发出，经过交换机再转发给路由器，层层跳跃最终到达目标服务器**
7. **服务器响应请求**
	* 服务器接收到 HTTP 请求后，检查 TCP 头部，确认目的端口，找到对应的应用层进程，处理请求并返回响应数据。
	- 数据通过 HTTP 协议封装，经过 TCP、IP、链路层等层次回到客户端。
8. **客户端渲染页面**
	* 浏览器收到响应报文后，解析其中的 HTML、CSS、JavaScript 等资源，渲染出网页页面。
	* 动态页面的 客户端渲染（CSR），动态拿到数据，js 渲染

## 请求在后端的负载均衡过程你了解吗？
1. DNS 负载均衡：
	- 在全局流量的第一层，**DNS 负载均衡**会根据用户的地理位置或其他因素（如可用性）将请求导向最合适的服务器机房。DNS 会返回不同的数据中心的 **虚拟 IP** 或直接返回 IP 地址。
2. 四层负载均衡（LVS 等）：
	- [[LVS（Linux Virtual Server）]] 等四层负载均衡基于 **传输层**（L4）转发流量，通常使用[DNAT（目的地址转写）](NAT) 将客户端的请求转发到后台的 Nginx 集群 或其他负载均衡器。
	- 四层负载均衡 主要基于 IP 地址 和 端口号 来转发请求，而不关心请求内容。其特点是高效、低延迟，适用于简单的负载均衡需求。
3. 七层负载均衡（Nginx、HAProxy 等）：
	- 七层负载均衡 基于 **应用层**（L7），通过检查 HTTP 请求的 URL、请求头、Cookie 等内容，进行更细粒度的流量分发。
	- Nginx 可以基于请求路径将流量转发到不同的后端服务。
## 场景细化到抖音刷视频呢？
1. URL 请求与 DNS 解析（一致）
2. TCP 握手（一致）
3. HTTP 请求（请求视频流）
	1. 视频切片
	2. 个性化参数 userId 地理位置 偏好等
4. DNS 和 [[CDN]]
5. 服务器响应
	1. 应用层协议可能是用 http/2 或 ws
6. TCP 分段
7. 客户端视频解码
8. 播放完，推荐新内容的请求
