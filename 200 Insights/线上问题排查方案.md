# 问题定位
## CPU
* 监控指标
	* 运行队列、CPU使用率、上下文切换
* 命令
	* top，一般关注使用率最高的进程
* 常见问题
	* CPU突然飙升
	* CPU使用居高不下
* 常见原因
	* 往往是业务逻辑问题导致，如死循环，频繁GC或上下文切换过多
* 排查方案
	* 使用top命令找到占用高的进程
	* 使用`top -H -p pid`找到占用高的线程
	* 拿到nid（native thread id），转为16进制（因为java native thread以16进制形式输出）
	* 使用`jstack pid | grep 'nid' -C5`命令查看堆栈信息
		* - `-C`（Context）：显示匹配行前后指定数量的行
		- `-A`（After）：显示匹配行后面几行
		- `-B`（Before）：显示匹配行前面几行
	*  使用可视化工具：Jprofiler、等
## 内存
* 常见问题
	* 内存溢出OOM
		* 程序申请内存，但内存不够
		* 排查方式：
			* 使用`jmap`命令生成dump文件`jmap -dump:live,format=b,file=heap.hprof <pid>`
			* JVM启动参数增加参数，当应用OOM自动生成dump文件
			* 查看dump文件中最大对象，查看具体引用情况
	* 内存泄漏
		* 定义：程序运行时分配内存给临时变量，用完之后没有被GC回收。长期这样很多内存都浪费了容易OOM
		* 常见原因：
			* 可以复用的对象每次都new，但是不回收。比如客户端连接
			* 文件流操作，但是没有正常关闭
		* 排查：由于最后结果是OOM，所以参照OOM
	* 垃圾回收导致服务卡顿
		* 常见原因：
			* 时间过长

# 工具

## jdk自带
* jstack
	* 用于输出Java进程中所有线程的调用栈信息（线程快照）
	* 举例子
		* 查死锁：`jstack <pid> | grep -A 30 "Found one Java-level deadlock"`
		* 查线程：`jstack pid | grep 'nid' -C5`
* jps
	* 列出当前系统中正在运行的所有java进程的PID和主类名
	* 用途：确定要分析的java应用的PID
* jstat
	* 用于查看JVM的GC，类加载，内存等实时统计信息
	* `jstat -gc <pid> 1000 10` 每1000ms打印一次GC信息，共打印10次
	* 用途：排查Full GC频繁，内存压力等问题
## 图形化监控
* JConsole
	* 官方轻量级的GUI监控
* JVisualVM
	* 更强大，支持dump倒入分析
* MAT
	* 用于分析dump文件.hprof，找到内存泄漏的对象位置
	* 因为有对象引用链
* **Arthas**（阿里开源诊断工具）
	* 运行时 Java 应用诊断工具，**无需重启服务**，非常适合生产环境排查问题
