  ### 1. 准备服务器

  1. 创建部署账号

     sudo adduser deploy
     sudo usermod -aG sudo deploy
     用这个账号来执行发布脚本。
     用这个账号来执行发布脚本。
  2. 生成/导入 SSH 公钥（可以在本机或专用机上执行）

     ssh-keygen -t ed25519 -C "github-deploy"

     把生成的私钥留在本地（后面要放进 GitHub Secrets），公钥内容 ~/.ssh/
  id_ed25519.pub 追加到服务器的 ~deploy/.ssh/authorized_keys。
  3. 开放 SSH 权限
      - 确认服务器 sshd 允许 deploy 用户登录。
      - 如果不是默认 22 端口，记下端口号，workflow 要用。
  4. 准备部署脚本或命令
     比如在服务器上放一个 ~/deploy_backend.sh，里面写：

     #!/bin/bash
     set -e
     cd /home/deploy/algocove-backend
     git pull
     ./gradlew bootJar
     sudo systemctl restart algocove-backend

     记得给脚本执行权限 chmod +x.

  ———

  ### 2. 在 GitHub 仓库里配置 Secrets

  进入仓库 Settings → Secrets → Actions，新建以下项：

  | Secret 名称 | 值 |
  |-------------|-----|
  | SERVER_HOST | 服务器 IP 或域名（例如 2.3.4.1 或 api.algocove.com） |
  | SERVER_USER | deploy |
  | SERVER_SSH_KEY | 刚才生成的私钥（id_ed25519 内容） |
  | SERVER_PORT (可选) | 如果用的不是 22 端口，填端口号 |

  如果要同时部署前端或多台服务器，也可以再加不同的 Secrets。

  ———

  ### 3. 编写 GitHub Actions Workflow

  在仓库 .github/workflows/deploy-backend.yml 写一个工作流，例如：

  name: Deploy Backend

  on:
    push:
      branches: [ main ]      # 或你要触发部署的分支

  jobs:
    deploy:
      runs-on: ubuntu-latest

      steps:
        - name: Checkout repo
          uses: actions/checkout@v4

        - name: Deploy via SSH
          uses: appleboy/ssh-action@v1.0.3
          with:
            host: ${{ secrets.SERVER_HOST }}
            username: ${{ secrets.SERVER_USER }}
            key: ${{ secrets.SERVER_SSH_KEY }}
            port: ${{ secrets.SERVER_PORT || 22 }}
            script: |
              cd /home/deploy/algocove-backend
              git fetch --all
              git checkout main
              git pull
              ./gradlew bootJar
              sudo systemctl restart algocove-backend

  说明：

  - runs-on: ubuntu-latest 表示用 GitHub 托管的 runner。
  - appleboy/ssh-action 负责建立 SSH 连接并执行你写在 script 里的命令。
  - 如果你已经有自己的部署脚本，只要在 script 里调用就行，比如 ./
  deploy_backend.sh。
  - 若要 deploy 多台机器，可加多个 step 或多个 job，对不同的 host/script。

  ———

  ### 4. 测试

  1. 提交一次变更到 main（或你配置的分支）。
  2. 在 GitHub 仓库的 Actions 页面，看 workflow 是否执行成功。
  3. 如果失败，查看步骤日志，重点检查：
      - SSH 是否连通（密钥/端口/用户名是否正确）。
      - 服务器上是否有对应目录/脚本。
      - gradlew 等命令是否可执行。

  调试通过后，后续推送到指定分支就会自动执行部署。

  ———

  ### 5. 可选提升

  - 加上 known_hosts 配置（可通过 ssh-keyscan 填入 Secrets，或在 workflow 中
  StrictHostKeyChecking no）。
  - 如果需要 ZERO-DOWNTIME，可以让脚本使用 Docker/PM2/usr/bin/supervisor 等
  管理方式。
  - 如果还要部署前端或静态文件，可以在同一个 workflow 里加第二个 job，同理通
  过 SSH 执行部署命令即可。

  这样就完成了 GitHub 托管 Runner + SSH 的部署链路，既不需要在服务器上装
  runner，又能自动化发布。
