### 基础

### maven中的打包方式jar和pom有什么区别？

1. jar 打包方式：

- 用于普通的 Java 项目模块
- 会将项目的代码、资源文件等打包成一个可执行或可引用的 JAR 文件
- 打包后的 JAR 包含：.class 文件、配置文件、资源文件等
- 通常用于具体的功能模块实现，比如你的 mockredbook-auth 模块

1. pom 打包方式：

- 专门用于父项目 (parent project) 或聚合项目 (aggregator project)
- 不会打包任何代码或资源文件
- 仅用于管理子模块、依赖和插件
- 主要职责是：
  - 统一管理依赖版本
  - 聚合管理多个子模块
  - 继承配置给子模块



### 自定义注解和切面

* 项目中使用到自定义注解和切面来实现API请求日志切面

* 自定义注解是什么？
  * 提供元编程方法，允许我们在不改变代码逻辑的情况下为代码添加元数据，这些元数据可以在编译时或运行时通过反射被访问。
* AOP面向切面编程
  * 是一个编程范式，提供一种能力，让开发者可以模块化多个对象的横切关注点（日志，事务管理，安全等）
  * 概念
    * 切点（Pointcuts）：定义在哪里应用切面
    * 通知（Advices）：定义在特定切点上要执行的代码。
      * 前置通知
      * 后置通知
      * 环绕通知：完全控制方法的执行，可以修改如惨，返回值，甚至组织执行
      * 返回通知：方法成功返回后执行
      * 异常通知：方法抛出异常后执行
    * 切面（aspect）：切面将切点和通知结合起来，定义了在何处和合适应用特定的逻辑

### 如何创建自定义注解？

* 创建自定义注解需要使用 `@interface` 关键字，并结合 **元注解**（`@Retention`、`@Target` 等）进行定义。
* 元注解
  * `@Retention(RetentionPolicy.RUNTIME)`: 指定注解的保留策略，即注解在何时生效？
    * RUNTIME表示在运行时保留，可通过**反射**在运行时被访问和解析。需要反射解析必须使用RUNTIME（@Autowired，@Transactional）
    * CLASS，编译进字节码文件，运行时不可见（@Entity）
    * SOURCE，仅源码可见，编译后丢弃（@Override、@SuppressWarnings）
  * `@Target({ElementType.METHOD})`：指定注解的目标元素，即可以在哪里使用该注解。
    * METHOD：只能用在方法上
    * TYPE： 类、接口、枚举
    * FIELD：成员变量上
  * `@Documented`指定被注解的元素是否会出现在生成的Java文档中



### AOP切面类

* @Aspect
  * 声明该类为一个切面类
* @Pointcut
  * 定义一个企鹅电，后面跟一个表达式，表达式可以定义为切某个注解，也可以切某个package下的方法
* 定义通知类型
  * **@Before**: 在切点之前，织入相关代码；
  * **@After**: 在切点之后，织入相关代码;
  * **@AfterReturning**: 在切点返回内容后，织入相关代码，一般用于对返回值做些加工处理的场景；
  * **@AfterThrowing**: 用来处理当织入的代码抛出异常后的逻辑处理;
  * **@Around**: 环绕，可以在切入点前后织入代码，并且可以自由的控制何时执行切点；
    * `public Object doAround(ProceedingJoinPoint joinPoint) throws Throwable {`
    * 参数来自于 **Spring AOP**，它是**Spring 框架在运行时自动注入**的方法参数，主要用于 `@Around`（环绕通知）的切面方法
      * `joinPoint.getSignature()`:获取方法的签名（方法名、参数等信息）
      * `joinPoint.getTarget()`:获取被代理对象

### 切面类注意

✅ **如果定义了 `@Pointcut`，那么 `@Before`、`@After`、`@Around` 这些通知可以直接引用 `@Pointcut` 方法，而不需要再写 `execution(...)` 之类的切点表达式！**





### 为什么使用@AutoConfiguration来控制切面类的装配呢？

* Spring Boot 采用 **自动装配机制**（Auto Configuration）来让 **某些功能组件按需启用**，而不是**强制加载**。对于 `Aspect`（切面类），**如果不希望它默认生效，而是让用户选择是否启用，就需要 `@AutoConfiguration`**。
* 与@Component区别
  * **`@Component` 只能通过 Spring 扫描注册，无法与 `spring.factories` 配合做条件加载。**
  * `@AutoConfiguration` 可以在 Spring Boot 自动装配机制下生效**，支持**按需启用**。使用@ConditionalOnProperty 来控制是否启用
*  Spring Boot **Starter 组件**开发时
  * **在某些 Spring Boot 项目里自动生效**，但在其他项目里可以**选择不开启**，那么 `@AutoConfiguration` + `@ConditionalOnProperty` **就是最佳选择**



### 为什么要创建资源文件org.springframework.boot.autoconfigure.AutoConfiguration.imports填写切面类配置类的路径呢？

* Spring Boot 3.x 之后，Spring 官方**废弃了 `spring.factories`** 方式，推荐使用 `org.springframework.boot.autoconfigure.AutoConfiguration.imports`。

* 因为 Spring Boot **不会自动扫描 `@AutoConfiguration`**，必须在 `org.springframework.boot.autoconfigure.AutoConfiguration.imports` 文件中**手动注册**你的自动配置类，Spring Boot 才能在启动时自动加载它。

### Optional类是什么？

* `Optional<T>` 是 **Java 8** 引入的一个容器类，用于表示**可能为空的对象**。
  它的主要作用是**避免 `NullPointerException`（NPE）**，并提供更优雅的方式来处理 `null` 值。
* 创建

```java
// 方式 1：创建一个非空 Optional
Optional<String> optional = Optional.of("Hello");

// 方式 2：创建一个可为空的 Optional
Optional<String> optionalNullable = Optional.ofNullable(null);

// 方式 3：创建一个空的 Optional
Optional<String> emptyOptional = Optional.empty();

```

* `isPresent()` 检查值是否存在 返回boolean

* `ifPresent`存在直接执行

  * ```java
    optional.ifPresent(System.out::println); // Hello
    ```



### 全局异常处理类的方法使用的注解@ExceptionHandler({ xxx.class })，和@ResponseBody

* `@ExceptionHandler` 作用于 **方法级别**，用于指定**该方法可以捕获哪些异常**。
  - 当控制器（`Controller`）中的方法抛出指定类型的异常时，Spring 会自动调用 `@ExceptionHandler` 标注的方法来处理这个异常。

* `@ResponseBody` **表示方法的返回值会作为 HTTP 响应体（Response Body）直接返回**，而不是一个视图
  * **Spring MVC 默认是基于视图（View）的框架**，如果没有特殊指定，**Spring 会把返回值解析成一个视图名称**，然后去 `templates/` 或 `WEB-INF/views/` 目录下查找对应的模板文件（如 `*.html`、`*.jsp`）。

### `@ControllerAdvice` 是什么？

`@ControllerAdvice` 是 Spring 提供的**全局异常处理机制**，它的作用是**拦截整个 Spring MVC 控制层（`@Controller`）的异常，并统一处理**。

### 为什么在JsonUtil中要创建静态的ObjectMapper实例？

* `ObjectMapper` 是 **Jackson 库** 提供的一个核心类，主要用于 **Java 对象和 JSON 之间的相互转换**（序列化和反序列化）。

`private static final ObjectMapper INSTANCE = new ObjectMapper();`

* **`ObjectMapper` 创建成本高**，避免**每次调用都创建新实例**，提高性能。
* **`static` 让它成为类级别的共享对象，所有方法复用，减少内存占用**。
* **`final` 确保 `ObjectMapper` 只被初始化一次，防止修改实例**。
* **大部分情况下是线程安全的**，可以在多线程环境下安全使用。





### RBAC 权限控制模型

* Role-Based Access Control
* 将用户与角色进行关联，并将权限分配给角色，而不是直接分配给用户

* RBAC 0
  * 最基本的基于用户-角色-权限的模型
* RBAC 1
  * 基于角色的层次模型
  * 特点
    * 角色继承：一个角色可以继承另一个角色的所有权限
    * 权限传递：继承关系是传递的
* RBAC 2
  * 基于约束的RBAC模型，建立在 RBAC 0 基础之上，但是增加了**约束**
  * 特点
    * **互斥角色**：某些角色不能同时赋予同一个人
    * **先决条件**：用户要获得某个角色，必须先拥有另一个角色
    * **基数约束**：限制某个角色可以被赋予的用户数量

* RBAC 3 = RBAC1 + RBAC2
* 用户组
  * 同一用户组提前分配好角色

### 声明式注解事务失效原因

* 方法可见性：`@Transactional` 仅在 `public` 方法上生效。
* 自调用：当类中的方法调用同一个类中的另一个 `@Transactional` 方法时，事务可能不会生效。这是因为事务注解是通过 AOP 实现的，而 Spring 的 AOP 代理机制在这种情况下不会被触发
* **异常处理**：只有 `RuntimeException` 和 `Error` 类型的异常会触发事务回滚。如果你抛出的是 `checked exception`，事务不会回滚，除非你明确指定 `rollbackFor` 属性。
* **代理对象**：确保你是在 Spring 管理的代理对象上调用方法。如果你直接使用 `new` 关键字实例化对象，Spring 的 AOP 代理机制将不会被应用。

### 使用redis 分布式锁

* 为什么使用
  * 在生产环境中，子服务必然是以集群的方式部署，那么，就会带来一个问题，每个服务启动后，都会跑一次 `PushRolePermissions2RedisRunner` , 就来带来多次同步 Redis 缓存的问题

分布式锁是确保在分布式系统中多个节点能够协调一致地访问共享资源的一种机制。Redis 分布式锁通过 Redis 的原子操作，确保在高并发情况下，对共享资源的访问是互斥的。

- 可以使用 Redis 的 `SETNX` 命令来实现。如果键不存在，则设置键值并返回 1（表示加锁成功）；如果键已存在，则返回 0（表示加锁失败）。
- 多个子服务同时操作 Redis , 第一个加锁成功，则可以同步权限数据；后续的子服务都会加锁失败，若加锁失败，则不同步权限数据；
- 另外，结合 `EXPIRE` 命令为锁设置一个过期时间，比如 1 天，防止死锁。则在 1 天内，无论启动多少次认证服务，均只会同步一次数据。

###  @RefreshScope 是啥

* 在 Nacos 配置中心的场景下，`@RefreshScope` 的主要功能包括：
  1. **动态刷新配置**：当 Nacos 配置中心的配置发生变化时，应用中的配置会自动更新，避免了手动重启应用的繁琐过程。

  3. **与 Spring Cloud 集成**：`@RefreshScope` 与 Spring Cloud 的配置管理机制紧密集成，能够无缝地处理配置更新事件。

# Cassandra

**Apache Cassandra** 是一个 **分布式 NoSQL 数据库**，专为 **大规模数据存储和高可用性** 设计，最初由 Facebook 开发，后开源并捐赠给 Apache 基金会。

### 基本概念

- **节点（Node）**：Cassandra 集群中的每个服务器称为一个节点。每个节点都存储数据，并且相互之间没有主从关系，所有节点都是对等的。
- **集群（Cluster）**：由多个节点组成的分布式系统称为集群。集群中的节点共同工作，处理读写请求并存储数据。
- **数据中心（Data Center）**：集群中的节点可以分布在多个数据中心，每个数据中心包含若干个节点。数据中心的划分有助于实现跨地域的高可用性。
- **键空间（Keyspace）**：键空间是一个逻辑容器，用于管理多个表，可以理解为 MySQL 中的库。另外，键空间定义了数据复制的策略。
- **表（Table）**：表是数据存储的基本单位，由行和列组成。每张表都有一个唯一的名称和定义。
- **主键（Primary Key）**：每行数据都有一个唯一的主键。主键由分区键和可选的列组成，用于唯一标识数据行。
- **分区键（Partition Key）**：Cassandra 使用分区键的哈希值将数据分布到不同的节点上，从而实现负载均衡和数据的水平扩展。分区键可以是单个列或多列的组合（复合分区键）。

###  笔记发布流程

笔记发布处理流程打字如下：

- 请求笔记发布接口前，用户填写笔记内容时，先将笔记图片或视频上传至 OSS 对象存储服务，拿到资源的访问直链；
- 用户填写好笔记标题、内容、话题等等后，点击发布按钮，请求由网关转发给笔记服务；
- 笔记服务请求分布式 ID 生成服务，为该条笔记生成一个全局唯一 ID;

### 笔记点赞接口处理

- 判断被点赞的笔记是否真实存在，若不存在，则返回提示信息：笔记不存在；
- 判断笔记是否被点赞过，若已点赞，则返回提示信息：该篇笔记已点赞；
- 判断用户的 ZSET 点赞列表是否存在，若存在，再判断 ZSET 的大小已经达到 100（每页10条，共10页的数据）：
  - 若达到，则向 ZSET 中添加最新的点赞笔记，并将点赞时间最久的那篇，移除出 ZSET;
  - 若未达到，则直接向 ZSET 中添加最新的点赞笔记；
- 发送 MQ, 让消费者去将点赞记录存入 `t_note_like` 表中；

## 数据结构

### 基本数据类型

* 文本
  * `text`: 用于全文搜索；支持分词和倒排索引
    * 标题
  * `keyword`:用于精确匹配，不参与分词
    * 状态值，分类
* 数值
  * 整型
    * integer 32
    * long 64
  * 浮点型
    * float 单精度
    * double
* 布尔
  * boolean
* 日期
  * `date`
    * 默认格式strict_date_optional_time`，例如 `2024-11-16T15:30:00Z

### 复杂数据类型

* 对象类型

  * `object` 存JSON对象，可以直接访问其对象。将数组中的每个对象存储为一个独立的子文档

  

* 嵌套类型
  * `nested`: 存数组中的复杂对象



### 地理数据类型

* 地理点类型
  * `geo_point`: 存经纬度
* 地理形状类型
  * `geo_shape`:存地理图形

### 特殊数据类型

* ip地址
  * `ip`: v4 或 v6
* 范围类型
  * `range`: 存储数值，日期或IP的范围
  * 包括：`integer_range`、`float_range`、`long_range`、`date_range`、`ip_range`



## 倒排索引

* 是什么
  * 关键词到文档的映射
  * 存储每个关键词以及包含这些关键词的文档信息

### 构建过程

* 分词
  * 将文本内容拆分为独立的单词或短语
* 归一化
  * 转小些
  * 去除停用词：对查询结果影响不大的词 --》“的”
  * 词干提取：归于基本形式 running-〉run
* 创建倒排索引
  * 根据分词结果，为每个词项建立一个到文档的映射关系

### 搜索过程

* 解析查询条件
  * 将查询关键词解析为分词结果
* 查找倒排索引
  * 查找词对应到的文档列表
* 合并结果
  * 根据查询类型合并文档列表，不指定则默认为OR
    * 若为AND，返回两个列表交集
    * OR，并集
* 排序结果
  * 根据相关性评分（TF-IDF），排序返回

### ES分词器

* 组件
  * 字符过滤：预处理如删除html标记等
  * 分词：将文本拆分为一系列的tokens，例如按照空格或标点切分
  * 此项过滤：处理分词结果，如转小写，去停用词，词干提取等
* 常见分词器
  * 默认-standard分词器
    * 多语言，按标准分词规则切分
  * simple分词器
    * 仅按非字母字符切分，且将所有词项转为小写
  * whitespace
    * 空格切分
  * keyword
    * 整个输入作为单独词项
  * stop
    * 在standard基础上，去掉常用的停用词is the...
