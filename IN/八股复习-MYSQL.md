## SQL基础

### **SQL 和 NoSQL 的区别**

1. **定义与代表性数据库**
   - **SQL（关系型数据库）**
      指采用结构化查询语言（SQL）操作的关系型数据库，数据以行和列组成的二维表形式存储。
      常见代表有：MySQL、PostgreSQL、Oracle、SQL Server 等。
   - **NoSQL（非关系型数据库）**
      指不采用传统关系模型的数据存储方式，数据结构灵活，适用于非结构化或半结构化数据。
      常见代表有：MongoDB（文档型）、Redis（键值型）、Cassandra（列式）、Neo4j（图形数据库）等。
2. **数据结构与存储方式**
   - **SQL：** 结构化数据，固定表结构，每行表示一条记录，每列表示一个字段属性。
   - **NoSQL：** 灵活结构，数据可以存为 JSON 文档、键值对、列族或图形结构，无需预定义模式。
3. **事务特性（ACID vs BASE）**
   - **SQL 支持 ACID：**
     - 原子性（Atomicity）
     - 一致性（Consistency）
     - 隔离性（Isolation）
     - 持久性（Durability）
        适用于金融、订单等对数据一致性要求极高的应用场景。
   - **NoSQL 倾向 BASE：**
     - Basically Available（基本可用）
     - Soft state（软状态）
     - Eventually consistent（最终一致性）
        更适合对一致性要求不高但需高可用和大规模分布的应用，如社交媒体、缓存系统等。
4. **扩展性对比**
   - **SQL：** 水平扩展较难，数据之间存在关联，扩展常需解决分布式 JOIN、事务管理等复杂问题。
   - **NoSQL：** 天生易扩展，数据间无强关联，适合分布式架构。支持主从复制、哨兵、分片集群等机制（如 Redis）。
5. **适用场景**
   - **SQL：** 数据结构稳定、关系复杂、强一致性需求高的场景。如银行系统、ERP系统。
   - **NoSQL：** 数据结构灵活、关系弱、大规模并发和高性能需求的场景。如社交网络、缓存、日志分析、实时分析等。

### **数据库三大范式（1NF、2NF、3NF）**

1. **第一范式（1NF）：原子性**
   - 定义：数据库表中的**每一列都是不可再分的原子数据项**，即不能出现如“多个值合在一个字段”这种情况。
   - 示例问题：字段中包含“家庭信息”列（如：3人，北京）或“学校信息”列（如：硕士，研二），这些不是原子值，违背 1NF。
   - 正确做法：将合并字段拆分为多个字段，如“家庭人口”、“户籍”、“学历”、“所在年级”。
   - 总结口令：**字段不可拆，字段值必须原子化。**

2. **第二范式（2NF）：消除部分依赖**

   - 定义：在满足 1NF 的基础上，**非主属性必须完全依赖于主键**，而不能只是依赖主键的一部分（适用于联合主键）。

   - 示例问题：订单表使用“订单号 + 产品编号”作为联合主键，但“订单金额”和“订单时间”仅依赖“订单号”，不依赖“产品编号”。
     - 正确做法：将订单表拆为两张表：
       1. 订单信息表（订单号，订单金额，订单时间）
       2. 订单明细表（订单号，产品编号，产品数量，产品折扣，产品价格）

   - 总结口令：**非主属性依赖主键整体，不能只依赖一部分。**

3. **第三范式（3NF）：消除传递依赖**

   - 定义：在满足 2NF 的基础上，**非主属性不能传递依赖于主键**。即非主属性不能依赖于另一个非主属性。

   - 示例问题：学生表中，“班主任性别”和“班主任年龄”依赖于“班主任姓名”，而“班主任姓名”又依赖于“学号”。

   - 正确做法：将班主任信息拆为独立表，通过“班主任姓名”或“班级编号”进行关联。

   - 总结口令：**非主属性不能依赖另一个非主属性。**

### **MySQL 的四种连接查询（JOIN）方式**

1. **内连接（INNER JOIN）**
   - 内连接只返回两个表中**匹配成功**的记录（交集）。
   - 含义：只显示分配了部门的员工。
2. **左外连接（LEFT JOIN）**
   - 左外连接返回左表的所有记录，即使右表中没有匹配，未匹配部分用 NULL 填充。
   - 含义：显示所有员工及其部门信息，未分配部门的员工也会显示，部门字段为 NULL。
3. **右外连接（RIGHT JOIN）**
   - 右外连接返回右表的所有记录，即使左表中没有匹配，未匹配部分用 NULL 填充。
   - 含义：显示所有部门及其员工信息，未分配员工的部门也会显示，员工字段为 NULL。
4. **全外连接（FULL JOIN）**
   - 全外连接返回两个表中的所有记录，包括匹配和不匹配的部分。MySQL 不直接支持 FULL JOIN。
   - 解决方式：使用 `LEFT JOIN` 和 `RIGHT JOIN` 的 `UNION` 实现。
   - 含义：显示所有员工与部门的组合，即使没有对应关系。

### **INT(1) 和 INT(10) 在 MySQL 中的区别**

1. **区别在于显示宽度，不影响存储大小**
   - 所有 `INT` 类型（无论是 `INT`、`INT(1)` 还是 `INT(10)`）的存储空间固定为 **4 字节**，可以表示的范围是一样的。
   - 括号中的数字（如 `1` 或 `10`）表示的是**显示宽度**，与实际存储无关。
2. **显示宽度的用途：仅影响格式化输出**
   - 显示宽度在**使用 `ZEROFILL`** 时才会生效，用于在显示时**左侧补零**。
   - 例如：
     - `INT(4) ZEROFILL` 存入值 `5`，显示为 `0005`
     - 若存入 `12345`，即使超过指定宽度，也会正常存储和显示（不会截断）
3. **注意事项**
   - 如果没有加 `ZEROFILL`，显示宽度是无效的。
   - 从 MySQL 8.0.17 开始，**`INT(M)` 语法的显示宽度已被废弃**（除了和 `ZEROFILL` 搭配时），不再建议使用。

### **SQL 查询语句的执行顺序**

1. **FROM**
   - 确定数据来源的主表，开始执行查询过程。
2. **ON**
   - 如果涉及连接操作，首先根据连接条件筛选匹配的行。
3. **JOIN**
   - 执行表连接操作（如 INNER JOIN、LEFT JOIN 等）。
4. **WHERE**
   - 对上一步的结果进行条件过滤，排除不满足条件的行。
5. **GROUP BY**
   - 对筛选后的数据进行分组，为聚合函数准备基础。
6. **聚合函数（如 COUNT、SUM、AVG 等）**
   - 对每个分组进行聚合计算。
7. **WITH CUBE / ROLLUP（可选）**
   - 扩展分组汇总的层级，生成更丰富的聚合结果。
8. **HAVING**
   - 对分组后的聚合结果进行条件过滤。
9. **SELECT**
   - 确定最终要返回的列，并可进行计算或别名处理。
10. **DISTINCT**
    - 对返回结果去重。

11. **ORDER BY**
    - 对最终结果进行排序。

12. **LIMIT / OFFSET**
    - 限制返回结果的行数，实现分页或截取部分数据。

### **MySQL 实现可重入锁的逻辑**

1. **表结构设计**
   - 使用一张锁表保存锁的状态，包括锁名、持有线程、重入次数等字段。
   - 通过设置主键和默认值实现对锁记录的唯一性和初始控制。
2. **加锁操作流程**
   1. 开启事务
   2. 查询锁记录是否存在，并加锁
      - 如果记录不存在，说明锁未被占用，直接插入记录，表示成功加锁
      - 如果记录存在，且持有者是当前线程，说明是重入，增加重入次数
   3. 提交事务
3. **解锁操作流程**
   1. 开启事务
   2. 查询锁记录是否存在，并加锁
      - 如果记录存在，持有者是当前线程，且重入次数大于 1，则减少重入次数
      - 如果记录存在，持有者是当前线程，且重入次数小于等于 1，则删除锁记录，表示释放锁
   3. 提交事务
4. **关键控制点**
   - 所有锁操作需包裹在事务中，确保并发安全
   - 使用 `FOR UPDATE` 加锁语句避免并发线程同时修改
   - 通过 `reentry_count` 实现可重入性，允许同一线程多次加锁而不阻塞自身

## 存储引擎

### **MySQL 执行一条 SQL 请求的过程**

1. **连接器**
   - 建立连接，管理连接，校验用户身份。
2. **查询缓存**
   - 查询语句如果命中查询缓存则直接返回，否则继续往下执行。MySQL 8.0 已删除该模块。
3. **解析 SQL**
   - 通过解析器对 SQL 查询语句进行词法分析、语法分析，然后构建语法树，方便后续模块读取表名、字段、语句类型。
4. **执行 SQL**
   - 执行 SQL 共有三个阶段：
     1. 预处理阶段：检查表或字段是否存在；将 select * 中的 * 符号扩展为表上的所有列。
     2. 优化阶段：基于查询成本的考虑，选择查询成本最小的执行计划。
     3. 执行阶段：根据执行计划执行 SQL 查询语句，从存储引擎读取记录，返回给客户端。

### **MySQL 常见存储引擎介绍**

1. **InnoDB**
   - 是 MySQL 默认的存储引擎。
   - 支持事务，具备 ACID 特性，支持行级锁与外键约束。
   - 适用于高并发的读写场景，能够较好地保障数据一致性和完整性。
   - 支持崩溃恢复和自动提交机制，安全性更强。
2. **MyISAM**
   - 是 MySQL 早期常见的非事务型存储引擎。
   - 占用空间小，查询性能较高，适合以读为主的场景。
   - 不支持事务、不支持行级锁与外键，只有表级锁。
   - 在高并发写入、数据一致性要求较高的场景下不适合使用。
3. **Memory（Heap）**
   - 将数据存储在内存中，访问速度极快。
   - 适用于临时计算、排行榜、缓存等读性能要求极高的场景。
   - 不支持事务、行级锁和外键。
   - 数据在服务器重启或崩溃后会全部丢失，仅结构保留。

### **MySQL 为什么选择 InnoDB 作为默认存储引擎**

1. **事务支持**
   - InnoDB 支持事务处理，具备 ACID（原子性、一致性、隔离性、持久性）属性。
   - 能确保关键业务数据的安全性，防止事务中断后数据混乱。
   - MyISAM 不支持事务，无法满足强一致性的需求。
2. **并发性能**
   - InnoDB 使用行级锁，粒度更细，可支持更高并发访问。
   - 支持多版本并发控制（MVCC），减少锁冲突，提高读写效率。
   - MyISAM 只支持表级锁，写操作会阻塞整个表，效率较低。
3. **崩溃恢复能力**
   - InnoDB 通过重做日志（redo log）实现崩溃恢复。
   - 当数据库异常关闭时，可通过日志回放确保数据不会丢失。
   - MyISAM 不支持崩溃恢复，断电或异常关闭容易导致数据损坏。
4. **数据完整性保障**
   - InnoDB 支持外键约束，可保证表间数据一致性。
   - 有助于防止主从数据不一致或逻辑错误。
   - MyISAM 不支持外键，依赖业务逻辑自行维护。

### **InnoDB 与 MyISAM 的区别**

1. **事务支持**
   - InnoDB 支持事务，符合 ACID 特性，适用于对数据一致性要求较高的场景。
   - MyISAM 不支持事务，是 MySQL 将默认存储引擎由 MyISAM 转为 InnoDB 的重要原因之一。
2. **索引结构**
   - InnoDB 使用聚簇索引，数据行与主键索引存储在一起，主键检索效率高。
   - 辅助索引需通过主键间接定位数据，主键不应过大。
   - MyISAM 使用非聚簇索引，索引与数据文件分离，索引指向数据的物理地址。
3. **锁粒度**
   - InnoDB 支持行级锁，锁粒度小，支持高并发访问。
   - MyISAM 仅支持表级锁，锁冲突大，更新时可能阻塞整个表，影响性能。
4. **count(\*) 的效率**
   - InnoDB 执行 `select count(*)` 时需扫描全表，效率低。
   - MyISAM 会记录行数，执行计数操作时只需读取变量，速度快。

## 索引

### **什么是索引？有什么好处？**

1. **索引的本质**
   - 索引类似于书的目录，用于加快数据定位。**索引是一种数据结构**
   - 它本身是存储在数据库中的一个结构，指向真实数据所在的位置。
2. **索引的作用**
   - 减少不必要的数据扫描，提高查询效率。
   - 能在不增加额外数据读取的前提下快速定位目标行。
3. **有无索引的查询效率对比**
   - 如果没有索引，查询需要进行全表扫描，时间复杂度为 O(n)。
   - 如果使用了索引，可以基于二分查找等方式定位，查询效率显著提升。
4. **索引的数据结构**
   - MySQL 中大多数索引基于 B+ 树实现。
   - B+ 树的搜索复杂度为 O(log d N)，其中 d 表示每个节点的最大子节点数量。

### **MySQL 索引的分类方式**

MySQL 索引可从四个角度分类：

1. **按数据结构分类**
2. **按物理存储分类**
3. **按字段特性分类**
4. **按字段个数分类**

#### **1. 按数据结构分类**

1. **B+Tree 索引**
   - 最常见的索引结构，InnoDB、MyISAM 和 Memory 引擎均支持。
   - 查询效率高，支持范围查找、排序、最左匹配等操作。
2. **Hash 索引**
   - 基于哈希表构建，等值查询效率高。
   - 只支持等值匹配，不支持范围查找、排序等操作。
   - InnoDB 不支持，但在内存结构中使用了自适应哈希索引；Memory 引擎支持。
3. **Full-text 全文索引**
   - 支持对文本字段进行关键词匹配。
   - MyISAM 原生支持，InnoDB 从 MySQL 5.6 开始支持；Memory 不支持。

#### **2. 按物理存储分类**

1. **聚簇索引（主键索引）**
   - 数据与索引存储在一起，叶子节点保存的是数据本身。
   - 每个表只能有一个聚簇索引，默认使用主键作为聚簇索引。
   - 若无主键，则选第一个非空唯一索引；都没有则创建隐藏 id 作为聚簇索引。
2. **二级索引（辅助索引）**
   - 索引结构中叶子节点保存的是主键值，而非真实数据。
   - 查询时需先查辅助索引，再通过主键回表。

#### **3. 按字段特性分类**

1. **主键索引**
   - 建立在主键字段上，值唯一且不允许为 NULL。
   - 建表时可直接通过 `PRIMARY KEY` 创建。
2. **唯一索引**
   - 建立在 UNIQUE 字段上，值唯一但允许为 NULL。
   - 一张表可有多个唯一索引。
3. **普通索引**
   - 无唯一性限制，仅用于加快查询速度。
   - 支持重复值，字段可为 NULL。
4. **前缀索引**
   - 只索引字符串前几个字符，可减少空间占用。
   - 适用于 char、varchar、binary、varbinary 字段。

#### **4. 按字段个数分类**

1. **单列索引**
   - 只基于一个字段创建，如主键索引、普通索引等。
2. **联合索引（复合索引）**
   - 基于多个字段组合创建的索引。
   - B+Tree 中索引按字段依次排序。

#### **联合索引的最左匹配原则**

- 联合索引中的字段有顺序，必须从左往右依次使用才能命中索引。
- 比如创建索引 (a, b, c)：
  - 可以命中索引：`where a=1`、`where a=1 and b=2`、`where a=1 and b=2 and c=3`
  - 无法命中索引：`where b=2`、`where c=3`、`where b=2 and c=3`
- 最左匹配规则在范围查询中也适用，一旦遇到范围查询就会停止匹配后续字段。
  - 比如：`where a > 10 and b = 2`，无法使用 b 字段。

### **MySQL 聚簇索引与非聚簇索引的区别**

1. **数据存储方式**
   - 聚簇索引：索引的叶子节点存储的是整行数据，数据按照主键顺序与索引一体存储。
   - 非聚簇索引：索引的叶子节点存储的是指向数据行的指针或主键，数据行单独存放在数据文件中。
   - InnoDB 是聚簇索引结构，MyISAM 是非聚簇索引结构。
2. **索引与数据的关系**
   - 聚簇索引与数据紧密耦合，查询时可直接从聚簇索引中取出数据。
   - 非聚簇索引需先查索引，再通过主键去数据区查找完整数据，称为“回表”。
3. **唯一性约束**
   - 每个 InnoDB 表只能有一个聚簇索引，因为数据只能按照一种顺序物理存储。
   - 而 MyISAM 中的非聚簇索引不限制数量，因为索引与数据是分离的。
4. **查询效率**
   - 查询主键时：聚簇索引效率高，直接定位数据。
   - 查询普通字段时：非聚簇索引可能只查索引（如果是覆盖索引），否则需“回表”。
   - 聚簇索引对主键查询快，对更新、插入操作性能稍逊，因为可能涉及页分裂。
   - 非聚簇索引读取灵活，但因为回表过程多了一步，整体效率相对低。

### 什么字段适合作为主键？

1. **具有唯一性且非空**
   - 主键必须唯一标识一行数据，并且不能为 `NULL`，这是主键的基本要求。
2. **最好具备递增趋势**
   - 如果主键是无序、随机生成的（如 UUID），会导致 B+ 树频繁分裂，影响聚簇索引的插入效率和性能。
   - 有序主键（如自增 ID）可以减少页分裂，提高插入性能。
3. **不建议使用业务字段作为主键**
   - 比如：会员卡号、订单号、手机号、学号等字段可能会因为业务变动出现重复或更改。
   - 一旦主键被业务耦合，未来变更代价极大，不利于维护。
4. **推荐使用自增字段作为主键**
   - 在单机系统中，`AUTO_INCREMENT` 是最常见且稳定的主键方案。
   - 但在分布式环境下（多台服务器分别写入），自增 ID 可能会重复，因此需要引入分布式 ID 生成方案（如雪花算法、UUID+哈希前缀、数据库段 ID 等）。

### 性别字段不建议加索引的原因：

1. **选择性低，过滤效果差**
   - 性别字段通常只有“男”“女”两种取值，区分度接近 0，索引选择性差。
2. **优化器不会使用索引**
   - 查询命中数据过多时，全表扫描反而比走索引更快，优化器会自动忽略低效索引。
3. **回表代价大，性能更差**
   - 使用二级索引还需回表查主键对应的完整数据，命中行数多时非常慢。
4. **加索引占空间，增删改还要维护**
   - 无用索引不仅浪费磁盘，还影响写入性能。
5. **不适合单独加索引**
   - 可以作为联合索引的一部分，但单独加索引几乎没有实际意义。

### 为什么自增 ID 更快？UUID 在 B+ 树中存储是有序的吗？

1. **自增 ID 更快的原因**
   - **顺序写入，页填充率高**：InnoDB 使用聚簇索引，主键顺序写入能保证新记录追加到最后，页面填充充分，减少空间浪费。
   - **定位快速，无额外开销**：新记录紧挨旧记录，插入位置明确，无需额外计算。
   - **减少页分裂和碎片**：连续写入降低了页分裂概率，维护成本低，性能稳定。
2. **UUID 插入慢的原因**
   - **非顺序写入，容易导致页分裂**：UUID 是无序的，新值插入 B+ 树中间位置，频繁引发页分裂，降低写入性能。
   - **占用空间大，降低页效率**：UUID 通常为 36 字符，作为主键时索引项变大，单页能存储的记录变少，树层级更高。
   - **比较开销大**：字符型主键比较成本高（逐字符比较），比整型 ID 更慢。
   - **磁盘 IO 次数多**：树更高、页更少，每次查找命中磁盘次数变多，整体性能下降。

### MySQL 中的索引是怎么实现的？

1. **InnoDB 使用 B+Tree 作为索引结构**
   - InnoDB 的主键索引和普通索引底层结构都是 **B+Tree（多路平衡查找树）**。
   - B+Tree 的特点是：**非叶子节点只存索引，叶子节点才存储数据，并且所有数据在叶子节点按主键顺序排列**。
2. **B+Tree 的结构特点**
   - **多路搜索树**：每个节点可有多个子节点，避免二叉树过深导致查询慢。
   - **所有数据存在叶子节点中**，非叶子节点只起导航作用。
   - **叶子节点之间有双向链表**，方便范围查询。
   - 索引项在树中是**按主键有序存储**，支持高效查找和范围扫描。
3. **索引是磁盘级结构，查找效率高**
   - B+Tree 是为磁盘设计的结构，每次从磁盘加载一个节点；
   - 节点中存储多个 key，减少 I/O 次数；
   - 理论上**树高为 3~4 层**即可支持百万级以上的数据查找。
4. **聚簇索引与二级索引**
   - 主键索引为**聚簇索引**，叶子节点直接存整行数据；
   - 普通索引为**二级索引**，叶子节点存的是主键值，查数据需**回表**。

### 数据页和叶子节点是什么关系？

1. **叶子节点其实就是数据页**
   - 在 InnoDB 中，B+ 树的**叶子节点本质上就是存储数据的“数据页”**（data page）；
   - 每一个叶子节点对应一个数据页，里面存放了按主键顺序排列的**实际记录行**。
2. **非叶子节点存索引，叶子节点存数据**
   - B+ 树中，非叶子节点（中间节点）只存储索引值和指向子节点的指针；
   - 只有叶子节点才真正存放完整的行记录，**每个叶子节点 = 一个数据页**。
3. **为什么说“数据页”而不是一直说“叶子节点”？**
   - 因为查找记录时，定位到叶子节点后，**实际操作的单位是数据页**；
   - 页结构（如页目录、槽、记录分组、n_owned 等）属于**InnoDB 页管理的概念**，不属于 B+ 树本身；
   - 所以当我们说“在页内查找记录”，说的是 InnoDB 在叶子节点（即数据页）内的进一步定位优化。
4. **总结**
   - **叶子节点 = 存放数据的数据页**；
   - **数据页 = 物理结构**，InnoDB 用它来组织磁盘上的数据块；
   - 页目录等优化机制，是 InnoDB 在叶子节点内部做的结构优化，用来加快记录定位。

### 到了 B+ 树的叶子节点后，数据是怎么查找的？

1. **叶子节点是主键有序的单向链表**
   - 叶子节点记录按主键有序排列；
   - 节点之间通过**单向链表**连接，便于范围查询；
   - 但如果没有优化，查找记录需要从头开始遍历，效率较低。
2. **InnoDB 为每个数据页建立了“页目录”提升查找效率**
   - 页目录类似书的目录，是对页内记录的一个**分组索引**；
   - 每组中存储的是该组最后一条记录的偏移地址；
   - 所有组的偏移地址按顺序存放，形成一个由多个“槽（slot）”组成的目录。
3. **页目录的创建机制**
   - 所有记录分组，每组记录按主键有序；
   - 每组最后一条记录头部会保存该组记录数（n_owned）；
   - 槽中存储的就是每组最后一条记录的偏移量，页目录就是由这些槽组成。
4. **查找流程：先定位槽，再查组内记录**
   - 先通过**二分法查页目录槽**，定位要找的记录在哪一组；
   - 再在目标槽指向的组内顺序查找，直到定位到目标记录；
   - 这样避免了对整个页内记录链表的全遍历，**显著提高查找效率**。
5. **示例说明**
   - 假设页内有 5 个槽（编号 0~4），我要查主键为 11 的记录；
   - 先二分查槽（中间是 2，槽 2 最大主键是 8），因为 11 > 8，向右查；
   - 二分找到槽 3（最大主键是 12），11 < 12，落在槽 3；
   - 再从槽 3 所指组内顺序扫描，定位主键为 11 的记录，完成查找。

### B+ 树的主要特性有哪些？

1. **所有叶子节点在同一层**
   - 保证所有数据项的查找路径长度一致，I/O 延迟相同，**查询效率更稳定**。
   - 所有叶子节点之间通过指针连接形成**有序链表**，适合范围查询和排序扫描。
2. **非叶子节点只存键值，不存数据**
   - 非叶子节点仅存键值和子节点指针，用于导航查找路径，不存实际数据。
   - 相比 B 树，非叶子节点更轻量，树的**分支因子更高、树层更少、查找更快**。
3. **叶子节点才存储数据记录**
   - 与 B 树不同，B+ 树只有叶子节点才存储完整的数据记录或其指针。
   - **所有查询都必须访问到叶子节点**，才能获取到目标数据。
4. **支持范围查询**
   - 所有叶子节点通过链表连接，支持**有序遍历**；
   - 查询范围数据时，顺着链表查找即可，**不需要频繁回到根节点**。
5. **自平衡**
   - 插入、删除后自动触发分裂或合并，保证树结构平衡；
   - 每个非根节点的子节点数量范围是 `[⌈M/2⌉, M]`，其中 M 是阶数；（B+ 树的阶数 M，指的是一个节点最多能有的子节点数）；根节点可以少于⌈M/2⌉
   - **保持高度稳定，避免退化成链表，查询效率始终有保障**。

### B 树 和 B+ 树 的区别对比表

| 对比项             | B 树                                      | B+ 树                                           |
| ------------------ | ----------------------------------------- | ----------------------------------------------- |
| **数据存储位置**   | 所有节点都存数据（叶子节点 + 非叶子节点） | 只有**叶子节点**存数据，非叶子节点只存索引      |
| **查询路径**       | 查找可能在任何节点命中                    | 所有查询**必须走到叶子节点**，路径一致          |
| **范围查询效率**   | 不支持链式范围扫描，需多次回到根节点      | 叶子节点链表连接，支持顺序/范围查询，**效率高** |
| **叶子节点结构**   | 叶子节点之间无连接                        | 所有叶子节点通过**链表连接**                    |
| **内存命中效率**   | 节点大、数据分散，缓存命中率低            | 索引更集中、数据统一在底层，**缓存效果更好**    |
| **查询性能稳定性** | 路径不固定，查找性能不稳定                | 路径固定，**查找性能稳定**                      |
| **索引冗余**       | 索引项中含数据，**占空间多**              | 索引项仅存键值，**更节省空间，分支因子更高**    |
| **适用场景**       | 内存型数据库或不需要频繁范围查询的场景    | **磁盘数据库（如 MySQL InnoDB）优先使用 B+ 树** |

### B+ 树的叶子节点链表是单向还是双向的？

1. **InnoDB 中的 B+ 树叶子节点链表是双向的**
   - 每个叶子节点都包含**指向前一个叶子节点和后一个叶子节点的指针**；
   - 这样可以支持**顺序扫描 + 逆序扫描**，也方便实现 `ORDER BY DESC` 等倒序操作。
2. **为什么需要双向链表？**
   - B+ 树天然适合范围查询，但如果是**从大到小排序**的场景（如分页倒序），双向链表能显著提升效率；
   - 单向链表只能向右遍历，不支持快速向前扫描。
3. **InnoDB 的 B+ 树实现特点**
   - **数据页之间通过双向链表连接**，每个数据页默认大小为 16KB；
   - 叶子节点就是数据页，存储完整的用户记录和辅助信息；
   - 数据都存放在主键索引（聚簇索引）的叶子节点中，二级索引的叶子节点只存主键值，需要回表查数据。

### 联合索引的实现原理

1. **联合索引定义**
   - 联合索引是把多个字段按顺序组合成一个索引项；
   - 如 `CREATE INDEX idx ON table(a, b, c);`，表示优先按 a 排序，a 相同时按 b，再按 c。
2. **联合索引在 B+ 树中的结构**
   - 联合索引的 key 是多个字段组成的复合值；
   - 在 B+ 树中**按字段顺序从左到右排序**：先全局按 a 排序，a 相同的记录内部再按 b 排序，依此类推。
3. **索引 key 的有序性**
   - 联合索引中只有**最左字段是全局有序的**；
   - 后续字段只有在前面字段值固定时，才是局部有序。

### 最左匹配原则

1. **定义**
   - 联合索引只能在**where 子句中从最左字段开始连续使用**时，才能命中索引；
   - 一旦跳过某个字段，就无法继续利用后面的字段索引。
2. **可以命中索引的写法示例（以索引 a, b, c 为例）**
   - `WHERE a = 1`
   - `WHERE a = 1 AND b = 2`
   - `WHERE a = 1 AND b = 2 AND c = 3`
3. **不能命中索引的写法（不符合最左前缀）**
   - `WHERE b = 2`
   - `WHERE c = 3`
   - `WHERE b = 2 AND c = 3`
   - 原因：**索引是按 a 开头排序的，跳过 a 无法利用有序性**
4. **注意：WHERE 子句顺序不影响索引使用**
   - 比如 `WHERE b=2 AND a=1` 依然能用索引，因为**优化器**会自动重排条件；
   - 关键是是否包含了从左到右连续的字段。

### 创建联合索引时需要注意什么？

1. **字段顺序要根据“区分度”从高到低排列**

   - 联合索引匹配时是从最左字段开始逐级匹配的；
   - 所以**越靠前的字段越容易参与过滤，越能提升索引利用率**；
   - 最好把“**区分度高**”的字段放在索引前面。

2. **什么是区分度？**

   - 区分度 = 某字段不同值的数量 / 表总行数

     ```sql
     SELECT COUNT(DISTINCT column) / COUNT(*) FROM table;
     ```

   - 表示该字段能把数据分成多少组，**区分度越高，过滤效果越好**。

3. **低区分度字段示例**

   - 如 `性别（male/female）`、`是否启用（0/1）`，这些字段只有几个取值，**不适合建索引或排在索引前面**；
   - 它们过滤能力差，加了索引也可能会被优化器忽略，直接走全表扫描。

4. **高区分度字段示例**

   - 如 `用户ID`、`邮箱`、`订单号`、`UUID` 这类字段，每一行几乎唯一；
   - 适合建索引，适合排在联合索引靠前的位置。

5. **一个实用建议（面试可用）**

   > 在实际工作中创建联合索引时，我们通常优先把**高区分度、查询频率高、能过滤大量数据的字段**放在前面，
   >  再将排序字段、低选择性字段或辅助查询字段放在后面。

### 联合索引命中判断表（以索引 `(a, b, c)` 为例）

| 查询条件                     | 是否命中索引 | 命中字段                | 说明                                       |
| ---------------------------- | ------------ | ----------------------- | ------------------------------------------ |
| `a = 1`                      | ✅ 是         | a                       | 完全符合最左匹配                           |
| `a = 1 AND b = 2`            | ✅ 是         | a, b                    | 连续命中前两个字段                         |
| `a = 1 AND b = 2 AND c = 3`  | ✅ 是         | a, b, c                 | 全部命中                                   |
| `a = 1 AND c = 3`            | ✅ 是（部分） | a（主），c（可下推）    | c 不命中索引，但可**索引下推优化（ICP）**  |
| `b = 2 AND a = 1`            | ✅ 是         | a, b                    | 顺序无关，优化器会重排                     |
| `b = 2`                      | ❌ 否         | 无                      | 跳过 a，最左匹配失效                       |
| `c = 3`                      | ❌ 否         | 无                      | 跳过 a，最左匹配失效                       |
| `a = 1 AND b > 10`           | ✅ 是         | a, b                    | 支持范围查询，命中前两个字段               |
| `a = 1 AND b > 10 AND c = 5` | ✅ 是（部分） | a, b（主），c（可下推） | b 是范围，c 无法参与索引排序，但可下推过滤 |

### MySQL 中索引失效的常见情况（6 种）

1. **模糊查询前置通配符（左模糊）**
   - ❌ `LIKE '%abc'` 或 `LIKE '%abc%'` 会导致索引失效；
   - ✅ `LIKE 'abc%'` 才能利用索引；
   - **原因：前缀不定，无法使用 B+ 树有序特性定位范围。**
2. **查询中对索引列使用函数**
   - ❌ `WHERE DATE(create_time) = '2024-01-01'`
   - ✅ `WHERE create_time >= '2024-01-01' AND create_time < '2024-01-02'`
   - **函数作用于索引对应的列，相当于索引字段被“包住”，优化器无法利用索引。**
3. **查询中对索引列进行表达式计算**
   - ❌ `WHERE age + 1 = 30`
   - ✅ `WHERE age = 29`
   - **表达式作用在列上，会导致索引失效，和函数原理一样。**
4. **类型不一致导致隐式转换**
   - ❌ `WHERE phone = 1234567890` （phone 是字符串）
   - ✅ `WHERE phone = '1234567890'`
   - **字段是字符串，条件是数字，会触发隐式类型转换，相当于使用了函数。**
5. **联合索引未遵循最左前缀原则**
   - 索引为 `(a, b, c)`，以下失效：
     - ❌ `WHERE b = 2`
     - ❌ `WHERE c = 3`
   - 以下命中：
     - ✅ `WHERE a = 1 AND b = 2`
   - **只有从最左字段连续使用时，联合索引才能生效。**
6. **WHERE 子句中含 OR，且 OR 的任一分支未使用索引**
   - ❌ `WHERE a = 1 OR name LIKE '%Tom%'`
   - ✅ `WHERE a = 1 AND name LIKE 'Tom%'`
   - **OR 查询中，只要有一个分支不能用索引，整个 WHERE 可能放弃索引使用。**
     - 在 OR 查询中，如果某些分支不能使用索引，MySQL 优化器会评估执行成本，**可能选择索引合并（index merge），也可能直接放弃所有索引走全表扫描**，因此我们说“可能放弃索引”。

* 索引失效大多是因为函数、类型转换、表达式、字段顺序错误或无法利用索引的结构

### 什么情况下会回表查询？

1. **主键索引与二级索引的区别**
   - 主键索引（聚簇索引）的 B+Tree 叶子节点存放的是整行数据；
   - 二级索引（辅助索引）的 B+Tree 叶子节点存放的是索引字段值和主键值，不包含整行数据。
2. **什么时候需要回表**
   - 如果查询使用了二级索引，但**查询字段不在该索引中**，就需要先通过二级索引查出主键值；
   - 然后再通过主键值去主键索引中查整行数据；
   - 这个“先查索引再回主键”的过程就是回表。
3. **什么时候不需要回表**
   - 如果查询的字段都能在二级索引中找到（即查询字段 ⊆ 索引字段），则不需要回表；
   - 这种情况称为“覆盖索引”。
4. **回表的判断标准**
   - 使用的是二级索引；
   - 查询字段不在二级索引中；
   - 需要主键去主键索引查整行数据。
5. **实际例子**
   - 表结构：`CREATE INDEX idx_name ON user(name);`
   - 查询一：`SELECT id FROM user WHERE name = 'Tom';`
      → 查询字段 `id` 是主键，会包含在二级索引中，不回表。
   - 查询二：`SELECT age FROM user WHERE name = 'Tom';`
      → `age` 不在二级索引中，必须回表到主键索引中查 age 字段。

### 什么是覆盖索引？

1. **定义**

   覆盖索引是指：**一个索引包含了查询语句中涉及的所有列**，
    所以查询可以**完全从索引中获取结果**，不需要回表访问原始数据行。

2. **优势**

   - 减少回表次数，提升查询效率；
   - 降低磁盘 I/O；
   - 提高缓存命中率（因为索引页更小，更容易驻留在内存中）。

3. **实现条件**

   - 查询用到的字段**全部包含在某个二级索引中**；
   - 不涉及 `SELECT *`；
   - 索引字段的**顺序可以不完全匹配**，只要字段都包含即可。

### 插入数据后索引会发生什么变化？

1. **B+树结构可能发生调整**
   - 新插入的数据需要保持索引的有序性；
   - 数据可能被插入到某个叶子节点中；
   - 如果该节点已满，可能会触发叶子节点的分裂操作；
   - 分裂后可能导致父节点也分裂，层层递归，最终可能引起树的高度变化；
   - 这就是 B+ 树的自平衡机制。
2. **索引元信息会被更新**
   - 插入数据后，索引页中的记录数量、页面分布、槽位信息等都会更新；
   - 这些操作由数据库自动完成，以确保索引的正确性和可查性。
3. **保持一致性与查询效率**
   - 所有这些结构调整和信息更新，都是为了保证：
     - 索引结构始终保持有序；
     - 查询效率不下降；
     - 数据与索引一致，不发生“查不到或查错”的错误。

### 怎么决定建立哪些索引？

#### 1. **什么时候适合建索引？**

1. 字段具有唯一性或高区分度
   - 如商品编码、身份证号等；
   - 这类字段可以大幅提升查询效率。
2. 经常用于 `WHERE` 查询条件的字段
   - 能够提升表的整体查询速度；
   - 如果涉及多个字段查询，可考虑建立联合索引。
3. 经常用于 `GROUP BY`、`ORDER BY` 的字段
   - 建立索引后能避免排序操作；
   - 因为 B+Tree 索引是有序的，查询可以顺序返回数据，提高性能。

#### 2. **什么时候不需要建索引？**

1. `WHERE`、`GROUP BY`、`ORDER BY` 中未使用的字段
   - 没有参与查询过滤或排序，就不会被用到；
   - 建了也只是浪费空间，优化器不会使用。
2. 字段值重复度非常高（低区分度）
   - 如性别字段（只有男、女）；
   - 这种字段即使建索引也过滤不了多少数据，优化器可能选择全表扫描。
3. 表数据量非常小
   - 如小配置表、状态表等；
   - 查询全表反而比走索引快，不需要建索引。
4. 字段更新频繁
   - 如余额、点赞数、浏览量等字段；
   - 建索引会增加维护成本，频繁写入时应避免建立。

### 什么是前缀索引？

1. **定义**

   前缀索引是指：**只截取字符串字段的前 N 个字符来建立索引**，而不是整个字段。

2. **使用场景**

   - 用于**长字符串字段**（如 `VARCHAR(255)`、`TEXT`）；
   - 当字段前几位的区分度已经足够时，可以只索引部分前缀；
   - 能显著减少索引体积，提升索引命中率与缓存利用率。

3. **优点**

   - 减小索引项的大小，降低存储空间占用；
   - 减少磁盘 I/O，提高查询性能；
   - 可以在一个页中放入更多索引项。

4. **限制 / 缺点**

   - 只能用于**前缀等值查询**（如 `WHERE name LIKE 'abc%'`）；
   - 不支持**前缀范围查询或排序**，如 `ORDER BY name` 可能无法使用该索引；
   - 不适合区分度不足的前缀。

5. **使用示例**

   ```sql
   -- 对 name 字段的前 10 个字符建立索引
   CREATE INDEX idx_name_prefix ON user(name(10));
   ```

   - 表中 name 字段可能很长；
   - 但我们只关心前缀匹配，比如 `LIKE 'John%'`；
   - 此时用前缀索引就非常合适。

## 事务

### 事务的特性及其实现方式

1. **原子性（Atomicity）**

- 定义：事务中的所有操作要么全部执行成功，要么全部失败，不会停在中间某个状态。
- 举例：买商品付款，如果付款成功但商品没发货，就会回滚，像这个事务没发生一样。
- InnoDB 实现方式：通过 **undo log（回滚日志）** 实现。

2. **一致性（Consistency）**

- 定义：事务执行前后，数据库要满足所有完整性约束，数据保持一致性。
- 举例：转账后用户 A 少了 200 元，用户 B 多了 200 元，总余额不变。
- 实现方式：靠数据库的**完整性约束机制 + 原子性 + 隔离性 + 持久性**共同保障。

3. **隔离性（Isolation）**

- 定义：多个事务并发执行时，不会互相干扰，每个事务在逻辑上是“隔离”的。
- 举例：你下单买东西的事务，不会影响其他人下单。
- InnoDB 实现方式：
  - 默认采用 **MVCC（多版本并发控制）** 实现；
  - 高隔离级别下可使用 **行级锁机制（如 S 锁、X 锁）**。

4. **持久性（Durability）**

- 定义：一旦事务提交，所做修改就会永久保存在数据库中，即使宕机也不会丢失。
- InnoDB 实现方式：通过 **redo log（重做日志）** 实现。

### MySQL 并发事务下的三种读现象

#### 1. **脏读（Dirty Read）**

- **定义**：一个事务读到了另一个**未提交事务所修改的数据**。
- **危害**：如果对方事务回滚，读到的数据就是“错误的”，造成数据不一致。
- **示例**：
  - 事务 A 修改余额为 200 万但尚未提交；
  - 事务 B 此时读取余额，读到了“未提交”的 200 万；
  - 如果事务 A 回滚，事务 B 就读到了错误的余额。
- **防止方式**：
  - 设置隔离级别为：**读已提交（READ COMMITTED）或更高**。

------

#### 2. **不可重复读（Non-Repeatable Read）**

- **定义**：在**同一个事务中多次读取同一条记录**，前后读到的值不同。
- **本质**：目标记录被其他事务修改并提交了。
- **示例**：
  - 事务 A 先读余额为 100 万；
  - 事务 B 修改余额为 200 万并提交；
  - 事务 A 再次读取余额，变成了 200 万；
  - 同一事务内前后读值不一致。
- **防止方式**：
  - 设置隔离级别为：**可重复读（REPEATABLE READ）或串行化（SERIALIZABLE）**。

------

#### 3. **幻读（Phantom Read）**

- **定义**：在**同一个事务中两次查询某个范围的多条记录**，发现记录条数发生了变化。
- **本质**：是其他事务**插入或删除了“符合查询条件的新记录”**。
- **示例**：
  - 事务 A 第一次查询“余额大于 100 万”的记录，发现 5 条；
  - 事务 B 插入一条新的“余额大于 100 万”记录并提交；
  - 事务 A 再查，发现变成了 6 条，记录数量发生变化；
  - 出现“幻影数据”，称为幻读。
- **防止方式**：
  - 设置隔离级别为：**串行化（SERIALIZABLE）**；
  - 或使用**间隙锁（Next-Key Lock）**。

------

### 三种现象对比表（建议熟记）：

| 问题类型   | 发生原因                          | 查询内容变化 | 防止隔离级别             |
| ---------- | --------------------------------- | ------------ | ------------------------ |
| 脏读       | 读取了未提交事务修改的数据        | 值异常       | READ COMMITTED 或以上    |
| 不可重复读 | 同一记录被其他事务修改并提交      | 值变化       | REPEATABLE READ 或以上   |
| 幻读       | 其他事务插入/删除了符合条件的记录 | 记录条数变化 | SERIALIZABLE（或间隙锁） |

### 哪些场景不适合出现脏读？为什么？

* 银行系统、库存管理、订单系统等对数据一致性要求高的场景必须**避免脏读**，防止读取未提交的错误数据，确保数据真实可靠。

### MySQL 是如何解决并发问题的？

#### 1. **锁机制（Lock）**

- **作用**：保证并发操作下数据的一致性，防止“读写冲突”或“写写冲突”。
- **MySQL 支持多种锁**：
  - 行级锁（最细粒度）；
  - 表级锁；
  - 页级锁；
- **适用场景**：
  - 写操作冲突场景；
  - 严格控制并发修改的系统（如金融、库存）。

#### 2. **事务隔离级别（Isolation Level）**

- **作用**：限制事务之间的可见性，防止脏读、不可重复读、幻读。
- **四种隔离级别**（由低到高）：
  - 读未提交（Read Uncommitted）；
  - 读已提交（Read Committed）；
  - 可重复读（Repeatable Read）；
  - 串行化（Serializable）；
- **使用方式**：通过 `SET SESSION TRANSACTION ISOLATION LEVEL ...` 设置；
- **选择依据**：根据业务对一致性和性能的权衡。

#### 3. **MVCC（多版本并发控制）**

- **作用**：实现“读写不互斥”，提升并发性能；
- **原理**：为每条记录维护多个“版本快照”，读取时根据事务版本号决定可见性；
- **默认生效范围**：`REPEATABLE READ` 和 `READ COMMITTED` 下自动生效；
- **适用场景**：读多写少的系统（如用户查询、商品浏览等）。

### 事务隔离级别

1. **读未提交（Read Uncommitted）**

   - 可以读取其他事务**尚未提交的数据**。

   - 可能出现：**脏读、不可重复读、幻读**。

   - 底层实现：**不使用 MVCC 的 Read View**，直接读取最新版本的数据。

2. **读已提交（Read Committed）**

   - 只能读取其他事务**已提交的数据**。

   - 可以避免**脏读**，但仍可能发生**不可重复读、幻读**。

   - 底层实现：**每次执行 SQL 前**都会重新生成一个新的 Read View。

3. **可重复读（Repeatable Read）** MYSQL默认隔离级别

   - 一个事务中多次读取相同数据，结果保持一致。

   - 可以避免**脏读和不可重复读**，但仍可能出现**幻读**。

   - 是 **InnoDB 的默认隔离级别**。

   - 底层实现：**事务开始时生成一个 Read View**，整个事务期间都基于这个视图读取数据。

4. **串行化（Serializable）**

   - 所有读写操作都加锁，事务**严格串行执行**。

   - 可以完全避免**脏读、不可重复读、幻读**。

   - 一致性最强，但**性能最低**。

   - 底层实现：对读取的记录加共享锁，对写入加排它锁，**强制事务串行化执行**。

**四种隔离级别对应读现象总结：**

| 隔离级别 | 脏读 | 不可重复读 | 幻读 |
| -------- | ---- | ---------- | ---- |
| 读未提交 | ✅    | ✅          | ✅    |
| 读已提交 | ❌    | ✅          | ✅    |
| 可重复读 | ❌    | ❌          | ✅    |
| 串行化   | ❌    | ❌          | ❌    |

### 可重复读隔离级别下，A 事务提交的数据，在 B 事务能看见吗？

**不能。**

1. 可重复读是通过 **MVCC（多版本并发控制）** 实现的。
2. 当事务 B 开始后（执行 `BEGIN`），第一次执行查询语句时会创建一个 **Read View**。
3. 后续 B 事务中的所有查询，都基于这个 **Read View** 读取数据。
4. Read View 会通过 **undo log** 找到**事务 B 开始时能看到的版本**，不包含之后其他事务提交的数据。
5. 因此，在 B 事务中，即使 A 已经提交了新数据，B 查询时也 **看不到 A 的新记录**，每次查到的都是一样的旧数据。

### 可重复读的幻读问题？

1. **问题背景**
    可重复读（Repeatable Read）隔离级别下，虽然可以避免不可重复读，但不能完全避免幻读，尤其是在**插入新记录**的场景中仍可能发生幻读。

2. **可重复读的原理**
    MySQL 的可重复读是通过 MVCC（多版本并发控制）实现的：

   - 在执行第一条查询语句时生成一个 **Read View** 快照；

   - 之后事务内的所有查询都基于这个快照读取数据，保证一致性。

3. **幻读产生的原因**

   - 可重复读能保证读取老数据，但**不能阻止新数据被插入**；

   - 如果另一个事务插入了符合当前查询条件的新数据并提交，当前事务在后续操作中就可能“读到”这些新插入的数据，形成幻读。

4. **具体示例简述**

   - 事务 A 执行 `SELECT * FROM t WHERE id = 5`，此时 id=5 的记录不存在，Read View 生成；

   - 事务 B 插入 id=5 的记录并提交；

   - 事务 A 
     - 先update B写入的新数据
     - 再查询，`SELECT * FROM t WHERE id = 5`，就能读到刚插入的记录；

   - 幻读发生：**前后两次查询结果不一致，出现了“幻觉中的数据”**。

5. 事务 A 对刚提交的数据做了什么？

   * **事务 A 执行 `UPDATE t SET ... WHERE id = 5`**
     - 注意：虽然 A 的 Read View 看不到事务 B 插入的数据（因为是后来插入的事务），
     - **但 update 是当前读（Current Read）操作，不走 Read View，而是直接读当前版本的数据。**
     - 于是事务 A **找到了这条数据并加锁成功**。
   * **InnoDB 内部动作：**
     - 一旦事务 A 对这条记录加锁，**这条记录的 `trx_id`（也就是版本标识）就被覆盖成事务 A 的 ID**；
     - 意味着：**从现在开始，这条记录就是事务 A 自己写的了。**

6. **本质区别**

   - 幻读不是已有数据变了，而是**之前不存在的记录“突然出现”了**；

   - 不可重复读关注的是“旧记录值变了”，而幻读关注的是“多了新记录”。

7. **解决方式**

   - 通过串行化（Serializable）隔离级别彻底避免幻读；

   - 或手动加锁，如 InnoDB 的 **Next-Key Lock（间隙锁 + 行锁）** 来阻止其他事务插入新记录。

### **Mysql 设置了可重复读隔离级别后，怎么保证不发生幻读？**

- **幻读的定义**
   幻读是指同一个事务中，两次执行相同的查询语句时，返回的结果集中出现了“新增”或“消失”的行。通常发生在其他事务插入了满足查询条件的新记录时。
- **可重复读是否能避免幻读**
   在 MySQL 的 InnoDB 存储引擎中，**可重复读（REPEATABLE READ）默认开启了 next-key lock**，通过锁住记录及其前后的间隙来防止幻读。
- **什么是 next-key lock**
   next-key lock 是 **记录锁（Record Lock）+ 间隙锁（Gap Lock）** 的组合。它不仅锁住已经存在的记录，还锁住记录之间的“间隙”，防止其他事务在这些间隙中插入新记录。
- **如何避免幻读**
  - 在可重复读隔离级别下，InnoDB 自动使用 next-key lock 机制。
  - 使用 `SELECT ... FOR UPDATE` 或 `SELECT ... LOCK IN SHARE MODE` 等**加锁查询语句**，能够显式触发 next-key lock。
  - 即使查询条件未命中任何记录，InnoDB 也会对满足条件的“插入位置”加间隙锁，防止新插入。
- **总结关键点**
  - 幻读的本质是“新增记录导致查询结果不一致”；
  - InnoDB 的可重复读通过 **next-key lock** 自动防止幻读；
  - 使用加锁语句如 `SELECT ... FOR UPDATE` 可以增强对幻读的控制。

### **MySQL（InnoDB）中的锁分类**

- **按照作用对象分类**
  - **表级锁（Table Lock）**
     锁住整张表，其他事务不能对该表进行任何读写操作。开销小，效率高，但并发能力弱。
  - **行级锁（Row Lock）**
     只锁住某几行记录，允许其他事务访问表中未被锁定的行。并发能力强，但开销大、容易产生死锁。

- **按照加锁方式分类**
  - **共享锁（S 锁，Shared Lock）**
     允许多个事务**同时读取一行记录**，但**不允许修改**。
     通常由 `SELECT ... LOCK IN SHARE MODE` 或串行化隔离级别下的普通 `SELECT` 触发。
  - **排他锁（X 锁，Exclusive Lock）**
     一个事务**加了排他锁后，其他事务既不能读也不能写**。
     通常由 `INSERT`、`UPDATE`、`DELETE`、`SELECT ... FOR UPDATE` 等语句触发。

- **InnoDB 特有的三种行级锁（重点）**

  InnoDB 为了防止幻读，引入了三种更细致的行级锁：

  - **记录锁（Record Lock）**
     锁定单条已存在的记录，防止其他事务对该记录进行修改。
     例如：`select * from user where id = 1 for update` 会对 `id = 1` 加记录锁。
  - **间隙锁（Gap Lock）**
     锁定“两个记录之间的间隙”，防止其他事务在这个范围内插入新记录。
     例如：当查找条件未命中任何记录时，为防止别人在这“空隙”中插入，会加间隙锁。
  - **Next-Key Lock（临键锁）**
     是“记录锁 + 间隙锁”的组合，锁住当前记录和前面的间隙。
     例如：`select * from user where id > 10 and id < 20 for update`，会对符合条件的记录及其前后的间隙都加 next-key lock。

### **串行化隔离级别是通过什么实现的？**

- **串行化的目标**
   实现事务像“一个个排队执行”那样运行，从而**完全避免脏读、不可重复读、幻读**。本质上就是让并发变成“看起来像串行”。
- **InnoDB 是怎么做到的？**
   在 MySQL 的 InnoDB 存储引擎中，串行化隔离级别主要通过以下三种手段来实现串行效果：
  - **对普通 SELECT 加锁（不是快照读）**
     在串行化隔离级别下，普通的 `SELECT` 语句也会对读取到的记录加 **共享锁（S 锁）**，不再是非阻塞的快照读（不像可重复读那样通过 MVCC 实现一致性视图）。
  - **使用 next-key lock 锁定范围**
     所有加锁的读操作（比如 `SELECT`）会使用 **next-key lock（记录锁 + 间隙锁）**，锁住当前记录以及它前后的间隙，防止其他事务插入新记录，**避免幻读**。
  - **限制对加锁记录的写操作**
     一旦某事务用 SELECT 锁定了某些记录，其他事务就**不能对这些记录做增删改**，否则会阻塞或产生死锁，从而强制其他事务“排队”，模拟串行执行。
- **极端情况可能退化为表锁**
   在某些极端情况下（如范围太大、索引失效），InnoDB 可能会对整张表加锁，来确保串行效果。这虽然安全，但会极大影响并发性能。
- **总结关键词记忆法**
   串行化隔离级别实现方式 = **加锁 SELECT + next-key lock + 强制排队执行**

### **MVCC 实现原理**

- **什么是 MVCC？**
   MVCC（Multi-Version Concurrency Control，多版本并发控制）允许多个事务**同时读取同一行记录**，但每个事务读到的是**它启动时刻的版本快照**，从而实现 **非阻塞读（快照读）**。
- **MVCC 的作用**
  - 避免加锁，提高并发性能；
  - 在不加锁的情况下，防止脏读、不可重复读；
  - 是“读已提交”和“可重复读”隔离级别的核心机制。

**Read View 是 MVCC 的关键机制**

- **什么是 Read View？**
   Read View 是事务在执行查询时生成的一份快照，用于判断哪些记录版本对当前事务可见。它决定了一个事务能“看到”哪些已经提交或未提交的数据。
- **不同隔离级别下 Read View 的生成时机**
  - 读已提交（Read Committed）：**每条 SELECT 执行前都会重新生成 Read View**；
  - 可重复读（Repeatable Read）：**事务第一次执行 SELECT 时生成一个 Read View，整个事务期间都使用这个快照**。

**Read View 的四个重要字段**

- **creator_trx_id**：当前事务的事务 ID，即创建这个 Read View 的事务 ID。
- **m_ids**：当前活跃事务的 ID 列表（已开启但未提交）。注意是一个列表。
- **min_trx_id**：m_ids 中最小的事务 ID，表示当前系统中“已启动但未提交的最早事务”。
- **max_trx_id**：数据库分配的“下一个事务 ID”，是“当前所有事务中最大的 ID + 1”。

**隐藏列与版本链（undo 日志）**

- InnoDB 中每条记录都有两个隐藏字段：
  - `trx_id`：记录这条数据是被哪个事务修改的；
  - `roll_pointer`：指向 undo 日志中的旧版本记录，用于构建“版本链”。
- 每次对记录修改时，都会把旧数据写入 undo 日志，并通过 `roll_pointer` 指向旧版本，从而支持事务回滚 & 快照访问。

**Read View 如何判断记录是否可见？**

访问一条记录时，会根据它的 `trx_id` 和当前 Read View 的四个字段判断记录是否对当前事务可见：

- 如果 `trx_id < min_trx_id`：表示这条记录由**已提交事务**生成 → 可见；
- 如果 `trx_id ≥ max_trx_id`：表示是**还未开始的事务生成的记录** → 不可见；
- 如果 `trx_id ∈ m_ids`：表示是**还在运行的事务生成的记录** → 不可见；
- 如果 `trx_id ∉ m_ids` 且在 `[min_trx_id, max_trx_id)` 范围内：表示**已提交事务生成的记录** → 可见。
- 自己写的记录永远对自己可见，不走 Read View 判断流程；Read View 只负责判断**别人写的记录你能不能看**。

### **一条 UPDATE 是不是原子性的？为什么？**

- **是原子性的。**
   InnoDB 中的一条 `UPDATE` 操作**具有原子性**，即使它涉及多个步骤，要么全部完成，要么全部不生效。
- **原子性的保证机制：锁 + undo log 日志**
  - **行级锁保护并发安全**
     执行 `UPDATE` 时，InnoDB 会对目标行加上 **行级排他锁（X 锁）**，保证同一时间只有当前事务能修改这条记录，防止并发事务干扰。
  - **undo log 支持失败回滚**
     在更新记录前，InnoDB 会将旧值写入 `undo log`，一旦事务失败或被回滚，可以通过 `undo log` 恢复数据到更新前的状态。
- **更新过程简化流程如下**
  1. 查找记录并加排他锁；
  2. 记录旧数据到 undo log；
  3. 写入新值；
  4. 事务提交前，数据对其他事务不可见；
  5. 若失败或回滚，用 undo log 撤销操作。
- **注意：一条 UPDATE 是原子性的，但整个事务仍然是多个语句组成的整体事务。只有事务提交，数据才对其他事务可见。**

### **滥用事务，或一个事务中包含大量 SQL 的弊端**

- **资源长时间不释放**
  - 事务提交前，**锁、内存、undo log、buffer pool 等资源都无法释放**；
  - 如果事务执行时间过长，容易造成**资源占用过多或锁竞争严重**。
- **锁定范围大，容易导致死锁或锁等待超时**
  - 一个事务内执行多条 SQL，可能锁住大量行或范围；
  - 其他事务并发访问这些数据时容易阻塞，甚至出现死锁。
- **undo log 占用大量空间，回滚成本高**
  - 每条更新语句都会生成一条 undo log；
  - SQL 越多，生成的回滚记录越多，占用空间大；
  - 一旦事务失败，回滚时间可能很长，严重影响性能。
- **MVCC 多版本链变长，影响读取性能**
  - 事务未提交前，新旧数据版本都要保留；
  - 对其他事务来说，查询时需要遍历多个旧版本，增加 IO 和判断成本。
- **主从延迟增加，影响数据同步**
  - 主库提交事务后才会将 binlog 同步给从库；
  - 如果一个事务执行时间过长，从库必须等待主库完成整个事务，造成复制延迟；
  - 比如主库一个事务执行 10 分钟 → 从库同步延迟也可能达 10 分钟。

## 锁

### 讲一下 MySQL 里有哪些锁？

**一、全局锁**

- 通过 `FLUSH TABLES WITH READ LOCK` 命令将整个数据库置于只读状态。
- 会阻塞其他线程对数据库结构和数据的修改操作（如增删改）。
- 常用于**全库逻辑备份**，保证备份期间数据一致。
- 与物理备份不同，逻辑备份要求数据在导出期间不能变动。

**二、表级锁**

- MySQL 中表级锁包括以下几种：

  **1. 表锁（table lock）**

  - 使用 `LOCK TABLES` 语句对表加锁。
  - 会限制当前线程之外的其他线程进行读写操作。

  **2. 元数据锁（metadata lock，简称 MDL 锁）**

  - 当执行数据库中的表操作（查询、更新、结构变更等）时自动加上。
  - 对普通 CRUD 操作加的是 MDL 读锁。
  - 对表结构变更操作加的是 MDL 写锁。
  - 用于防止表结构在使用过程中被修改，确保元数据一致性。

  **3. 意向锁**

  - 执行插入、更新、删除等操作前先加意向锁（如意向独占锁）。
  - 用于快速判断表中是否已经被加了记录锁，提高锁冲突检测效率。
  - 由 InnoDB 自动加锁，用户无需手动干预。

**三、行级锁**

- 由 InnoDB 引擎支持，MyISAM 不支持行级锁。

- 锁粒度最小，**并发性最高**，但加锁成本相对较高。

  **1. 记录锁（Record Lock）**

  - 锁住某一条具体记录。
  - 分为共享锁（S 锁）与排他锁（X 锁），读读共享、写写互斥。

  **2. 间隙锁（Gap Lock）**

  - 锁住一个区间（不包含记录本身），用于解决幻读问题。
  - 主要在可重复读隔离级别下启用。

  **3. Next-Key Lock（临键锁）**

  - 是记录锁与间隙锁的组合。
  - 锁定一个范围，并锁定范围内已有记录本身。
  - 防止幻读，确保可重复读。

### 数据库的表锁和行锁有什么作用？

**一、表锁的作用**

- **整体控制**：
   表锁可以用来控制整个表的并发访问。当一个事务获取了表锁，其他事务将无法对该表进行任何读写操作，保障数据的完整性与一致性。
- **粒度大**：
   表锁的加锁范围大，锁定整张表，可能会影响其他操作，带来锁竞争和性能问题。
- **适用于大批量操作**：
   适合用于重建表、大量导入导出等批量数据处理场景，能够保证操作的完整性。

**二、行锁的作用**

- **细粒度控制**：
   行锁可以精确地控制对某些行的访问，不会影响表中其他数据的访问，提高系统并发能力与性能。
- **减少锁冲突**：
   相比表锁，行锁只锁定必要的数据行，避免了对整张表加锁，减少锁冲突概率。
- **适用于频繁单行操作**：
   适合高并发、频繁更新某些单独记录的场景，例如订单系统中的订单修改、删除等操作。

### **MySQL的binlog详解**

1. **binlog基本概念**
   - binlog是MySQL的Server层实现的日志
   - MySQL在完成一条更新操作后，Server层会生成一条binlog
   - 事务提交时，会将该事务执行过程中产生的所有binlog统一写入binlog文件
   - 所有存储引擎都可以使用binlog
2. **binlog的写入特性**
   - 采用追加写入方式，写满一个文件后创建新文件继续写入
   - 不会覆盖以前的日志，保存的是全量日志
   - 主要用于数据备份恢复和主从复制
   - binlog文件大小可配置，达到上限后会自动切换到新文件
3. **binlog的记录内容**
   - 记录所有**数据库表结构变更和表数据修改**的日志
   - 不会记录查询类的操作，如SELECT和SHOW操作
   - 可用于数据恢复和时间点还原
4. **binlog的三种格式类型**
   - **STATEMENT格式**
     - 记录每一条修改数据的SQL语句（相当于记录了逻辑操作）
     - 可视为逻辑日志，从库根据SQL语句重现
     - 存在动态函数问题：如使用uuid()或now()这类函数时，主从执行结果可能不一致
     - 优点是日志量较小，节省存储空间和网络带宽
   - **ROW格式**
     - 记录行数据最终被修改成什么样了（记录数据变化的结果）
     - 不能称为逻辑日志，不存在STATEMENT格式的动态函数问题
     - 缺点是每行数据变化都会被记录，如批量update语句可能产生大量记录
     - 使binlog文件过大，而STATEMENT格式下只记录一条update语句
     - 优点是主从复制的数据一致性更高
   - **MIXED格式**
     - 结合了STATEMENT和ROW两种模式的优点
     - 会根据不同情况自动选择使用ROW模式或STATEMENT模式
     - 对于可能导致主从不一致的操作使用ROW格式，其他情况使用STATEMENT格式
     - 平衡了日志大小和复制准确性
5. **binlog的应用场景**
   - 数据备份与恢复：可以用于时间点恢复
   - 主从复制：实现数据的分布式部署
   - 数据审计：跟踪数据变更历史

### Undo Log 日志的作用是什么？

**一、基本作用**

- undo log 是一种**撤销日志**，用于在事务未提交前，记录修改前的数据版本。
- 它**保障了事务的原子性（Atomicity）**，是实现 ACID 中原子性的重要机制。
- 一旦事务出现异常或主动回滚，可以通过 undo log 撤销已执行的操作，恢复数据。

**二、undo log 的工作流程**

- 在事务执行期间，MySQL 会先把数据修改前的版本记录到 undo log。
- 若事务发生回滚，InnoDB 会读取 undo log 中的数据，执行相应的反向操作，撤销之前的修改。

**三、不同操作下 undo log 的记录内容**

- **插入操作**：记录主键值，回滚时执行删除操作。
- **删除操作**：记录整条被删除的记录，回滚时重新插入。
- **更新操作**：记录更新前的旧值，回滚时将新值改回旧值。

**四、举例说明**

- 若删除了一条记录，undo log 会记录该记录的完整内容。
- 回滚时会从 undo log 中读取原始数据，再执行 `INSERT` 操作恢复数据。

### **redo log与undo log的关系与区别**

1. **两种日志的基本定义**
   - **redo log**：InnoDB引擎的重做日志，记录数据页的物理修改，用于实现事务的持久性
   - **undo log**：InnoDB引擎的回滚日志，记录事务修改前的数据，用于实现事务的原子性和MVCC
2. **为什么有了undo log还需要redo log**
   - Buffer Pool虽提高读写效率，但基于内存，断电会导致数据丢失
   - undo log无法解决数据持久性问题，只能实现回滚操作
   - redo log通过WAL技术保证已提交事务的持久化，解决崩溃恢复问题
3. **WAL技术原理**
   - Write-Ahead Logging：先写日志，再写数据
   - 更新数据时：先修改内存中的Buffer Pool（标记为脏页）→ 记录redo log → 定期将脏页刷新到磁盘
   - 优势：将随机写转变为顺序写，大幅提升性能
4. **两种日志的核心区别**
   - 记录内容
     - redo log记录"事务完成后"的数据状态（更新后的值）
     - undo log记录"事务开始前"的数据状态（更新前的值）
   - 作用目标
     - redo log保证持久性（Durability）
     - undo log保证原子性（Atomicity）
   - 应用场景
     - redo log用于系统崩溃后的数据恢复
     - undo log用于事务回滚和MVCC并发控制
5. **redo log的特点**
   - 物理日志：记录具体数据页的具体修改（如对某表空间某数据页某偏移量做了什么更新）
   - 顺序追加写入，不需要随机寻址，I/O效率高
   - 具有crash-safe能力，保证已提交事务不丢失
6. **事务恢复机制**
   - 事务未提交崩溃：通过undo log回滚事务
   - 事务已提交崩溃：通过redo log恢复事务
   - 整个过程保证了事务的ACID特性

### redo log 怎么保证持久性的？

**一、作用概述**

- redo log 是 InnoDB 用于**保证事务持久性（Durability）**的关键机制。
- 它在崩溃恢复时用于将已提交事务的数据重新写入磁盘。

**二、持久性保证方式**

- **WAL（Write-Ahead Logging）机制**
  - 所有修改操作先记录到 redo log，再写入磁盘。
  - 即使宕机，系统也能通过 redo log 记录恢复数据。
- **顺序写入机制**
  - redo log 采用追加写的方式，将日志顺序写入文件尾部。
  - 减少随机写，提高磁盘写入性能。
- **Checkpoint 机制**
  - MySQL 定期将内存数据刷盘，同时记录当前 LSN（Log Sequence Number）到磁盘。
  - 恢复时根据 LSN 定位，从 redo log 的正确位置开始重做，确保操作按顺序执行。

### 能不能只用 binlog 不用 redo log？

**不能**

- **binlog（归档日志）**
  - 属于 Server 层日志，记录的是 SQL 语句或逻辑变更。
  - 无法记录事务执行过程中哪些具体的数据页（脏页）尚未刷入磁盘。
- **redo log（重做日志）**
  - 属于存储引擎层（如 InnoDB 引擎）的日志。
  - 专门记录事务对数据页的物理修改过程。
  - 可以记录哪些数据页还没有刷盘，崩溃恢复时通过重做这些操作恢复数据，确保事务持久性。

**结论：**

- 如果只用 binlog 而不用 redo log，发生宕机时，MySQL 无法恢复尚未刷盘的脏页数据，会导致数据丢失，无法保障事务的持久性。

### binlog 两阶段提交过程是怎样的？

**一、为什么要用两阶段提交？**

- redo log 与 binlog 分别属于引擎层与 Server 层，需要同时持久化，单独提交可能导致两个日志不一致。
- 为保证 redo log 和 binlog 一致性，MySQL 内部使用两阶段提交（XA事务）进行协调。

**二、两阶段提交具体过程**

- **第一阶段：Prepare 阶段**
  - MySQL 向 InnoDB 写入 redo log 并刷盘（确保持久）。
  - redo log 的事务状态标记为 prepare。
  - 此阶段未写入 binlog。
- **第二阶段：Commit 阶段**
  - 将事务对应的 binlog 写入磁盘（sync_binlog=1保证磁盘持久）。
  - MySQL 收到 binlog 写成功后，调用 InnoDB 引擎的事务提交接口。
  - 引擎将 redo log 事务状态改为 commit。

**三、崩溃恢复如何保证一致？**

- 若崩溃发生在 Prepare 阶段（redo log 刷盘、binlog 未刷盘）：
  - 恢复时在 binlog 找不到对应事务，事务回滚。
- 若崩溃发生在 Commit 阶段（redo log 与 binlog 都刷盘完成）：
  - 恢复时能在 binlog 找到事务，则事务提交。

**四、关键点总结**

- 两阶段提交保证 redo log 与 binlog 的数据一致性。
- 以 binlog 写入成功作为事务提交成功的标识。

### update语句的具体执行过程是怎样的？

**一、查询待更新记录**

- 执行器调用引擎接口，按主键从存储引擎获取记录：
  - 若数据在 Buffer Pool 中，直接返回给执行器。
  - 若不在 Buffer Pool，从磁盘加载进 Buffer Pool 后返回。

**二、引擎层执行更新**

- 判断记录新旧值：
  - 如果更新值相同，不进行修改操作。
  - 如果更新值不同，执行后续更新流程。

**三、事务开启，记录 undo log**

- 事务开启时，InnoDB 记录修改前数据到 undo log（旧值）。
- undo log 存于 Buffer Pool 的 Undo 页中（同时 undo 页本身也记录到 redo log）。

**四、记录 redo log（WAL机制）**

- InnoDB 在内存修改数据页（标记为脏页），并记录到 redo log（未立即刷盘）。
- redo log 采用 WAL 技术，先记录 redo log，之后异步将脏页刷盘。

**五、写入 binlog**

- 执行器生成 binlog 并缓存到 binlog cache 中。
- 事务提交时一次性将 binlog cache 写入 binlog 文件并持久化磁盘。

**六、两阶段提交事务**

- **prepare阶段**：
  - redo log 刷盘并标记事务状态为 prepare。
- **commit阶段**：
  - binlog 刷盘。
  - redo log 状态改为 commit。

**七、完成更新**

- 至此，整个更新操作完成，数据得到持久化。

### Buffer Pool是啥？

Buffer Pool 是 **InnoDB 存储引擎**中的一个重要内存结构，用于缓存磁盘上的数据和索引页。简单理解：

- **作用**：
  - 存储频繁访问的 **数据页和索引页**，减少磁盘 I/O。
  - 提升数据库性能（读写效率）。
- **工作原理**：
  - 当数据首次被查询时，会从磁盘加载到 Buffer Pool 中。
  - 后续访问该数据时，直接从 Buffer Pool（内存）中读取，速度更快。
  - 数据被修改时，也先修改 Buffer Pool 中的数据页（标记为脏页），随后再异步写回磁盘。
- **关键点**：
  - Buffer Pool 大小会直接影响 MySQL 性能，通常设置为系统内存的 60% ~ 80%。
  - Buffer Pool 使用 LRU 算法管理数据页，保证内存使用效率。

总结：
 Buffer Pool 就是 MySQL（InnoDB）用来提高效率的缓存池，用于减少慢速的磁盘操作，加快数据的读写速度。

### MySQL 如何保障数据不丢失？

**关键机制是 redo log：**

- 每次修改数据（CRUD 操作）时，MySQL（InnoDB）都会**隐式地开启事务**。
- **执行事务时：**
  - InnoDB 首先在内存（Buffer Pool）中修改对应的数据页（通常为聚簇索引的叶子节点）。
  - 修改的数据（修改前后状态）会先被记录到内存的 **redo log buffer** 中。
- **事务提交时：**
  - InnoDB 将 redo log buffer 中的日志刷新到磁盘（redo log 文件）。
  - 此时事务正式完成提交，即使数据库突然宕机，也能通过 redo log 恢复未刷盘的数据页。
- 事务提交时，redo log 会**立即刷盘**，此时即使系统宕机，数据也不会丢失。
- 系统重启后，可以通过 redo log 将未及时刷盘的数据恢复，保障数据持久性。

**重要补充：隐式事务**

- 即使不显式使用 `BEGIN`、`COMMIT` 事务语句，InnoDB 也会自动为每条独立的 CRUD 语句创建一个隐式事务并自动提交。
- 因此，即便单条简单 CRUD 操作，也一定有 redo log 参与数据持久性保障。

### 为什么要写 Redo Log，而不是直接写到 B+树里？

**一、性能原因**

- Redo log 是**顺序写磁盘**，而 B+树的数据页是**随机写磁盘**。
- 顺序写的磁盘性能远远高于随机写，可以极大提升事务提交效率。

**二、恢复能力**

- Redo log 记录的是**物理级别的修改**（修改哪个数据页的哪个位置，改了什么数据）。
- 出现数据库宕机时，可以通过 Redo log 快速恢复未完成刷盘的数据页。
- 保证了事务的**持久性**，即使系统崩溃，已提交的事务数据也不会丢失。

**三、具体实现原理**

- 修改数据页前，先将变更信息记录到 redo log 中。
- 数据页之后异步刷盘，不必实时写磁盘。
- 崩溃恢复时，根据 redo log 记录完成数据页的恢复（redo 操作），确保数据安全。

### MySQL 两次写（Double Write Buffer）了解吗？

**一、为什么需要 Double Write Buffer？**

- Linux 文件系统页（OS Page）大小为 4KB，而 InnoDB 数据页默认大小为 16KB。
- 写入磁盘时，InnoDB 的一页数据（16KB）需要分成 4 个操作系统页（4KB）写入。
- 多页写入非原子操作，如果写一半宕机，导致数据页损坏（部分写入问题）。
- Redo Log 无法解决这种物理数据页损坏问题，因为 redo log 只记录页的修改内容，并不保存完整的数据页副本。

**二、Double Write Buffer 的具体过程**

1. 脏页数据先通过内存拷贝到内存中的 **Double Write Buffer**（约 2MB）。
2. 然后，以顺序写方式刷入磁盘上的共享表空间（共写两次，每次 1MB）。
3. 完成后，再从内存中的 Double Write Buffer 将数据页随机写入对应的磁盘位置（.ibd文件）。

**三、Double Write Buffer 出现异常情况恢复机制**

发生宕机可能情况：

- **情况一**：步骤①前宕机。数据未写入磁盘，有 redo log 可恢复。
- **情况二**：步骤①后，步骤②前宕机。Double Write Buffer 内存数据丢失，有 redo log 可恢复。
- **情况三**：步骤②完成，步骤③前宕机。共享表空间有完整的数据页副本，可通过 Double Write Buffer 中的备份副本恢复损坏的数据页。

**四、为什么 redo log 无法替代 Double Write Buffer？**

- Redo log 只记录对数据页的**修改操作**（物理修改），不保存数据页副本。
- 如果数据页本身损坏，redo log 再怎么重做操作也无法修复数据损坏。
- 因此需要有完整的数据页备份（Double Write Buffer）用于物理恢复。

总结：

- Double Write Buffer 是针对页数据写入磁盘时的物理保障机制。
- Redo log 是对数据修改操作的日志记录，无法替代物理数据页副本的作用。

## 性能调优

### MySQL的EXPLAIN有什么作用？

**一、基本作用**

- 用于分析SQL语句的**执行计划**，判断语句执行效率。
- 帮助开发者定位SQL查询是否使用索引、是否进行全表扫描，以及是否产生临时表或排序。

**二、关键字段含义**

- **possible_keys**：可能使用到的索引。
- **key**：实际使用的索引，NULL则表示没有使用索引。
- **key_len**：索引使用的长度（越短越好）。
- **rows**：扫描的行数（越少越好）。
- **type**：扫描方式（重要指标，从好到坏排序见下文）。
- **Extra**：额外的重要信息（如filesort、temporary、using index等）。

**三、type字段扫描方式（从最好到最差）**

1. **const**（常量扫描）
   - 查询条件直接匹配主键或唯一索引，只扫描一行。
   - 速度最快、效率最高。
2. **eq_ref**（唯一索引扫描）
   - 通常出现在多表JOIN时使用主键或唯一索引，返回一条数据。
   - 效率非常高，多用于表连接。
3. **ref**（非唯一索引扫描）
   - 非唯一索引上的扫描，可能返回多条记录。
   - 范围小，不用全表扫描，性能较好。
4. **range**（索引范围扫描）
   - 用于where条件中使用范围查询（如between、in、<、>）。
   - 使用索引范围内的数据，性能较好。
5. **index**（全索引扫描）
   - 扫描整个索引树，不扫描数据文件。
   - 比全表扫描好一点，但性能依旧一般。
6. **ALL**（全表扫描）
   - 最差的方式，全表逐行扫描，没有使用索引。
   - 性能最低，需尽量避免。

**四、Extra字段重要信息**

- **Using filesort**
   查询包含ORDER BY或GROUP BY，但未能通过索引直接排序，导致MySQL需要额外排序，性能差。
- **Using temporary**
   查询需要建立临时表存储中间结果（通常出现在GROUP BY时），性能差，需尽量避免。
- **Using index**
   查询的数据直接来自索引，无需访问数据表，效率高（索引覆盖）。

**总结：**

通过EXPLAIN，可以准确定位SQL性能问题，分析执行效率瓶颈，优化SQL语句和索引设计。

### MySQL 主从复制的原理和过程

**一、主从复制的作用**

- 提高系统性能（读写分离）
- 数据备份，提高容灾能力

**二、主从复制的核心机制**

- 基于 **binlog** 日志完成复制。
- 主库所有数据变更记录到 binlog，从库读取 binlog 重做操作，最终实现数据一致。

**三、具体复制过程（核心三步骤）**

**1、写入binlog（主库）**

- 主库收到事务请求，先记录到 **binlog 日志**。
- 提交事务后，主库更新存储引擎数据并返回客户端成功响应。

**2、同步binlog（从库I/O线程）**

- 从库开启专门的 **I/O线程** 连接主库的 **log dump线程**。
- I/O线程从主库获取 binlog 内容并写入从库的 **relay log（中继日志）**。
- 写完 relay log 后，返回主库“同步成功”的响应。

**3、回放binlog（从库SQL线程）**

- 从库有一个 **SQL线程** 专门回放 relay log 中的记录。
- 从库执行 relay log 中的SQL语句或事件，更新自己的存储引擎数据。
- 最终从库数据与主库达到一致。

**四、重要特点（异步复制）**

- 主库执行事务后立即返回客户端，不等待从库完成复制。
- 所以复制过程是**异步的**，主库与从库之间有微小延迟。

**总结：** 主从复制分成 **主库记录binlog → 从库同步binlog到relay log → 从库回放relay log** 三个步骤完成。

### 分库和分表是什么？有什么区别？

**一、定义与作用**

- **分库**：
   将一个大数据库按一定规则拆分成多个独立的小数据库，每个数据库实例只存储一部分数据。
  - 核心作用：解决**单机数据库容量和性能瓶颈**。
- **分表**：
   将单个数据库中的一张大表拆分成多张小表，每个表只存储一部分数据。
  - 核心作用：解决单表**数据量过大导致查询效率低下**的问题。

**二、具体实现方式**

分库分表可以从两个维度进行：

1. **垂直分库（纵向拆分）**
   - 按照**业务模块或功能划分**，如订单库、用户库、支付库等。
   - 减轻单个库压力，提升系统扩展性。
   - 未彻底解决单表过大的问题。
2. **水平分库（横向拆分）**
   - 根据**某个字段（如用户ID或订单号）**进行水平拆分，把数据分散到不同的数据库实例中。
   - 可以有效解决单机瓶颈，实现数据库层面的分布式扩展。
3. **垂直分表（纵向拆分）**
   - 将一张表中的多个字段（尤其是大字段或访问频率低的字段）拆分为多个表。
   - 减小单表宽度，提高数据查询效率，提升内存缓存效率。
4. **水平分表（横向拆分）**
   - 在**同一个数据库实例**内，根据特定规则（如取模）将数据拆分为多张结构相同的表。
   - 解决单表数据量过大的问题，但数据仍然在单一数据库实例内，未完全实现分布式扩展。



**三、总结区分**

- **垂直分库**：按业务拆分，解决业务层面的压力分担。
- **水平分库**：按数据划分，实现数据库层面的分布式扩展。
- **垂直分表**：减少单表宽度，优化IO、提高查询效率。
- **水平分表**：减少单表数据量，解决单表性能问题，但未解决单库容量限制。
