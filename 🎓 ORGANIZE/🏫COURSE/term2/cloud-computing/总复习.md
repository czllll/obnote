# **Chapter 1: Introduction to Cloud Computing**

## **1. Foundational Concepts**

### 1.1 What is Cloud Computing?

Cloud computing is the **on-demand delivery** of IT resources (compute, storage, network) over the Internet with **pay-as-you-go** pricing. It transforms computing into a **utility-like service** (类似水电的公用服务).

### 1.2 NIST Definition

NIST defines cloud computing with **five essential characteristics**:

- **On-demand self-service** – Users can provision computing resources without human interaction.
- **Broad network access** – Services accessible from anywhere via standard protocols.
- **Resource pooling** – Resources are dynamically assigned from a shared pool.
- **Rapid elasticity** – Automatic and scalable provisioning.
- **Measured service** – Usage is tracked and billed accordingly.

------

## **2. Technological Drivers: Exponential Growth**

Cloud became feasible due to exponential advances across three domains:

| Domain  | Law                              | Trend                                               |
| ------- | -------------------------------- | --------------------------------------------------- |
| Compute | Moore’s Law (18-month doubling)  | >200B transistors in single GPU                     |
| Network | Gilder’s Law (9-month doubling)  | 400 Gbps ports standard                             |
| Storage | Kryder’s Law (12-month doubling) | Global data exceeds **zettabytes (ZB, 10²¹ bytes)** |



> These trends drive **scale**, **efficiency**, and **low cost**, forming the bedrock of cloud economics.

------

## **3. Cloud Service Models (Three Layers)**

| Model                     | User Responsibility       | Examples                |
| ------------------------- | ------------------------- | ----------------------- |
| **IaaS** (Infrastructure) | OS, runtime, applications | Amazon EC2, Azure VM    |
| **PaaS** (Platform)       | Application logic only    | Google App Engine       |
| **SaaS** (Software)       | None (fully managed)      | Salesforce, Google Docs |



> IaaS offers **maximum control**, SaaS offers **maximum abstraction**.

------

## **4. Cloud Deployment Models (Four Types)**

| Model               | Scope                        | Use Case                                    |
| ------------------- | ---------------------------- | ------------------------------------------- |
| **Private Cloud**   | Single organization          | Banks, governments                          |
| **Public Cloud**    | Open to public               | Startups, global services                   |
| **Community Cloud** | Shared among specific groups | Academic consortia, national science clouds |
| **Hybrid Cloud**    | Mix of above                 | Enterprises transitioning to cloud          |



> **Hybrid deployment** is often a stepping stone toward cloud-native architectures.

------

## **5. Core Enablers: Virtualisation and Automation**

### 5.1 Virtualisation

- **Hypervisor** (虚拟机监控器): Allows multiple OS instances to share the same hardware.
- Enables **multi-tenancy** and resource consolidation.

### 5.2 Containers

- OS-level virtualisation with lightweight isolation.
- Faster boot time and higher density than VMs.
- Common tools: Docker, containerd, Kubernetes.

### 5.3 Software-Defined Infrastructure

- **SDN (Software-Defined Networking)**: Abstracts control plane from hardware.
- **SDS (Software-Defined Storage)**: Virtualises storage hardware.

------

## **6. Layered Cloud Architecture**

A standard model includes:

1. **Physical Layer** – Servers, disks, switches.
2. **Virtualisation Layer** – VMs, containers, networks.
3. **Service Layer** – IaaS/PaaS/SaaS resources.
4. **Service Management Layer** – Monitoring, SLA enforcement, billing.
5. **Application/User Layer** – End-user interfaces and workloads.

> **Broker-Provider-Consumer** roles are logically separated in the NIST reference architecture.

------

## **7. Hyperscale Data Center Architecture**

### Key Features:

- 10,000–100,000 servers per site
- **PUE (Power Usage Effectiveness)** often < 1.2
- Advanced cooling: **liquid cooling**, hot/cold aisle separation
- Multi-region replication for compliance and latency

### Compliance considerations:

- **Data sovereignty** (数据主权): Enforcing legal control over data location and access
- **Geofencing** (地理围栏): Limiting access based on region

------

## **8. Benefits of Cloud Adoption**

### Technical Benefits

- Elastic scalability
- Global access
- Faster provisioning
- Automated monitoring
- Centralised security enforcement

### Economic Benefits

- **OpEx model** replaces CapEx (运营开支 vs 资本开支)
- Reduced time to market
- Pay-per-use reduces idle cost
- No upfront investment for global rollout

------

## **9. Real-World Example: Netflix on AWS**

| Aspect           | Implementation                                 |
| ---------------- | ---------------------------------------------- |
| Scalability      | Thousands of EC2 Auto Scaling Groups           |
| Storage          | Over 330 PB on Amazon S3                       |
| Machine Learning | Personalised recommendations via SageMaker     |
| Resilience       | Chaos Monkey injects faults to test robustness |
| Serverless       | Lambda for video processing pipelines          |



> Netflix is a textbook example of **cloud-native architecture at hyperscale**.

------

## **10. Sample Exam/Interview Questions with Model Answers**

### Q1: How does Serverless differ from IaaS in terms of NIST traits?

**A:** Serverless pushes elasticity and metering further—developers don’t manage infrastructure; execution scales per request and billing is per invocation. IaaS requires explicit provisioning and management of VMs.

### Q2: Is a PUE lower than 1.0 possible?

**A:** Not under standard definitions. PUE = IT power / total power; 1.0 means 100% efficiency. Though heat reuse improves sustainability, it doesn’t lower PUE below 1.0.

### Q3: Why use Service Mesh in microservice-based cloud systems?

**A:** It provides traffic control, observability, and security at the network layer, decoupled from application code, enabling fine-grained control and resilience.

### Q4: What are the trade-offs between multi-cloud and single-cloud strategies?

**A:** Multi-cloud avoids vendor lock-in and supports jurisdictional compliance, but increases operational complexity. Single-cloud enables deeper integration and easier optimisation.

### Q5: What are methods for tenant isolation in SaaS platforms?

**A:**

- **Logical isolation**: tenant ID per row or schema
- **Physical isolation**: separate DB instances
- **Encryption isolation**: tenant-specific encryption keys
- **Access control isolation**: role-based, fine-grained access policies (e.g., via OAuth/OIDC)

------

## 1 Foundational Recap

Cloud computing turns IT into an **on-demand utility (按需公用事业)**; this chapter refines the classification of *what* is offered—**service taxonomy (服务分类法)**—and *where* it runs—**deployment topology (部署拓扑)**.

------

## 2 Cloud Service Taxonomy (Three-Layer Stack 堆叠)

| Layer                                                    | Consumer Focus                      | Provider Scope                         | Example Services                         |
| -------------------------------------------------------- | ----------------------------------- | -------------------------------------- | ---------------------------------------- |
| **SaaS** *(Software-as-a-Service, 软件即服务)*           | Use complete applications           | Manage everything below GUI            | Google Workspace, Salesforce             |
| **PaaS** *(Platform-as-a-Service, 平台即服务)*           | Develop / deploy code               | Manage OS, middleware, scaling         | AWS Elastic Beanstalk, Azure App Service |
| **IaaS** *(Infrastructure-as-a-Service, 基础设施即服务)* | Provision virtual servers & storage | Provide virtualised hardware & network | Amazon EC2, Google Compute Engine        |

------

## 3 Deployment Typologies (部署类型)

### 3.1 Classical Trio (经典三分法)

- **Public Cloud (公有云)** – owned by a provider, shared by many tenants.
- **Private Cloud (私有云)** – hosted in a company data-centre, single tenant.
- **Hybrid Cloud (混合云)** – workload placement split across private & public zones.

### 3.2 Extended Forms (扩展形态)

| Term                | 中文注释    | Essence                                                      |
| ------------------- | ----------- | ------------------------------------------------------------ |
| **Multi-cloud**     | 多云环境    | One organisation consumes several unrelated clouds.          |
| **Federated cloud** | 联合云      | Multiple providers inter-operate, exposing a unified resource pool. |
| **Cloud bursting**  | 云突发/溢出 | Private cloud off-loads peak traffic to public cloud.        |
| **Broker**          | 云经纪人    | Third-party that aggregates & arbitrates multi-provider resources. |

------

## 4 Shared Responsibility Model (共享责任模型)

Responsibilities shift with the service layer: the provider secures and maintains what it controls, while the customer secures what it deploys. **Responsibility boundary (责任边界)** graphics show the split among **data, application, runtime, middleware, OS, virtualisation, network, storage and physical** layers.

------

## 5 Use-Case Patterns (使用场景模式)

| Pattern                                                      | Flow                                     | Typical Example                      |
| ------------------------------------------------------------ | ---------------------------------------- | ------------------------------------ |
| **End-user → Cloud** *(终端用户直达云)*                      | Direct SaaS consumption                  | Streaming video, webmail             |
| **Enterprise → Cloud** *(企业上云)*                          | Internal IT processes                    | HR, finance on cloud ERP             |
| **Enterprise → Cloud → End-user** *(企业通过云服务终端)*     | Org uses cloud to deliver public service | Food-delivery app backend            |
| **Enterprise A → Cloud → Enterprise B** *(企业间云对接)*     | Two firms share cloud resources          | Supplier–retailer inventory API      |
| **Hybrid** via **Federated Provider (联合提供方)** or **Broker (经纪人)** | Mix internal + external clouds           | Retailer handling Black Friday surge |

------

## 6 Resource Virtualisation & Virtual Infrastructure Management (资源虚拟化与虚拟基础设施管理)

- **Resource virtualisation** 抽象 compute, storage, and network into logical units.
- **VIM (Virtual Infrastructure Manager, 虚拟基础设施管理器)** orchestrates these units, exposing APIs upward.
- Typical stack: **hypervisor (虚拟机监控器)** → VIM (OpenStack Nova, VMware vCenter) → **cloud controller (云控制器)**.

------

## 7 Classical vs Cloud Computing (传统 IT vs 云计算)

| Criterion                       | Classical (Buy & Own 自购自建) | Cloud (Subscribe & Use 订阅即用) |
| ------------------------------- | ------------------------------ | -------------------------------- |
| Up-front **CapEx** *(资本开支)* | High                           | None                             |
| Provisioning time               | Weeks–Months                   | Minutes                          |
| Peak sizing                     | Over-provisioned               | Elastic pay-per-use              |
| Management overhead             | In-house                       | Shared / automated               |

------

## 8 Sample Exam / Interview Q & A (典型问答)

1. **How does cloud bursting support cost optimisation?** *(云突发如何优化成本)*
    |Answer|By moving peak workloads to public cloud, a firm avoids over-provisioning private capacity and pays only for transient spikes.|
2. **Three security challenges in multi-cloud?** *(多云安全挑战)*
    |Answer|Unified IAM (身份与访问管理), consistent logging/monitoring (日志与监控一致性), configuration drift (配置漂移).|
3. **Role of a cloud broker?** *(云经纪人职责)*
    |Answer|Aggregates SLAs, offers single catalogue, places workloads, mediates billing.|
4. **Contrast SaaS and PaaS in the responsibility model.** *(SaaS 与 PaaS 责任差异)*
    |Answer|SaaS provider handles app patching; PaaS customer manages code and runtime configs.|
5. **Example of enterprise-to-cloud-to-enterprise.** *(E2C2E 案例)*
    |Answer|Manufacturer and retailer exchanging inventory via cloud-hosted EDI.|

------

## 9 Key Takeaways (关键要点)

- **Service ≠ Deployment** – *What* versus *Where*.
- **Virtualisation (虚拟化) + Orchestration (编排)** enables elasticity.
- **Responsibility shifts upward** along IaaS → PaaS → SaaS.





# Chapter 3 – Cloud Platform Architecture over Virtualised Datacenters (Part I: Compute & Storage)

*Structured, layered notes with **中文注释** for all uncommon terms*

------

## 0 Snapshot Overview

> This lecture explains **warehouse‑scale datacenters (仓库级数据中心)**, their compute & storage subsystems, and why **hardware heterogeneity (硬件异构性)** plus accelerators matter.

------

## 1 Hardware Benefits of Cloud (硬件优势)

1. **Illusion of infinite resources** – users no longer pre‑plan capacity.
2. **No up‑front commitment (零前期投入)** – start small, scale later.
3. **Short‑term, usage‑based billing (按用量计费)** – encourages conservation.

------

## 2 From Clusters to Warehouse‑Scale Computing (WSC)

| Stage                           | Scale             | Characteristics                               |
| ------------------------------- | ----------------- | --------------------------------------------- |
| **Server Cluster (服务器集群)** | 10² – 10³ nodes   | Single organisation, specialised interconnect |
| **Data Center (数据中心)**      | 10³ – 10⁴ servers | High power & cooling demand                   |
| **Warehouse‑Scale DC**          | 10⁵ – 10⁶ servers | Request‑level parallelism, massive redundancy |

### Design Considerations (设计要点)

- **Cost‑performance (性价比)** – small savings × large scale.
- **Energy efficiency (能效)** – power & cooling dominate Opex.
- **Redundancy for dependability (冗余可靠性)**.
- **Mixed workloads** – interactive vs batch.

------

## 3 Resource Hierarchy: Servers, Racks, Containers

1. **Server (服务器)** – consumes up to 500 W, produces heat.
2. **Rack (机架)** – ~42 servers, ~21 kW.
3. **Cluster Container (集装箱式机房)** – ~1,000 servers, drop‑in power/network/cooling, accelerates build‑out.

Growth strategy: **system→rack→container** enables incremental scaling.

------

## 4 Heterogeneity Levels (异构性层级)

| Level                                  | Example               | Impact                        |
| -------------------------------------- | --------------------- | ----------------------------- |
| **Macro** – diverse nodes / networks   | x86 + ARM + GPU nodes | Optimise per workload         |
| **Micro** – deep memory + accelerators | CPU cache, SSD, FPGA  | Tailor latency / throughput   |
| **Software** – OS, middleware          | Linux, Windows, K8s   | Runtime portability vs tuning |

------

## 5 Processor Landscape & Accelerators (处理器与加速器)

### 5.1 General‑Purpose Processors (GPP 通用处理器)

- **Microprocessor** for PCs/servers: flexible, multi‑task.
- **Microcontroller (MCU 微控制器)** for embedded tasks: low cost/power.

### 5.2 Application‑Specific Processors (专用处理器)

| Type     | 中文注释         | Typical Role                    |
| -------- | ---------------- | ------------------------------- |
| **DSP**  | 数字信号处理器   | Real‑time math (audio, video)   |
| **ASIP** | 专用指令集处理器 | Custom instruction mix          |
| **ASIC** | 专用集成电路     | Fixed logic, highest efficiency |

### 5.3 Accelerators / Coprocessors (加速器/协处理器)

| Accelerator | 中文       | Strength                                |
| ----------- | -------- | --------------------------------------- |
| **GPU**     | 图形处理器    | SIMT, massive parallelism for ML/DL     |
| **FPGA**    | 现场可编程门阵列 | Reconfigurable hardware ● lower latency |
| **TPU**     | 张量处理单元   | Google ML ASIC, systolic arrays         |

**CPU vs GPU**: CPU optimises latency; GPU optimises throughput with many cores and latency‑tolerant memory design.

------

## 6 Public‑Cloud Compute Instance Spectrum (云实例类型)

- **EC2 General** (汎用型): T‑, M‑families → balanced workload.
- **EC2 GPU** (图形/计算加速): G4, P5 → AI, rendering.
- **EC2 FPGA F1** (可编程逻辑): custom hardware pipelines.
- Trend: providers expose ASIC‑based instances (Google Cloud TPU‑v5e).

------

## 7 Storage Architectures in Cloud (云存储架构)

| Option                        | 中文注释     | Usage                                |
| ----------------------------- | ------------ | ------------------------------------ |
| **SAN**                       | 存储区域网络 | Fibre Channel blocks for databases   |
| **NAS**                       | 网络附加存储 | File shares over Ethernet            |
| **Local disks + replication** | 本地盘+副本  | GFS/HDFS, 3× replicas for resilience |

**Design note:** WSCs favour cheap local disks + software redundancy to minimise SAN costs.

------

## 8 Workload & Utilisation Dilemma (工作负载与利用率难题)

- Data centers are **always on but not always busy**.
- Peak load can be ×10 average; provisioning for peaks wastes resources otherwise.
- **Server utilisation (利用率)** often 5‑20 %; elastic cloud sharing mitigates waste.

------

## 9 Sample Exam / Interview Q & A (典型问答)

1. **Why do warehouse‑scale datacenters pursue energy efficiency so aggressively?**
    *Answer:* Power and cooling dominate operational cost; small percentage savings scale to millions of dollars and enable higher compute density within the same power envelope.
2. **Compare FPGA and ASIC as accelerators in terms of flexibility and performance.**
    *Answer:* FPGA is reconfigurable post‑manufacture, offers medium efficiency; ASIC is fixed‑function, delivers maximum performance & power efficiency but zero flexibility once fabricated.
3. **Explain cloud bursting from a compute perspective.**
    *Answer:* Burst workload VM instances are spun up in public cloud during spikes, leveraging “illusion of infinite resources” while base load remains on‑prem/private.
4. **What workload characteristics favour GPU over CPU?**
    *Answer:* Highly parallel, data‑parallel operations such as matrix multiplication in deep learning; SIMT excels at throughput rather than low‑latency serial tasks.
5. **Why is workload‑driven heterogeneity important in modern clouds?**
    *Answer:* Matching specialised hardware (GPU, TPU, FPGA) to specific workloads maximises performance‑per‑watt and reduces total cost.

------

## 10 Key Takeaways (关键要点)

- Cloud hardware delivers **scale, elasticity, and pay‑as‑you‑go economics**.
- **Warehouse‑scale design** focuses on cost, energy, and redundancy at massive scale.
- **Heterogeneity & accelerators** optimise diverse workloads (AI, signal processing).
- Storage favours **local disks + replication** over expensive SAN in hyperscale.
- Proper utilisation management balances **service quality vs resource waste**.

# Chapter 4 – Cloud Platform Architecture over Virtualised Datacenters (Part II: Networking)

*Structured, layered notes with **中文注释** for all uncommon terms*

------

## 0 Snapshot Overview

This lecture covers the **network layer (网络层)** of hyperscale clouds: datacentre topologies, high‑speed interconnects, and the rise of **software‑defined networking (SDN, 软件定义网络)** plus **network‑function virtualisation (NFV, 网络功能虚拟化)**.

------

## 1 System‑Level Architecture Perspective (系统层视角)

A cloud platform is a **service‑oriented stack** sitting atop a virtualised datacentre. At the network level it must provide:

- **East–West traffic (东西向流量)** for intra‑service RPCs and storage replication.
- **North–South traffic (南北向流量)** for client ingress/egress via edge and load balancers.
- **Rich interconnect (丰富互联)** among racks to sustain multi‑Tb s⁻¹ aggregate throughput.

------

## 2 Datacentre Network (DCN) Topology

| Layer                    | 中文注释   | Role                                        |
| ------------------------ | ---------- | ------------------------------------------- |
| **Server Rack**          | 机架       | Houses 40–50 servers.                       |
| **ToR Switch**           | 架顶交换机 | First‑hop 10/25/40 GbE fan‑in.              |
| **Tier‑1 / Aggregation** | 汇聚层     | Cross‑rack traffic, 100/400 GbE.            |
| **Tier‑2 / Core**        | 核心层     | Connects pods, interfaces to border router. |
| **Load Balancer**        | 负载均衡器 | Layer‑7 routing, hides DC internals.        |

> **Rich interconnection (冗余互联)** increases throughput and resilience by providing multiple paths between any two racks.

------

## 3 Ethernet Evolution (以太网演进)

- **400 GbE** optics are mainstream; **800 GbE** for datacentre interconnect (DCI) is emerging.
- Next target is **1.6 Tb s⁻¹** per link. Challenges: PAM‑4 signalling, forward‑error correction (FEC), and silicon photonics.
- Trend: **energy per bit** must decrease despite higher rates—drives innovation in DSP and laser tech.

------

## 4 InfiniBand & DPUs (高速互连与数据处理单元)

### 4.1 InfiniBand Overview

High‑performance fabric originally for HPC, now adopted in cloud GPU clusters.

| Generation | Speed      | 中文注释           |
| ---------- | ---------- | ------------------ |
| **FDR**    | 56 Gb s⁻¹  | Fourteen Data Rate |
| **HDR**    | 200 Gb s⁻¹ | High Dynamic Range |
| **NDR**    | 400 Gb s⁻¹ | Next Data Rate     |
| **XDR**    | 800 Gb s⁻¹ | eXtreme Data Rate  |

### 4.2 DPUs (数据处理单元)

- **BlueField DPU** offloads networking, security, storage virtualisation from CPU.
- Supports **in‑network acceleration (网络内加速)** such as SHARP reductions for MPI.

------

## 5 Software‑Defined Networking (SDN)

### 5.1 Plane Separation (平面分离)

| Plane                | 中文     | Function                                           |
| -------------------- | -------- | -------------------------------------------------- |
| **Data Plane**       | 数据平面 | High‑speed packet forwarding in ASIC.              |
| **Control Plane**    | 控制平面 | Routing/flow decisions, centralised in controller. |
| **Management Plane** | 管理平面 | Configuration & monitoring APIs.                   |

### 5.2 SDN Architecture Components

- **SDN Switch** – Open vSwitch, maintains **flow tables (流表)**.
- **Controller** – ONOS, OpenDaylight; provides **global network view (全局视图)**.
- **Northbound API** – REST/JSON to apps.
- **Southbound API** – OpenFlow to switches.

### 5.3 SDN Benefits in Cloud

- Centralised policy & security.
- **Network virtualisation (网络虚拟化)** → tenant VPCs.
- Traffic engineering, dynamic load balancing.
- Automation & rapid provisioning.

------

## 6 Network Function Virtualisation (NFV) & VNFs

- **NFV** shifts functions (firewalls, LB, IDS) from dedicated boxes to VMs/containers.
- **VNF (虚拟网络功能)** chains are orchestrated for **service function chaining (服务链)**.
- Containers cut start‑up latency, vital for **event‑driven (事件驱动)** network services and 5G edge workloads.

------

## 7 Emerging Use Cases (新兴用例)

| Scenario                    | 中文注释     | Value                                                      |
| --------------------------- | ------------ | ---------------------------------------------------------- |
| **Cloud bursting**          | 云突发       | Spin‑up extra capacity in public cloud.                    |

## 0 Snapshot Overview

Virtualisation (虚拟化) provides the **abstraction layer (抽象层)** between physical hardware and cloud services, enabling multi-tenant isolation, flexible provisioning and workload mobility.

## 1 Virtual Machine vs Traditional Computer

| Aspect               | Traditional Bare-Metal   | Virtual Machine (VM 虚拟机) |
| -------------------- | ------------------------ | --------------------------- |
| OS-to-HW binding     | 1 : 1 rigid              | N : 1 via hypervisor        |
| Workload isolation   | Process-level            | VM-level (stronger)         |
| Resource utilisation | Low (< 20 % typical)     | Higher via consolidation    |
| Fault containment    | Kernel crash → host halt | Crash isolated to VM        |

**Key enabler:** **Hypervisor / Virtual Machine Monitor (VMM 虚拟机监控器)** partitions hardware transparently.

## 2 Implementation Levels of Virtualisation (五层实现)

| Level                                  | 中文注释     | Example Tech              |
| -------------------------------------- | ------------ | ------------------------- |

| **Library / API**                      | 用户态库级   | vCUDA, Wine               |
| **Operating-System**                   | 操作系统级   | LXC, OpenVZ (containers)  |

| **Instruction-Set Architecture (ISA)** | 指令集架构级 | QEMU (binary translation) |

------

## 3 Machine-Level Virtualisation Styles

### 3.1 Type 2 (Hosted) Hypervisor – 托管式

Runs atop a host OS (e.g., VirtualBox). **Pros:** ease of use. **Cons:** extra overhead.

### 3.2 Type 1 (Bare-Metal) Hypervisor – 裸金属

Runs directly on hardware (e.g., VMware ESXi, Xen, KVM). Lower latency, higher performance.

------

## 4 Hypervisor-Based Virtualisation Techniques

| Technique                            | 中文注释       | Mechanism                                                    | Trade-off                                           |
| ------------------------------------ | -------------- | ------------------------------------------------------------ | --------------------------------------------------- |
| **Full Virtualisation**              | 全虚拟化       | Binary translation traps sensitive x86 instructions; unmodified guest OS | Maximum compatibility, performance hit              |
| **Para-Virtualisation**              | 半虚拟化       | Guest OS patched to replace non-virtualisable instructions with **hypercalls (超级调用)** | Higher perf, but OS modification & maintenance cost |
| **Hardware-Assisted Virtualisation** | 硬件辅助虚拟化 | CPU extensions (Intel VT-x, AMD-V) add new root mode Ring 0P | Near-native speed, requires modern CPUs             |

> **Ring Architecture (特权级环):** x86 provides Rings 0–3; hypervisors leverage extra privilege levels or trapping to manage guests.

------

## 5 Operating-System Level Virtualisation (Containers)

- Shared kernel; instances called **containers (容器)**.
- Examples: Linux namespaces + cgroups, LXC, Docker, Podman.
- **Advantages:** lightweight, second-scale start, density.
- **Limitations:** same kernel across tenants → weaker isolation than VMs; no heterogeneous OS per host.

------

## 6 Virtual Machine Managers & Infrastructure Orchestration

### 6.1 Hypervisor Products (主流超管)

- **VMware ESXi** – commercial, feature-rich.

| Resource monitoring                    | 资源监控       |
| Access control                         | 访问控制       |
| Federation across sites                | 跨站点联邦     |

**Examples:** OpenStack Nova, OpenNebula, oVirt.

### 6.3 Resource Provisioning Flow (资源供应流程)

1. VIM selects host → 2. Hypervisor allocates vCPU/Memory → 3. VM cluster serves distributed application.

- **libvirt** – unified API supporting KVM, Xen, ESXi, LXC.

## 8 Sample Exam / Interview Q & A (典型问答)

1. **Why does binary translation incur overhead in full virtualisation?**
    *Answer:* Sensitive instructions are trapped and emulated in software, adding extra execution cycles versus direct CPU execution.
2. **Give two reasons containers start faster than VMs.**
    *Answer:* They share the host kernel (no OS boot) and require only namespace/cgroup setup instead of BIOS + hypervisor init.

5. **Explain how hardware-assisted virtualisation improves I/O virtualisation.**
    *Answer:* Features like Intel VT-d enable direct device assignment, bypassing software emulation and reducing latency.

------

## 9 Key Takeaways (关键要点)

- Virtualisation underpins cloud elasticity and multi-tenancy.
- Five implementation levels span from ISA emulation to application sandboxes.
- Hypervisor techniques evolve: full → para → hardware-assisted for efficiency.
- Containers complement, not replace, VMs—each suits different isolation/perf needs.
- VIMs orchestrate VM clusters, enabling **Infrastructure-as-a-Service (IaaS)**.

# Chapter 8 – VM Scheduling & Cloud Resource Management

*Structured, layered notes (English-major) with **中文注释** for all uncommon terms*

------

## 0 Snapshot Overview

Virtual‑machine (VM) scheduling determines **where and when** each VM runs in a cloud, directly impacting **performance (性能)**, **cost (成本)**, and **energy efficiency (能效)**. This chapter explores resource‑management policies, classic scheduling algorithms, live migration challenges, and scale‑up/out strategies.

------

## 1 Architectural Context (体系结构层次)

```
Physical Hosts → Hypervisors → VIM Scheduler → VM Instances → Applications
```

- Scheduler resides in **core middleware (核心中间件)**, below user‑level PaaS/SaaS and above hypervisors.
- Must balance **multi‑objective optimisation (多目标优化)** across throughput, SLA, and power.

------

## 2 Resource‑Management Functions (管理职能)

| Function                | 中文注释     | Purpose                                           |
| ----------------------- | ------------ | ------------------------------------------------- |
| **Admission Control**   | 接纳控制     | Reject or queue requests violating global policy. |
| **Capacity Allocation** | 容量分配     | Grant vCPU/Memory for each VM.                    |
| **Load Balancing**      | 负载均衡     | Spread workload evenly to avoid hotspots.         |
| **Energy Optimisation** | 能耗优化     | Consolidate VMs; power‑down idle hosts.           |
| **QoS Enforcement**     | 服务质量保证 | Meet SLA latency/throughput targets.              |

------

## 3 Scheduling Granularity (调度粒度)

1. **Host Level** – place VMs on physical nodes.
2. **VM Level** – allocate vCPUs/NUMA, set shares.
3. **Application Level** – threads/processes within VM.

> Cloud schedulers often operate hierarchically: host choice → vCPU pinning → guest OS thread pool.

------

## 4 Classic Scheduling Algorithms (经典算法)

| Algorithm                         | 中文注释      | Behaviour                            | Suitable Workloads            |
| --------------------------------- | ------------- | ------------------------------------ | ----------------------------- |
| **Round‑Robin (RR)**              | 轮询          | Cycles through hosts sequentially.   | Homogeneous, stateless.       |
| **First‑Come‑First‑Serve (FCFS)** | 先来先服务    | Allocate to first capable host.      | Simple batch, low variance.   |
| **Shortest‑Job‑First (SJF)**      | 最短作业优先  | Prefers small VMs to cut turnaround. | Predictable job sizes.        |
| **Priority‑Based**                | 优先级调度    | Rank by SLA criticality.             | Mixed SLAs.                   |
| **Best‑Fit / First‑Fit**          | 最佳/首次适配 | Bin‑packing to minimise waste.       | Consolidation, energy saving. |

### Heuristic Families (启发式家族)

- **Power‑aware** – minimise Watts per VM.
- **Performance‑aware** – minimise SLA violations.
- **Network‑aware** – reduce data‑transfer latency.
- **Hybrid meta‑heuristics (元启发式)** – GA, PSO, ACO for NP‑hard placement.

------

## 5 Scheduler in OpenNebula (案例)

- Implements **match‑making (匹配)** + **rank policy (评分策略)**.
- RANK formula may include free CPU, host load, energy cost.
- Pluggable policies allow custom weighting via **requirement/rank expressions (需求/评分表达式)**.

------

## 6 Live Migration (实时迁移)

| Aspect       | 中文注释     | Key Points                                         |
| ------------ | ------------ | -------------------------------------------------- |
| **When**     | 触发时机     | Overload/underload detection, maintenance windows. |
| **Which VM** | 迁移对象选择 | Heaviest, newest, or minimum data size.            |
| **Where**    | 迁移目标选择 | Host with spare capacity & network proximity.      |

### Migration Techniques (迁移技术)

- **Pre‑copy (预拷贝)** – iterative memory copy, small downtime.
- **Post‑copy (后拷贝)** – resume at target, fetch pages on demand.

### Data‑Plane Issues (数据面问题)

- **Memory state** size GB–TB.
- **Network sessions** require ARP spoof or VPN tunnelling.
- **Disk state** large; rely on shared SAN/NAS or block replication.

------

## 7 Energy‑Efficient Consolidation (能耗整合)

- Cluster monitors utilisation → if low, **live migrate** VMs to fewer hosts.
- Idle hosts enter **sleep or power‑off (休眠/关机)** modes.
- Case study: 485 W vs 552 W scenario saves ≈12 %.

------

## 8 Scaling Strategies (扩展策略)

| Type          | 中文注释 | Mechanism                    | Trade‑off                                     |
| ------------- | -------- | ---------------------------- | --------------------------------------------- |
| **Scale‑Out** | 横向扩展 | Add more VM replicas.        | Needs load balancer; eventual consistency.    |
| **Scale‑Up**  | 纵向扩展 | Resize VM to bigger flavour. | Limited by host capacity; potential downtime. |

Auto‑scalers monitor CPU, queue length, or custom metrics to trigger actions.

------

## 9 Sample Exam / Interview Q & A (典型问答)

1. **Explain policy vs mechanism in cloud schedulers.**
    *Answer:* Policy defines *what* goals (e.g., minimise energy); mechanism defines *how* (e.g., best‑fit algorithm) to achieve them.
2. **Why might pre‑copy migration fail under high memory churn?**
    *Answer:* Dirty pages change faster than copy loop, causing endless iterations; switchover downtime grows or migration aborts.
3. **List two metrics a power‑aware scheduler uses.**
    *Answer:* Host wattage per utilisation, thermal headroom, or PUE factor.
4. **Describe FCFS with First‑Fit heuristic in one sentence.**
    *Answer:* Queue requests in arrival order and place each on the first host whose free resources satisfy the VM’s requirements.
5. **Give one advantage of scaling out over scaling up.**
    *Answer:* Horizontal replicas avoid single‑VM bottlenecks and support rolling updates without downtime.

------

## 10 Key Takeaways (关键要点)

# Chapter 9 – Kubernetes: Container Orchestration at Cloud Scale

*Structured, layered notes (English‑major) with **中文注释** for uncommon terms*

------

## 0 Snapshot Overview

Kubernetes (a.k.a. **k8s**, “k + 8 letters + s”) is an **open‑source container‑orchestration platform (容器编排平台)** originating from Google, now governed by CNCF. It provides a **declarative API (声明式 API)** to deploy, scale and manage containerised workloads on clusters of virtual or physical machines.

------

## 1 Conceptual Position in the Cloud Stack

```
Physical / VM → Container Runtime → **Kubernetes** → Micro‑services / CI‑CD
```

- Fills the **core‑middleware layer (核心中间件层)**, just above the container engine (Docker/CRI‑O/containerd) and below user‑level PaaS/SaaS.

------

## 2 High‑Level Architecture

| Tier                   | 中文注释        | Key Elements                                                 |
| ---------------------- | --------------- | ------------------------------------------------------------ |
| **Control Plane**      | 控制平面        | `kube‑apiserver`, `etcd`, `kube‑scheduler`, `kube‑controller‑manager` |
| **Data Plane / Nodes** | 数据平面 / 节点 | `kubelet`, `kube‑proxy`, Container Runtime                   |

### 2.1 Cluster (集群)

A set of nodes working as one unit, coordinated by the **Master (主节点)**, persisting state in **etcd** (分布式键值存储, SSOT).

### 2.2 Pod (最小调度单元)

- One or more co‑located containers sharing **network namespace (网络命名空间)** and **volumes (共享卷)**.

### 2.3 Key Interfaces

| Acronym | 中文注释       | Function                                            |
| ------- | -------------- | --------------------------------------------------- |
| **CRI** | 容器运行时接口 | Pluggable runtimes (Docker Shim, containerd, CRI‑O) |
| **CNI** | 容器网络接口   | Config network plugins (Calico, Flannel)            |
| **OCI** | 开放容器规范   | Image & runtime standard guaranteeing portability   |

## 3 Control‑Plane Components

| Component                   | 中文注释   | Role                                                   |
| --------------------------- | ---------- | ------------------------------------------------------ |
| **kube‑apiserver**          | API 服务器 | REST front end, authN/Z, admission control             |
| **etcd**                    | 分布式存储 | Strongly‑consistent KV store for cluster state         |
| **kube‑scheduler**          | 调度器     | Assigns Pods to Nodes (default = best‑fit bin‑packing) |
| **kube‑controller‑manager** | 控制器管理 | Reconciles actual vs desired state via control loops   |

------

## 4 Node Components

| Component             | 中文注释   | Purpose                                                |
| --------------------- | ---------- | ------------------------------------------------------ |
| **kubelet**           | 节点代理   | Manages Pod lifecycle on host                          |

| **Container Runtime** | 容器运行时 | Pulls, unpacks & executes images                       |

------

## 5 Workload Primitives & Objects

| Object                 | 中文注释    | Highlights                                       |
| ---------------------- | ------- | ------------------------------------------------ |
| **Deployment**         | 部署      | Manages replica sets, performs rolling updates   |
| **ReplicaSet**         | 副本集     | Ensures N identical Pods running                 |
| **Service**            | 服务      | Stable virtual IP/DNS, label‑selects target Pods |
| **Namespace**          | 命名空间    | Logical tenancy & policy boundary                |
| **ConfigMap / Secret** | 配置 / 密钥 | Decouple env configs & credentials               |

------

## 6 Scaling Mechanisms

### 6.1 Horizontal Pod Autoscaler (HPA 水平自动伸缩)

- Control loop monitors CPU/custom metrics; adjusts **replica count (副本数)**.

### 6.2 Vertical Pod Autoscaler (VPA 垂直自动伸缩)

- Recommender analyses live usage, Updater evicts Pods, Admission Controller rewrites **resource requests (资源请求)**.

### 6.3 Cluster Autoscaler (CA 集群自动伸缩)

- Adds/removes nodes when Pods are **unschedulable (无法调度)** or cluster is under‑utilised. Works alongside HPA/VPA.

### 6.4 ML‑Based Predictive Scaling (机器学习预测伸缩)

- Uses AR, LSTM, HTM or RL models to predict **QPS (每秒查询)**; proactively scales Pods via custom controller.

------

## 7 Networking Model

- Every Pod gets its own IP; all Pods can talk flat L3, no NAT.
- Services implement **cluster‑IP (集群内 IP)**, **NodePort**, **LoadBalancer**, or **Ingress (入口控制)**.
- Overlay or routed plugin provides cross‑node connectivity.

------

## 8 Observability & Reliability

| Probe              | 中文注释 | Use                                    |
| ------------------ | -------- | -------------------------------------- |
| **livenessProbe**  | 存活探针 | Restart deadlocked container           |
| **readinessProbe** | 就绪探针 | Remove Pod from Service LB until ready |
| **startupProbe**   | 启动探针 | Give slow‑start apps extra time        |

Controllers restart failed Pods, ensuring self‑healing and high availability.

1. **Why is Kubernetes considered declarative?**
    *Answer:* Users submit desired‑state objects (YAML manifests); controllers continuously reconcile actual state to match the declaration, rather than executing imperative steps.
2. **Contrast HPA and VPA in resource management.**
    *Answer:* HPA scales replica count based on external metrics; VPA resizes CPU/memory of existing Pods, evicting & recreating them.

    *Answer:* Services expose stable endpoints decoupled from Pod IPs, allowing Pods to be replaced or rescheduled without clients knowing.

    *Answer:* `kubelet` heartbeats stop; after grace period controllers mark its Pods for eviction, rescheduling them to healthy nodes.

    *Answer:* Benefit: lowers cost by removing idle nodes; Drawback: node provisioning time (minutes) may lag behind sudden spikes.

## 10 Key Takeaways (关键要点)

- Kubernetes abstracts clusters into **Pods, Services, Deployments**, offering declarative control & self‑healing.
- Control plane stores state in **etcd**; scheduler plus controllers reconcile towards desired state.
- Autoscaling spans **Pod (HPA/VPA)** and **Node (CA)** layers, with emerging ML predictors for proactive scaling.
- CNI, CRI and OCI standards ensure plug‑and‑play networking, runtimes and images.
- Mastering k8s primitives is foundational for modern **cloud‑native (云原生)** architectures and DevOps workflows.

# Chapter 11 – Serverless Architectures & Function‑as‑a‑Service (FaaS)

------

## 0 Snapshot Overview

Serverless computing hides servers from developers and bills **per‑invocation (按调用计费)**. The core mechanism is **Function‑as‑a‑Service (FaaS)**. This chapter details execution models, benefits, cold‑start trade‑offs, vendor platforms, open‑source stacks, workflow orchestration, and research challenges.

------

## 1 Architectural Layer in the Cloud Stack (位置)

```
Physical / VM → Container Runtime → Core Middleware (Kubernetes, Service Mesh) → **Serverless Runtime (FaaS Gateway)** → User Functions → Application Workflows
```

- Sits above container orchestration, below user‑level middleware. Provides **event‑driven (事件驱动)** execution and automatic scaling.

------

## 2 Definition & Service Model

| Layer    | Unit of Deployment  | Billing Model               | Example                     |
| -------- | ------------------- | --------------------------- | --------------------------- |
| **IaaS** | VM image            | Per VM‑hour                 | EC2, Azure VM               |
| **PaaS** | Application package | Per app‑instance‑hour       | App Engine, Beanstalk       |
| **FaaS** | Function code       | Per invocation + ms runtime | AWS Lambda, Azure Functions |

**Serverless ≠ FaaS**—serverless is the architecture; FaaS is the mechanism by which compute is realised.

------

## 3 Execution Pipeline (执行管线)

1. **Event Trigger (事件触发器)**—HTTP, queue, blob, cron, IoT.
2. **API Gateway** authenticates request, applies throttling.
3. **FaaS Scheduler** checks **function cache (函数缓存)** for warm container.
4. If **warm**, route request → execute.
5. If **cold**, pull image / layer, start sandbox, initialise runtime → execute.
6. **Metering & Logs** shipped to monitoring backend; container kept warm until idle‑timeout.

------

## 4 Benefits & Constraints

### 4.1 Benefits

- **Zero server management**
- **Automatic horizontal scaling (连续弹性)**
- **Never pay for idle (无空闲付费)**
- Built‑in high availability across AZs

### 4.2 Constraints

| Constraint               | 中文注释       | Impact                                     |
| ------------------------ | -------------- | ------------------------------------------ |
| **Cold start latency**   | 冷启动延迟     | Adds 100 ms–>1 s; bad for low‑latency APIs |
| **Statelessness**        | 无状态限制     | State must persist in DB / cache           |
| **Execution time limit** | 运行时长上限   | 15 min (Lambda), 540 sec (GCF)             |
| **Size limit**           | 代码包大小限制 | 250 MB (unzipped) for Lambda               |

------

## 5 Application Patterns

| Pattern                   | Typical Use                  | Note                                   |
| ------------------------- | ---------------------------- | -------------------------------------- |
| **Single‑function**       | Alexa skill, image thumbnail | Simple trigger → action                |
| **Event Streaming**       | IoT sensor data, Kafka topic | High concurrency, short exec           |
| **Data Processing**       | Map‑style batch, ETL         | Fan‑out via SNS/SQS; consolidate in DB |
| **Serverless Backend**    | Mobile/Web API               | Combine Lambda + API Gateway           |
| **Parallel ML Inference** | Batch classify images        | GPU‑backed Lambda or EKS Fargate       |

------

## 6 Cold Start vs Warm Start

| Stage             | Delay Source             | Optimisation                           |
| ----------------- | ------------------------ | -------------------------------------- |
| **Platform cold** | Container image download | Provisioned Concurrency, image caching |
| **Runtime cold**  | Language runtime init    | Custom runtime stripping, SnapStart    |
| **App cold**      | Dependency load          | Layer‑based deps, lazy import          |

- **Provisioned Concurrency (预置并发)** keeps N instances warm; bill per‑hour.

------

## 7 Commercial & Open‑Source Platforms

| Platform                            | 中文注释 | Language Support        | Unique Feature                              |
| ----------------------------------- | -------- | ----------------------- | ------------------------------------------- |

------

## 8 Serverless Workflows & Orchestration

| Workflow Engine              | 中文注释 | Definition DSL                | Parallelism       | Error Handling    |
| ---------------------------- | -------- | ----------------------------- | ----------------- | ----------------- |
| **AWS Step Functions**       | 步骤函数 | Amazon States Language (JSON) | ✔ Map/Parallel    | Retry, catch      |
| **Azure Durable Functions**  | 持久函数 | C#/JS orchestrator            | ✔ Fan‑out/in      | Built‑in patterns |

## 9 Key Metrics for Evaluation

| Metric                     | 中文注释   | Why It Matters                                         |
| -------------------------- | ---------- | ------------------------------------------------------ |

## 11 Sample Exam / Interview Q & A (典型问答)

### 4.3 Bare Metal vs Cloud VM Execution (裸机 vs 云虚拟机)

## 4 Parallelisation Challenges (并行化挑战)

| Challenge           | 问题描述             | MapReduce Solution                                           |
| ------------------- | -------------------- | ------------------------------------------------------------ |
| **Scheduling**      | 把任务分配给工作节点 | Master assigns map/reduce tasks; re-queues on failure        |
| **Availability**    | 任务多于可用节点     | Task queue; speculative backup tasks                         |
| **Dependencies**    | 共享中间结果         | Shuffle & sort groups keys for reducers                      |
| **Fault-tolerance** | 节点故障             | Heartbeats + re-execution; DFS replication                   |
| **Synchronisation** | 并发一致性           | Implicit via shuffle barrier; combines output deterministically |

------

## 5 MapReduce Fundamentals

### 5.1 Programming Interface

```text
map (k, v)  →  list<k', v'>
reduce (k', list<v'>)  →  list<k', v''>
partition (k', p) → bucket  // optional
combine   (k', list<v'>) → list<k', v'> // mini-reduce
```

- **Map** transforms; **Reduce** aggregates; optional **Combine** reduces network traffic.

### 5.2 Runtime Responsibilities

1. **Input splitting** across M map tasks.
2. **Local mapping** generates (k', v') pairs.
3. **Shuffle & Sort** groups identical keys.
4. **Reduce tasks** process partitions; write HDFS output.
5. Master tracks progress, restarts stragglers.

------

## 6 Data-Flow Diagram (五步流程)

```
Input → Map → Shuffle/Sort → Reduce → Output
```

- **Partition** decides which reducer receives a key.
- **Combine** executes after map, before shuffle, as in-memory optimisation.

------

## 7 Worked Examples

### 7.1 Word Count (词频统计)

- **Map:** emit (word, 1) for each token.
- **Combine/Reduce:** sum counts per word.
- **Result:** (word, count) pairs enabling tops-N queries.

### 7.2 Facebook “Friends in Common”

1. Map emits key = (sorted pair of users), value = friend list.
2. Shuffle groups pair keys.
3. Reduce intersects lists → mutual friends.
4. Pre-compute daily; serve via low-latency KV store.

> Performance note: 1 B users; single machine ~10 h; 10 000-node cluster ≈ 4 s.

------

## 8 Hadoop Implementation Stack

| Layer                | 中文注释       | Component                                  |
| -------------------- | -------------- | ------------------------------------------ |
| **Distributed FS**   | 分布式文件系统 | HDFS (NameNode + DataNodes)                |
| **Resource Mgmt**    | 资源管理       | YARN (ResourceManager + NodeManager)       |
| **Execution Engine** | 执行引擎       | MapReduce v2 (MRv2) on YARN                |
| **High-level API**   | 高级接口       | Hive (SQL), Pig (dataflow), Spark (memory) |

**Data Locality Principle (数据本地性):** launch map tasks on nodes hosting HDFS blocks; “move compute to data”.

------

## 9 Distributed File System Essentials

| Feature                | HDFS 实现            | Why Important                           |
| ---------------------- | ------------------ | --------------------------------------- |
| **Block storage**      | 128 MB default     | Large sequential IO, fewer metadata ops |
| **Replication factor** | 默认 3 副本            | Fault tolerance, rack awareness         |

## 10 Sample Exam / Interview Q & A (典型问答)

1. **How does the shuffle phase guarantee that all identical keys reach the same reducer?**
    *Answer:* Hash-based partitioner maps each key to a reducer index; runtime sorts key streams before pushing to reducer.
2. **Why are combiners considered “optional hints”?**
    *Answer:* Framework may skip or run combiners multiple times; they must be associative & commutative to ensure correctness.
3. **List two drawbacks of vanilla MapReduce for iterative ML algorithms.**
    *Answer:* Repeated disk IO between iterations; lacks in-memory data sharing leading to high latency—Spark mitigates via RDD caching.
4. **Explain speculative execution and when it triggers.**
    *Answer:* Master launches duplicate task on a different node when progress of original lags beyond threshold, countering stragglers.
5. **Compare HDFS with a SAN in terms of cost and scalability.**
    *Answer:* HDFS leverages commodity disks and horizontal scaling; SAN centralises storage, incurs higher cost and single-point bandwidth bottlenecks.

- Layered big-data architecture separates ingestion, storage, analysis, and consumption; business layers handle governance & QoS.

# Chapter 15 – Evolution from Hadoop 1 to Hadoop 3 and Apache Spark

## 0 Snapshot Overview

This chapter traces Hadoop’s journey from its **MapReduce‑centric v1 architecture** through HDFS Federation and **YARN (Yet Another Resource Negotiator)** in v2 to today’s Hadoop 3.x, then pivots to Apache **Spark**, the in‑memory data‑processing engine that supersedes MapReduce for many workloads.

------

## 1 Learning Objectives (学习目标)

1. Identify limitations of **Hadoop v1 (原始 Hadoop)** and how federation, HA, and YARN address them.
2. Explain YARN’s three‑tier resource‑management model.

4. Understand Spark’s layered architecture, RDD concept, and runtime integration with YARN/stand‑alone/Mesos.

## 2 Limitations of Original Hadoop v1

| Limitation                  | 中文注释          | Impact                                                       |
| --------------------------- | ----------------- | ------------------------------------------------------------ |
| **Single JobTracker**       | 单点 JobTracker   | Scalability cap (~5 K nodes, 40 K tasks) & SPOF              |
| **Static map/reduce slots** | 固定槽位          | Resource under‑utilisation (map slots idle while reduce full) |
| **NameNode RAM bottleneck** | NameNode 内存瓶颈 | Metadata in RAM → 50–100 M files ceiling                     |
| **Single NameNode**         | 单 NameNode       | Cluster outage on failure                                    |

> Yahoo once estimated Hadoop 1 limit ≈ 5 000 nodes; modern clouds exceed 10×.

------

## 3 Key Innovations in Hadoop v2 (HDFS Federation + YARN)

### 3.1 HDFS Federation (联邦)

- **Multiple independent NameNodes (多 NameNode)** share common DataNodes.
- Horizontal namespace scaling; DataNode registers with all NameNodes.

### 3.2 NameNode High Availability (双活 NameNode)

- Active/Stand‑by pair with **JournalNode quorum (日志节点仲裁)**; automatic fail‑over.

### 3.3 YARN Architecture

| Component                  | 中文注释    | Responsibility                                            |
| -------------------------- | ----------- | --------------------------------------------------------- |
| **ResourceManager (RM)**   | 资源管理器  | Global arbitration of containers                          |
| **NodeManager (NM)**       | 节点管理器  | Per‑node container launch, resource & log collection      |
| **ApplicationMaster (AM)** | 应用管理器  | Per‑app scheduler → negotiates containers, monitors tasks |
| **Container**              | 容器 (YARN) | Bundles CPU + memory + disk + network share               |

**Flow:** Client → RM → allocates container for AM → AM negotiates more containers → tasks run under NMs.

------

## 4 Hadoop 3.x Enhancements

| Feature                      | 中文注释          | Benefit                                      |
| ---------------------------- | ----------------- | -------------------------------------------- |
| **Erasure Coding**           | 擦除编码          | 50 % storage saving vs 3× replication        |
| **GPU support**              | GPU 调度          | Deep‑learning & compute accelerators in YARN |
| **YARN‑Docker**              | Docker 容器运行时 | Consistent runtime & dependency isolation    |
| **HTTP 2 NameNode protocol** | 更快拍照          | Efficiency, security (TLS).                  |

Hadoop 3.4.1 released **18 Oct 2024** continuing incremental updates.

------

## 5 MapReduce vs Spark: Conceptual Shift

| Aspect          | MapReduce                        | Spark                                                     |
| --------------- | -------------------------------- | --------------------------------------------------------- |
| Data in memory? | Disk between stages              | **Resilient Distributed Dataset (RDD)** keeps data in RAM |
| Iterative alg.  | Re‑read from HDFS each iteration | Cache → orders of magnitude faster                        |
| API level       | map / reduce pair                | Rich DAG API (map, filter, join, window, ML)              |
| Runtime         | MapReduce v2 on YARN             | Stand‑alone, **YARN**, Mesos, Kubernetes                  |
| Speed           | Baseline                         | ~10× disk, ~100× in‑memory                                |

------

## 6 Spark Architecture Layers

```
Client API (Scala, Java, Python, R) → Core Engine (scheduler, DAG) → Libraries
        └─ SQL / DataFrame
        └─ Streaming (Structured)
        └─ MLlib
        └─ GraphX
Resource Mgmt: Stand‑alone │ YARN │ Mesos
Storage: HDFS │ S3 │ GCS │ JDBC │ Cassandra
```

- **Spark Core** – task scheduling, memory mgmt, failure recovery.
- **RDD** – immutable, partitioned collection; lineage enables **fault‑tolerant recompute**.

------

## 7 Spark Execution Flow

1. **Driver Program (驱动程序)** builds DAG → submits to **Cluster Manager**.
2. Cluster launches **Executor** containers (JVMs).
3. Driver sends **tasks** to Executors via RPC.
4. Executors run tasks, cache partitions, return results.

### Caching Levels

- `MEMORY_ONLY`, `MEMORY_AND_DISK`, `OFF_HEAP`.
- Persist intermediate RDDs to accelerate iterative ML.

## 10 Sample Exam / Interview Q & A (典型问答)

1. **What architectural separation in YARN resolved JobTracker bottlenecks?**
    *Answer:* Split global RM (resource allocation) from per‑app AM (task scheduling), eliminating single‑threaded JobTracker duties.

## 4 Measuring Energy in Computing Systems (测量方法)

## 5 Energy‑Efficiency Solution Space (六大策略)

### 5.1 Data‑Centre Design (机房设计)

- Site selection near **cheap renewables (可再生能源)**, free‑air or liquid cooling.
- Floor layout, hot/cold aisles, high‑efficiency UPS, **power‑usage effectiveness ≤ 1.2**.

### 5.2 DVFS / DVS (动态电压与频率调整)

| Technique | 中文注释                  | Effect                                                       |
| --------- | ------------------------- | ------------------------------------------------------------ |
| **DVS**   | Dynamic Voltage Scaling   | ↓ V → quadratic drop in dynamic power, ↑ latency             |
| **DFS**   | Dynamic Frequency Scaling | ↓ f lowers dynamic power linearly                            |
| **DVFS**  | Joint voltage+freq        | 10 % ↓ f can yield **17 % energy saving** with < 8 % perf loss |

- **Power‑aware scheduling** chooses per‑task DVFS setting to meet deadlines at minimum energy.

### 5.3 Server Consolidation (服务器整合)

- VM live‑migration packs VMs onto fewer hosts; idle hosts enter **sleep/off**.
- Challenges: hot‑cold aisle disruption, reboot risk, admin policies.

### 5.4 Virtualisation & Energy‑Aware Scheduling (能耗感知调度)

- Host overload/under‑load detection → migrate/select VMs (e.g., **MMT—Minimum Migration Time**).
- **Power‑Aware Best‑Fit** heuristic solves bin‑packing under SLA.

### 5.5 Data Analytics for Inefficiency Detection (数据分析)

- DCIM collects CPU/Memory/NIC metrics → ML identifies resource waste (observed 4 – 14 % CPU, 1 – 8 % Memory across server types).
- User demand forecasting enables **over‑provisioning / over‑booking** tactics.

### 5.6 Software‑Level EE Design (软件节能设计)

- Identify **energy hotspots (能耗热点)** during coding via IDE plug‑ins.
- Adopt adaptive patterns: **monitor–analyse–plan–execute (MAPE)** loop, autoscale threads, select heterogeneous cores.

## 10 Sample Exam / Interview Q & A (典型问答)

1. **Why does DVFS save quadratic energy but only linear performance?**
    *Answer:* Dynamic power ∝ V² f; lowering V & f reduces energy quadratically with voltage but execution time increases roughly linearly with frequency drop, yielding net gain.*
2. **Explain PUE limitations as a sole metric.**
    *Answer:* PUE captures facility overhead but ignores IT workload usefulness; a low‑PUE centre running inefficient code is still wasteful.*
3. **List two challenges of aggressive server consolidation.**
    *Answer:* Hot/cold aisle imbalance increases cooling cost; reboot latency risks SLA violation if powered‑off hosts fail to start.*
4. **How does speculative backup in DVFS scheduling help deadlines?**
    *Answer:* Launch a high‑frequency copy near deadline for slack‑aware tasks; if DVFS‑slowed primary overruns, backup ensures finish.*
5. **Give an example of energy‑aware software refactor at design stage.**
    *Answer:* Replace polling loops with event callbacks, reducing CPU wake‑ups and idle power draw.*

## 2 Background & Motivation

### 2.1 End of Classical Scaling

- CMOS transistor density improvements slow → cost per transistor no longer falls dramatically.
- Vendors pursue **domain‑specific accelerators**—GPU, TPU, DPU, custom ASICs—to maximise perf/W for target kernels.

### 2.2 Extreme Heterogeneity (极端异构)

DOE ASCR report (2018) projects supercomputers with **dozens of accelerator types**; challenge: organising and interconnecting such diversity efficiently fileciteturn21file0.

> **Question:** How do we physically arrange specialised dies so that applications reach required memory bandwidth and inter‑device connectivity?

### 2.3 Disaggregation Definition

> **Disaggregation** separates tightly‑coupled server components into **poolable, network‑attached resource blades (资源板卡)** that can be **composed on demand** by software orchestration.

## 4 Node‑ vs Rack‑Scale Disaggregation

### 4.1 Disaggregated Node

- Breaks single motherboard into chiplets/tiles over advanced packaging—e.g., **Intel Foveros**, AMD Infinity Fabric.
- Domain of chip architects.

### 4.2 Disaggregated Rack (机架级分解)

- Treats a rack as one **resource‑centric computer** where:
  - CPU sleds, GPU sleds, Memory sleds, NVMe sleds connect via **low‑latency optical fabric**.
  - Software defines **virtual machines with arbitrary ratios** (e.g., 1 CPU : 1 TB DRAM).
- Requires two fabrics — intra‑rack high‑speed; inter‑rack commodity Ethernet fileciteturn21file0.

> Current prototypes achieve 1–10 GB s⁻¹ link; still 1–2 orders below local DDR5 (100+ GB s⁻¹).

## 6 Workload‑Driven Case for Disaggregation

### 6.1 Memory‑Hungry Analytics

### 6.2 Accelerator Density

- AI training often limited by GPU count per node; rack‑scale pool offers tens of GPUs attached to modest CPU control plane.

### 6.3 Cold‑Storage Heavy

- Video archive services require many drives but little CPU; decoupling reduces stranded compute.

## 8 Challenges & Open Questions

- **Bandwidth Gap:** remote‑memory ≥ DDR local? need optical + CXL 3.1.
- **Latency‑Sensitive Apps:** HPC codes may suffer; solutions: near‑memory caching, NIC‑offload prefetch.
- **Software Stack:** OS/hypervisor, Kubernetes & VIM need awareness of composable resources.
- **Fault Domain Expansion:** component failure affects many virtual servers; requires fine‑grained redundancy.
- **Cost/Power Trade‑off:** optical fabric adds cost; evaluate TCO vs stranded resource savings.

## 9 Sample Exam / Interview Q & A

1. **Why does disaggregation promise higher utilisation but risk higher latency?**
    *Answer:* Pooling resources decreases stranded capacity, enabling flexible ratios; however, remote access traverses fabric adding micro‑second latency vs on‑board buses.*
2. **Explain how CXL enables memory pooling in rack‑scale designs.**
    *Answer:* CXL Type‑3 devices present memory behind a coherent interface; CPUs map remote DRAM as cache‑coherent address space via CXL switch.*
3. **Give two reasons optical interconnects are preferred over copper for rack disaggregation.**
    *Answer:* Higher bandwidth‑distance product without signal degradation; lower power/bit above 50 Gb s⁻¹ links.*
4. **What is resource fragmentation and how does disaggregation mitigate it?**
    *Answer:* Fragmentation = unused capacity trapped in fixed‑ratio servers; pooling lets orchestration compose right‑sized sets per workload.*
5. **List one software challenge when integrating disaggregated GPU pools into Kubernetes.**
    *Answer:* Scheduler must account for remote‑GPU topology and bandwidth, extending device‑plugin and NUMA‑aware placement policies.*

## 10 Key Takeaways (关键要点)

- Disaggregation rearranges compute, memory, storage into **network‑attached pools** enabling extreme heterogeneity and flexible provisioning.
- Node‑level (chiplet) and rack‑level approaches complement each other; both depend on **high‑bandwidth, low‑latency fabrics** such as CXL and silicon photonics.
- Workload perspective—memory‑hungry analytics, GPU‑dense AI, storage‑heavy cold data—drives ROI for disaggregated DCs.
- Open challenges: bandwidth gap to local DRAM, software orchestration, fault domains, cost‑power trade‑offs.
- Research continues via EU dReDBox, DOE extreme heterogeneity, and vendor prototypes aiming to mainstream composable infrastructure in the next decade.
