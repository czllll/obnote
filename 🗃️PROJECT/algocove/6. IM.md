			* 多了一层转发，latancy大

		* 方案：

# O3改进点
## 消息存储

> 你描述的「两条索引（发件箱 + 收件箱）＋一条公共内容记录」的三表模型，确实是很多 IM/社交产品在 **早期** 或 **中小规模** 阶段常用的做法；它最大优点是易理解、逻辑清晰，也天然满足了“各删各的”这一业务要求。不过，在 **真正的高并发、海量数据** 场景下，工业界往往会在此基础上做 **4 个方向的演进**：

1. **去 Join 化 → 一条消息两行存**（或直接落对象存储）
    
2. **分库分表 / 分区键设计**
    
3. **状态拆分：读回执、撤回、@ 提醒等放入专门的 state / event 表**
    
4. **冷热分层 + TTL / 归档**
    

下面逐点展开，并给出常见的落地方式与坑点，供你对照自身需求做取舍。

---

## 1 ️. “公共内容表” VS “一条消息两行”

|方案|读路径|写路径|优点|缺点 & 触发点|
|---|---|---|---|---|
|**你的三表方案**（message & user_message_idx）|查询=一次索引（走 user_id+msg_id PK）+一次主键 Join|1 行 message + 2 行 idx|存储节省，结构直观|- 热表 Join 成本随并发增大- message 表成为“中心热点”- 租户级扩容复杂|
|**两行方案**（直接把内容复制进 user_message）|查询=一次主键|写入=2 行|- 读极致简单，横向拆分容易- 避免跨分片 Join|- 存储翻倍（图片/文件需外部存储）- 删除／编辑需双写|

> **业界趋势**：
> 
> - **微信**早期是内容表 + 索引表，后来读瓶颈出现后，把 Text/Blob 拆到 COS/HDFS，索引表只保留 header & pointer；
>     
> - **Slack / Discord** 直接走「一条消息两行」+ **S3** 附件；
>     
> - 如果你的单聊/群聊页一次要 load 上千条历史，读放大比存储更痛，通常会优先考虑“两行方案 + 指针存正文”。
>     

---

## 2 ️. 分库分表策略

|分区键|典型选择|场景 & 注意点|
|---|---|---|
|**user_id + 逻辑分片号**|“user_id % N” 或一致性 hash|**Inbox/Outbox** 场景最常见；删除、撤回、读回执天然走单分片。|
|**conversation_id + 时间**|“conv_id % N, order by timestamp”|群聊或 Thread 模型；可以保证同一会话消息顺序性。|
|**Snowflake or Sonyflake 全局自增 ID**|分片=ID 高位|基础设施支持强的公司常用；全局唯一 + 顺序性，可做拉链表、乐观并发控制。|

**常见坑**

- 会话迁移：用户拉黑或合并帐号时，如何把历史消息迁移到新分片？
    
- 跨分片 Join：比如搜索“含关键词 A 的会话”必须扫所有分片，通常要再建 **ElasticSearch/Lucene** 异构索引。
    

---

## 3 ️. 状态拆分

> 把“消息本体”与“消息状态”解耦几乎是所有大厂的共识。

- **read_state / unread_count** 单独落 KV（Redis / DynamoDB / TiKV 等）；
    
- **撤回 / 举报 / 收回附件** 等生成一条 event_log；
    
- **@mention / reaction** 可走 Notification 表或 Streaming (Kafka)；
    
- 这样既减少热写，又方便做增量推送。
    

---

## 4 ️. 冷热分层 & 归档

|层级|存多久|存储介质|查询方式|
|---|---|---|---|
|**Hot**|30~60 天|MySQL / TiDB / Cassandra|直接主库|
|**Warm**|~1 年|HBase / Bigtable|后台异步迁移；用户查询走异步回流|
|**Cold / Archive**|1 年+|S3/GCS + Parquet|需异步拉回，或只做合规审计|

企业 IM（钉钉、Slack Enterprise Grid）通常还要做“合规审计 API”→ 直接在 Cold 层跑 Spark SQL。

---

## 什么时候保留三表设计就足够？

- DAU < **200 万**，写 QPS < **几千 / 分片**；
    
- 读瓶颈主要靠 Redis 缓存（PK 缓存即可，不用分布式 Join）；
    
- 消息体尺寸不大（纯文本 / 小图片，不用额外对象存储）；
    
- 未来 1~2 年内不计划大规模出海或 B 端运营。
    

如果你符合以上条件，三表方案很省心。只要**注意索引命中 & 消息体大小**——MySQL 单行最好 < 1 MB，就能跑得很稳。

---

### 小结 & 建议

1. **功能完整优先**：先用三表模型把删/撤/搜索做到位。
    
2. **监控瓶颈**：关注“跨表 Join 延迟、message 表热点、缓存击穿”三个指标。
    
3. **渐进演进**：读放大出现→复制消息到 user_message；存储压力出现→正文拆到对象存储；合规/搜索→异构索引或归档。
    
4. **治理流程**：上线“数据清理 & 归档”后台任务，避免粗暴 DELETE 阻塞线上。
    

> **一句话**：你的设计是工业界“第一步正确姿势”。随着 **QPS、消息体大小、合规需求** 增长，再按上面 4 个方向逐层演进即可。祝落地顺利!



# IM 模块需求分析
## 架构分析
* IM模块为algocove系统中的一个业务模块，使得系统有用户间实时通信的能力，V1阶段仅做用户间的单聊实现
* 涉及中间件与业务划分如下：
	* im-biz 业务模块：写实际的业务逻辑。
	* gateway 系统网关提供入口路由
	* redis：提供用户在线离线的记录以及ack的存储
	* kafka：消息队列保证消息有序性，避免手写业务上的重试了
	* mysql：db，提供消息、用户、用户关系等的数据存储
	* netty：底层通信框架，提供ws接口实现
* 业务流程如下：
	* 若两用户均在线
	* 用户A在页面点击用户B主页发起聊天：
		* 与server建立ws连接
		* 发送消息
	* server接收到消息
		* 落库
		* 发送kafka消息
	* B用户在线
		* kafka转发消息到B
		* B接收到消息
		* 前端显示

# 开发流程
## 建立消息实体和db tables
```sql
CREATE TABLE t_im_message (
  msg_id      BIGINT       PRIMARY KEY,
  chat_id     BIGINT       NOT NULL,
  seq         BIGINT       NOT NULL,
  sender_uid  VARCHAR(32)  NOT NULL,              
  content     MEDIUMTEXT   NOT NULL,
  created_time  DATETIME     DEFAULT CURRENT_TIMESTAMP,
  KEY idx_chat_seq (chat_id, seq),
  KEY idx_sender_ts (sender_uid, created_at)
);

CREATE TABLE t_im_conv_user (
  uid         VARCHAR(32)  NOT NULL,              
  chat_id     BIGINT       NOT NULL,
  unread_cnt  INT          NOT NULL DEFAULT 0,
  read_seq    BIGINT       NOT NULL DEFAULT 0,
  last_msg_id BIGINT       NOT NULL,
  updated_time  DATETIME     DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  PRIMARY KEY (uid, chat_id),
  KEY idx_recent (uid, updated_at)
);

```











- **建两张表**（SQL 如上）
    
- **初始化 Redis 键**（用户首次登录时写 `online:{uid}`）
    
- **写 WebSocket Handler** —— 收/推/ack
    
- **写 `/msg` Controller** —— 落库 + Kafka producer
    
- **Kafka Consumer** —— 批写 DB + 推送 or 离线池
    
- **前端**：全局 WS，聊天窗口按实体渲染、ack 已读
    
- **定时任务**：每 1 min 把 `conv:{uid}` / `total_unread:{uid}` flush 回 DB
