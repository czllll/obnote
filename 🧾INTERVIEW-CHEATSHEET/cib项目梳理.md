###  财资管理平台: 负责银行存款投资模块迭代开发及缺陷修复，实现多银行账户管理。 

#### 问题

* **说一下业务流程？**

  * 这个业务是投融资模块下的一个子模块
    * 普通用户可以进行存款申请，存款登记。
    * 用户在进行存款申请时，需要填入账户，金额，存入方式等信息。同时系统会生成一个存款申请单据包含唯一的存款申请id号。
    * 更高级的用户审批通过后，财务人员进行实际的资金存入。
    * 用户再进行存款登记，登记时要同时登记发生实际存款的银行流水号（必填）。同样，更高级用户进行审批，通过后生成一笔存款单据。
    * 存款单据信息包含
      * 唯一的单据号
      * 对应的银行流水号
      * 金额
      * 账号
      * 日期
      * 经办人/审批人等信息
    * 支出同理

* **这个模块实际的银行资金存入支出吗**？**还是就是个类似于台账的功能**？

  * 实际上不涉及实际的资金存入和支出，是一个类似于台账的功能

* **和银行接口是如何交互的？**

  * 系统定期通过银企直联同步流水数据，本模块同步数据然后进行对账
  * 有异常的单据提示异常，并标记给人工处理
    * 异常情况

* **实际的对账是如何做的呢？**

  * 事实上我门有专门的对账模块，我是通过feign调用的对账模块提供的接口
    * 我传入银行流水号，申请单据号，金额等信息进行匹配
    * 对账模块主要做了什么事情呢？
      * 检查传入的信息是否在同步到db的银行流水中有对应的信息

* **如果没有及时登记资金，也不会有冲突和提示对吧？**

  * 这部分是由“对账模块”来做的

    * **银行流水定期核对机制**（通常由专门的对账模块实现）
      专门的对账模块通常会定期（比如每日或每周）扫描『银行流水库』与各业务模块（如你的存款模块）单据的匹配情况。

      对于有银行流水却一直没有对应系统单据的情况，系统会：

      - 生成**『未登记资金入账』异常**，通知财务人员。
      - 提醒财务人员在系统中进行相应的登记和处理。

* **如果是每日同步一次银行流水数据的话，是不是有可能在存款模块，用户进行了存款登记，但是对账时，由于还没有同步流水信息过来，就会报错？**

  * 用户登记存款时，**不再调用实时对账**。
  * 存款登记单据**暂存**为`待对账`状态。
  * 【定时任务】每日银行流水数据同步后，**定时任务**自动进行批量匹配。
    * 同步银行流水数据
    * 自动进行批量资金对账
      * 系统会批量取出状态为`『待对账』`的存款登记单据；
        * 并用多线程（线程池）并发地进行自动对账
  * 如果匹配成功，自动标记单据为`已对账`；否则标记为`对账异常`。
  * `对账异常`单据进入**人工复核流程**，财务人员人工检查并处理（例如核实流水号是否输错）。

* **你做的需求中最有挑战性的是哪个？**

  * 我做过的需求中最有挑战性的一个，是**投融资模块中统一权限校验功能的实现**。

  * 我们投融资模块中有很多审批接口和业务接口，只有管理组以上的用户才有审批权限。之前我们是直接在每个方法里单独写权限判断代码，代码冗余严重，维护性差。

  * 为了统一解决这个问题，我设计了一个通用的**权限校验starter组件**，采用了**Spring AOP + 自定义注解**的方案：

    - 首先，通过自定义的权限注解，标注在Controller接口或业务方法上，明确限定哪些接口需要特定角色以上的权限访问。
    - 然后，通过统一的AOP切面动态拦截所有标注权限注解的方法，从登录上下文中自动获取当前用户角色，进行统一的权限校验。

  * 这样，业务代码无需再单独编写权限校验逻辑，只需要简单打上注解即可。这种方式极大提高了代码的复用性，降低了维护成本。

    

* **并发量大不大**

  * 我们的系统是企业内部的系统，一般并发量不会很高
  * 但是也考虑了可能出现的并发场景。比如，每天早上定时批量同步流水数据后，可能会产生并发请求：如批量对账操作
    * 从银行同步银行流水数据
    * 系统匹配用户登记的存款单据和银行流水单据
    * 匹配后更新单据状态，如 待对账 --> 已对账
  * 也会有并发风险：
    * 定时任务可能会重复执行，同一单据可能被多次匹配
    * 多个线程对同一部单据进行对账，可能造成数据更新混乱
  * 解决
    * 资金单据状态更新一般用『乐观锁』防止并发修改；
      * 在单据表中增加`version`字段，每次更新版本号+1
      * 更新时增加版本号判断，确保更新的单据是最新版本
    * 使用Redis分布式锁，防止定时任务触发的自动对账时重复处理；
      * 集群部署或者同一任务多次触发会造成重复对账
      * 使用redis setnx 实现分布式锁，只有获取到锁的节点会执行定时任务，其他节点跳过，避免重复执行

### **银企前置机客户端开发**：开发并维护客户端银企直联接口，支持用户在本地管理银企直联相关信息。

#### 问题

* **什么是银企直联？**
  * 允许企业通过银行提供的直连接口（API、SFTP等协议）直接访问银行账户，无需通过银行官网人工操作
* **什么是银企前置机？**
  * 位于企业财资管理系统（TMS/ERP）与银行系统之间
  * 作用
    * 对接企业内部系统（ERP、财务系统）
    * 与银行API交互，发送指令、获取数据
    * 数据转换（不同银行接口格式不同，进行标准化）
  * 组成
    * admin
    * server
      * 负责与银行系统交互，处理企业支付查询对账请求
    * client
      * 供企业用户使用，允许用户本地管理银行配置，发起交易等

* **业务流程是怎样的？**

  * 这一部分主要是客户端的改造。配置、维护银行信息，供前置机server读取使用

  * 流程

    * 用户在GUI选择管理银企直联信息（crud银行）
    * 客户端调用本低API，讲配置存入DB
    * 前置Server读取这些数据，在实际对接银行时使用
    * 用户可以在客户端查看当前所有已配置的银行信息

    

    

    

### **银企前置机服务端开发**：集成多家银行银企插件，使前置机具备访问他行银企直联信息并调用相关服务的能力。

#### 问题

* 业务流程是怎样的？
  * **前置机 Server 读取企业已配置的银行信息**（从本地数据库获取 `bank_config`）。
  * **根据银行类型，加载对应的银行插件**（调用不同银行的 SDK）。
  * **封装请求参数，调用银行 SDK 或 API**（不同银行的格式可能不同）。
  * **解析银行返回的数据，转换成标准格式**（保证前置机 Server 处理时格式统一）。
  * **返回结果给前置机 Server，供业务系统使用**。
* 使用场景？
  * 企业用户在**客户端**选择银行，输入 API Key、证书等信息，前置机 Server 读取这些配置，加载**对应的银行插件**进行请求。
  * **不同银行的 SDK 可能有不同的入参、返回格式，插件需要进行转换。**
  * **前置机 Server 只需要调用标准化接口，不用关心不同银行的差异。**
* 技术实现
  * 核心任务：插件化架构+适配不同银行SDK
    * 设计：由于不同银行的SDK格式不同，设计插件机制，前置机 Server 通过 `BankPluginLoader` 动态加载合适的插件。
  * 实现
    * 使用Java SPI（Service Provider Interface）实现插件动态加载
    * 统一接口封装-格式转换
    * 银行SDK适配
      * 不同银行的 API 格式不一样（XML/JSON），所以我们做了适配层，把所有银行 API 统一成标准接口
      * 前置机 Server 只需要调用 `queryAccountInfo(accountNo)`，而不需要关心银行的 API 差异
