## 应用层

### HTTP 常见状态码总结

1. **HTTP 状态码的分类**

   HTTP 状态码按首位数字共分为 5 类：

   - **1xx（信息性）**：表示请求已被接收，继续处理。常用于协议交换的中间状态，实际使用较少。
   - **2xx（成功）**：表示请求已成功被服务器接收、理解并处理。
   - **3xx（重定向）**：表示请求的资源位置发生变化，需要客户端进一步操作。
   - **4xx（客户端错误）**：客户端提交的请求有错误，服务器无法处理。
   - **5xx（服务器错误）**：服务器处理请求时内部发生错误，属于服务端问题。

2. **常见状态码详解**

   - **1xx 信息性响应**
     - `100 Continue`：表示一切正常，客户端可继续发送请求主体。
     - `101 Switching Protocols`：服务器同意客户端更改协议（如 HTTP 升级为 WebSocket）。
   - **2xx 成功响应**
     - `200 OK`：请求成功。
     - `201 Created`：请求成功并创建了新资源。
     - `202 Accepted`：请求已接受但尚未处理完成。
     - `204 No Content`：请求成功，但无返回内容（常用于 DELETE）。
   - **3xx 重定向响应**
     - `301 Moved Permanently`：永久重定向，请使用新的 URL。
     - `302 Found`：临时重定向，资源临时搬迁。
     - `303 See Other`：请求应通过另一个 URI 使用 GET 获取资源。
     - `304 Not Modified`：缓存生效，资源未修改，客户端可使用本地副本。
     - `307 Temporary Redirect`：临时重定向，方法不变。
     - `308 Permanent Redirect`：永久重定向，方法不变。
   - **4xx 客户端错误**
     - `400 Bad Request`：请求语法错误，服务器无法理解。
     - `401 Unauthorized`：未认证，需提供凭证（如登录）。
     - `403 Forbidden`：已认证但无权访问。
     - `404 Not Found`：请求资源不存在。
     - `405 Method Not Allowed`：请求方法不被允许。
     - `408 Request Timeout`：请求超时。
     - `409 Conflict`：请求与资源当前状态冲突。
     - `429 Too Many Requests`：请求频率过高，被限流。
   - **5xx 服务器错误**
     - `500 Internal Server Error`：服务器内部错误。
     - `501 Not Implemented`：服务器不支持请求的方法。
     - `502 Bad Gateway`：网关或代理服务器从上游服务器收到无效响应。
     - `503 Service Unavailable`：服务器暂时无法处理请求（如超载、维护）。
     - `504 Gateway Timeout`：网关或代理请求上游服务器超时。

### HTTP 状态码 301 和 302 区别总结

1. **重定向的基本概念**
   - 当客户端请求的资源地址发生变更，服务器会返回 3xx 状态码，指示客户端访问新的地址（URL）。
   - 响应中会携带 `Location` 头，指向新的资源地址，浏览器会自动跳转。
2. **301 Moved Permanently（永久重定向）**
   - 表示请求的资源已被永久移动到新的位置。
   - 客户端应使用新的 URL 进行访问，**今后都应使用新地址**。
   - 浏览器会缓存该重定向结果，**下次访问会自动使用新地址**。
   - 常用于网站域名迁移、SEO 优化（权重会传递到新地址）。
3. **302 Found（临时重定向）**
   - 表示请求的资源临时被移动到另一个位置，**原 URL 仍有效，未来可能还会使用**。
   - 浏览器不应缓存重定向地址。
   - 原则上客户端应继续使用原 URL 发起请求。
   - 在 HTTP/1.0 中，302 是最常用的临时重定向状态码，但其行为在某些浏览器中存在歧义（如会把 POST 改成 GET）。
4. **扩展补充：303 与 307、308**
   - `303 See Other`：建议客户端使用 **GET 方法** 访问新地址，常用于表单 POST 后跳转。
   - `307 Temporary Redirect`：明确要求客户端使用 **原始方法**（如 POST）重定向，解决了 302 的歧义问题。
   - `308 Permanent Redirect`：与 301 类似，也是永久重定向，但保留原始请求方法。
5. **实际开发中的注意点**
   - SEO 推荐使用 **301** 做永久重定向，传递页面权重。
   - 表单提交后跳转建议使用 **303**，避免表单重复提交。
   - 对于严格遵循 HTTP 规范的 API，推荐使用 **307/308** 来避免请求方法改变。

### HTTP 状态码 502 和 504 区别总结

1. **基本概念**

   都属于 **5xx 服务器错误**，通常发生在使用代理或网关（如 Nginx、CDN）时，涉及多个服务器之间的通信问题。

2. **502 Bad Gateway（错误网关）**

   - 网关或代理服务器尝试请求上游服务器时，**收到的是无效响应**。
   - 本质是：**上游服务器响应了，但响应内容异常或格式错误，无法处理。**
   - 常见场景：
     - 上游服务器崩溃、重启中、挂掉了。
     - 返回了非法的 HTTP 响应头。
     - 程序 BUG，返回内容不合法。
     - Nginx 转发错误配置。

3. **504 Gateway Timeout（网关超时）**

   - 网关或代理服务器等待上游服务器响应时，**长时间无响应导致超时**。
   - 本质是：**上游服务器根本没有返回任何响应。**
   - 常见场景：
     - 上游服务器处理时间过长（如数据库慢查询）。
     - Nginx 设置的超时时间过短。
     - 上游服务已挂或卡死，没有响应。

4. **形象举例：以 Nginx 为代理**

   - 用户请求 → Nginx → 后端服务（如 Tomcat）

   | 场景                     | 状态码 | 说明               |
   | ------------------------ | ------ | ------------------ |
   | Nginx 收到非法响应       | 502    | 有响应但格式错误   |
   | Nginx 等太久都没收到响应 | 504    | 完全没响应，超时了 |

5. **实际处理建议**

   - **遇到 502：**
     - 检查上游服务器是否崩溃、重启中。
     - 检查接口返回是否符合 HTTP 协议。
     - 检查网关配置是否正确。
   - **遇到 504：**
     - 优化上游处理逻辑，避免长时间卡顿。
     - 检查数据库慢查询、死循环等问题。
     - 调整 Nginx 的 `proxy_read_timeout`、`fastcgi_read_timeout` 等超时配置。

### HTTP 请求的常见方法总结

1. **GET：获取资源**
   - 用于向服务器请求指定资源。
   - 常用于获取数据，**无副作用**（幂等）。
   - 请求参数通过 URL 传递。
   - 通常不会改变服务器状态。
   - **浏览器缓存友好，支持书签、历史记录。**
2. **POST：创建资源**
   - 用于向服务器提交数据，创建新资源或进行处理。
   - 请求体中包含数据（如表单、JSON）。
   - 非幂等，每次请求都可能产生不同结果。
   - 常用于表单提交、用户注册、发评论等操作。
3. **PUT：更新资源（整体替换）**
   - 用于更新指定资源，**通常是完整更新**。
   - 客户端提供完整的资源表示，服务器用其替换旧资源。
   - 幂等：多次请求结果一致。
   - 若资源不存在，部分服务端会创建资源（行为不统一）。
4. **PATCH：更新资源（局部修改）** 🔥【补充】
   - 用于对资源进行**部分更新**。
   - 与 PUT 的区别在于 **只修改变化部分**，而不是整体替换。
   - 幂等性不强制，依实现而异。
   - 常用于用户信息修改、配置更新等场景。
5. **DELETE：删除资源**
   - 请求服务器删除指定资源。
   - 幂等：多次删除同一资源结果相同。
   - 可能需要权限验证或进行逻辑删除。
6. **HEAD：获取资源头部信息**
   - 类似 GET，但不返回响应体（仅返回响应头）。
   - 用于检查资源是否存在、获取元信息（如内容长度、类型）。
   - 可用于断点续传前检查资源状态。
7. **OPTIONS：探测通信能力**
   - 用于查询服务器支持哪些请求方法。
   - 常用于 **CORS 预检请求**，验证跨域是否被允许。
   - 一般不会带数据，也不会修改服务器状态。
8. **其他（了解即可）**
   - `CONNECT`：用于建立网络隧道，常用于 HTTPS。
   - `TRACE`：用于诊断请求链路，容易被滥用，一般禁用。

### GET 与 POST 请求的区别总结

1. **基本定义**

   - **GET**：用于向服务器获取指定资源，参数一般拼接在 URL 中，请求无副作用。
   - **POST**：用于向服务器提交数据，通常包含在请求体（body）中，可用于创建或处理资源。

2. **参数传递位置**

   - **GET**：参数通过 URL 传递，拼接在 `?` 后面，公开且有长度限制（浏览器/服务器限制）。
   - **POST**：参数放在请求体（body）中，内容更丰富，基本无长度限制，支持各种数据格式。

3. **幂等性与安全性**

   - **GET**
     - 幂等：多次请求结果相同。
     - 安全：只获取数据，不改变服务器状态。
     - 可被缓存、保存在浏览器历史中、作为书签保存。
   - **POST**
     - 非幂等：多次请求可能导致重复创建资源。
     - 不安全：通常会修改服务器资源状态。
     - 默认不会被缓存，也无法作为书签保存。

4. **缓存机制**

   - **GET**：可被浏览器缓存、CDN 缓存，提升性能。
   - **POST**：通常不会被缓存。

5. **使用场景**

   | 使用场景                   | 推荐方法 | 原因                        |
   | -------------------------- | -------- | --------------------------- |
   | 查询、获取数据             | GET      | 无副作用，支持缓存和书签    |
   | 提交表单、注册登录         | POST     | 数据放在 body，支持多种格式 |
   | 文件上传                   | POST     | 支持传输大数据              |
   | 查询中含敏感信息（如密码） | POST     | 不暴露参数在 URL 中         |

6. **实际开发中的注意事项**

   - 虽然从规范上讲 GET 是“安全、幂等”的，但如果使用 GET 实现了更新或删除操作，会破坏语义，导致缓存失效或安全隐患。
   - 虽然 POST 理论上可以做任何事，但应合理区分 GET/POST 的语义，保持接口清晰一致。
   - Nginx、浏览器、CDN 等中间件对 GET/POST 的行为也不同，设计接口时需注意兼容性与行为一致性。

### HTTP 的长连接（Keep-Alive）总结

1. **什么是长连接（Keep-Alive）**

   - **HTTP 长连接**指的是：在一次 TCP 连接建立后，客户端和服务器可以在该连接上 **复用多个 HTTP 请求/响应**，无需每次请求都重新建立 TCP 连接。

   - 通常通过在请求头中加入：

     ```
     Connection: keep-alive
     ```

     来启用长连接。

2. **对比短连接（非 Keep-Alive）**

   | 类型       | 特点                                           | 缺点                         |
   | ---------- | ---------------------------------------------- | ---------------------------- |
   | **短连接** | 每次请求都新建 TCP 连接，请求完就断开          | 频繁建立/关闭 TCP，开销大    |
   | **长连接** | TCP 连接在多个请求间复用，连接持续保持一段时间 | 如果连接过多会占用服务端资源 |

3. **长连接的优势**

   - 避免频繁的 TCP 握手与挥手，**提升性能**。
   - 降低网络延迟，提升网页加载速度。
   - 特别适用于**一个页面中有多个资源请求**（如图片、CSS、JS 等）。

4. **连接的关闭方式**

   - 客户端或服务端在请求头中显式声明关闭：

     ```
     Connection: close
     ```

   - 服务器在 **超时未收到请求** 或连接空闲时自动关闭。

5. **长连接与 HTTP 版本的关系**

   - **HTTP/1.0** 默认使用短连接，需要显式声明 `Connection: keep-alive`。
   - **HTTP/1.1** 默认开启长连接，除非声明 `Connection: close`。
   - **HTTP/2 及以上**：使用 **一个 TCP 连接实现多路复用（multiplexing）**，比 HTTP/1.1 的长连接更高效，彻底解决了“队头阻塞”问题。

6. **长连接的典型应用场景**

   - 现代网页中需加载大量资源的场景。
   - API 调用频繁的前后端通信。
   - 持续交互的接口（可结合 WebSocket 实现全双工通信）。

### HTTP/1.1 中请求拆包的机制

1. **为什么需要拆包机制**

   - HTTP 是基于 TCP 的协议，而 **TCP 是流式协议**，数据可能被拆分成多个包发送。
   - 所以服务器收到的数据流并不是“一整个请求”，需要 **明确知道请求正文的长度**，才能正确读取和解析请求体。

2. **HTTP/1.1 中的拆包方式：使用 `Content-Length`**

   - 客户端在发送 HTTP 请求时，会在请求头中添加：

     ```
     Content-Length: <字节数>
     ```

   - 该字段表示请求体（body）的长度，以 **字节数** 为单位。

   - 服务器接收请求后，会根据该长度从数据流中读取相应字节，直到完整读取整个请求体为止。

3. **拆包流程示意**

   - 例：若请求体大小为 1000 字节，则添加请求头：

     ```
     Content-Length: 1000
     ```

   - 服务器据此读取数据流中 **接下来的 1000 个字节**，作为请求体内容。

   - 读取完成后，即可开始处理该请求。

4. **Content-Length 的作用**

   - 用于解决 **TCP 流数据边界不明确**的问题。
   - 避免出现读取不完整（丢包）或读取过多（截断下一条请求）的错误。
   - 保证服务器可以完整、准确地接收请求体。

5. **注意事项**

   - 只有当请求中 **包含请求体（如 POST、PUT）** 时，才需携带 `Content-Length`。
   - 若未携带 `Content-Length`，服务端将无法判断正文长度，可能导致阻塞或解析失败。
   - 若使用了分块传输（chunked transfer encoding），则不使用 `Content-Length`，改用 `Transfer-Encoding: chunked`。

### HTTP 断点续传总结

1. **什么是断点续传**

   - 断点续传是 HTTP/1.1 的特性，允许客户端在下载中断后，**从中断的位置继续下载**，而不必从头开始。
   - 它通过 HTTP 请求头中的 `Range` 与响应头中的 `Content-Range` 等字段实现。

2. **断点续传的作用场景**

   - 下载大文件时网络中断或暂停，重连后希望从断点继续下载。
   - 视频播放器、下载工具、CDN 文件传输等场景中广泛应用。
   - 可以节省带宽，提高下载效率，提升用户体验。

3. **断点续传的基本流程**

   假设要下载一个 1024KB 的视频文件：

   1. 客户端发送请求，服务端在响应头中返回：

      ```
      Accept-Ranges: bytes
      ```

      表示支持按字节范围请求资源。

   2. 若客户端下载到 512KB 时网络断开，之后重新请求，携带请求头：

      ```
      Range: bytes=512000-
      ```

      表示从 512KB 开始续传。

   3. 服务器响应：

      ```
      HTTP/1.1 206 Partial Content
      Content-Range: bytes 512000-1023999/1024000
      Content-Length: 512000
      ```

      表示只返回 512KB 后半部分内容。

   4. 客户端拼接前半部分与后半部分，完成文件恢复。

4. **关键 HTTP 头字段说明**

   - **Accept-Ranges: bytes**
     - 服务端声明支持“范围请求”功能，单位为字节。
     - 没有该字段则说明不支持断点续传。
   - **Range: bytes=start-end**
     - 由客户端发送，表示希望获取指定区间的内容。
     - 可以只写开始如：`Range: bytes=512000-`，表示从某处到结尾。
   - **Content-Range: bytes start-end/total**
     - 由服务端返回，表示实际响应的数据区间及完整资源的总长度。
   - **Content-Length**
     - 实际返回的数据长度，而不是资源的总长度。

5. **相关状态码**

   - `206 Partial Content`：服务器成功处理了带 Range 的请求，返回部分内容。
   - `416 Requested Range Not Satisfiable`：客户端请求的范围无效（超出资源大小等）。

6. **断点续传的局限性**

   - 服务端必须支持 Range 请求，否则无法实现断点续传。
   - 文件内容若有变更，断点续传的数据可能不一致。
   - 静态资源服务器/CDN 通常都支持；动态接口需开发配合。

### HTTP 为什么不安全？（与 HTTPS 对比）

1. **HTTP 是明文传输协议**

   - HTTP 协议传输的数据不会加密，所有请求与响应内容在传输过程中**裸露在网络中**，容易被第三方窃取、篡改或伪造。
   - 通常运行在 TCP 协议之上，**不具备任何安全机制**。

2. **三大安全风险**

   - **窃听风险**：攻击者可在通信链路中监听到请求内容，例如账号、密码、用户敏感信息。
   - **篡改风险**：中间人可拦截并修改请求或响应内容，例如强制插入广告、篡改页面、投放木马。
   - **伪装风险**：攻击者可以伪造网站，例如伪装成银行或购物网站，诱导用户输入账号密码进行钓鱼攻击。

3. **HTTPS 如何解决上述风险**

   HTTPS = HTTP + SSL/TLS，加密层位于 HTTP 与 TCP 之间。

   - **信息加密**：传输内容经过加密，即使被监听也无法解密。
   - **校验机制**：保证通信内容未被篡改，一旦修改传输内容，校验就会失败。
   - **身份认证**：通过数字证书验证服务器身份，防止用户被钓鱼或重定向到伪造网站。

4. **HTTPS 带来的安全增强点**

   | 安全能力     | 描述                                          |
   | ------------ | --------------------------------------------- |
   | **加密性**   | 防止第三方窃听内容                            |
   | **完整性**   | 防止中间人篡改请求/响应数据                   |
   | **身份验证** | 通过 CA 签发证书确认对方身份（服务器/客户端） |

5. **小贴士：HTTPS ≠ 绝对安全**

   - 如果用户访问了钓鱼网站、或服务器证书被伪造（如未验证证书链），HTTPS 也无法完全保障安全。
   - 仍需配合**浏览器警告机制、CA 认证体系、防钓鱼训练、服务器配置加固**等手段。

### HTTP和HTTPS 的区别？

**1. HTTP与HTTPS核心区别**
1.1 **协议安全性**

- HTTP：明文传输（✘ 原描述正确，但需强调风险：易被窃听/篡改）
- HTTPS：= HTTP + SSL/TLS（✔ 修正说明：**TLS是SSL的升级版**，现代多用TLS）
  - 补充：加密方式含 **对称加密（传输数据） + 非对称加密（交换密钥）**

1.2 **连接建立过程**

- HTTP：TCP三次握手 → 直接传输（✔ 原描述正确）
- HTTPS：TCP三次握手 → **TLS四次握手**（★重要细节）
  - 补充步骤：
    1. 客户端发送TLS版本、支持的加密算法
    2. 服务端返回证书+公钥
    3. 客户端验证证书→生成会话密钥→用公钥加密发送
    4. 服务端用私钥解密→确认加密通信

1.3 **默认端口**

- HTTP：80（✔）
- HTTPS：443（✔）
  - 补充：**端口可自定义**，但非标准端口需显式声明（如http://domain:8080）

1.4 **身份验证**

- HTTPS必须使用CA证书（★修正：**自签名证书也可用**，但会触发浏览器警告）
  - 补充流程：证书包含 **公钥+服务器信息+数字签名**，通过CA链验证真实性

**2. 扩展考点**
2.1 **性能对比**

- HTTPS多出TLS握手和加密计算，首次连接延迟高约200-400ms
- HTTP/2需基于HTTPS，可复用连接降低延迟

2.2 **应用场景**

- HTTP：内部网络、无需敏感数据的服务（如新闻站）
- HTTPS：登录/支付/API接口，**现代浏览器强制要求HTTPS**（如Chrome标记HTTP为不安全）

### HTTPS握手过程说一下？

传统的 TLS 握手早期多使用 RSA 算法实现密钥交换，但现代更推荐 **ECDHE（支持前向保密）**。在部署 TLS 证书时，证书文件包含服务端的公钥，会在握手阶段传递给客户端，而私钥必须严格保密（*若泄漏会导致历史通信被解密*）。

### **RSA密钥交换的四次握手流程**

1. **第一次握手 - ClientHello**
   - 客户端发送：
     - 支持的TLS版本（如TLS 1.2）
     - **客户端随机数（Client Random）**
     - 支持的加密套件（如`RSA_AES_128_GCM_SHA256`）
2. **第二次握手 - ServerHello**
   - 服务端响应：
     - 选定的TLS版本和加密套件
     - **服务端随机数（Server Random）**
     - **数字证书**（含RSA公钥，需验证CA签名）
     - *（双向认证时会请求客户端证书）*
3. **第三次握手 - ClientKeyExchange**
   - 客户端操作：
     - 验证证书（*检查CA链、有效期、域名匹配*）
     - 生成**预主密钥（Premaster Secret）** → **用证书公钥加密**后发送
     - 发送**Finished消息**（*加密的握手摘要，防篡改*）
4. **第四次握手 - ServerFinished**
   - 服务端操作：
     - **用私钥解密预主密钥**
     - 双方生成**会话密钥**（*公式：`Master Secret = PRF(Client Random + Server Random + Premaster Secret)`*）
     - 发送**Finished消息**确认加密通道建立

### Http1.1和2.0的区别是什么？

1. **头部压缩（Header Compression）**
   - HTTP/1.1 每次请求都要带完整的 Header，重复字段造成浪费。
   - HTTP/2 使用 **HPACK 算法** 对头部进行压缩，客户端和服务器各维护一张动态表，存储已发送过的头字段，只需发送索引号即可。
   - 优势：
     - 减少冗余数据传输。
     - 降低网络带宽开销。
     - 提高传输速度。
2. **二进制分帧（Binary Framing Layer）**
   - HTTP/1.1 是基于文本的协议，解析复杂。
   - HTTP/2 采用 **二进制格式** 传输，通信双方传输的是 **帧（Frame）**，每个帧都属于某个流（Stream）。
   - 常见帧类型包括：**头信息帧（Headers Frame）** 和 **数据帧（Data Frame）**。
   - 优势：
     - 简化协议解析。
     - 提高解析效率。
     - 更适合计算机处理。
3. **多路复用（Multiplexing）**
   - HTTP/1.1 中同一 TCP 连接一次只能处理一个请求，多个请求必须排队，容易出现“队头阻塞（Head-of-line Blocking）”。
   - HTTP/2 引入 **Stream（流）** 的概念，允许多个请求/响应并发在一条 TCP 连接上交错传输，互不影响。
   - 优势：
     - 消除队头阻塞问题。
     - 提高资源利用率与传输效率。
4. **服务器推送（Server Push）**
   - HTTP/1.1 是被动响应模式，必须客户端先发请求。
   - HTTP/2 支持服务器主动推送资源（如 CSS、JS、图片等）给客户端，无需客户端显式请求。
   - 优势：
     - 减少请求延迟。
     - 加快页面加载速度。
5. **连接复用 + 更少的连接数**
   - HTTP/1.1 为实现并发请求，通常建立多个 TCP 连接（浏览器限制同域名并发数）。
   - HTTP/2 通过多路复用只需一个连接即可处理多个请求，减少连接开销与拥塞。
6. **流量控制（Flow Control）**
   - HTTP/2 支持对单个流设置独立的流量控制策略，避免一个流占用所有带宽。
7. **优先级（Priority）**
   - HTTP/2 中的流可以分配权重和依赖关系，允许客户端指定请求的优先级，服务器可据此优化资源调度。

### HTTP进行TCP连接之后，在什么情况下会中断？

**1. 主动关闭连接（四次挥手）**

- **触发条件**：
  - 客户端或服务端主动调用`close()`系统调用
  - HTTP请求完成后，客户端或服务端主动关闭连接（如HTTP/1.0默认关闭）
- **具体流程**：
  1. 主动关闭方发送`FIN`报文（第一次挥手）
  2. 被动关闭方回复`ACK`（第二次挥手）
  3. 被动关闭方发送`FIN`（第三次挥手）
  4. 主动关闭方回复`ACK`（第四次挥手）
- **补充说明**：
  - **TIME_WAIT状态**：主动关闭方会等待`2MSL`（报文最大生存时间）再彻底关闭，确保最后一个`ACK`到达对方
  - **HTTP/1.1优化**：默认启用`Keep-Alive`，避免频繁四次挥手

**2. 超时重传失败（被动中断）**

- **触发条件**：
  - 发送方未收到`ACK`响应，触发超时重传机制
  - 达到最大重传次数（默认`tcp_retries2`，通常为15次）
- **关键参数**：
  - **RTO（重传超时时间）**：动态计算，基于RTT（往返时间）
  - **最大重传次数**：由系统参数决定（Linux默认为15次）
- **补充场景**：
  - 网络链路故障（如网线断开、路由器宕机）
  - 接收方进程崩溃（无法回复`ACK`）

**3. 空闲超时关闭**

- **HTTP Keep-Alive超时**：
  - 服务器或客户端设置的`Keep-Alive`超时时间到期（如Nginx默认`keepalive_timeout 75s`）
  - 长时间无HTTP请求/响应时触发
- **TCP Keepalive机制**：
  - 系统级保活探测（默认2小时空闲后发送探测包）
  - 探测失败后断开连接
- **补充说明**：
  - **HTTP/1.1**：默认启用`Connection: keep-alive`，需依赖应用层或传输层超时设置
  - **HTTP/2**：多路复用减少连接重建，但仍有空闲超时控制

**4. 其他中断情况（补充）**

- **异常中断**：
  - 接收方发送`RST`报文（如端口未监听、缓冲区溢出）
  - 中间设备（防火墙/NAT）会话超时强制断开
- **应用层控制**：
  - HTTP响应头`Connection: close`显式关闭
  - WebSocket等长连接协议的心跳机制失败

### HTTP、SOCKET和TCP的区别？

1. **核心概念**
   HTTP是应用层协议，负责定义客户端和服务器之间的数据交换格式；Socket是操作系统提供的网络通信接口，实现进程间数据传输；TCP是传输层协议，确保数据可靠、有序地传输。

2. **关键区别**

   1. **协议层级**

      - HTTP：应用层协议，处理具体业务逻辑（如网页请求）。
      - Socket：介于应用层和传输层之间的编程接口。
      - TCP：传输层协议，负责端到端的可靠传输。

      1. **功能特点**
         - HTTP：无状态、基于请求-响应模型，依赖TCP实现可靠性。
         - Socket：支持多种协议（TCP/UDP），提供通用的网络操作函数（如send/recv）。
         - TCP：通过三次握手、流量控制、重传机制保证数据完整。

   1. **交互对象**
      - HTTP：浏览器与Web服务器之间的通信规范。
        - Socket：任意网络进程间的通信端点（需指定IP和端口）。
        - TCP：主机传输层实体间的连接管理。

3. **协作流程示例**

   * 浏览器生成HTTP请求（如GET /index.html）。

   * 通过Socket API调用TCP发送请求（操作系统添加TCP头部）。

   * TCP协议处理分片、重传、确认等底层细节。

   * 数据经网络层（IP）和物理层传输到服务器。

4. **常见误区**

   - 误区：Socket是协议 → 实际是编程接口。

   - 误区：HTTP可脱离TCP → HTTP/1-2依赖TCP，HTTP/3改用QUIC（基于UDP）。

   - 误区：TCP Socket与UDP Socket无关 → 两者共享相同API，通过参数区分。

### DNS（Domain Name System）域名系统详解

1. **基本概念**
   DNS是互联网中将域名转换为IP地址的分布式数据库系统。它解决了人们需要记忆复杂IP地址的问题，通过层级化的域名设计实现高效查询。

2. **域名结构特点**

   - 书写格式：从低到高，如"[www.server.com](http://www.server.com/)."

   - 层级划分：
     - 根域名（.）：全球13组根服务器
       - 顶级域（.com/.net等）：由ICANN管理
       - 二级域（server.com）：用户可注册部分
       - 子域名（www.等）：用户自定义

3. **域名解析流程**
   * 客户端查询本地DNS缓存
   * 未命中则向递归DNS服务器查询
   * 递归服务器从根域名开始迭代查询：
     a) 根服务器返回顶级域服务器地址
     b) 顶级域服务器返回权威服务器地址
     c) 权威服务器返回最终解析结果
   * 结果返回客户端并缓存

4. **技术特点**

   - 分布式架构：全球超过1000台根镜像服务器

   - 缓存机制：TTL控制缓存时间，提高效率

   - 查询类型：递归查询与迭代查询结合

   - 协议支持：同时支持UDP和TCP的53端口

5. **补充说明**

   - 完整域名末尾的"."表示根域，通常省略

   - DNS同时支持正向解析（域名→IP）和反向解析（IP→域名）

   - 新兴技术：DNSSEC提供安全验证，防止DNS欺骗

### DNS域名解析工作流程

1. **客户端发起 DNS 请求**
   - 用户在浏览器中输入网址（如 `www.server.com`）后，客户端会发起 DNS 查询请求。
   - 请求会被发送给配置在本地系统中的 **本地域名解析器（Local DNS Resolver）**，通常是运营商的 DNS 服务器。
2. **本地 DNS 查询缓存**
   - 本地 DNS 首先在自身缓存中查找是否已有 `www.server.com` 的 IP 地址。
   - 如果找到，直接返回给客户端，解析流程结束。
   - 如果未命中缓存，则进入递归查询流程。
3. **本地 DNS 递归查询根 DNS**
   - 本地 DNS 向 **根域名服务器（Root DNS Server）** 发起请求：“请告诉我 `www.server.com` 的 IP 地址。”
   - 根 DNS 不会直接告诉你目标 IP，但会告诉你哪个顶级域名服务器（TLD Server）可以回答，例如 `.com` 域的服务器地址。
4. **本地 DNS 向顶级域名服务器查询**
   - 本地 DNS 拿到 `.com` 顶级域名服务器地址后，继续发请求：“请告诉我 `www.server.com` 的 IP 地址。”
   - 顶级域名服务器会告诉你：这个域名由某个 **权威 DNS 服务器（Authoritative DNS Server）** 管理，并返回其地址。
5. **本地 DNS 向权威 DNS 服务器查询**
   - 本地 DNS 继续向权威 DNS 服务器发请求，查询 `www.server.com` 的真实 IP。
   - 权威 DNS 服务器是该域名的最终解释者，具有该域的准确解析记录。
6. **权威 DNS 返回解析结果**
   - 权威 DNS 服务器将 `www.server.com` 对应的 IP 地址（如 `X.X.X.X`）返回给本地 DNS。
7. **本地 DNS 缓存并返回结果**
   - 本地 DNS 将解析结果暂存在缓存中（设置有效期），以便下次请求能快速响应。
   - 然后将 IP 地址返回给客户端。
8. **客户端与目标建立连接**
   - 客户端收到 IP 地址后，使用该 IP 与服务器建立 TCP 连接，开始进行 HTTP 等后续通信。

### **DNS 底层协议：TCP 与 UDP**

1. **默认使用 UDP**
   - DNS 查询默认使用 UDP 协议，端口号为 53。
   - UDP 属于无连接协议，适合 DNS 查询这种频繁、体积小、响应快的场景。
2. **UDP 的优点**
   - **低延迟**：无连接特性省略握手过程，减少时延。
   - **简单快速**：没有连接维护与流量控制机制，传输开销小。
   - **轻量级**：报文头部占用资源少，适用于频繁小请求。
3. **使用 TCP 的场景**
   - **响应数据过大**：当 DNS 响应超过 UDP 的报文大小限制（IPv4 为 512 字节，IPv6 为 1232 字节），客户端会自动改用 TCP 重新发送请求。
   - **区域传输（Zone Transfer）**：主从 DNS 服务器之间同步数据时使用 TCP。
   - **防攻击策略**：部分服务器为防止 UDP 放大攻击，强制使用 TCP。
4. **UDP 的风险与容错机制**
   - UDP 可能发生丢包、乱序等问题，不保证可靠性。
   - DNS 协议通过超时重传、查询重试、结果缓存等机制提高整体可用性与正确性。
5. **总结**
   - 常规 DNS 查询首选 UDP，保障查询速度与效率。
   - 在特定条件下自动切换为 TCP，保证完整性与稳定性。

### HTTP到底是不是无状态的？

1. **本质是无状态协议**
   - 每个 HTTP 请求都是独立的，请求之间没有上下文关系。
   - 服务器不会主动保存客户端的历史请求或会话信息。
   - 每次请求必须携带所有必要的信息（如认证凭据），以便服务器正确处理。
2. **无状态的含义**
   - 不保留用户状态，无法区分是否为同一用户或会话。
   - 有利于服务器的**扩展性和简洁性**，便于实现分布式架构。
3. **状态保持的实现机制**
   - 尽管 HTTP 是无状态协议，但可以通过额外机制实现“伪状态”。
   - 常用方式包括：
     - **Cookie**：由服务器下发至客户端，客户端在后续请求中自动携带。
     - **Session**：服务器端保存用户状态信息，客户端仅携带 Session ID。
     - **Token（如 JWT）**：在客户端保存状态数据，服务器通过校验 Token 识别用户身份。
4. **总结**
   - HTTP 协议本身无状态，**状态保持依赖外部机制实现**。
   - Cookie、Session、Token 是实际 Web 应用中维持状态的重要手段。

### 携带Cookie的HTTP请求是有状态还是无状态的？Cookie是HTTP协议簇的一部分，那为什么还说HTTP是无状态的？

**HTTP 与 Cookie 的状态关系**

1. **HTTP 本质无状态**
   - 每个请求都是独立的，与前后请求无关。
   - 服务器不会主动保存客户端的上下文状态或会话信息。
   - 为了理解请求，客户端必须在每次请求中提供完整信息。
2. **Cookie 实现状态保持**
   - Cookie 是浏览器自动携带的一段小数据，用于存储会话标识等信息。
   - 服务器通过读取 Cookie 实现用户识别、状态跟踪、会话管理等功能。
   - 属于一种“**在无状态协议基础上的补充机制**”，并不能改变 HTTP 协议无状态的本质。
3. **Cookie 与 HTTP 协议的关系**
   - Cookie 是 HTTP 协议标准中的一部分，由响应头 `Set-Cookie` 设置，由请求头 `Cookie` 携带。
   - 虽属于协议规范，但其作用是**在应用层补充状态管理**能力，而不是由传输协议原生提供状态。
4. **为什么仍说 HTTP 是无状态的**
   - 无状态是指协议层不保留状态，**状态保持完全依赖额外机制（如 Cookie、Session、Token）**。
   - 即使 HTTP 请求携带 Cookie，本质上依然是独立请求，服务器需通过 Cookie 主动识别用户状态。
   - 这体现了状态管理的“外挂”性质，而非协议原生支持。
5. **总结**
   - HTTP 是无状态协议。
   - Cookie 是实现状态保持的一种机制，属于应用层的补充手段。
   - 使用 Cookie 后的 HTTP 请求可以“看似有状态”，但协议本身依然无状态。

### **Cookie、Session、Token 的区别**

1. **存储位置**
   - **Cookie**：数据存储在客户端（浏览器），服务器通过响应头 `Set-Cookie` 下发，浏览器自动携带。
   - **Session**：数据存储在服务器端，客户端只保存一个标识（Session ID），通过 Cookie 或 URL 参数传回服务端查询状态。
   - **Token**：数据存储在客户端，服务端通常不存储用户状态，通过解密或验证 Token 来识别用户。
2. **数据容量**
   - **Cookie**：单个大小限制约 4KB，数量也有限（每个域名通常不超过 20 个）。
   - **Session**：存储在服务端，理论上可存储任意数据，容量依赖服务端内存/存储。
   - **Token**：通常是一个加密后的字符串，体积小但包含所有身份信息。
3. **安全性**
   - **Cookie**：容易被劫持或伪造，存在 XSS 风险，但可通过设置 `HttpOnly`、`Secure`、`SameSite` 属性加强安全。
   - **Session**：相对安全，状态存在服务端，但仍需防范 Session 劫持与固定攻击。
   - **Token**：不依赖服务端存储，但存在泄露风险，一旦被截获即可伪造身份，需加密、设置过期时间。
4. **生命周期**
   - **Cookie**：可设置过期时间；也可设为临时 Cookie（随浏览器关闭即删除）。
   - **Session**：生命周期由服务器设置，用户关闭浏览器或超时后失效。
   - **Token**：一般有明确的过期时间；如需刷新，需要配合 Refresh Token。
5. **性能影响**
   - **Cookie**：每次请求都自动携带，尤其大数据量时可能降低性能。
   - **Session**：需要服务端频繁查找状态，可能增加服务器压力，适合小规模应用。
   - **Token**：请求无需查表验证，但体积可能大，且无法主动失效。
6. **扩展总结**
   - **Cookie**：状态存在客户端，适合保存小型数据如登录凭证、偏好设置等。
   - **Session**：状态存在服务端，适合保存结构化会话数据。
   - **Token**：无状态、跨平台、前后端分离场景常用，适合 RESTful 接口。

### **客户端禁用 Cookie 后 Session 的可用性问题**

1. **默认情况**

   - 大多数 Web 服务器通过 Cookie 传递 Session ID。
   - 若客户端禁用 Cookie，则服务器无法将 Session ID 返回给客户端，也无法从后续请求中识别该 ID，导致无法建立会话。

2. **替代方案**

   - 尽管禁用 Cookie 会导致 Session 失效，但可以通过以下方式绕过限制，实现会话保持：

   1. **URL 重写**
      - 在每次需要携带状态的请求中，将 Session ID 添加为 URL 参数。
      - 例如：`http://example.com/page → http://example.com/page;jsessionid=XXXXX`
      - 优点：浏览器不依赖 Cookie 也能传递 Session ID。
      - 缺点：
        - URL 不整洁，容易被用户复制粘贴传播，造成 Session ID 泄露风险。
        - 不适合敏感场景，存在 CSRF 攻击隐患。
   2. **隐藏表单字段**
      - 在 HTML 表单中嵌入一个隐藏字段用于保存 Session ID。
      - 表单提交时，浏览器将 Session ID 一并提交至服务端，服务器据此识别用户状态。
      - 缺点：
        - 仅适用于基于表单的 POST 请求。
        - 不适合链接跳转和 Ajax 场景。

3. **总结**

   - 禁用 Cookie 会限制 Session 使用。
   - URL 重写和隐藏字段可作为替代手段，但都存在安全性、用户体验和通用性上的不足。

### **Cookie 与 localStorage 的区别**

1. **存储容量**
   - Cookie：每个大小限制约为 4KB，总数也有限。
   - localStorage：每个域名可使用 5~10MB 左右，更适合存储大量数据。
2. **数据发送方式**
   - Cookie：每次请求时都会自动携带并发送给服务器。
   - localStorage：数据仅保存在浏览器本地，不自动随请求发送，适用于前端数据缓存。
3. **生命周期**
   - Cookie：可设置过期时间，到期后自动删除；若不设置，则为会话 Cookie，浏览器关闭即失效。
   - localStorage：持久存储，除非手动删除，否则永久保留。
4. **安全性**
   - Cookie：存在被窃取或篡改的风险，尤其在未设置 `HttpOnly`、`Secure` 时。
   - localStorage：仅限浏览器端访问，不与请求一同发送，安全性相对更高，但仍需防范 XSS 攻击。
5. **作用范围**
   - Cookie：可用于客户端和服务器之间的数据通信，适合身份识别、登录状态维持等。
   - localStorage：用于同源页面之间的数据共享，不支持跨域访问。

### **什么数据应存在 Cookie，什么数据应存在 localStorage**

- 使用 Cookie 存储：
  - 身份认证信息（如 token，但应配合 `HttpOnly` 增强安全性）
  - 跨页面、跨子系统需要传递给服务器的数据
  - 有过期需求或需自动发送的状态数据
- 使用 localStorage 存储：
  - UI 状态、前端缓存、用户偏好设置等本地数据
  - 不需要发送到服务器的数据
  - 大容量、持久保存的数据

### **JWT 与传统会话认证方式的区别**

1. **无状态性**
   - JWT 是无状态令牌，服务器不需要保存会话数据。
   - 所有用户身份信息（如 ID、角色、权限等）都已加密包含在 JWT 中，服务端只需解密验证即可使用。
   - 适合分布式架构与微服务系统，方便横向扩展与跨域访问。
2. **安全性**
   - JWT 通常使用密钥对令牌进行签名（如 HMAC、RSA），防止篡改。
   - 只有拥有密钥的服务器才能验证 JWT 的有效性，保障数据完整性与真实性。
   - 相较于传统 Session 和 Cookie 验证方式，JWT 更能抵御会话伪造、会话固定等攻击。
   - 注意：JWT 本身不能防止被盗用，需配合 HTTPS、安全存储、短有效期等手段。
3. **跨域支持**
   - JWT 不依赖 Cookie，可通过请求头（如 `Authorization: Bearer <token>`）或 URL 参数传输。
   - 支持不同子域、不同系统之间的身份验证，适用于单点登录（SSO）和跨源访问场景。
   - 避免了 Cookie 在跨域场景下的 SameSite 限制问题。
4. **与传统 Session 的对比总结**
   - Session 需在服务器端保存状态，适合单体应用，扩展性差。
   - JWT 不需服务器保存状态，扩展性好，但安全管理更复杂。

### **JWT 的结构组成**

JWT（JSON Web Token）由三部分组成：**头部（Header）**、**载荷（Payload）**、**签名（Signature）**。格式如下：

```
Header.Payload.Signature
```

1. **头部（Header）**

   - 声明令牌的类型（通常为 `JWT`）和所使用的签名算法（如 `HS256`、`RS256`）。

   - 示例：

     ```
     CopyEdit
     {
       "alg": "HS256",
       "typ": "JWT"
     }
     ```

2. **载荷（Payload）**

   - 存放实际传递的声明（Claims），包含身份信息和其他元数据。

   - 分为三类：

     - **注册声明（Registered Claims）**：如 `iss`（签发者）、`exp`（过期时间）、`sub`（用户 ID）、`iat`（签发时间）等。
     - **公共声明（Public Claims）**：由双方约定的字段。
     - **私有声明（Private Claims）**：自定义字段，如用户角色、权限等。

   - 示例：

     ```
     {
       "sub": "1234567890",
       "name": "George White",
       "admin": true,
       "iat": 1516239022
     }
     ```

3. **签名（Signature）**

   - 对 `Header` 和 `Payload` 进行 Base64 编码后，用密钥和指定算法生成签名，防止数据被篡改。

   - 示例计算方式：

     ```
     HMACSHA256(
       base64UrlEncode(header) + "." + base64UrlEncode(payload),
       secret
     )
     ```

**补充说明**

- JWT 是使用 Base64URL 编码，不是加密，**可以被解码但不能被伪造**（伪造会因签名不匹配被拒）。
- JWT 是无状态的，适合前后端分离、微服务、单点登录（SSO）等场景。

### JWT 令牌为什么能解决集群部署，什么是集群部署？

1. **什么是集群部署**
   - 指多个服务器共同对外提供服务，负载均衡器将用户请求分发给任意一台服务器处理。
   - 在传统的 Session 模型中，用户状态通常保存在服务器内存或数据库中。
   - 如果用户请求被分发到另一台服务器，将因无法获取原 Session 而面临状态丢失、需重新登录或引入共享缓存（如 Redis）的复杂方案。
2. **传统 Session 的问题**
   - 会话信息依赖服务器端存储，造成状态无法跨服务器共享。
   - 为解决此问题通常要引入 Redis 等中间件做 Session 共享，增加系统复杂度和运维成本。
3. **JWT 的优势**
   - JWT 是自包含（self-contained）的令牌，用户登录认证后生成的令牌中已包含身份信息、权限信息及会话状态。
   - 客户端在每次请求时携带 JWT，服务器可通过本地验证解析出用户身份。
   - 无需服务器之间共享状态，无需会话同步或引入额外存储。
4. **如何解决集群问题**
   - 每台服务器都能独立验证 JWT，处理请求无需依赖外部组件。
   - 提升系统的**伸缩性**与**容错性**，适用于微服务、负载均衡等场景。
5. **总结**
   - 传统 Session 模型：服务器需维护状态，不利于横向扩展。
   - JWT 模型：令牌中自带状态信息，天然支持分布式架构，简化集群部署。

### **JWT 的缺点与解决方案**

1. **不可主动失效**
   - JWT 是无状态令牌，服务端不保存其状态。
   - 一旦签发，在过期之前始终有效，**无法提前撤销**。
   - 即使用户登出、被禁用、权限更改，JWT 依然可被继续使用，存在安全隐患。
2. **常见问题场景**
   - 用户主动退出登录，但 JWT 未过期。
   - 管理员想强制某个用户立即失效其令牌。
   - 改变权限或封号操作后，旧 JWT 仍可能继续被使用。
3. **黑名单机制**
   - 可在业务层引入“**JWT 黑名单机制**”进行弥补。
   - 实现方式：
     - 将需要立即失效的 JWT 标记为无效，写入 Redis 等内存数据库维护的黑名单中。
     - 每次收到 JWT 请求时，先检查是否存在于黑名单中，若存在则拒绝访问。
   - 缺点：
     - 增加一部分服务端存储和逻辑判断，牺牲一部分无状态性。
4. **可选方案**
   - 设置较短的 JWT 有效期（如 15 分钟）+ Refresh Token 机制进行续签。
   - 结合黑名单策略处理高风险操作后的强制失效场景。
5. **总结**
   - JWT 的缺点在于“无法主动废弃”。
   - 黑名单机制可作为补偿措施。
   - 安全设计上应结合短有效期 + 黑名单 + Refresh Token 综合管理。

### **JWT 泄露后的应对方式**

1. **立即标记失效**
   - 一旦检测到 JWT 泄露或存在安全风险，服务端应立即将该令牌标记为失效。
   - 拒绝任何携带该令牌的请求，阻止攻击者继续使用。
   - 实现方式包括修改用户权限或内部标记机制。
2. **刷新令牌机制**
   - JWT 一般设有较短有效期，需要定期刷新。
   - 可通过 **Refresh Token** 获取新的 JWT，同时使旧令牌作废。
   - 即便旧令牌泄露，由于有效期短，也能快速失效，降低风险。
3. **黑名单机制**
   - 服务端维护一个黑名单（如 Redis 中的 Set），存储被吊销或泄露的 JWT。
   - 每次收到请求时，先校验令牌是否在黑名单中，若存在则拒绝访问。
   - 有效防止被盗令牌继续被使用，但会牺牲部分无状态优势，带来性能开销。

### **前端存储 JWT 的常用方式**

1. **Local Storage（本地存储）**
   - **优点**
     - 存储空间较大（通常为 5MB）。
     - 不会自动随每次 HTTP 请求发送，性能开销小。
   - **缺点**
     - 容易受到 XSS（跨站脚本攻击）威胁。
     - 所有脚本都可以访问，安全性较低。
2. **Session Storage（会话存储）**
   - **优点**
     - 数据仅在当前浏览器窗口/标签页中有效，关闭即清除，有助于减少泄露风险。
     - 不会自动发送到服务器，避免不必要的流量。
   - **缺点**
     - 不支持跨标签页共享。
     - 用户刷新页面或重新打开可能导致重新认证。
3. **Cookie**
   - **优点**
     - 可设置 `HttpOnly`，防止 JavaScript 访问，增强对 XSS 攻击的防护。
     - 可设置 `Secure` 标志，仅在 HTTPS 下传输。
   - **缺点**
     - 存储大小有限（约 4KB）。
     - 每次请求都会自动携带，可能影响性能。
     - 若未正确设置 SameSite 和 CSRF Token，可能会受到 CSRF 攻击。

**总结建议**

- **安全性优先**：建议使用 Cookie + `HttpOnly` + `Secure` 配置，防止脚本劫持。
- **灵活性优先**：需要手动控制发送、存储和清除 JWT 的场景，可考虑 Local Storage。
- **一次性会话**：对安全要求较高且仅需单窗口登录场景，可考虑 Session Storage。

### **为什么有 HTTP 协议了，还需要 RPC？**

1. **本质区别**
   - HTTP 是**通信协议**，属于传输层之上的标准。
   - RPC（Remote Procedure Call）是一种**调用方式**，强调让开发者像调用本地函数一样调用远程服务。
   - gRPC、Thrift 等是对 RPC 的具体协议实现，底层可基于 TCP、HTTP/2 等协议。
2. **适用场景差异**
   - HTTP 常用于 **B/S 架构**（浏览器/服务器），更适合面向用户的 Web 应用场景。
   - RPC 多用于 **C/S 架构**（客户端/服务器），适合服务间通信，特别是微服务之间。
3. **发展趋势**
   - 当前 C/S 和 B/S 架构已经趋于融合，很多服务既支持网页也支持移动端。
   - 外部接口多用 HTTP（RESTful API），而内部服务通信往往使用 RPC，提高性能和可维护性。
4. **性能与效率**
   - RPC 协议相较于 HTTP/1.1，序列化效率更高、通信开销更小、性能更优。
   - gRPC 使用 HTTP/2 + ProtoBuf，相较 HTTP/1.1 更适合高并发、低延迟的服务调用。
5. **HTTP/2 的影响**
   - HTTP/2 的多路复用、头部压缩等特性弥补了部分性能差距。
   - 但由于发展时间尚短，生态尚未完全成熟，短期内难以完全取代 RPC。

**总结**

- HTTP 是通信协议，RPC 是调用方式，**二者不是替代关系**。
- 外部接口偏向使用 HTTP，内部服务偏向使用高效的 RPC。
- gRPC 等现代 RPC 框架与 HTTP/2 趋于融合，但仍各有优势，需结合场景选用。

### **HTTP 长连接 vs WebSocket 区别**

1. **通信模式**
   - **HTTP/1.1 长连接**：
     - 使用 TCP 连接，但通信是**半双工**的（同一时间只能单向传输）。
     - 客户端请求、服务器响应，服务器无法主动推送。
   - **WebSocket**：
     - 也是基于 TCP，但支持**全双工通信**，客户端和服务端可以互相主动发送消息。
     - 更适合高频、实时的交互场景。
2. **连接方式**
   - **HTTP 长连接**：
     - 多次请求共用一个 TCP 连接，提升资源利用率（例如 keep-alive）。
     - 每次仍需遵循完整的 HTTP 请求-响应结构。
   - **WebSocket**：
     - 通过 HTTP 协议进行一次握手后，升级为 WebSocket 协议，后续通信无需 HTTP 协议格式。
3. **应用场景**
   - **HTTP 长连接适用**：
     - 登录页面、表单提交等请求少、交互简单的场景。
     - 可通过**轮询**或**长轮询（Comet）**模拟服务器推送。
   - **WebSocket 适用**：
     - 即时通讯、网页游戏、在线协作等需要频繁数据交互、低延迟推送的场景。
     - 例如聊天系统、股票行情、在线客服等。
4. **推送能力**
   - **HTTP 长连接**：服务器不能主动发消息，只能等待请求。
   - **WebSocket**：服务器可主动向客户端推送消息，降低响应延迟。

------

**总结**

| 对比项   | HTTP 长连接       | WebSocket           |
| -------- | ----------------- | ------------------- |
| 通信方式 | 半双工            | 全双工              |
| 建立连接 | HTTP 协议，不升级 | HTTP 握手后升级协议 |
| 推送能力 | 需轮询/长轮询实现 | 原生支持服务器推送  |
| 应用场景 | 请求-响应场景     | 实时双向通信场景    |

### **Nginx 支持的负载均衡算法**

1. **轮询（Round Robin）**
   - 将请求按顺序依次分配到后端服务器。
   - 实现简单，但无法处理节点性能差异或某节点卡顿的问题。
2. **IP 哈希（IP Hash）**
   - 根据客户端 IP 计算哈希值，分配请求到固定的服务器。
   - 适用于需要会话保持（同一用户始终访问同一台服务器）的场景。
3. **URL 哈希（URL Hash）**
   - 根据请求 URL 的哈希结果分配服务器。
   - 可用于静态资源缓存一致性需求，如某个资源始终由同一台服务器处理。
4. **最短响应时间（Least Time）**
   - 根据后端服务器响应时间动态分配，请求优先发给响应时间短的服务器。
   - 适用于后端性能不均、请求延迟敏感的场景。
5. **加权轮询（Weighted Round Robin）**
   - 在轮询基础上为不同服务器设置权重，性能好的服务器权重更高，获得更多请求。
   - 适用于后端服务器性能差异明显的场景，提升资源利用率。

### **Nginx 属于七层网络结构中的哪一层？**

- Nginx 工作于 **OSI 模型的第七层（应用层）**，是一种**七层负载均衡器**。
- 可以基于请求内容（如 URL、Header、参数）做转发或路由处理。

## 传输层

### **TCP 报文段头部字段解析**

TCP 报文头部的**最小长度为 20 字节**，若含有可选字段则更长。各字段含义如下：

1. **源端口号（Source Port）** - 16 位
   - 表示发送方的端口，用于标识本地进程。
2. **目的端口号（Destination Port）** - 16 位
   - 表示接收方的端口，用于标识远端进程。
3. **序列号（Sequence Number）** - 32 位
   - 表示该报文段**数据的起始字节序号**，用于字节流的有序传输和确认。
4. **确认号（Acknowledgment Number）** - 32 位
   - 表示期望接收对方下一个字节的序号，即**已成功接收的数据最后一个字节 + 1**。
   - 仅当 ACK 标志位为 1 时该字段有效。
5. **数据偏移（Data Offset）/首部长度** - 4 位
   - 表示 TCP 头部的长度，以 4 字节为单位。
   - 用于指出数据部分从哪个字节开始（最少为 5，即 20 字节）。
6. **保留字段（Reserved）** - 3 位
   - 保留为今后使用，通常填 0。
7. **控制位（Flags）/标志位** - 9 位
   - 关键标志如下（常见组合决定连接行为）：
     - **URG**：紧急指针有效
     - **ACK**：确认号有效
     - **PSH**：提示接收方应尽快交付数据
     - **RST**：重置连接
     - **SYN**：发起连接
     - **FIN**：断开连接
8. **窗口大小（Window Size）** - 16 位
   - 表示当前接收方可接收的字节数（用于流量控制）。
9. **校验和（Checksum）** - 16 位
   - 覆盖头部和数据，用于**差错检测**，由发送方计算、接收方验证。
10. **紧急指针（Urgent Pointer）** - 16 位

- 仅在 URG 为 1 时有效，表示紧急数据的结束位置。

1. **选项字段（Options，可变长度）**

- 可包含如最大报文段长度 MSS、窗口扩大因子、时间戳等。
- 该字段不固定，常见于连接建立时协商。

1. **填充字段（Padding）**

- 为了保证 TCP 头部长度是 4 字节的整数倍而填充 0。

### **TCP 三次握手过程**

TCP 三次握手用于**建立可靠连接**，确保双方都具备收发能力。过程如下：

1. **第一次握手（SYN）**
   - **客户端 → 服务端**
   - 客户端发送一个 **SYN 报文段**，随机生成一个初始序列号 `seq = x`。
   - 表示客户端请求建立连接，进入 **SYN_SENT** 状态。
2. **第二次握手（SYN + ACK）**
   - **服务端 → 客户端**
   - 服务端收到客户端 SYN 后，发送一个 **SYN + ACK 报文段**，表示同意连接。
   - 包含两个内容：
     - SYN，服务端也发送自己的初始序列号 `seq = y`
     - ACK，确认号 `ack = x + 1`
   - 服务端进入 **SYN_RECEIVED** 状态。
3. **第三次握手（ACK）**
   - **客户端 → 服务端**
   - 客户端收到服务端 SYN+ACK 后，发送 **ACK 报文段**。
   - 确认号为 `ack = y + 1`，自己的序列号为 `seq = x + 1`
   - 发送完后，客户端进入 **ESTABLISHED（连接已建立）** 状态。
   - 服务端收到这个 ACK 后，也进入 **ESTABLISHED** 状态，连接正式建立。

### **为什么 TCP 需要三次握手建立连接**

1. **防止历史连接导致混乱（主要原因）**
   - 三次握手的核心目的是：**防止旧的、失效的连接请求（历史 SYN 报文）被错误接收，导致建立无效连接。**
   - 场景模拟：
     - 客户端第一次发送 `SYN(seq=90)`，但报文丢失；
     - 客户端重试后发送新 `SYN(seq=100)`；
     - 若旧报文后续又突然到达服务器，服务器若未验证客户端是否还有效，可能错误建立连接。
   - 三次握手中，**第三次由客户端确认 ACK 是关键**，它能防止服务端与无效客户端建立连接。
   - RFC793 明确指出：三次握手的目的是“避免历史连接引发混淆”。
2. **同步双方的初始序列号**
   - TCP 是面向字节流的协议，**每个方向都需要各自的初始序列号（ISN）**，以实现可靠传输。
   - 只有通过三次交互，才能确保：
     - 双方均已收到对方的 ISN；
     - 数据收发能正确标识、去重、确认；
     - 避免只同步一端导致确认错乱。
3. **避免资源浪费**
   - 如果只有两次握手，服务器在收到 SYN 后就进入 ESTABLISHED 状态，会分配缓存、资源等；
   - 若客户端未收到服务端的 ACK，就会一直重发 SYN；
   - 此时服务端每次都分配资源，**可能创建多个无效连接，浪费资源甚至被利用进行攻击（如 SYN Flood）**。

### **TCP 三次握手中第三次 ACK 丢失会发生什么？**

1. **连接状态变化**
   - 客户端在发送第三次 ACK 报文（确认服务端的 SYN）后，**立即进入 ESTABLISHED 状态**，认为连接已建立成功。
   - 此时可以开始发送数据（虽然服务端还未确认）。
2. **服务端的行为**
   - 服务端在发送完第二次握手的 `SYN+ACK` 后，**等待客户端的 ACK 报文（第三次握手）**。
   - 如果 ACK 丢失，服务端会以为客户端没有收到 `SYN+ACK`，于是**启动重传机制**。
3. **ACK 丢失处理机制**
   - ACK 报文本身**不会被重传**，**但 ACK 丢了时，触发的是服务端重传前一个报文（SYN+ACK）**。
   - 服务端会多次重发 `SYN+ACK`，等待客户端重新回应 ACK。
4. **最大重传次数（以 Linux 默认为例）**
   - 参数 `tcp_synack_retries` 默认为 **2**。
   - 表示服务端最多重传两次 `SYN+ACK` 报文。
   - 超过最大重传次数后，若仍未收到 ACK，服务端会**断开连接（放弃建立）**。

**总结流程**

- 客户端发送第三次 ACK → 丢失
- 服务端未收到 ACK → 重传 SYN+ACK
- 客户端已处于 ESTABLISHED 状态，会忽略重传 SYN+ACK，或再次发送 ACK
- 若服务端仍未收到 ACK → 达到最大重传次数后断开连接

**核心结论**

- 第三次 ACK 丢失不会影响客户端状态（已建立），
- 服务端通过超时重传机制尝试补救，最终仍未收到则主动关闭连接。

### **三次握手与 accept 的关系以及 accept 做了哪些事情**

1. **三次握手与 accept 的关系**
   - TCP 三次握手用于在内核中建立连接，但**不会直接交给应用程序使用**。
   - 应用层需要调用 `accept()` 才能**将已完成三次握手的连接“领取”出来**，用于后续数据收发。
   - 因此，**三次握手是建立连接的前提，accept 是应用程序真正获取连接的入口**。
2. **连接队列的划分**
   - listen 函数调用后，内核会为该监听 socket 创建两个连接队列：
     - 半连接队列（SYN 队列）：保存收到 SYN 后、等待三次握手完成的连接（状态为 SYN_RECV）。
     - 全连接队列（accept 队列）：保存三次握手已完成、等待应用调用 accept 的连接（状态为 ESTABLISHED）。
   - 三次握手完成后，连接才会进入全连接队列，等待应用 accept。
3. **accept 做的事情**
   - 从内核的全连接队列中**取出一个已完成连接**。
   - 为该连接**创建新的 socket 描述符**。
   - 将该 socket 返回给用户进程，用于后续收发数据。
   - 一个 accept 对应一个客户端连接，典型的用于服务端处理新连接。
4. **整体流程总结**
   - 客户端调用 connect() 发起连接，进入 SYN_SENT 状态。
   - 服务端收到 SYN，放入半连接队列，进入 SYN_RECV 状态。
   - 三次握手完成，连接进入 ESTABLISHED 状态，并加入全连接队列。
   - 服务端调用 accept()，从全连接队列中取出连接，创建 socket 句柄给应用程序使用。

### **客户端发送的第一个 SYN 报文服务器没收到会怎样？**

1. **触发重传机制**

   - 当客户端发送第一个 SYN 报文后进入 SYN_SENT 状态，等待服务端的 SYN-ACK 报文。
   - 如果在超时时间内未收到回应，客户端会触发超时重传机制，重发 SYN 报文。
   - 注意：每次重传的 SYN 报文其序列号是一样的。

2. **重传时间间隔策略**

   - 每次超时重传时间为上一次的 2 倍，属于指数回退策略。
   - 默认的重传时间间隔：
     - 第 1 次重传：1 秒后
     - 第 2 次重传：2 秒后
     - 第 3 次重传：4 秒后
     - 第 4 次重传：8 秒后
     - 第 5 次重传：16 秒后
     - 最后等待：32 秒后主动放弃连接

3. **重传次数由内核参数控制**

   - Linux 系统中由 `tcp_syn_retries` 参数控制最大重传次数，默认值为 5。

   - 可通过以下命令查看：

     ```
     cat /proc/sys/net/ipv4/tcp_syn_retries
     ```

4. **连接放弃时间**

   - 如果 5 次重传仍无回应，则客户端放弃连接请求，TCP 建立失败。
   - 总超时时间约为 1+2+4+8+16+32 = 63 秒，大约一分钟。

### **服务器收到第一个 SYN 报文，回复的 SYN+ACK 报文丢失会怎么样？**

1. **第二次握手的作用**
   - 第二次握手的 SYN+ACK 报文包含两个作用：
     - 是对客户端第一个 SYN 的确认（ACK），用于响应第一次握手；
     - 是服务端发起连接的 SYN，用于自身建立连接。
2. **客户端未收到 SYN+ACK 的处理**
   - 如果客户端在超时时间内未收到第二次握手（SYN+ACK）报文，会认为自己的 SYN 报文丢失。
   - 客户端会触发超时重传机制，**重发 SYN 报文**，也就是**重新发起第一次握手**。
   - 重传次数由 `tcp_syn_retries` 参数控制，默认值为 5。
3. **服务器未收到第三次握手的处理**
   - 如果 SYN+ACK 报文（第二次握手）丢失，服务器无法收到客户端的 ACK（第三次握手）。
   - 服务器会触发超时重传机制，**重发 SYN+ACK 报文**。
   - 重传次数由 `tcp_synack_retries` 参数控制，默认值为 5。
4. **双方都可能重传**
   - 客户端在超时后会重发 SYN（第一次握手），由 `tcp_syn_retries` 控制。
   - 服务器在超时后会重发 SYN+ACK（第二次握手），由 `tcp_synack_retries` 控制。
   - 所以**第二次握手丢失后，客户端和服务端都在进行重试，直到连接成功或重试超限放弃**。

### **假设客户端重传了 SYN 报文，服务端又收到重复的 SYN 报文怎么办？**

1. **识别 SYN 报文重复**
   - TCP 是面向连接的可靠协议，**每个连接由四元组（源 IP、源端口、目标 IP、目标端口）唯一标识**。
   - 如果服务端在等待第三次握手（即处于 SYN_RECV 状态）时又收到相同四元组的 SYN 报文，说明客户端在重传。
2. **服务端处理方式**
   - 服务端会将这个重传的 SYN 报文识别为“连接重试”。
   - 不会重复创建连接结构，而是**继续重发 SYN+ACK 报文（第二次握手）**。
   - 此机制确保即使第二次握手的响应报文丢失，客户端仍能接收到新的响应。
3. **实现连接可靠性的关键机制**
   - 客户端通过超时重传 SYN，保证服务端能收到连接请求。
   - 服务端通过重复发送 SYN+ACK，保证客户端最终能完成三次握手。
   - 这是一种冗余但有效的方式，避免在三次握手过程中因网络丢包导致连接失败。

### **如何应对 SYN Flood 攻击？**

1. **调整 netdev_max_backlog**

   - 控制网卡接收队列长度，当内核来不及处理数据包时使用。

   - 默认值为 1000，可增大以应对流量高峰，例如：

     ```
     net.core.netdev_max_backlog = 10000
     ```

2. **扩大 TCP 半连接队列**

   - 半连接队列大小受多个参数控制，需同时调整以下三项：
     - `net.ipv4.tcp_max_syn_backlog`
     - `listen()` 函数的 backlog 参数
     - `net.core.somaxconn`

3. **开启 tcp_syncookies**

   - 启用 Syncookie 机制，当半连接队列满时仍能正常建立连接。

   - 启用方式：

     ```
     echo 1 > /proc/sys/net/ipv4/tcp_syncookies
     ```

   - 参数含义：

     - 0：关闭
     - 1：半连接队列满才启用
     - 2：始终启用（不推荐）

4. **减少 SYN+ACK 重传次数**

   - 减少处于 `SYN_RECV` 状态的 TCP 连接的持续时间，加快资源释放。

   - 通过内核参数 `tcp_synack_retries` 设置，默认值为 5，可设为 2：

     ```
     echo 2 > /proc/sys/net/ipv4/tcp_synack_retries
     ```

### **TCP 四次挥手的详细过程**

1. **第一次挥手（客户端发起）**
   - 客户端主动调用关闭连接的函数，发送 **FIN 报文**，表示自己不再发送数据。
   - 此时客户端进入 **FIN_WAIT_1** 状态。
2. **第二次挥手（服务端确认）**
   - 服务端收到 FIN 报文后，立刻回复一个 **ACK 确认报文**。
   - 服务端进入 **CLOSE_WAIT** 状态。
   - 同时，TCP 协议栈会将 FIN 插入到接收缓冲区，应用程序通过 `read()` 读取数据后，**会在已接收数据之后读到 EOF**。
3. **第三次挥手（服务端关闭）**
   - 当服务端应用程序读到 EOF 后，若还有数据要发送，则继续写数据。
   - 写完数据后，调用关闭连接函数，发送 **FIN 报文**。
   - 若没有数据要发送，也可以直接调用关闭连接函数，发送 FIN 报文。
   - 此时服务端进入 **LAST_ACK** 状态。
4. **第四次挥手（客户端确认）**
   - 客户端收到服务端的 FIN 报文后，回复 **ACK 报文**。
   - 客户端进入 **TIME_WAIT** 状态，等待 2MSL（最大报文寿命）后彻底关闭，进入 **CLOSED** 状态。
   - 服务端收到 ACK 后，进入 **CLOSED** 状态。

### **为什么 TCP 四次挥手中间的两次不能变成一次？**

1. **服务端不能立即发送 FIN 报文**
   - 当服务端收到客户端的 FIN 报文后，虽然内核会立即返回 **ACK 确认报文**；
   - **但服务端应用程序可能还有数据要发送**，因此：
     - 不能立刻发出 FIN 报文；
     - 需要把 **发送 FIN 报文的控制权交给服务端应用程序**。
2. **控制权由服务端应用层决定**
   - 若服务端还有数据要发送：
     - 应用程序会先完成发送，再调用关闭连接函数，发出 FIN 报文；
   - 若没有数据要发送：
     - 应用程序可以直接调用关闭函数，发出 FIN 报文。
3. **为什么不能合并为一次？**
   - TCP 三次握手的控制权在内核，而 **四次挥手中第二次和第三次之间的控制权在服务端应用程序**；
   - 由于是否立即关闭连接取决于应用程序的逻辑，不能由内核直接决定；
   - 因此 ACK 和 FIN 报文无法合并，必须分开发送，导致形成四次挥手。

### **为什么第二次和第三次挥手能合并？**

当被动关闭方在 TCP 挥手过程中，**没有数据要发送** 且 **启用了 TCP 延迟确认机制**，那么第二次和第三次挥手就会**合并传输**，这样就出现了“三次挥手”。

**合并的条件：**

- 被动关闭方没有数据要发送
- 被动关闭方启用了 TCP 延迟确认机制

**具体流程：**

- 第一次挥手：主动关闭方发送 `FIN` 报文
- 第二次挥手：被动关闭方收到 `FIN` 后，原本应先回复 `ACK`，再发送 `FIN`
- 如果启用了延迟确认机制，且没有数据要发送，`ACK` 和 `FIN` 会合并发送，成为 `FIN, ACK` 报文
- 第三次挥手：主动方接收到 `FIN, ACK` 后，发送 `ACK`，连接关闭

**合并后的效果：**

- 原本需要 4 个报文的四次挥手，变成了 3 个报文
  1. `FIN`
  2. `FIN, ACK`
  3. `ACK`

**总结：**

第二次和第三次挥手之所以能合并，是因为确认和关闭的控制权都在被动关闭方一侧，
 如果没有数据要发送，可以一次性发出 `FIN, ACK`，提升效率。

### **第三次挥手一直没发，会发生什么？**

- 主动关闭方收到对方 ACK 报文后，进入 **FIN_WAIT2** 状态，表示发送通道已经关闭，等待对方发送 FIN 报文以完成连接关闭。
- 如果连接是通过 `shutdown` 函数关闭的，那么连接可以**长期停留在 FIN_WAIT2** 状态，因为仍可能接收/发送数据。
- 如果连接是通过 `close` 函数关闭的，此时不能再收发数据，因此不能长时间处于此状态。
- 为防止资源占用，Linux 提供内核参数 **`tcp_fin_timeout`** 来控制 **FIN_WAIT2** 状态的持续时长，默认值为 **60 秒**。

**示例命令：**

```
echo 60 > /proc/sys/net/ipv4/tcp_fin_timeout
```

**总结：**

- 如果 60 秒内没有收到对方的 FIN 报文，连接就会被**强制关闭**，避免资源长时间被占用。
- 该机制适用于“孤儿连接”（即通过 `close` 关闭的一端），防止对方第三次挥手丢失导致连接卡住。

### **第二次和第三次挥手之间，主动断开的那端能干什么？**

- 如果主动断开的一方调用的是 `shutdown` 函数来关闭连接，**只关闭发送功能**，而**未关闭接收能力**：
  - 那么在第二次和第三次挥手之间，主动断开的一方 **仍然可以接收数据**。

### 断开连接时客户端 FIN 包丢失，服务端的状态是什么？

- **客户端状态变化：**
   客户端调用 `close()` 函数后，发送 FIN 报文，进入 `FIN_WAIT_1` 状态。
- **第一次挥手丢失：**
   如果服务端未收到该 FIN 报文，客户端将根据重传机制，最多重发 `tcp_orphan_retries` 次（默认是 8 次），每次重传间隔是上一次的 2 倍时间。
- **重传失败后的处理：**
   当重传超过最大次数仍未收到 ACK，客户端会认为连接异常中断，直接进入 `CLOSE` 状态。
- **服务端状态：**
   服务端因为没有收到 FIN 报文，仍然保持在 `ESTABLISHED` 状态。
- **结果：**
   双方连接状态不一致，客户端已关闭，服务端仍保持连接，造成资源无法及时释放。

### 为什么四次挥手之后要等2MSL?

1. **什么是 MSL（Maximum Segment Lifetime）**

   MSL 是报文段在网络上的最⼤存活时间，超过这个时间后，报文将被丢弃。
    它是一个时间值，而非 TTL 中的“跳数”，常用于 TCP 协议中的连接终止处理。

2. **MSL 与 TTL 的区别**

   - TTL（Time To Live）是 IP 层字段，单位是跳数，每经过一个路由器 TTL 减一，为 0 就丢弃。
   - MSL 是时间单位，表示报文在网络中最多能存活多久。
   - 为了确保一个报文彻底消失，MSL 应该大于或等于 TTL 消耗为 0 所需的时间。

3. **Linux 中 MSL 的典型设置**

   - TTL 的默认值通常为 64。
   - Linux 将 MSL 设置为 30 秒，认为经过 64 个路由器所用时间不应超过 30 秒。
   - 因此，若 30 秒后未收到应答，则认为报文已从网络中消失。

4. **为什么 TIME_WAIT 状态要等 2 倍 MSL**

   - 在四次挥手中，主动关闭连接的一方进入 TIME_WAIT 状态。
   - 网络中可能仍有延迟到达的报文（如最后一个 ACK 或 FIN），为了避免这些“迟到包”干扰后续连接，需等待足够时间。
   - 若不等待，可能出现旧连接的残留报文影响新连接（尤其是在使用相同四元组的连接中）。
   - “来一回一”，即一个报文往返一次最多需要 2MSL，才能确保对方收到了 ACK，且网络中所有报文都消失。

5. **2MSL 的另一个重要作用：保证连接可靠关闭**

   - 如果最后一个 ACK 丢失，对方会重发 FIN。
   - 等待 2MSL 可以确保自己还能接收到对方重发的 FIN，并重新发送 ACK。
   - 若不等待，可能对方未收到 ACK 而认为连接未关闭，造成状态异常。

6. **2MSL 等待的总结**

   - 等待 2MSL 是为了防止已关闭连接的残留报文干扰新连接。
   - 也为了保证可靠地结束连接，处理 ACK 或 FIN 的重传问题。
   - 时间足够长可以**至少容忍一次报文丢失**并确保重传成功。

### 服务端出现大量的timewait有哪些原因?

1. **HTTP 没有使用长连接**
   - 在 HTTP/1.0 中默认不启用长连接，若要启用需要在请求头中添加 `Connection: Keep-Alive`，响应头也需加该字段。
   - 在 HTTP/1.1 中默认启用 Keep-Alive，除非请求或响应中显式添加 `Connection: close`。
   - 一旦没有使用长连接，TCP 连接在一次请求/响应后就立即断开。
   - 实际情况中，为了释放资源，往往是由服务器主动断开连接，因此 TCP 四次挥手由服务器最后发送 ACK，进入 TIME_WAIT 状态。
2. **HTTP 长连接超时**
   - HTTP 长连接允许一个 TCP 连接中复用多个请求和响应，减少连接建立和释放的成本。
   - 若客户端在一个请求后不再发送请求，服务器无法立即关闭连接，但仍需保持连接状态，直到超时。
   - 为了避免资源长期被占用，服务端如 nginx 等通常配置了 `keepalive_timeout` 参数，设定 HTTP 长连接最大空闲时间。
   - 一旦超时未收到新的请求，服务器会主动断开连接，进入 TIME_WAIT 状态。
   - 如果高并发下很多连接都在等待超时后断开，服务器上就可能出现大量 TIME_WAIT 状态。
3. **HTTP 长连接的请求数达到上限**
   - Web 服务器为避免某些连接长期占用资源，通常设置参数限制一个长连接最多可处理的请求数量。
   - 以 nginx 为例，`keepalive_requests` 参数指定一个连接最多处理的请求数，默认值为 100。
   - 达到该上限后，nginx 会主动断开该连接，从而产生 TIME_WAIT。
   - 在高 QPS 场景下（如超过 10,000 QPS），若该参数仍为默认值，nginx 会频繁关闭连接，TIME_WAIT 数量激增。
4. **TIME_WAIT 出现在服务器的原因**
   - TCP 中主动关闭连接的一方会进入 TIME_WAIT 状态。
   - HTTP 协议中，客户端或服务器都可主动关闭 TCP 连接。
   - 在实际 Web 服务实现中，通常是服务器主动关闭连接（不管是谁禁用了 Keep-Alive），因此 TIME_WAIT 出现在服务器端。
   - 如果客户端能主动关闭连接，TIME_WAIT 状态将出现在客户端而非服务器端。
5. **如何缓解服务器上的 TIME_WAIT 问题**
   - **启用并优化 HTTP 长连接**：避免每次请求都重建连接，减少 TIME_WAIT 产生频率。
   - **合理配置超时时间与最大请求数**：
     - 调大 `keepalive_timeout`（适中）避免频繁超时断开；
     - 调大 `keepalive_requests` 避免频繁因达到请求上限而断开。
   - **使用端口复用机制**：如 Linux 的 `SO_REUSEPORT`、`SO_REUSEADDR`，或开启 netfilter 中的 `tcp_tw_reuse` 和 `tcp_tw_recycle`（但这两个已被弃用，慎用）。
   - **用连接池复用连接**，避免连接频繁建立与释放。

### TCP和UDP区别是什么？

1. **是否面向连接**
   - TCP 是**面向连接**的协议，在传输数据前必须建立连接（三次握手）。
   - UDP 是**无连接**协议，发送数据前不需要建立连接，传输开销更小。
2. **通信方式**
   - TCP 是一对一的**点对点通信**。
   - UDP 支持一对一、一对多、多对多等多种交互模式，适用于广播和多播场景。
3. **可靠性**
   - TCP 提供可靠的数据传输，具备确认应答机制，确保数据**不丢失、不重复、按序到达**。
   - UDP 是**尽最大努力交付**，不保证数据可靠传输、顺序、完整性，但性能较高。
   - 若需可靠性，可在 UDP 上实现应用层协议，如 **QUIC**、**RTP** 等。
4. **拥塞控制与流量控制**
   - TCP 有拥塞控制和流量控制机制，如慢启动、拥塞避免、快速重传、滑动窗口等，保证网络稳定和数据传输安全。
   - UDP **没有**拥塞控制与流量控制，即使网络拥塞或丢包，也照常发送数据，适合对实时性要求高的场景，如视频会议、直播、游戏等。
5. **首部开销**
   - TCP 首部长度最少为 **20 字节**，若使用选项字段则会更长。
   - UDP 首部固定为 **8 字节**，结构简单，开销小，处理效率高。
6. **传输方式**
   - TCP 采用**字节流**传输，没有明显边界，数据被看作连续的流，适合大文件传输。
   - UDP 是**报文（数据报）**传输，有边界，一个数据报就是一个消息，适合一次性传输的短消息。
7. **顺序与乱序处理**
   - TCP 通过序列号和确认应答，**保证有序传输**。
   - UDP 不保证顺序，可能出现**乱序、丢包**，需由上层协议或应用自己处理。

### TCP为什么可靠传输

1. **连接管理**
   - TCP 通过三次握手建立连接，四次挥手断开连接，确保双方具备发送与接收能力。
   - 建立可靠连接是保障后续可靠数据传输的基础。
2. **序列号机制**
   - TCP 会给每个字节分配一个**序列号（Sequence Number）**，用于数据包编号。
   - 作用：
     - 保证数据的**有序传输与重组**；
     - 防止数据**重复接收或丢失**；
     - 支持**累计确认**（即多个包可一次性确认）；
     - 实现数据分段、按序还原。
3. **确认应答（ACK）机制**
   - TCP 是**全双工可靠传输**，每接收一次数据，接收方都会发送一个 **ACK 确认报文段**，带有确认号。
   - 确认号表示“期望接收的下一个字节序号”，表示前面的数据已收到。
   - 如果超时未收到 ACK，发送端将触发**超时重传机制**。
4. **超时重传**
   - 两种典型场景：
     1. 数据包丢失：未在指定时间内收到确认，重传数据。
     2. 确认包丢失：接收方已收到数据，但 ACK 丢了，发送方未收到确认，也会触发重传。
   - 重传的数据由接收方通过序列号识别并丢弃重复部分，确保传输正确性。
5. **流量控制**
   - 目的：防止发送方速度过快，导致接收方缓冲区溢出。
   - TCP 使用 **滑动窗口机制** 实现流量控制。
     - 接收方在 ACK 中告知窗口大小（可接受的数据量）。
     - 发送方依据窗口调整发送速率，避免数据拥塞和丢包。
   - 实现了按接收能力动态控制发送速度。
6. **拥塞控制**
   - 网络出现拥塞时（如丢包），发送端需降低发送速率，避免进一步加剧网络负载。
   - TCP 拥塞控制通过维护一个 **拥塞窗口（Congestion Window）** 实现。
   - 拥塞控制方法：
     - 慢启动（slow start）
     - 拥塞避免（congestion avoidance）
     - 快速重传（fast retransmit）
     - 快速恢复（fast recovery）
   - 拥塞控制与滑动窗口共同决定实际发送速率，取其最小值。

### tcp粘包怎么解决？

1. **什么是 TCP 粘包问题**

   TCP 是面向字节流的协议，不保留消息边界。发送方连续发送多个数据包时，接收方可能一次性接收多个包，或者一个包被拆成多次接收，从而产生粘包或拆包问题。本质原因在于接收方无法判断消息的开始与结束边界。

2. **粘包产生的常见原因**

   - 发送方连续发送多个小包，TCP 为提高效率将多个包合并发送
   - 接收方读取缓冲区大小设置不合理，一次性读取多个包或不完整包
   - 应用层读取数据速度不够，导致多个包堆积后一次读取
   - 使用了 Nagle 算法，延迟发送导致合并发送

3. **三种解决 TCP 粘包的方式**

   粘包的本质是边界不清晰，解决方案的核心是让接收方明确每条消息的边界。常见有三种分包方案。

4. **固定长度分包**

   每条用户消息采用固定长度格式，例如规定每条消息为 64 字节，接收方每读取 64 字节就认为是一条完整消息
    优点：实现简单，不需要额外协议字段
    缺点：灵活性差，实际使用较少，容易造成空间浪费

5. **特殊字符作为边界**

   在两条消息之间插入特定分隔符字符串（如 `\r\n`、`$` 等），接收方读到特殊字符即认为本条消息已完整
    HTTP 就是通过回车换行作为边界的协议示例
    注意：若消息中包含分隔符，需要转义处理，否则会被误识别为边界

6. **自定义消息结构**

   自定义协议格式，由固定长度的包头和变长的包体组成，包头中包含描述消息体长度的字段
    例如以下结构：

   ```
   struct {
       uint32_t message_length;
       char message_data[];
   } message;
   ```

   解析流程如下：

   - 首先读取包头（如 4 字节），解析出 message_length
   - 然后读取指定长度的 message_data
   - 拼接成一条完整消息，供业务逻辑处理

7. **三种方案对比总结**

   - 固定长度：简单但灵活性差，适用于定长协议
   - 特殊字符：适合文本协议，易于实现但需转义特殊内容
   - 自定义结构：通用性强，是目前最常见和推荐的方案，广泛应用于 RPC 框架、二进制协议等场景

### 怎么用udp实现http？

**1. 基础架构**

- 基于UDP的**QUIC**协议是当前最成熟的解决方案，已被HTTP/3标准采用
- 在保留UDP高效特性的同时，通过创新设计实现了可靠传输

**2. 核心实现机制**
**2.1 连接管理**

- 采用64位Connection ID唯一标识连接
- 实现0-RTT快速重连功能
- 支持网络环境变化时的无缝迁移

**2.2 可靠性保障**

- 引入独立于网络路径的数据包编号
- 设计双向ACK确认机制
- 实现智能选择性重传策略

**2.3 传输优化**

- 原生支持多路复用传输
- 集成前向纠错(FEC)技术
- 采用自适应拥塞控制算法

**3. 协议栈对比分析**

- 传统HTTP/2 over TCP TLS协议栈：[HTTP] → [TLS] → [TCP] → [IP]
- HTTP/3 over QUIC协议栈：[HTTP] → [QUIC加密] → [UDP] → [IP]

**4. 典型应用场景**

- 移动设备网络切换环境
- 高延迟通信网络
- 实时性要求高的应用

**5. 技术说明**

- QUIC协议通过创新设计，在UDP基础上实现了超越TCP的可靠传输能力
- 充分体现了端到端设计原则的灵活性

### 什么是QUIC协议？

**QUIC 协议是什么**

QUIC（Quick UDP Internet Connections）是由 Google 提出并推动、最终被 IETF 标准化的基于 **UDP** 的新一代传输层网络协议。
 它旨在**替代 TCP + TLS + HTTP/2** 的组合，提升 Web 应用的性能，尤其在移动、弱网环境下表现更优。

**QUIC 的主要特点**

- **基于 UDP**：在 UDP 上实现可靠传输，绕过 TCP 固有的传输延迟瓶颈。
- **集成 TLS 加密**：QUIC 将 TLS 加密集成在协议内部，默认使用 TLS 1.3，实现“加密传输即默认”。
- **连接建立更快**：首次连接只需一次 RTT（甚至 0-RTT）即可完成握手，明显快于 TCP 的三次握手 + TLS 握手。
- **多路复用**：支持多个流并发传输，且**一个流的丢包不会阻塞其他流**（解决了 TCP 的队头阻塞问题）。
- **连接迁移支持**：客户端 IP/端口发生变化（如 Wi-Fi 切换 4G）时连接仍可维持，不需要重新建立连接。

**QUIC 与 TCP 的对比**

| 特性           | TCP                   | QUIC                     |
| -------------- | --------------------- | ------------------------ |
| 连接建立       | 三次握手 + TLS 多 RTT | 一次握手，甚至 0-RTT     |
| 加密支持       | 需配合 TLS            | 协议内集成 TLS 1.3       |
| 多路复用       | 有队头阻塞            | 无队头阻塞               |
| 传输基础       | 基于 IP 层，使用 TCP  | 基于 UDP 层              |
| 中间设备兼容性 | 兼容性强              | 初期对防火墙、NAT 有挑战 |
| 连接迁移       | 不支持                | 支持（基于连接 ID）      |

**QUIC 的应用场景**

- 移动设备访问 Web（如从 4G 切换到 Wi-Fi）
- 视频流媒体、游戏、实时通信等对时延敏感的场景
- 现代浏览器与 Web 服务器之间的通信（HTTP/3 正是运行在 QUIC 上）

**QUIC 的发展现状**

- 已被 IETF 正式标准化（RFC 9000、RFC 9001、RFC 9002）
- Google、Facebook、Cloudflare 等大型服务商已大规模部署
- Chrome、Firefox、Edge、Safari 等主流浏览器均已支持
- HTTP/3 是基于 QUIC 的最新 HTTP 协议版本，未来将全面替代 HTTP/2

### 介绍一下TCP拥塞控制？

1. **什么是 TCP 拥塞控制**

   TCP 拥塞控制是为了避免过多数据注入网络导致网络拥塞而引发的性能问题。
    在共享网络环境中，若网络负载过高，会出现延迟、丢包、重传等情况，造成网络阻塞。
    为了保证网络的稳定性与公平性，TCP 通过拥塞控制机制，在网络拥塞时自动减慢发送速率。

2. **拥塞控制的核心思想**

   控制发送方的数据注入速率，避免“发送方发送过快导致网络拥塞”的情况发生。
    通过“拥塞窗口（cwnd）”控制发送方的可发送数据量。

3. **发送窗口的计算公式**

   TCP 实际允许发送的数据量由以下三者中最小值决定：

   - 拥塞窗口 cwnd：根据网络状况动态变化，由拥塞控制算法维护
   - 接收窗口 rwnd：由接收方通知，表示其可接收的缓冲空间
   - 实际发送窗口 swnd = min(cwnd, rwnd)

4. **cwnd 的调整规则**

   - 当网络状况良好，没有丢包或延迟时，cwnd 会逐渐增加
   - 一旦检测到网络拥塞（如超时或快速重传），cwnd 会减小
   - cwnd 的变化反映了 TCP 对网络状态的判断，是拥塞控制的核心手段

5. **如何判断发生了拥塞**

   TCP 并不能直接检测网络是否拥塞，而是通过**ACK 应答的延迟或丢失**来间接判断
    发送方如果在设定时间内未收到 ACK，应答，就认为发生了拥塞，启动拥塞控制逻辑

6. **TCP 拥塞控制的四个算法阶段**

   - **慢启动（Slow Start）**：
      初始 cwnd 值较小，每收到一个 ACK，cwnd 增加 1，呈指数级增长，加快发送速度以尽快达到带宽极限
   - **拥塞避免（Congestion Avoidance）**：
      当 cwnd 超过阈值（ssthresh）后进入该阶段，cwnd 线性增长，每 RTT 增加约一个 MSS，避免过快注入数据
   - **拥塞发生（Congestion Detection）**：
      若发生丢包或超时，表示拥塞发生，需收敛流量，设置 ssthresh = cwnd / 2，并将 cwnd 重置为 1，重新进入慢启动
   - **快速恢复（Fast Recovery）**：
      在收到三个重复 ACK 后，触发快速重传并进入快速恢复，cwnd 减小但不回到初始值，继续在较低速率下传输数据

## 网络场景

### 打开百度发生了什么？

1. **DNS 解析**
   - 用户在浏览器地址栏输入 `www.baidu.com` 后，首先需要将域名转换为 IP 地址。
   - 浏览器会查询本地缓存 → 系统缓存 → hosts 文件 → DNS 服务器。
   - 最终通过 **递归查询** 或 **迭代查询** 获取百度服务器的公网 IP 地址（如 `220.181.38.148`）。
2. **建立 TCP 连接（三次握手）**
   - 客户端使用 DNS 解析得到的 IP 与服务器建立连接，默认使用端口 80（HTTP）或 443（HTTPS）。
   - TCP 三次握手过程如下：
     1. 客户端发送 SYN 报文，进入 SYN_SEND 状态。
     2. 服务器收到后回复 SYN+ACK，进入 SYN_RECV 状态。
     3. 客户端收到后发送 ACK，连接建立，双方进入 ESTABLISHED 状态。
3. **建立 TLS 连接（HTTPS 场景）**
   - 若使用 HTTPS，会在 TCP 建立后执行 **TLS 握手**：
     - 客户端发送 Client Hello，协商加密算法等。
     - 服务端返回证书、公钥、Server Hello。
     - 客户端验证证书、生成对称密钥，并通过公钥加密传给服务端。
     - 双方确认加密方式后，开始加密通信。
4. **浏览器发送 HTTP 请求**
   - 建立连接后，浏览器构造一个 HTTP 请求报文，请求百度首页：
     - 方法：`GET / HTTP/1.1`
     - 请求头包含：Host、User-Agent、Accept-Encoding、Cookie 等信息
   - 浏览器可能携带缓存控制字段，如 `If-Modified-Since`、`ETag` 等用于资源协商。
5. **服务器响应 HTTP 请求**
   - 百度服务器处理请求后，返回首页的 HTML 页面数据：
     - 状态行如：`HTTP/1.1 200 OK`
     - 响应头包含：Content-Type、Content-Length、Set-Cookie 等
     - 响应体即为 HTML 页面内容
   - 若有缓存命中，服务器可能返回 `304 Not Modified`
6. **浏览器解析 HTML 页面**
   - 浏览器开始解析 HTML 页面结构，遇到 CSS、JS、图片等资源时，会**并发发送多个请求**获取它们。
   - 浏览器会维护连接池、复用 TCP 连接（HTTP/1.1 Keep-Alive，或使用 HTTP/2 多路复用）。
7. **资源加载与渲染**
   - 浏览器根据 HTML 结构构建 DOM 树、根据 CSS 构建 CSSOM 树，并合成 Render Tree。
   - 执行 JavaScript 脚本，处理异步加载逻辑。
   - 最终将页面绘制到屏幕上，展示出百度首页。
8. **TCP 连接释放（四次挥手）**
   - 页面加载完毕或连接空闲超时后，TCP 连接可能会被关闭。
   - TCP 连接关闭使用四次挥手机制：
     1. 主动方发送 FIN，进入 FIN_WAIT_1；
     2. 被动方返回 ACK，进入 FIN_WAIT_2；
     3. 被动方再发送 FIN，主动方返回 ACK；
     4. 被动方进入 CLOSED，主动方进入 TIME_WAIT，等待 2MSL 后关闭。

### 请求抖音与请求百度网页的网络过程区别？

1. **连接建立方式不同**
   - 请求百度网页：浏览器通常使用 **TCP + TLS + HTTP/2 或 HTTP/1.1**，首次连接需进行三次握手和 TLS 握手，整体建立过程需 **2 次 RTT**。
   - 请求抖音：抖音 App **优先使用 QUIC 协议（基于 UDP 的 HTTP/3）**，连接建立只需 **1 次 RTT 或 0-RTT**；部分 Android/iOS 设备还支持 **TCP Fast Open（TFO）**，可在握手时携带数据，跳过完整三次握手流程。
2. **协议层级不同**
   - 百度网页使用的是浏览器内置的标准协议栈（HTTP/1.1 或 HTTP/2），升级到 HTTP/3 需浏览器支持 + Alt-Svc 协议协商。
   - 抖音 App 自带网络协议栈，可直接使用 HTTP/3 协议访问服务端，具有更高自由度和优化能力。
3. **请求行为和使用模型不同**
   - 百度网页以**单次请求、一次性响应为主**（获取 HTML 和资源），连接短暂。
   - 抖音 App 会**长时间维持连接**，进行持续的滑动加载视频、点赞、评论等交互，属于**重度长连接、多路并发、数据流式拉取模型**。
4. **多路复用使用方式不同**
   - 百度网页通过 HTTP/2 多路复用加载资源，多个请求共享连接但仍可能受限于首帧阻塞。
   - 抖音通过 **QUIC 的多路复用能力**，将视频数据流、行为上报、评论请求等分流处理，**避免队头阻塞**，提升响应性能。
5. **缓存与预加载策略不同**
   - 百度网页主要依赖浏览器的 HTTP 缓存机制，加载完后释放连接。
   - 抖音内置 CDN + 分片缓存机制，且支持**滑动预加载、边播边下**，连接生命周期更长，链路利用率更高。
6. **网络适应能力不同**
   - 百度网页适配能力有限，连接失败或慢速通常重试一次或提示错误。
   - 抖音支持 **弱网优化机制**：如动态降级 HTTP/3 → HTTP/2、限速模式、连接重选、CDN 节点切换等，能自动应对信号差、延迟高等问题。

### **网页非常慢转圈圈的时候，排查网络问题要从哪些角度入手？**

1. **确认是客户端问题还是服务端问题**
   - 首先测试：浏览器是否可以访问其他网站？
     - 若不行，说明客户端网络自身存在问题；
     - 检查是否联网、Wi-Fi 是否正常、是否插好网线；
   - 若其他网站正常，说明本地网络无问题，可以继续排查 DNS、TCP 等环节。
2. **排查 DNS 是否解析成功**
   - 若域名无法解析出 IP 地址：
     - 检查域名是否拼错；
     - 检查本地 DNS 设置是否正确。
   - 若 DNS 能解析出 IP，则继续判断：
     - 是否成功建立了 TCP 三次握手？
     - 是否成功发出了 HTTP 请求？
     - 是否收到了 HTTP 响应（即使没渲染出页面，也可能有响应码）？
3. **查看 HTTP 响应码进一步定位问题**
   - 若响应码为 404：可能是 URL 地址拼写错误或资源不存在；
   - 若响应码为 500：说明服务器端执行代码时出错；
   - 若响应码为 200：请求成功，可能是前端页面渲染出问题，可打开开发者工具（F12）查看报错信息。
4. **排查网络延迟与拥塞问题**
   - 若网络一切正常但页面很慢显示出来，可能是网速太慢或丢包严重；
   - 查看客户端是否带宽被占满、是否存在 TCP 重传、丢包等性能问题。

### 如何判断 server A 与 server B 的 TCP 连接是否正常？异常怎么处理？

1. **问题背景**
   - TCP 是面向连接的协议，server A 与 server B 建立连接后，即使 B 崩溃或断电，A 也无法立即感知，连接仍可能长时间停留在 `ESTABLISHED` 状态，白白占用资源。
   - 为了解决这种“连接假死”问题，TCP 提供了**保活机制（Keepalive）**。
2. **什么是 TCP 保活机制**
   - 保活机制用于在连接长时间无数据交互时，主动探测对方是否还活着。
   - 若设置了保活，TCP 会周期性向对方发送保活探测包，并观察是否有响应。
   - 若连续多次探测失败，内核认为连接已断开，将向上层应用报告连接失效。
3. **Linux 系统中的默认保活参数**
   - `net.ipv4.tcp_keepalive_time = 7200`：空闲 7200 秒（2 小时）后启动保活探测
   - `net.ipv4.tcp_keepalive_intvl = 75`：每隔 75 秒发送一次探测包
   - `net.ipv4.tcp_keepalive_probes = 9`：最多发送 9 次，仍无响应则判定连接失效
   - 判断连接失效的最小时间为：`7200 + 75 × 9 = 7875 秒 ≈ 2 小时 11 分 15 秒`
4. **启用保活机制的注意事项**
   - 应用程序必须在 socket 上显式设置 `SO_KEEPALIVE` 选项，否则即使内核配置了保活参数，机制也不会启用。
   - 应用场景如：长连接 HTTP、数据库连接、微服务调用等。
5. **保活机制探测的几种处理结果**
   - **对端正常**：对端响应保活包，连接保持，等待下一次保活探测。
   - **对端宕机但未关闭连接**：保活包无响应，连续探测失败后，内核主动断开连接。
   - **对端异常关闭（如直接拔网线）**：系统可能收不到 FIN 包或 RST 包，必须依赖保活机制来探测。
   - **网络断开/防火墙丢弃探测包**：也会导致保活探测失败，最终断连。
6. **TCP 保活机制与应用层超时机制的对比**
   - TCP 保活机制触发时间较长，适合备用安全机制，不适合高实时要求；
   - 应用层（如 Nginx、Java、Go 等）一般通过 **keepalive_timeout** 或 **业务心跳机制** 实现更快断开检测；
   - 比如：每隔 60 秒发送一次心跳请求，如果超过 3 次未响应，即视为断连，可快速释放资源。

7. **总结**
   * 在长连接场景中，TCP 的保活机制用于识别“连接假死”，可配合应用层心跳一起使用，提高连接可用性与资源释放效率。

### **服务端正常启动了，但是客户端请求不到，可能有哪些原因？如何排查？**

1. **客户端请求无响应时的排查方向**
   - **确认接口 IP 是否正确**：可使用 `ping` 命令测试服务端 IP 是否可达。
   - **确认端口是否开放**：使用 `telnet` 或 `nc` 工具测试服务端 IP 和端口是否可连通。
   - **检查防火墙或安全组策略**：确认服务器防火墙或云安全组没有关闭对应端口。
   - **检查代理设置是否干扰**：如客户端设置了代理，需确认代理是否拦截或导致请求失败。
   - **确认客户端配置是否正确**：浏览器、Postman 等工具是否配置了错误的请求头、路径等。
2. **客户端请求有响应但返回错误状态码的排查方式**
   - **400 Bad Request**：请求参数或格式错误，通常是接口使用不当或缺少必要字段。
   - **401 Unauthorized**：未授权，可能缺少必要的 header 信息，如 token、auth 字段等。
   - **403 Forbidden**：已登录但无权限，常见于用户权限不匹配、URL 无访问权限、中间件拦截等。
   - **404 Not Found**：接口路径错误或资源未部署，排查 URL 是否正确。
   - **500 Internal Server Error**：服务器内部异常，需登录服务器查看日志，确认后端代码是否报错。
   - **502 Bad Gateway / 503 Service Unavailable / 504 Gateway Timeout**：
     - 说明服务端网关（如 Nginx）无法转发请求到后端服务；
     - 若每次都报该错，说明后端服务未启动或连接失败；
     - 若偶发出现，说明服务压力大或响应慢。
3. **补充建议**
   - 登录服务端查看服务状态、日志，确认接口是否已启动、是否在监听正确端口。
   - 使用 curl、postman、浏览器等多种方式交叉验证，排除客户端工具本身问题。
   - 检查服务注册发现（如 Nacos、Consul）是否失效或接口下线。

## 网络攻击

### 什么是ddos攻击？怎么防范？

1. **DDoS 攻击的定义**
   - DDoS（Distributed Denial of Service，分布式拒绝服务）攻击，是指攻击者通过大量互联网流量**并发向目标服务器或其周边网络发起请求**，使其资源耗尽、服务瘫痪，导致正常用户无法访问。
   - 攻击流量通常来源于大量被感染的设备组成的“**僵尸网络（Botnet）**”，这些设备可以是计算机、物联网设备等，被远程控制后用于攻击。
2. **DDoS 攻击的原理**
   - 攻击者控制僵尸网络，通过每个节点不断向目标服务器发送请求或数据包，造成**带宽挤占、连接耗尽、CPU 资源消耗过载**。
   - 由于攻击流量往往来自真实设备和合法协议，难以通过传统方式识别和阻断。
3. **常见的 DDoS 攻击类型**
   - **网络层攻击**：如 UDP 放大攻击、NTP Flood 等，利用协议缺陷制造大量反射流量。
   - **传输层攻击**：如 SYN Flood 攻击，通过伪造半连接请求耗尽 TCP 连接资源。
   - **会话层攻击**：如 SSL Flood，目标是服务器的 SSL 握手资源，使其无法完成新连接。
   - **应用层攻击**：如 DNS Flood、HTTP Flood、模拟正常用户行为发起高频请求，直接拖垮服务器的应用处理能力。
4. **DDoS 防御手段**
   - **提升网络带宽与服务能力**：通过扩容网络、增加服务器集群抗压能力，增强基础设施防御力。
   - **使用防火墙与入侵检测系统**：
     - 配置边界防火墙规则，限制不必要端口；
     - 拦截异常 IP 的访问流量，降低攻击面。
   - **流量清洗与负载均衡**：
     - 借助专业 DDoS 防护服务，过滤恶意流量；
     - 合理分发请求至多个后端服务器，减缓单点压力。
   - **访问控制与限频策略**：
     - 限制单 IP 请求频率；
     - 配置访问白名单、验证码机制，防止请求集中或自动化攻击。

### **什么是 SQL 注入？如何防范？**

1. **SQL 注入的定义**

   - SQL 注入是指攻击者通过向应用程序输入恶意构造的数据，使其被拼接进 SQL 查询语句中，从而执行非预期的数据库操作。
   - 产生原因通常是程序将用户输入直接拼接进 SQL 语句，**缺乏输入验证和转义**，导致恶意代码被数据库执行。

2. **SQL 注入的示例**

   - 假设有如下查询语句：

     ```
     SELECT * FROM users WHERE username = '$input'
     ```

     若用户输入为：`' OR '1'='1`，则实际执行的是：

     ```
     SELECT * FROM users WHERE username = '' OR '1'='1'
     ```

     这样会绕过认证，甚至查询整个用户表，造成严重信息泄露。

3. **防范 SQL 注入的常见方法**

   - **输入验证与转义**：
      在将用户输入用于 SQL 查询前，校验格式、类型、长度，并转义特殊字符，防止恶意语句构造。
   - **使用参数化查询**：
      使用预编译 SQL（如 PreparedStatement、ORM 框架等）传入用户参数，避免将输入作为 SQL 语句的一部分拼接。
   - **限制数据库权限**：
      给数据库账户分配最小权限原则，禁止非必要的 `DROP`、`UPDATE` 等敏感操作，防止攻击者滥用高权限。
   - **输入过滤机制**：
      过滤输入中的关键字（如 `--`、`' OR '`、`UNION` 等），结合正则或白名单，拒绝非法输入。

### **什么是 CSRF 攻击？如何防范？**

1. **CSRF 攻击的定义**
   - CSRF（Cross-Site Request Forgery，跨站请求伪造）是指攻击者诱导用户在已登录状态下，向受信任网站发送非法请求，从而执行非用户本意的操作。
   - 攻击者通常构造一个**看似合法的 HTTP 请求**，诱导用户点击或自动触发，让浏览器携带登录状态（如 Cookie）向目标网站发起操作请求。
2. **CSRF 攻击的特点**
   - 利用的是用户的身份和认证状态，而不是系统的漏洞；
   - 通常依赖浏览器自动携带身份凭据（如 Cookie、Session）；
   - 攻击请求看起来合法，甚至可以是 GET、POST、图片请求、隐藏 iframe 等形式；
   - 受影响操作包括转账、改密、删帖等敏感接口。

3. **CSRF 防御手段**

   - **验证用户会话**：在服务端校验用户请求中的 Session ID、Token 等是否与当前登录会话匹配。

   - **使用双重验证**：除 Cookie 外增加验证码、短信码、扫码等验证，提高操作敏感性的防护强度。

   - **设置 CSP 策略**：通过 Content-Security-Policy 限制脚本来源，降低攻击者构造页面诱导用户的风险。

   - **禁止表单自动提交**：添加确认按钮，禁止自动触发，避免被嵌入恶意页面后自动提交。

   - **检查 Referer 或 Origin 头**：服务端校验请求来源，确保请求来自本站，而非第三方站点。

### 总结一句话：

* CSRF 攻击本质是“攻击者伪造请求，用户不知情地执行”，防御重点是“确保请求发起人就是用户本人”。

### **什么是 XSS 攻击？如何防范？**

1. **XSS 攻击的定义**
   - XSS（Cross-Site Scripting，跨站脚本攻击）是指攻击者在网页中注入恶意 JavaScript 代码，当用户访问页面时，代码会在其浏览器中执行，导致信息泄露、权限劫持、页面篡改等问题。
2. **XSS 攻击的目的**
   - 窃取用户信息（如 Cookie、Token、会话 ID）；
   - 模拟用户操作（如发帖、点赞、转账）；
   - 劫持页面内容、诱导钓鱼攻击；
   - 执行浏览器端恶意代码，进一步攻击用户设备。
3. **XSS 攻击的分类**
   - **存储型 XSS**（持久型）：恶意脚本被存储在服务器端数据库中，当其他用户访问时加载执行。
   - **反射型 XSS**（非持久型）：恶意脚本通过 URL 或请求参数传入，立即被响应页面反射并执行。
   - **DOM 型 XSS**：恶意脚本不经服务器处理，而是通过修改 DOM 结构在客户端执行，依赖前端 JS 的处理逻辑漏洞。
4. **防范 XSS 攻击的常见手段**
   - **输入验证**：
     - 对所有用户输入进行合法性校验，过滤特殊字符；
     - 禁止插入 `<script>`、`onerror=` 等 HTML/JS 标签。
   - **输出编码**：
     - 在将用户数据输出到 HTML 页面前进行编码（如 HTML 转义、URL 编码），防止脚本注入。
   - **设置 CSP 策略**（Content Security Policy）：
     - 限制网页中可执行的脚本来源，减少被注入脚本运行的可能性。
   - **使用 HttpOnly Cookie**：
     - 设置 Cookie 为 `HttpOnly`，防止 JavaScript 读取，降低敏感信息泄露风险。

### **什么是 DNS 劫持？**

1. **DNS 劫持的定义**
   - DNS 劫持是指攻击者在用户进行域名解析过程中，**篡改 DNS 响应结果**，将用户请求的域名解析到攻击者控制的虚假 IP 地址，从而达到**诱导、欺骗或监听用户流量**的目的。
2. **DNS 劫持的原理**
   - 当用户访问某网站（如 `www.example.com`）时，会先向 DNS 服务器发起域名解析请求；
   - 攻击者通过**篡改 DNS 缓存、污染本地 Hosts 文件、控制上游 DNS 服务器等方式**，将该域名解析到恶意 IP；
   - 用户在不知情的情况下访问了仿冒页面、钓鱼网站或恶意广告页面。
3. **DNS 劫持的实现方式**
   - **本地劫持**：修改用户设备中的 Hosts 文件或 DNS 配置；
   - **网络层劫持**：在运营商或中间设备处拦截并伪造 DNS 响应；
   - **服务器端劫持**：攻击或控制用户配置使用的 DNS 服务器，返回恶意地址；
   - **劫持 DNS 流量**：通过 ARP 欺骗、中间人攻击等方式篡改 DNS 查询结果。
