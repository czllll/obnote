## 数据结构

### 讲一下redis底层数据结构？

#### **Redis常用的五种数据类型及底层数据结构**

- **String（字符串）**
  - 底层实现：SDS（简单动态字符串）、int 整数编码（用于节省空间）。
  - 存储方式：单个键值对应一个简单文本字符串（最多512MB）。
  - 场景：缓存对象、常规计数、分布式锁、共享session等。
- **List（列表）**
  - 底层实现：快速链表（quicklist，由ziplist压缩列表和linkedlist双向链表组合而成）。
  - 存储方式：有序集合，可包含重复元素，允许两端快速插入/删除。
  - 场景：消息队列（但存在需要自行实现唯一ID和无法消费组模式消费的缺点）、最近访问记录、时间线等。
- **Set（集合）**
  - 底层实现：intset（整数集合）或hashtable（哈希表）。
  - 存储方式：无序集合，元素唯一性，不允许重复。
  - 场景：标签管理、关注列表、点赞、抽奖活动去重、好友推荐等。
- **Hash（散列）**
  - 底层实现：ziplist（压缩列表，数据少时使用）或hashtable（哈希表，数据量大时使用）。
  - 存储方式：键值对集合（field-value对），适合对象存储。
  - 场景：缓存对象、购物车、用户信息存储等。
- **Zset（有序集合）**
  - 底层实现：ziplist（压缩列表）或skiplist（跳跃表）+ hashtable（哈希表）。
  - 存储方式：元素唯一，带分数(score)，按分数进行排序。
  - 场景：排行榜、热度统计（如热门搜索）、定时任务等。

#### **Redis新增数据类型及应用场景**

- **BitMap（二进制位图，Redis 2.2版本新增）**
  - 存储方式：位操作，存储0/1状态。
  - 场景：签到、判断用户登录状态、连续签到天数统计。
- **HyperLogLog（基数统计，Redis 2.8版本新增）**
  - 存储方式：概率性数据结构，近似去重统计（误差0.81%以内）。
  - 场景：网站UV统计、页面独立访客数统计。
- **GEO（地理位置，Redis 3.2版本新增）**
  - 存储方式：Zset变种，通过经纬度进行存储和距离计算。
  - 场景：地理位置相关场景，如附近的人、滴滴打车、地图查找等。
- **Stream（消息流，Redis 5.0版本新增）**
  - 存储方式：类似日志追加方式，基于消息ID自动生成和持久化，支持消息消费确认、消费组。
  - 场景：消息队列，支持持久化消费、可回溯数据、分组消费（相比List更高级）。

### 用过zset吗？

**ZSet（有序集合）的基本概念**

- 元素唯一，每个元素关联一个 `score`（分数），按照 `score` 排序。
- 底层数据结构：**跳跃表（skiplist） + 哈希表（hash table）**。
- 典型应用：排行榜、按权重排序的数据（如文章点赞量、商品热度、用户积分等）。

**ZSet 常用命令及示例**

- **添加元素：ZADD**

  - 语法：`ZADD key score member`

  - 示例：为文章添加点赞量

    ```sql
    ZADD user:xiaolin:ranking 200 article:1
    ZADD user:xiaolin:ranking 40 article:2
    ZADD user:xiaolin:ranking 100 article:3
    ZADD user:xiaolin:ranking 50 article:4
    ZADD user:xiaolin:ranking 150 article:5
    ```

- **增加已有元素的分数：ZINCRBY**

  - 语法：`ZINCRBY key increment member`

  - 示例：article:4 增加 1 个赞

    ```sql
    ZINCRBY user:xiaolin:ranking 1 article:4
    ```

- **获取元素的分数：ZSCORE**

  - 语法：`ZSCORE key member`

  - 示例：查询 article:4 的点赞数

    ```sql
    ZSCORE user:xiaolin:ranking article:4
    ```

- **按分数倒序获取指定排名区间内的元素：ZREVRANGE**

  - 语法：`ZREVRANGE key start stop [WITHSCORES]`

  - 示例：获取前 3 篇文章及其点赞数

    ```sql
    ZREVRANGE user:xiaolin:ranking 0 2 WITHSCORES
    ```

- **按分数区间获取成员：ZRANGEBYSCORE**

  - 语法：`ZRANGEBYSCORE key min max [WITHSCORES]`

  - 示例：获取点赞数在 100 到 200 的文章

    ```sql
    ZRANGEBYSCORE user:xiaolin:ranking 100 200 WITHSCORES
    ```

**补充说明与注意事项**

- ZSet 中的 `score` 是浮点数，可用于表示权重、热度、时间戳等。
- 同一个成员重复添加会更新其分数。
- **ZREVRANGE** 是倒序（大到小），而 **ZRANGE** 是正序（小到大）。
- 若需分页查询排行榜，可结合 `ZREVRANGE` 的 `start-stop` 实现。

### zset底层实现

1. **ZSet 的底层数据结构**
   - ZSet（有序集合）由 **两种底层结构** 实现，具体取决于集合大小和元素长度：
     - **早期版本（Redis < 7.0）**
       - 当元素个数 < 128 且每个元素 < 64 字节：
         - 使用 **压缩列表（ziplist）**
       - 否则使用：
         - **跳跃表（skiplist） + 哈希表（dict）**：
           - 跳跃表按 `score` 排序，支持范围查找；
           - 哈希表用于快速定位成员值和对应 `score`。
     - **Redis 7.0 之后的变化**
       - **压缩列表已被弃用**
       - 替代为 **listpack（列表包）**，作为小型数据的底层结构
2. **总结各结构特点**
   - **压缩列表（ziplist）**（已废弃）
     - 连续内存，节省空间，适合小数据量
     - 插入/删除性能差，更新代价高
   - **listpack（Redis 7.0 起）**
     - 更高效的紧凑型内存结构，替代 ziplist
     - 支持快速遍历，性能优于 ziplist
   - **跳跃表（skiplist）+ 哈希表**
     - 支持高效范围查询（跳跃表）
     - O(logN) 插入和查询性能
     - 哈希表保证快速按成员查找和去重
3. **底层结构选择逻辑（面试可背）**
   - 若元素个数 < 128 且每个元素长度 < 64 字节：
     - Redis < 7.0 使用 **ziplist**
     - Redis ≥ 7.0 使用 **listpack**
   - 否则统一使用 **跳跃表 + 哈希表**

### 跳表是怎么实现的？

1. **跳表的基本概念**

   - 跳表是一种 **多层级的有序链表结构**，每一层是下一层的「快捷方式」。
   - 支持快速插入、删除和查找，时间复杂度为 `O(logN)`。
   - Redis 使用跳表作为 **ZSet 类型的底层结构**（当数据量大时）。

2. **跳表结构组成**

   - **节点（zskiplistNode）包含：**

     - `ele`: 元素本身（类型为 `sds`）
     - `score`: 排序用的分值（类型为 `double`）
     - `backward`: 后退指针，便于反向遍历
     - `level[]`: 层级数组，每一层包含：
       - `forward`: 前向指针（指向该层下一个节点）
       - `span`: 跨度，表示跳过的节点数量

   - **跳表结构图示：**

     ```
     L2 ──────────► 3 ───► NULL
     L1 ─────► 2 ─► 3 ─► 5 ─► NULL
     L0 ─► 1 ─► 2 ─► 3 ─► 4 ─► 5 ─► NULL
     ```

   - **各层级说明：**

     - L0 是所有节点都会出现的一层
     - L1、L2 是根据概率随机出现的高层级，用于加速查找

3. **跳表的查找过程（面试重点）**

   - 查找时从顶层向下查找，再在某层内横向跳跃：
     - 例如：查找值为 4，路径为 L2→3→L1→3→L0→4
   - 最终查找复杂度为 `O(logN)`，比链表的 `O(N)` 更高效

4. **跳表的层级生成策略**

   - 创建节点时，**随机生成层数**

   - 每增加一层的概率是 25%（即 1/4 概率）

   - 最大层数上限为 **64 层**

   - 生成逻辑：

     ```
     初始层数为 1；
     每次随机数 < 0.25 就增加一层；
     否则停止；
     ```

   - 这种结构使得层数高的节点越来越少，整体符合概率分布。

5. **跨度（span）的意义**

   - 跨度用于计算当前节点跳到下一个节点之间的“跨度”

   - 在计算排名（如 ZRANK）时非常重要

   - 图中如：

     ```
     跨度 1 表示：正好跳一个节点
     跨度 3 表示：跳过三个节点
     ```

6. **跳表头节点特殊说明**

   - 跳表的 **头节点** 在创建时，直接分配最大层级（64）
   - 用于作为整个跳表的起点，指向各层第一个实际节点

7. **跳表 vs 红黑树**

   - 跳表是 Redis 实现 ZSet 的选择，原因：
     - 插入、删除更简单
     - 代码实现更清晰
     - 在有序数据存储场景中性能表现好于红黑树

###  **跳表是怎么设置层高的？**

- 跳表节点的层数是通过**概率方式**随机生成的。
- **生成逻辑如下：**
  - 创建节点时，生成一个 `[0,1)` 范围的随机数；
  - 若该随机数 `< 0.25`（概率为 25%），则增加一层；
  - 然后继续生成下一个随机数，继续判断；
  - **直到某个随机数 ≥ 0.25 为止**，最终确定该节点的层数。
- **最大层数上限为 64 层**，即最多有 64 层索引。
- **这种方式的好处：**
  - 层数高的节点非常稀少，整体形成金字塔结构；
  - 插入与查找平均时间复杂度为 `O(logN)`，且实现简单。

### Redis为什么使用跳表而不是用B+树?

1. **Redis 为什么选跳表？**

   Redis 是**内存数据库**，相比磁盘型数据库，其对 **内存访问效率、结构简单性和写入性能要求更高**。跳表在以下方面比 B+ 树更适合：

2. **维度对比：跳表 vs B+ 树**

   - **内存访问：**

     - 跳表：符合 CPU 缓存局部性，指针跳跃少，内存访问高效。
     - B+ 树：节点结构复杂，层次深，缓存不友好。

     - **实现复杂度：**
       - 跳表：代码简单，无需复杂的旋转、节点合并/分裂逻辑。
       - B+ 树：需实现分裂合并、最小填充、兄弟借值等，逻辑复杂。

     - **写入性能：**
       - 跳表：插入/删除只调整部分指针，开销低。
       - B+ 树：可能引发父节点递归更新，成本较高。

   - **内存占用：**
     - 跳表：结构紧凑，无内存碎片。
       - B+ 树：预分配导致碎片多，占用高。

3. **深入分析：为什么跳表更适合 Redis？**

   - **1）内存访问结构友好**

     - Redis 不涉及磁盘 I/O，跳表层级结构+链表指针命中 CPU 缓存，访问更快。
     - B+ 树偏向磁盘结构优化，节点分页适合磁盘读取。

     - **2）实现简单，代码简洁**
       - Redis 跳表核心代码仅约 200 行。
       - B+ 树涉及复杂边界处理，代码动辄上千行。

     - **3）无锁/低锁支持并发**
       - 跳表可结合 CAS 或细粒度锁优化并发。
       - B+ 树往往涉及全局锁或页锁，粒度粗，性能瓶颈明显。

   - **4）跳表支持高效范围查询**
     - 支持双向链表结构，配合 `ZRANGE` 等操作效率高。
       - 与 B+ 树的叶子节点范围扫描效果相近。

### 压缩列表是怎么实现的？

1. **压缩列表的定义与作用**

- 压缩列表是 Redis 为了节省内存而设计的**连续内存块组成的顺序型数据结构**，常用于：
  - 小体量的 List、Hash、ZSet 的底层实现
- 特点：**结构紧凑、节省空间**

2. **压缩列表结构组成**

表头部分（固定 3 个字段）：

| 字段      | 含义                                             |
| --------- | ------------------------------------------------ |
| `zlbytes` | 整个压缩列表占用的内存字节数                     |
| `zltail`  | 表尾节点距离起始地址的偏移量（便于快速定位尾部） |
| `zllen`   | 压缩列表中 entry 节点的个数（最多 65535 个）     |
| `zlend`   | 列表结尾标识，固定值 `0xFF`（255）               |

节点部分（entry）：

每个 entry 包含以下三段：

- `prevlen`：上一个节点的长度（用于实现**反向遍历**）
- `encoding`：当前节点数据的编码方式和长度
- `data`：实际数据（字符串或整数）

3. **压缩列表的优点**

   - **紧凑存储**：连续内存、无额外结构开销，节省空间

   - **双向遍历**：通过 `prevlen` 可以实现反向查找

   - **适配多类型数据**：encoding 能根据数据类型自动调整分配大小

4. **压缩列表的缺点与局限**

   - **更新成本高（连锁更新）**：
     - 若插入一个新元素导致原有元素的 `prevlen` 改变，可能触发**多个节点的连锁更新**
     - 例如：插入一个长度超过 254 的元素，导致 `prevlen` 从 1 字节变为 5 字节

   - **查询复杂度高**：
     - 查询首尾元素复杂度是 `O(1)`
       - 查询中间元素只能顺序遍历，复杂度为 `O(N)`
       - 因此**不适合存储过多元素**

5. **适用场景**

   - 节点个数较少、数据体积较小时使用效果最佳

   - 用作：
     - 小型 Hash 的底层结构
       - 小型 ZSet 或 List 的底层结构

6. **演进替代：listpack 和 quicklist**
   - Redis 后续版本对压缩列表做出改进：
     - **quicklist（Redis 3.2）**：用于 List 类型，解决连锁更新问题，结合多个 ziplist 组成双向链表
     - **listpack（Redis 5.0）**：替代 ziplist，用于 Hash 和 ZSet，结构更简单、压缩效率更高

### 介绍下redis的listPack？

1. **定义与作用**

   - listpack 是 Redis 5.0 引入的一种紧凑型数据结构，目的是替代早期的压缩列表（ziplist）。

   - 设计初衷是解决 ziplist 在更新数据时容易触发“连锁更新”的问题，同时继续保持结构紧凑、节省内存。

2. **整体结构组成**
   - listpack 由连续内存块构成，结构如下：
     - 总字节数：记录整个 listpack 所占用的内存大小。
     - 元素数量：记录 listpack 中元素的个数。
     - entry1 ~ entryN：每一个元素节点。
     - 结尾标识：固定为 `0xFF`，表示结构结束。

3. **每个节点（entry）的结构**

   - encoding：定义元素的编码方式和数据类型。

   - data：实际存储的数据，类型可能是整数或字符串。

   - len：表示当前 entry 的总长度（即 encoding + data 的字节总和）。

4. **与 ziplist 的对比优势**

   - 不再记录上一个节点的长度（ziplist 的 prevlen 字段），避免新增或修改节点时引起“级联修改”。

   - 结构更加清晰，插入和删除效率更高，特别适合写操作频繁的场景。

   - 在小数据量的条件下与 ziplist 一样节省空间，但写入性能显著提升。

5. **应用场景**

   - 用作 Hash 类型的底层结构（当字段较少时）。

   - 用作 ZSet 类型的底层结构（元素和分数都较小时）。

   - 用作 Stream 类型中消息的内部存储结构（消息紧凑存储，写入高效）。

6. **总结**

   - listpack 是压缩列表的升级替代方案，完全避免了连锁更新问题。

   - 在 Redis 5.0+ 中成为轻量级复合结构的默认实现，兼具空间效率与写入性能。

### **哈希表是怎么扩容的**

1. **基本结构**
   - Redis 中的哈希表底层由 `dict` 结构实现，内部有两个哈希表数组：`ht[0]` 和 `ht[1]`。
   - 正常情况下只使用 `ht[0]`，当需要扩容或缩容时，会启用 `ht[1]` 并进入 rehash 状态。
2. **触发 rehash 的时机**
   - 哈希表中的元素个数大于槽位数，并且当前不在 rehash 状态。
   - 删除大量元素导致负载因子过小，也可能触发缩容。
   - 手动调用 `redis-cli` 执行一些命令或后台自动触发。
3. **rehash 的基本流程**
   - 为 `ht[1]` 分配新的空间，通常为 `ht[0]` 大小的 2 倍。
   - 将 `ht[0]` 中的数据**逐步迁移**到 `ht[1]`。
   - 迁移完成后，释放 `ht[0]` 的空间，并将 `ht[1]` 设置为新的 `ht[0]`，将 `ht[1]` 置空。
4. **渐进式 rehash 的机制**
   - 为避免一次性迁移大量数据带来的阻塞，Redis 采用渐进式 rehash。
   - 每次对哈希表执行写操作（增删改）时，都会额外迁移 `ht[0]` 的一部分数据到 `ht[1]`。
   - 查询操作时，如果没在 `ht[0]` 查到，会继续去 `ht[1]` 查找。
   - 渐进过程分批进行，直到数据全部搬完。
5. **渐进式 rehash 的细节说明**
   - 在 rehash 期间，Redis 会在每次操作中执行少量数据的搬迁。
   - 插入新数据只会插入到 `ht[1]` 中，`ht[0]` 不再接收新数据。
   - 查询时需同时查询 `ht[0]` 和 `ht[1]`，直到迁移完成。
   - 删除或更新时也需要判断该 key 在两个表中的位置。
6. **为什么要使用渐进式 rehash**
   - 如果使用一次性 rehash，在数据量很大时会造成阻塞，影响正常服务请求。
   - 渐进式方式将 rehash 分散到多次请求中，避免了卡顿。
7. **rehash 期间的性能特性**
   - 插入写入的性能会略有下降，因为同时要迁移数据。
   - 查询操作变慢，因为可能需要在两个哈希表中查找。
   - 但整体对服务影响较小，适用于高并发环境。

### **哈希表扩容的时候，有请求怎么查**

1. 执行渐进式 rehash 时，Redis 同时维护两个哈希表：`ht[0]` 和 `ht[1]`。
2. 查询一个 key 时，Redis 会：
   - 先在 `ht[0]` 中查找；
   - 如果没找到，再到 `ht[1]` 中查找；
3. 这个过程保证了查询操作的正确性，即使数据正在迁移中。

### **String 是使用什么存储的？为什么不用 C 语言中的字符串？**

1. **Redis 的字符串使用 SDS 存储**
   - Redis 中的 String 类型使用的是 SDS（Simple Dynamic String）结构，而不是传统的 C 语言字符数组（char*）。
   - SDS 提供了更高效、更安全的字符串操作能力。
2. **SDS 数据结构组成**
   - `len`：实际字符串长度（不包含结尾的 `\0`），用于 O(1) 获取字符串长度。
   - `alloc`：已分配空间的总长度（不包含结尾的 `\0`），用于判断是否需要扩容。
   - `flags`：表示 SDS 类型（如 sdshdr5、sdshdr8、...sdshdr64）。
   - `buf[]`：字节数组，用于保存实际的数据（可为文本或二进制数据）。
3. **SDS 相比 C 字符串的优势**
   - **O(1) 获取字符串长度**
     - C 语言中 `strlen` 为 O(N)，SDS 通过 `len` 变量直接返回，复杂度为 O(1)。
   - **二进制安全**
     - C 字符串以 `\0` 作为结束符，无法存储中间包含 `\0` 的二进制数据。
     - SDS 使用 `len` 记录长度，支持任意格式的二进制内容。
   - **避免缓冲区溢出**
     - C 字符串函数如 `strcat` 可能因未判断空间大小造成溢出。
     - SDS 内部通过 `alloc - len` 判断剩余空间，若不足会自动扩容。
   - **自动扩容机制**
     - 如果修改时空间不够，SDS 会自动将空间扩展至所需大小，无需手动处理。
     - 避免因开发者疏忽导致缓冲区问题。
4. **总结**
   - SDS 在原始 C 字符数组的基础上添加了 `len`、`alloc` 和 `flags` 三个元信息字段。
   - 它解决了 C 字符串中存在的三个核心问题：
     - 无法 O(1) 获取长度；
     - 无法存储二进制数据；
     - 容易发生内存越界。

## 线程模型

### **Redis 为什么快**

1. **单线程模型，减少上下文切换**
   - Redis 使用单线程处理所有网络 I/O 和命令执行，避免了多线程切换带来的上下文开销。
   - 同时也规避了多线程竞争、加锁等问题，程序更简单，性能更稳定。
2. **所有操作在内存中完成**
   - Redis 是内存数据库，所有数据都保存在内存中，访问速度远高于磁盘。
   - 内存访问速度以纳秒计，远优于磁盘或数据库层的 I/O。
3. **高效的数据结构设计**
   - Redis 各种数据结构（如 SDS、跳表、压缩列表、哈希表等）都为高性能访问专门优化。
   - 操作复杂度低，插入、查找、删除大多为 O(1) 或 O(logN)。
4. **I/O 多路复用机制**
   - Redis 使用 `select/epoll` 等多路复用技术，在单线程下支持并发处理多个 Socket 连接。
   - 内核监听多个客户端 socket，一旦有请求到达就交给 Redis 处理，实现高并发下的高效处理。
5. **CPU 不是瓶颈**
   - Redis 的瓶颈大多数是网络带宽或内存，而不是 CPU。
   - 单线程能充分利用 CPU 缓存局部性，避免线程上下文切换导致的缓存失效。
6. **基准测试结果**
   - 官方测试显示，Redis 单线程可处理超过 10 万 QPS，性能极高。
   - 图示显示在 10w 请求以内性能平稳，负载增加后下降也较慢，性能非常优秀。

### **Redis 哪些地方使用了多线程**

1. **Redis 主线程是单线程**

   - Redis 执行命令的主流程，包括：接收请求 → 解析 → 执行命令 → 返回响应，全部在一个主线程中完成。
   - 所以我们常说 Redis 是单线程的。

2. **Redis 启动时会创建后台线程（BIO）**

   - 用于执行一些慢操作，防止阻塞主线程。
   - Redis 2.6 版本后，默认会创建两个后台线程：
     - 关闭文件线程（`bio_close_file`）
     - AOF 刷盘线程（`bio_aof_fsync`）
   - Redis 4.0 后新增一个后台线程 `bio_lazy_free`，用于异步释放内存（如 `unlink`、`flushdb async` 等）。

3. **后台线程的任务机制**

   - 主线程创建后台任务，将任务放入队列。
   - 后台线程从队列中取出任务执行，实现异步解耦。
   - 这样不会阻塞主线程，如异步删除大 key 就使用了该机制。

4. **Redis 6.0 引入 I/O 多线程**

   - 由于网络 I/O 开销增大，Redis 6.0 开始支持多个线程同时处理客户端连接读写。

   - 默认多线程仅用于**写 socket**，要开启对**读 socket**的支持，需要配置：

     ```lua
     io-threads-do-reads yes
     ```

   - 启动的线程数量通过 `redis.conf` 中 `io-threads` 设置，例如：

     ```lua
     io-threads 4
     ```

5. **线程数量说明**

   - Redis 默认的 `io-threads` 为 4，表示总共 4 个线程（包含主线程），实际启动 3 个 I/O 线程。
   - 官方建议：线程数小于等于 CPU 核心数，4 核设 2~3，8 核设 6。

6. **Redis 启动默认创建的线程（除主线程外）**

   - `bio_close_file`：关闭文件任务线程
   - `bio_aof_fsync`：AOF 刷盘线程
   - `bio_lazy_free`：异步释放内存线程
   - `io_thd_1`、`io_thd_2`、`io_thd_3`：I/O 处理线程
   - 共 **6 个子线程**，不包括主线程

### Redis怎么实现的io多路复用？

1. **Redis为何需要I/O多路复用**

   - Redis采用单线程模型，所有操作顺序执行。
   - 若直接使用阻塞I/O，单个客户端I/O操作会阻塞整个进程，无法服务其他客户端。
   - I/O多路复用通过单线程监控多个连接（Socket），解决阻塞问题，实现高并发。

2. **核心概念**

   - **多路**：指多个客户端网络连接（Socket）。
   - **复用**：指单线程通过轮询机制管理多个I/O流。

3. **实现机制**

   - **Socket与文件描述符（FD）**：
     - 每个客户端连接对应一个Socket，生成唯一的文件描述符（FD）。
   - **事件注册与监听**：
     - I/O多路复用程序（如`epoll`）将FD注册到监听队列。
     - 监听多个FD的读写事件（如`accept`、`read`、`write`）。
   - **事件驱动处理**：
     - 当FD就绪（有数据到达），触发文件事件处理器执行对应操作。

4. **工作流程（以epoll为例）**

   - **步骤1**：客户端连接Redis，生成Socket并绑定FD。
   - **步骤2**：Redis通过`epoll`将FD加入监听列表。
   - **步骤3**：`epoll`轮询监听FD，就绪时通知事件处理器。
   - **步骤4**：单线程处理就绪事件（如执行命令、返回结果）。

5. **技术优势**

   - **非阻塞**：单线程避免阻塞，快速响应多个客户端请求。
   - **高性能**：通过事件驱动减少无效轮询，降低CPU开销。
   - **Reactor模式**：基于事件分发，实现高效并发处理。

6. **与传统阻塞I/O对比**

   | **对比项** | **阻塞I/O**            | **I/O多路复用**          |
   | :--------- | :--------------------- | :----------------------- |
   | 线程模型   | 多线程（每连接一线程） | 单线程监控多连接         |
   | 资源消耗   | 高（线程上下文切换）   | 低（单线程+事件驱动）    |
   | 适用场景   | 低并发                 | 高并发（如Redis、Nginx） |

### **Redis 网络模型是怎样的？**

Redis 6.0 版本之前，采用的是单 Reactor 单线程模式：

- Reactor 负责 select 监听客户端请求，并分发给相应处理器
- Acceptor 处理客户端连接
- Handler 进行 read 和业务处理，最后 send 给客户端

**它的缺点为：**

1. 无法充分利用多核 CPU 性能
2. Handler 处理业务时会阻塞全过程，引起延迟

**适用场景**：业务非常简单快速。

Redis 是用 C 语言实现的，主要操作在内存中完成，性能瓶颈不在 CPU，因此前 6.0 版本采用单 Reactor 单进程模型。

**Redis 6.0 之后对网络 I/O 进行了改进：**

- 将网络 I/O 改为多线程处理，同时添加 io-threads 参数
- 命令执行仍由单线程处理，避免加销系统处理复杂并发纠结

**综合：**

- Redis 6.0 之前是单 Reactor 单线程，缺点是不能利用多核 CPU，业务阻塞引起延迟
- Redis 6.0 之后网络 I/O 多线程，命令仍保持单线程模型



## 事务

### 如何实现redis 原子性？

1. **Redis原子性的实现原理**

   - **单线程模型**：Redis采用单线程处理命令，天然避免多线程竞争问题，确保单条命令的原子性。
   - **Lua脚本**：多条命令需保证原子性时，可将操作封装为Lua脚本，Redis会将其作为整体执行（中途不穿插其他命令）。

2. **单命令原子性示例**

   - 如`INCR`、`SETNX`等命令本身是原子操作。
   - 单线程顺序执行，无需额外控制。

3. **多命令原子性实现（Lua脚本）**

   - **核心机制**：

     - Lua脚本执行期间，Redis不会处理其他客户端请求。
     - 脚本内所有操作要么全部成功，要么全部失败。

   - **典型场景**：

     - **分布式锁**：判断锁归属与释放锁需原子化。

     ```
     -- 判断锁归属并释放（KEYS[1]=锁key, ARGV[1]=唯一标识）
     if redis.call("get", KEYS[1]) == ARGV[1] then
         return redis.call("del", KEYS[1])
     else
         return 0
     end
     ```

   - **优势**：

     - 避免锁误删（如A的锁被B释放）。
     - 减少网络开销（多个操作合并为一个请求）。

4. **与事务（MULTI/EXEC）的区别**

   | **对比项** | **Lua脚本**            | **事务（MULTI/EXEC）**             |
   | :--------- | :--------------------- | :--------------------------------- |
   | 原子性保证 | 强原子性（整体执行）   | 弱原子性（仅命令队列顺序执行）     |
   | 中断可能性 | 不可中断               | 执行EXEC前可能被其他客户端命令打断 |
   | 适用场景   | 复杂逻辑（如条件判断） | 简单批量操作                       |

5. **注意事项**

   - **Lua脚本复杂度**：避免长时间运行的脚本阻塞Redis（默认有执行时间限制）。
   - **错误处理**：Lua脚本中需自行处理异常（如键不存在）。

6. **记忆技巧**

   - **单命令**：“单线程顺序跑，天然原子不用锁”。
   - **Lua脚本**：“多操作包Lua，整体执行不怕堵”。

### 除了lua有没有什么也能保证redis的原子性？

1. **Redis事务**的原子性实现

   - **MULTI/EXEC机制**：
     - 使用`MULTI`开启事务，后续命令入队但不立即执行。
     - `EXEC`触发事务执行，队列中的命令按顺序一次性执行。
   - **原子性表现**：
     - 事务正常执行时（无错误），所有命令作为一个整体完成。
     - **局限性**：事务中某条命令失败时，**不会回滚已执行的命令**（Redis不支持事务回滚）。

2. **事务执行流程示例**

   ```shell
   # 开启事务
   127.0.0.1:6379> MULTI
   OK
   
   # 命令入队（LPOP操作错误类型，但入队时不报错）
   127.0.0.1:6379> LPOP a:stock  # a:stock为非列表类型
   QUEUED
   
   # 正确命令入队
   127.0.0.1:6379> DECR b:stock
   QUEUED
   
   # 执行事务（第一条失败，第二条成功）
   127.0.0.1:6379> EXEC
   1) (error) WRONGTYPE Operation against a key holding the wrong kind of value
   2) (integer) 8
   ```

   - **结果说明**：
     - `LPOP`因类型错误失败，但`DECR`仍成功执行。
     - Redis仅保证命令顺序执行，不保证全部成功。

3. **与Lua脚本的对比**

   | **特性**       | **Redis事务（MULTI/EXEC）**      | **Lua脚本**                  |
   | :------------- | :------------------------------- | :--------------------------- |
   | **原子性强度** | 弱原子性（命令独立执行，无回滚） | 强原子性（整体执行或失败）   |
   | **错误处理**   | 部分失败不影响其他命令           | 脚本错误则全部不执行         |
   | **适用场景**   | 简单批量操作                     | 需条件判断或严格原子性的场景 |

4. **事务的注意事项**

   - **无回滚机制**：Redis设计哲学强调简单性，故意省略回滚功能。
   - **入队时检查**：仅检查命令语法错误（如参数数量），不检查逻辑错误（如操作类型不匹配）。
   - **WATCH命令**：可配合`WATCH`实现乐观锁，监控键值变化，若被修改则事务不执行。

5. **其他保证原子性的方式**

   - **WATCH+MULTI**：乐观锁机制，监控关键键，若事务执行前键被修改，则事务放弃。

     ```
     WATCH key
     MULTI
     SET key new_value
     EXEC  # 若key被其他客户端修改，此处返回nil
     ```

6. **记忆技巧**

   - **事务口诀**：“MULTI排队EXEC跑，失败不管继续搞”。
   - **Lua优势**：“Lua打包强原子，错误全回无漏网”。

## 日志

### Redis有哪2种持久化方式？分别的优缺点是什么？

#### **1. AOF（Append-Only File）日志**

- **实现方式**：

  - 记录**写操作命令**（如`SET name xiaolin`），以追加方式写入文件。
  - 重启时重新执行AOF文件中的命令恢复数据。

- **写回策略**（`appendfsync`配置项）：

  | **策略**   | **写回时机**       | **优点**                 | **缺点**             |
  | :--------- | :----------------- | :----------------------- | :------------------- |
  | `Always`   | 每条命令后同步写盘 | 数据零丢失（最高可靠性） | 性能差（频繁磁盘IO） |
  | `Everysec` | 每秒批量写盘       | 性能与可靠性平衡         | 宕机可能丢失1秒数据  |
  | `No`       | 由操作系统控制写盘 | 性能最佳                 | 宕机可能丢失大量数据 |

- **优点**：

  - 数据安全性高（支持多种同步策略）。
  - 支持**AOF重写**（压缩日志，移除冗余命令）。
  - 损坏可修复（`redis-check-aof`工具）。

- **缺点**：

  - 文件体积大（记录所有写操作）。
  - 恢复速度慢（需逐条执行命令）。

#### **2. RDB（Redis Database）快照**

- **实现方式**：
  - 定期生成内存数据的**二进制快照**（默认文件`dump.rdb`）。
  - 恢复时直接加载快照文件到内存。
- **触发命令**：
  - `SAVE`：阻塞主线程生成快照（影响性能）。
  - `BGSAVE`：fork子进程生成快照（推荐，非阻塞）。
- **优点**：
  - 文件体积小（二进制压缩格式）。
  - 恢复速度快（直接加载数据）。
  - 对性能影响小（子进程生成快照）。
- **缺点**：
  - 数据可能丢失（两次快照间的数据未持久化）。
  - 频繁快照可能阻塞主线程（fork子进程开销）。

#### **3. AOF vs RDB核心对比**

| **对比项**     | **AOF**              | **RDB**            |
| :------------- | :------------------- | :----------------- |
| **数据安全性** | 高（可配置同步策略） | 低（依赖快照间隔） |
| **文件体积**   | 大（记录命令）       | 小（二进制压缩）   |
| **恢复速度**   | 慢（需重放命令）     | 快（直接加载数据） |
| **性能影响**   | 高（频繁磁盘IO）     | 低（子进程处理）   |

#### **4. 如何选择？**

- **追求高可靠性**：优先使用`AOF`（`appendfsync always`）。
- **追求高性能**：使用`RDB`或`AOF + Everysec`。
- **混合持久化**（Redis 4.0+）：
  - 开启`AOF`，并定期使用`BGSAVE`生成RDB快照。
  - 重启时先加载RDB快照，再重放AOF增量命令。

#### **5. 记忆技巧**

- **AOF**：“日志追加保安全，重放命令恢复慢”。
- **RDB**：“快照一键存内存，丢了数据别喊疼”。

## 缓存淘汰和过期删除

### 过期删除策略和内存淘汰策略有什么区别？

1. **内存淘汰策略（Memory Eviction Policy）**

   - 是在**内存达到 maxmemory 限制时触发**的机制。
   - 目的是**释放内存空间**，以便能继续写入新的数据。
   - Redis 提供了多种淘汰策略（由 `maxmemory-policy` 配置）：
     - `noeviction`：写操作报错，不再淘汰键。
     - `allkeys-lru`：对所有键使用 LRU 算法淘汰最少使用的键。
     - `volatile-lru`：只对设置了过期时间的键使用 LRU 淘汰。
     - `allkeys-random`：随机淘汰所有键。
     - `volatile-random`：随机淘汰设置了过期时间的键。
     - `volatile-ttl`：优先淘汰 TTL 值（剩余生存时间）较小的键。
   - 特点：**仅在内存不足时触发**，和键是否过期无关。

2. **过期键删除策略（Key Expiration Policy）**

   - 是针对**设置了过期时间的键**，在键过期时进行删除的机制。
   - Redis 采用两种方式结合的策略来删除过期键：
     1. **定期删除（定时任务）**
        - Redis 每隔一段时间**随机抽取一部分设置了过期时间的键**，检查并删除已过期的键。
        - 控制 CPU 占用，避免影响主线程性能。
     2. **惰性删除（访问触发）**
        - 每当访问某个键时，**先检查其是否过期**，如果过期则在访问时立刻删除。
   - 特点：**只对设置了过期时间的键有效**，并非因内存不足而触发。

3. **两者的区别总结**

   | 对比项               | 内存淘汰策略                   | 过期键删除策略              |
   | -------------------- | ------------------------------ | --------------------------- |
   | 触发条件             | 内存达到限制                   | 键的过期时间已到            |
   | 是否需要设置过期时间 | 否（所有键）                   | 是（仅对设置了 TTL 的键）   |
   | 目的                 | 腾出内存，继续接收写请求       | 保证过期数据及时删除        |
   | 删除方式             | 主动触发，按照策略删除一部分键 | 定时 + 惰性检查并删除过期键 |

### 介绍一下Redis 内存淘汰策略

1. **内存淘汰策略概述**

   - Redis 的内存淘汰策略用于**在内存使用超过 `maxmemory` 限制时**，决定如何处理新的写入请求。
   - 若没有设置淘汰策略并超出内存限制，Redis 会返回错误。
   - Redis 提供了 **8 种内存淘汰策略**，可通过配置项 `maxmemory-policy` 设置。
   - 这 8 种策略分为两大类：
     - **不进行数据淘汰（1 种）**
     - **进行数据淘汰（7 种）**

2. **不进行数据淘汰的策略**

   - **noeviction**
     - Redis 3.0 之后的默认策略。
     - 内存超出 `maxmemory` 限制时，**不再淘汰任何键**，而是直接返回错误。
     - 如果只是执行查询或删除等读操作，Redis 可正常工作；一旦写入操作超限，将报错。

3. **进行数据淘汰的策略**

   - 可细分为以下两类：

   **3.1 在设置了过期时间的数据中进行淘汰**

   - **volatile-random**：随机淘汰设置了过期时间的任意键。
   - **volatile-ttl**：优先淘汰 TTL 剩余时间最短（即最快过期）的键。
   - **volatile-lru**：根据 LRU（Least Recently Used）原则，淘汰最久未使用的、设置了过期时间的键。
   - **volatile-lfu**：根据 LFU（Least Frequently Used）原则，淘汰最少使用的、设置了过期时间的键（Redis 4.0 新增）。

   **3.2 在所有数据范围内进行淘汰**

   - **allkeys-random**：随机淘汰任意键（无论是否设置过期时间）。
   - **allkeys-lru**：从所有键中淘汰最久未使用的键。
   - **allkeys-lfu**：从所有键中淘汰使用频率最低的键（Redis 4.0 新增）。

4. **策略选择建议**

   - 若希望 Redis 作为缓存系统，推荐使用：
     - `allkeys-lru`：命中率高，清除冷数据，最常用。
     - `allkeys-lfu`：适合有热点数据场景，保留访问频繁的键。
   - 若设置了过期时间，并希望只清除过期类数据，推荐：
     - `volatile-lru` 或 `volatile-ttl`

### 介绍一下Redis过期删除策略？

**1. 过期键的两种删除策略**

Redis采用**惰性删除 + 定期删除**组合策略，平衡CPU资源与内存效率。

**2. 惰性删除（Lazy Expiration）**

- **触发时机**：
  在客户端访问或修改某个key时，Redis会先调用`expireIfNeeded()`函数检查该key是否过期。

- **执行逻辑**：

  ```
  int expireIfNeeded(redisDb *db, robj *key) {
      if (!keyIsExpired(db,key)) return 0;  // 未过期直接返回
      if (server.lazyfree_lazy_expire) 
          dbAsyncDelete(db,key);  // 异步删除（Redis 4.0+）
      else 
          dbSyncDelete(db,key);   // 同步删除
      return 1;
  }
  ```

- **流程**：

  ```
  客户端请求 → 检查key是否过期 → 是 → 删除key并返回null  
                              否 → 返回key的值  
  ```

- **特点**：

  - **优点**：对CPU友好，只在访问时检查。
  - **缺点**：若key长期不被访问，可能导致内存泄漏。

**3. 定期删除（Active Expiration）**

- **触发机制**：

  - 默认每秒执行10次（通过`hz 10`配置），每次随机抽取**20个key**检查。
  - 源码中定义`ACTIVE_EXPIRE_CYCLE_LOOKUPS_PER_LOOP=20`。

- **执行流程**：

  ```
  1. 随机抽取20个key → 删除过期key  
  2. 若过期key占比 >25%（即超过5个），重复步骤1  
  3. 若执行时间超过25ms或过期key比例≤25%，终止本轮检查  
  ```

- **伪代码逻辑**：

  ```
  while True:
      keys = random_sample(20)  # 随机抽取20个key
      expired_keys = delete_expired(keys)
      if len(expired_keys) <= 5 or time_limit_exceeded(25ms):
          break
  ```

- **特点**：

  - **优点**：减少内存浪费，主动清理过期key。
  - **缺点**：若过期key过多，可能短暂占用CPU资源。

**4. 策略对比与设计思想**

| **策略** | **触发条件**   | **优点**     | **缺点**             |
| :------- | :------------- | :----------- | :------------------- |
| 惰性删除 | 访问key时触发  | CPU占用低    | 内存可能泄漏         |
| 定期删除 | 周期性随机抽查 | 主动释放内存 | 高频检查可能影响性能 |

- **设计目标**：在**内存效率**和**CPU开销**间取得平衡。
- **实际应用**：
  - 惰性删除确保冷数据不会立即占用CPU。
  - 定期删除避免内存被大量过期key占满。

**5. 配置参数**

- **定期删除频率**：`hz 10`（默认每秒10次，可调整）。
- **异步删除开关**：`lazyfree-lazy-expire yes`（Redis 4.0+支持）。

**6. 记忆技巧**

- **惰性删除**：“不查不删，省CPU；访问才判，过期就砍”。
- **定期删除**：“每秒抽20，过25%再查；超25ms停，防CPU爆炸”。

### Redis缓存失效会不会立即删除？

1. **Redis 的过期键不会被立即删除**
   - Redis 并不会在键一过期时立即删除该键，而是采用两种策略组合来删除过期键：
     - **惰性删除（惰性策略）**
     - **定期删除（定时策略）**
2. **惰性删除策略（Lazy Deletion）**
   - Redis 不会主动删除过期键，而是在访问某个键（如 GET、SET、DEL）时，**先检查其是否过期**：
     - 若未过期：正常返回结果。
     - 若已过期：**此时才将该键删除**，并返回空结果或错误。
   - 优点：节省 CPU 资源。
   - 缺点：若过期键一直未被访问，将长期占用内存空间，导致内存泄漏风险。
3. **定期删除策略（Periodic Deletion）**
   - Redis 每隔一定时间执行一次任务：
     - **随机抽取一部分设置了过期时间的键**进行过期检查。
     - 删除其中已过期的键。
     - 控制了检查频率和每次检查的 key 数量，避免占用过多 CPU。
   - 优点：能清理掉一部分惰性删除未命中的过期键，避免内存膨胀。
4. **为什么 Redis 不选择“定期 + 立即删除所有过期键”？**
   - 在存在大量过期键的情况下：
     - 如果立即清理全部，会消耗大量 CPU 资源。
     - 特别是在内存充足但 CPU 紧张的情况下，**会显著影响 Redis 的响应延迟和吞吐量**。
   - 因此 Redis 采用“惰性 + 定期”的平衡策略：
     - 减少资源消耗；
     - 保持良好性能；
     - 适配不同的场景。

### Redis主从同步中的增量和完全同步怎么实现？

1. **Redis 主从复制的两种同步方式**

   - **全量同步（Full Resync）**：用于第一次连接或数据严重不一致的情况。
   - **增量同步（Partial Resync）**：用于短暂断线恢复后的继续同步，不必重新传全部数据。

2. **全量同步（Full Resync）**

   **2.1 触发时机**

   - 从服务器初次连接主服务器；
   - 从服务器断线重连但找不到偏移量或数据不一致；
   - 主服务器长时间未同步导致数据差异过大。

   **2.2 同步过程分为三阶段**

   - **第一阶段：建立连接，协商同步**
     - 从服务器发送 `PSYNC` 或 `SYNC` 命令；
     - 主服务器响应 `FULLRESYNC` 表示将进行全量同步。
   - **第二阶段：主服务器将当前快照发给从服务器**
     1. 主服务器执行 `BGSAVE` 生成 RDB 快照；
     2. 将 RDB 文件发送给从服务器；
     3. 从服务器清空现有数据，载入新 RDB 文件。
   - **第三阶段：主服务器发送写操作命令**
     - 主服务器会在生成 RDB 快照期间，将新写入的命令记录到 `replication backlog buffer`；
     - RDB 传输完成后，再将这些命令发送给从服务器；
     - 确保主从数据一致性。

   **2.3 实现步骤小结**

   1. 从服务器发送 SYNC；
   2. 主服务器生成 RDB 快照；
   3. 主服务器传输 RDB 文件；
   4. 从服务器加载快照；
   5. 主服务器记录期间写命令；
   6. 主服务器将 backlog 中写命令传给从服务器。

3. **增量同步（Partial Resync）**

   **3.1 核心机制**

   - 基于 `PSYNC runID offset` 命令；
   - 利用运行 ID 与复制偏移量（offset）判断是否能继续同步；
   - 依赖两个关键组件：
     - `replication offset`：主从双方当前同步进度；
     - `repl_backlog_buffer`：主服务器环形缓冲区，保存最近执行的写操作命令。

   **3.2 同步过程**

   - 网络断开 → 从服务器重连 → 发送 `PSYNC runID offset`；
   - 若 offset 范围仍在主服务器的 `repl_backlog_buffer` 中：
     - 直接发送缺失数据 → 增量同步完成；
   - 否则，数据被覆盖 → 回退为全量同步。

   **3.3 repl_backlog_buffer 说明**

   - 默认大小为 1MB；
   - 是环形结构，数据满会覆盖旧数据；
   - 主服务器每次写操作都会同时写入这个缓冲区；
   - 可通过增大此区域来**减少频繁全量同步**的风险。

   **3.4 优化建议**

   - 在高频网络断开或主从复制波动较大场景中，建议：
     - 增大 `repl-backlog-size`；
     - 增加 `repl-timeout`；
     - 优化主从网络质量，减少重连频率。

4. **总结：全量 vs 增量同步**

   | 对比项       | 全量同步                     | 增量同步                        |
   | ------------ | ---------------------------- | ------------------------------- |
   | 触发时机     | 初次连接、数据丢失、差异过大 | 网络短暂断开，偏移量仍有效      |
   | 是否传输快照 | 是（RDB）                    | 否，只传增量写命令              |
   | 传输效率     | 慢，阻塞，网络消耗大         | 快，性能友好                    |
   | 关键机制     | RDB + backlog buffer         | PSYNC + backlog buffer + offset |
   | 风险点       | 快照生成时间久，占用资源     | backlog 数据覆盖会回退全量同步  |

### redis主从和集群可以保证数据一致性吗 ？

**1. CAP理论下的Redis定位**

Redis主从复制和集群属于**AP模型**（可用性 + 分区容忍性），在网络分区（如节点间通信中断）时：

- **优先保证可用性**：服务正常响应，但可能返回旧数据。
- **牺牲强一致性**：节点间数据可能出现短暂不一致。

**2. 主从复制的数据一致性**

- **异步复制**（默认）：
  - 主节点写入成功后立即返回客户端，再异步同步到从节点。
  - **问题**：若主节点宕机，未同步的写入会丢失（即使启用`replica-serve-stale-data no`，从节点仍可能提供旧数据）。
- **半同步复制**（Redis 5.0+，需配置）：
  - 主节点等待至少一个从节点确认接收数据后才返回成功。
  - **改进**：降低数据丢失风险，但性能下降。

**一致性场景示例**：

```
1. 客户端写入主节点A → A返回成功 → 数据未同步到从节点B时A宕机  
2. 故障切换后B成为新主节点 → 客户端读取到旧数据  
```

**3. Redis集群的数据一致性**

- **分片架构**：
  - 数据按槽（slot）分片，每个主节点负责部分槽。
  - 跨槽操作（如事务、Lua脚本）需所有相关节点在线，否则失败。
- **异步复制**：
  - 主节点写入后异步同步到从节点，同样存在数据丢失风险。
- **故障转移**：
  - 集群检测到主节点下线时，自动提升从节点为新主节点。
  - **脑裂问题**：网络分区可能导致多个主节点同时写入同一槽，数据冲突。

**一致性场景示例**：

```
1. 客户端向主节点A写入数据 → A同步到从节点A1前宕机  
2. 集群选举A1为新主节点 → 原A恢复后成为从节点，未同步的写入永久丢失  
```

**4. 如何提高一致性？**

| **方案**                  | **说明**                                                     | **缺点**                           |
| :------------------------ | :----------------------------------------------------------- | :--------------------------------- |
| **WAIT命令**              | 阻塞客户端直到N个从节点同步完成（例：`WAIT 1 5000`等待1个从节点，超时5秒） | 增加延迟，不保证持久化到磁盘       |
| **min-replicas-to-write** | 主节点至少需N个从节点在线才接受写入（配置项）                | 可用性降低（从节点不足时拒绝写入） |
| **Raft协议扩展**          | 如Redis Sentinel使用Raft选主，但数据同步仍依赖异步复制       | 非强一致，仅保证主节点唯一性       |

**5. 最终一致性 vs 强一致性**

- **Redis默认**：最终一致性（数据最终会同步，但时间不确定）。
- **强一致性需求**：需结合外部机制（如分布式锁、WAIT命令），或改用其他数据库（如ZooKeeper、Etcd）。

### 哨兵机制是什么？

1. **哨兵机制概述**

   - Redis 的主从结构为**读写分离**，写操作只能通过主节点（master）执行。
   - 当主节点宕机时，从节点虽能提供读服务，但**无法接收新的写入请求**，也无法接收主节点的复制指令。
   - 若不做处理，整个 Redis 服务将陷入“写挂死”状态。

2. **主从故障转移的挑战**

   - 传统方式需人工介入：
     - 选定从节点，手动将其升级为主节点；
     - 通知客户端和其他从节点切换连接；
     - 更新原主节点信息，成本高、易出错。

3. **哨兵机制（Sentinel）的作用**

   - Redis 从 **2.8 版本**起引入 Sentinel 哨兵机制，用于自动化处理主从故障。
   - 哨兵机制是由多个特殊 Redis 进程组成的集群，具备以下能力：
     - **自动监控主节点状态**；
     - **检测主节点故障并自动切换主从**；
     - **通知客户端更新主节点连接信息**。

4. **哨兵机制的三个核心职责**

   **4.1 监控（Monitor）**

   - 哨兵节点会持续对主节点和从节点进行健康检查；
   - 可配置心跳检测频率、超时阈值等参数；
   - 一旦主节点在超时时间内无响应，会被标记为“下线”。

   **4.2 选主（Elect）**

   - 若确定主节点已下线，多个哨兵通过投票协商，从现有从节点中选出一个新的主节点；
   - 使用**Raft 类似的投票机制**，确保最终只选出一个主节点；
   - 被选中的从节点将被升级为新的主节点。

   **4.3 通知（Notify）**

   - 哨兵将新的主节点信息广播给所有 Redis 客户端和其他 Redis 节点；
   - 确保集群中所有组件能够重新连上新的主节点；
   - 保障整个 Redis 集群的高可用性。

5. **哨兵节点本质**

   - 哨兵是一个运行在**独立进程中的 Redis 程序**；
   - 它既不是主节点也不是从节点，但它能与所有 Redis 节点通信；
   - 可以部署多个哨兵节点，组成哨兵集群，提升可靠性。

6. **哨兵机制示意总结**

   | 哨兵职责 | 说明                                 |
   | -------- | ------------------------------------ |
   | 监控     | 定期 ping 主从节点，检测是否宕机     |
   | 选主     | 主节点失效后，协调从节点投票选出新主 |
   | 通知     | 更新客户端和其他节点的主节点连接信息 |

### 哨兵机制的选主节点的算法介绍一下？

**1. 故障节点主观下线**

- **检测机制**：
  - 每个Sentinel节点定期向Redis集群所有节点发送心跳包（PING）。
  - 若主节点在`down-after-milliseconds`时间内未响应，Sentinel将其标记为**主观下线**。
- **特点**：
  - 仅代表当前Sentinel节点的局部判断，不触发故障转移。

**2. 故障节点客观下线**

- **确认流程**：
  - 发起主观下线的Sentinel节点会询问其他Sentinel节点对该主节点的状态判断。
  - 若超过`quorum`数量的Sentinel节点认为主节点主观下线，则标记为**客观下线**。
- **关键点**：
  - 仅当主节点客观下线时，才会触发后续的故障转移流程。

**3. Sentinel集群选举Leader**

- **选举规则**：
  - 每个Sentinel节点均可发起选举，请求其他节点投票。
  - 其他Sentinel节点若未同意过其他选举请求，则同意当前请求（投票+1）。
- **当选条件**：
  - 需获得至少`max(quorum, Sentinel节点数/2 + 1)`的票数。
  - **示例**：3个Sentinel节点，`quorum=2`，则需至少2票才能当选Leader。
- **失败处理**：
  - 若未达到票数要求，重新发起选举。

**4. Sentinel Leader选择新主节点**

- **筛选步骤**：
  1. **过滤故障节点**：排除网络不可用或状态异常的从节点。
  2. **优先级排序**：选择`slave-priority`（配置项）最高的从节点。
  3. **复制偏移量**：选择数据同步最接近原主节点的从节点（偏移量最大）。
  4. **RunID最小**：若以上条件相同，选择启动时生成的`runid`最小的从节点。
- **目标**：
  - 优先选择数据完整、性能稳定的从节点晋升为主节点。

**5. 故障转移完成**

- **新主节点生效**：
  - Sentinel Leader通知其他从节点复制新主节点。
  - 客户端更新路由信息，指向新主节点。

**6. 设计要点**

- **高可用性**：通过多Sentinel节点协作避免单点故障。
- **最终一致性**：新主节点的数据可能短暂落后原主节点（异步复制导致）。
- **配置建议**：
  - `quorum`值需根据Sentinel节点数量合理设置（通常为半数以上）。

### Redis集群的模式了解吗 优缺点了解吗？

1. **Redis 集群（Redis Cluster）概述**

   - Redis Cluster 是 Redis 提供的**分布式部署方案**，用于解决单机内存瓶颈与高可用问题。
   - 其核心思想是：**数据分片 + 去中心化节点通信 + 高可用主从机制**。

2. **Redis Cluster 的核心机制：哈希槽（Hash Slot）**

   - Redis Cluster 中一共定义了 **16384 个哈希槽（slot）**。
   - 每个 key 会被映射到一个哈希槽中，槽再由节点负责处理。
   - 哈希槽计算方式：
     1. 对 key 计算 CRC16 值，得到 16bit 整数；
     2. 对 16384 取模，结果 ∈ [0, 16383]，即为哈希槽编号。
   - 每个节点负责处理多个槽（slot），Redis Cluster 通过槽而不是 key 映射节点，避免频繁重分配。

3. **哈希槽的分配方式**

   - **自动平均分配**
     - 使用 `cluster create` 命令时，Redis 会自动将 16384 个槽平均分配给所有节点。
     - 例如有 4 个节点，则每个节点负责约 4096 个槽。
   - **手动指定分配**
     - 使用 `cluster addslots` 命令手动为每个节点分配槽；
     - 需确保 16384 个槽都被分配，否则 Redis 集群无法启动。

   **示例命令：**

   ```
   bash
   
   
   CopyEdit
   redis-cli -h 192.168.1.10 -p 6379 cluster addslots 0 1
   redis-cli -h 192.168.1.11 -p 6379 cluster addslots 2 3
   ```

4. **Redis Cluster 的优点**

   - **高可用性**：
     - 节点间采用主从结构，一旦主节点挂掉，可自动由从节点顶替，保障数据持久与服务连续。
   - **高并发性能**：
     - 数据分散在多个节点，读写压力被分担，适合应对海量数据与高并发访问。
   - **易扩展性**：
     - 支持动态添加/删除节点实现弹性扩容；
     - 还支持代理请求路由，增强可定制性。

5. **Redis Cluster 的缺点**

   - **部署与维护复杂**：
     - 涉及节点分布、槽位分配、主从搭配、容错策略等，部署门槛较高。
   - **同步一致性问题**：
     - 节点断线或宕机会导致数据复制异常；
     - 主从同步存在延迟，可能引发读写不一致，影响读请求的正确性。
   - **功能受限（跨节点限制）**：
     - 某些操作如多键事务、Lua 脚本、批量操作等，**不能跨槽或跨节点执行**；
     - 比如 `MGET key1 key2` 若不在同一槽上，将执行失败；
     - key 若频繁迁移节点，某些 TTL 或修改操作可能失效。

## 场景

### 为什么使用redis？

1. **高性能**

   - Redis 数据存储在**内存中**，读写速度极快；

   - 相比 MySQL 等基于磁盘的关系型数据库，访问延迟大幅降低；

   - 常用于将热点数据缓存于 Redis，提升数据读取效率；

   - 应用场景：**缓存查询结果、热点数据访问、会话数据等**。

**示例：数据库缓存加速流程**

```
第一次请求：
  Web 应用 → 读取数据库 → 数据写入 Redis 缓存 → 返回结果

后续请求：
  Web 应用 → 读取 Redis 缓存（命中）→ 返回结果（无需访问数据库）
```

- 需要注意缓存一致性问题，即：MySQL 中数据变化后，需要及时更新 Redis 中的缓存数据，避免出现脏读。

2. **高并发**

   - Redis 是基于单线程的 I/O 多路复用模型，性能强劲；

   - **单机 QPS 可轻松突破 10 万**，远高于 MySQL 的约 1 万；

   - 更适合应对大并发、高吞吐的业务场景；

   - 可承载高频访问请求，将数据库压力迁移至缓存层。

3. **典型应用场景**

   - 秒杀/抢购系统：库存状态写入 Redis，抗高并发；

   - 排行榜、计数器、点赞：使用 Redis 原子自增/有序集合快速处理；

   - 限流系统：通过 Redis 实现滑动窗口或令牌桶限流逻辑。

### **Redis 比 MySQL 快的原因分析**

**1. 内存存储 vs 磁盘存储**

- **Redis**：
  - 数据存储在内存中，读写速度极快（纳秒级）。
  - 仅通过**异步持久化**（如RDB/AOF）将数据写入磁盘，避免阻塞主线程。
- **MySQL**：
  - 数据主要存储在磁盘上，即使有缓冲池（Buffer Pool），仍需频繁触发磁盘I/O。
  - 写操作需保证**ACID**，必须刷盘（如redo log），导致延迟较高。

**性能对比**：
Redis的读写速度通常是MySQL的**10~100倍**（内存访问 vs 磁盘I/O）。

**2. 数据结构与查询复杂度**

- **Redis**：
  - 提供**简单数据结构**（如String、Hash、List），操作时间复杂度低：
    - Hash查询：`O(1)`（直接通过key访问）。
    - 有序集合（Zset）范围查询：`O(log N)`（跳表实现）。
  - 无复杂关联查询，适合**单一键值操作**。
- **MySQL**：
  - 基于**B+树索引**，查询复杂度为`O(log N)`。
  - 需处理**多表关联**、事务隔离等，额外开销大。

**适用场景**：

- Redis适合**高频单键访问**（如缓存、计数器）。
- MySQL适合**复杂查询**（如JOIN、聚合分析）。

**3. 线程模型**

- **Redis**：
  - **单线程模型**（6.0+支持多线程I/O，但核心逻辑仍单线程）：
    - 避免多线程竞争锁、上下文切换开销。
    - 无死锁问题，代码更简单高效。
  - 通过**I/O多路复用**（epoll）处理高并发连接。
- **MySQL**：
  - **多线程模型**（连接池+线程池）：
    - 每个查询可能涉及锁竞争（如行锁、表锁）。
    - 线程切换和同步机制带来额外开销。

**并发性能**：
Redis单线程可轻松支持**10万+ QPS**，MySQL通常仅**数千QPS**（需优化配置）。

**4. 其他优化因素**

- **协议与网络**：
  - Redis使用**RESP协议**（二进制安全），传输效率高。
  - MySQL使用**文本协议**（如SQL语句），解析开销大。
- **持久化机制**：
  - Redis可关闭持久化，纯内存操作。
  - MySQL必须保证**事务持久性**（如redo log刷盘）。

**5. 何时选择Redis或MySQL？**

| **场景**         | **Redis**            | **MySQL**             |
| :--------------- | :------------------- | :-------------------- |
| **高频读写**     | ✅ 缓存、会话存储     | ❌ 磁盘I/O瓶颈         |
| **复杂查询**     | ❌ 不支持JOIN、子查询 | ✅ 关系型数据分析      |
| **强一致性需求** | ❌ 最终一致性         | ✅ ACID事务支持        |
| **高并发低延迟** | ✅ 单线程+内存存储    | ❌ 锁竞争和磁盘I/O限制 |

### 本地缓存和Redis缓存区别？

1. **本地缓存与 Redis 缓存的区别**
   - 本地缓存：数据缓存在应用本地内存或服务器本地；
   - Redis 缓存：数据缓存在远端 Redis 服务中，属于分布式缓存；
   - 两者在访问速度、扩展性、部署成本等方面各有优劣。
2. **本地缓存的优点**
   - **访问速度极快**：缓存存在本地内存中，读取无网络通信，**延迟低、吞吐高**；
   - **网络依赖性低**：不依赖外部服务，网络异常时仍可访问本地缓存；
   - **系统资源消耗低**：避免频繁网络请求，减轻服务端与网络带宽负担；
   - **适用场景**：适用于请求量大、实时性要求高、缓存更新频率低的单体或轻量服务。
3. **本地缓存的缺点**
   - **扩展性差**：受限于单机内存，**不支持大数据量的缓存**；
   - **数据不一致**：多节点部署时，各节点本地缓存无法自动同步，容易造成缓存不一致；
   - **管理复杂**：需自己处理缓存清理策略、失效机制，缺乏集中化运维手段。
4. **Redis 缓存（分布式缓存）的优点**
   - **强扩展性**：Redis 支持水平扩展，能处理海量数据和高并发请求；
   - **数据一致性好**：支持发布订阅、事件通知、分布式锁、复制协议等机制保障一致性；
   - **集中管理维护方便**：部署独立服务统一运维，支持定期淘汰策略、持久化等；
   - **支持多种数据结构和操作**：如 List、Set、Hash、ZSet、Bitmaps 等；
   - **适用场景**：微服务架构、大型系统、分布式部署环境。
5. **Redis 缓存的缺点**
   - **网络依赖强**：读写 Redis 需要额外网络开销，**受网络稳定性影响较大**；
   - **访问延迟略高于本地缓存**：虽远程通信开销较小，但比本地访问稍慢；
   - **部署成本更高**：需额外部署 Redis 服务，涉及主从、集群、哨兵等组件配置；
   - **运维复杂度高**：需要处理分布式一致性、节点故障转移、扩容等问题。
6. **本地缓存 vs Redis 缓存对比表**

| 对比项     | 本地缓存           | Redis 缓存（分布式缓存） |
| ---------- | ------------------ | ------------------------ |
| 访问速度   | 极快（内存读取）   | 较快（网络通信 + 内存）  |
| 网络依赖   | 无需网络           | 依赖网络                 |
| 数据一致性 | 多节点一致性差     | 支持集群协议保障一致性   |
| 扩展性     | 差，受限于本地资源 | 强，支持水平扩展         |
| 部署维护   | 简单，无需额外服务 | 复杂，需部署 Redis 服务  |
| 适用场景   | 小规模、单体服务   | 分布式系统、海量数据     |

1. **选型建议
   - 若访问数据量小、读频繁写少，可用本地缓存（如 Guava Cache、Caffeine）；
   - 若需支持多服务共享缓存、缓存数据较大、需高可用或分布式场景，应选 Redis 缓存；
   - 实际开发中常用 “**本地缓存 + Redis 缓存**” 组合模式，提升缓存命中率并保障数据一致性。

### Redis应用场景？

1. **Redis 的常见应用场景**

   Redis 是一种**高性能的内存型键值数据库**，由于其读写速度极快、丰富的数据结构和功能支持，在以下典型场景中广泛应用：

2. **缓存系统（最常见场景）**

   - 将热点数据缓存在 Redis 内存中，**减轻数据库压力，提升访问速度**；
   - 适合页面缓存、接口缓存、查询结果缓存等；
   - 常结合设置过期时间（TTL）+ 持久化策略使用；
   - 注意处理缓存穿透、击穿、雪崩等问题。

3. **排行榜 / 排名系统**

   - 利用 Redis 提供的 **有序集合（ZSet）** 实现排行榜功能；
   - 支持按分数排序、范围查询等操作，常用于：
     - 游戏排行、积分榜、热搜榜等；
   - 示例命令：`ZADD`、`ZRANGE`、`ZREVRANGE`。

4. **分布式锁**

   - Redis 提供原子性操作命令（如 `SET key value NX EX`）；
   - 可用于实现**分布式锁机制**，保障多线程/多节点间的互斥访问；
   - 配合 `Lua` 脚本、RedLock 等实现高可用强一致的分布式锁；
   - 适用场景：库存控制、任务排他执行、订单超卖防控等。

5. **计数器 / 实时统计**

   - Redis 支持高效的自增自减操作，如 `INCR`、`DECR`；
   - 常用于实现网站访问量、点赞数、浏览量、活跃人数统计等；
   - 单线程 + 内存操作确保了**高并发下数据安全性与性能**。

6. **消息队列**

   - Redis 支持 `List` 结构和 `Pub/Sub（发布订阅）` 功能；
   - 可用作轻量级消息队列，支持实时消息推送与处理；
   - 示例场景：日志异步写入、任务分发、事件通知等；
   - 对比专业 MQ：适合中小型、非强可靠性需求的消息场景。

7. **会话存储（Session 缓存）**

   - 可用于存储用户 Session、Token、登录状态等；
   - 配合 TTL 设置实现会话自动过期；
   - 保证高并发登录、登出、验证操作的性能。

8. **限流器 / 滑动窗口 / 令牌桶**

   - 结合 Redis 的计数、自增、过期等功能；
   - 可实现 API 限流、登录限制、刷单防控等安全机制；
   - 支持滑动窗口算法、漏桶、令牌桶等模型实现。

### Redis除了缓存，还有哪些应用?

1. **Redis 实现消息队列**

   1.1 **使用 Pub/Sub 模式**

   - Redis 的发布订阅（Pub/Sub）是一种基于频道的消息模式，发布者将消息发送到频道，所有订阅该频道的客户端都能接收。
   - 优点是实现简单，支持多客户端订阅同一频道，适合实时通知类场景。
   - 缺点是不支持消息持久化、无法保证消息顺序和可靠性，订阅者离线会丢消息。

   1.2 **使用 List 结构**

   - 通常使用 LPUSH 将消息加入列表，用 BLPOP 或 BRPOP 阻塞方式消费，实现简单的任务队列（先进先出）。
   - 可结合 Redis 的 TTL 实现延迟队列。
   - 结合 Redis 事务可保证操作的原子性。
   - 客户端需自行实现消息确认、重试机制等，功能不如专业消息队列系统完备。

2. **Redis 实现分布式锁**

   2.1 **使用 set nx 实现**

   - 基于 SET 命令实现抢锁，命令格式：`SET key value NX PX milliseconds`
   - NX 表示 key 不存在时才设置；PX 表示过期时间（毫秒）。
   - 加锁成功后客户端持有锁，完成操作后需判断锁是否为自己设置的，再进行删除。
   - 需保证加锁与解锁操作的原子性，通常使用 Lua 脚本实现。

   2.2 **RedLock 算法**

   - Redis 作者提出的分布式锁算法，基于多个 Redis 节点进行加锁，提高分布式场景下的可靠性。
   - 客户端需在多数（如 3/5）节点加锁成功才算加锁成功。
   - 加锁时间必须小于所有实例中加锁成功所花的最长时间，以避免死锁。
   - 能有效应对单点故障和网络延迟带来的问题，提高锁的可用性和安全性。

### Redis支持并发操作吗？

1. **支持单命令并发操作**
   - Redis 单个命令具有原子性，意味着命令要么完全执行成功，要么完全不执行。
   - 即使多个客户端并发访问同一个 key，Redis 也能通过单线程顺序执行命令，保证操作一致性。
   - 因此，Redis 天然支持**高并发场景下的安全读写**。
2. **支持多命令并发操作（通过事务）**
   - Redis 支持事务，可以将多个命令放在一个事务中顺序执行。
   - 使用 MULTI 开启事务，EXEC 提交事务，WATCH 可监控键变化防止并发冲突。
   - 多条命令在事务中统一执行，避免并发时出现中间状态问题。
   - 适用于需要保持一组操作原子性的一致性场景。

### Redis分布式锁的实现原理？什么场景下用到分布式锁？

1. **什么场景下需要用到分布式锁**

   - 在分布式系统中，如果多个服务实例可能同时访问同一份共享资源（如库存、订单、任务队列等），就需要通过分布式锁机制来保证**同一时刻只有一个实例能够操作该资源**。
   - 常见使用场景包括：
     - 秒杀/抢购系统防止超卖；
     - 任务调度系统避免重复执行；
     - 分布式定时任务只运行一次；
     - 多节点控制流程互斥执行。

2. **Redis 分布式锁的实现原理**

   2.1 **加锁实现**

   - 使用 Redis 提供的原子命令 `SET key value NX PX ttl` 实现加锁逻辑。

   - 含义说明：

     - key 表示锁的标识；
     - value 是客户端生成的唯一标识，用于区分不同客户端；
     - NX 表示 key 不存在时才进行设置，确保互斥；
     - PX 设置过期时间，防止客户端异常导致死锁。

   - 示例：

     ```lua
     SET lock_key unique_value NX PX 10000
     ```

   2.2 **加锁成功与失败判断**

   - 如果返回 OK，表示加锁成功；
   - 如果返回 nil，表示锁已存在，加锁失败。

   2.3 **解锁实现**

   - 解锁前需要先判断当前锁的持有者是否是自己（通过比对 value）；

   - 如果是自己，再执行删除 key 操作，防止误删他人锁；

   - Redis 无法保证 “GET 判断 + DEL 删除” 的原子性，因此使用 Lua 脚本实现：

     ```lua
     if redis.call("get", KEYS[1]) == ARGV[1] then
         return redis.call("del", KEYS[1])
     else
         return 0
     end
     ```

   - Redis 执行 Lua 脚本是原子操作，从而保证了解锁过程的安全性和正确性。

3. **实现 Redis 分布式锁需满足的三个条件**

   - 加锁操作必须是原子性的；
   - 锁必须设置过期时间，防止死锁；
   - 解锁必须只能由加锁客户端执行，不能误删别人的锁。

### Redis的大Key问题是什么？

1. **什么是 Redis 的大 Key 问题**
   - Redis 大 Key 指的是某个 key 对应的 value 占用内存空间过大，导致 Redis 性能下降或运行异常；
   - 常见影响包括：内存不均、访问延迟、阻塞主线程、主从同步延迟等问题。
2. **多大的数据量才算是大 Key**
   - 没有绝对标准，通常认为以下两种情况构成大 Key：
     - 字符串类型的 value 大于 1MB；
     - 集合类（如 list、set、zset、hash）元素数量超过 1 万条。
   - 是否为大 Key，还需结合业务场景综合判断。
3. **为什么大 Key 是问题**
   - Redis 是单线程处理请求，当访问或删除大 Key 时会阻塞线程，影响其他请求；
   - 大 Key 在主从同步、持久化过程中也会造成延迟和阻塞；
   - 在集群模式下，大 Key 还会造成槽位负载不均，影响集群扩展性和稳定性。
4. **如何判断和规避大 Key 问题**
   - 使用 `MEMORY USAGE key` 命令查看单个 key 占用内存；
   - 对集合类 key 尽量拆分或分页处理；
   - 结合业务性能要求，设定合理的大 Key 阈值，例如高并发下不超过 100KB；
   - 使用监控工具或慢查询日志及时发现大 Key 风险。

### 大Key问题的带来的问题？

1. **内存占用过高**
   - 大 Key 会占用大量内存，可能导致可用内存不足，从而触发内存淘汰策略；
   - 在极端情况下，可能导致 Redis 实例崩溃，影响系统稳定性。
2. **性能下降**
   - 大 Key 会造成内存碎片增多，影响 Redis 的访问效率；
   - 对大 Key 的操作（如读取、写入、删除）会消耗较多 CPU 与内存资源，进一步降低系统整体性能。
3. **阻塞其他操作**
   - 删除或访问大 Key 的操作耗时较长，可能阻塞 Redis 主线程，导致其他客户端请求无法及时响应；
   - 可能影响系统的吞吐量与响应时间。
4. **网络拥塞**
   - 每次访问大 Key 会产生大量数据传输，易造成网络带宽拥堵；
   - 例如：一个大 Key 为 1MB，若每秒访问 1000 次，就会产生 1000MB 的流量。
5. **主从同步延迟**
   - Redis 配置主从同步时，大 Key 会导致同步过程传输数据量大，从而产生延迟；
   - 增加主从数据不一致风险。
6. **数据倾斜**
   - 在 Redis 集群模式下，大 Key 会导致内存使用不均，某些节点内存负载过重；
   - 破坏哈希槽分布的均衡性，甚至引发 key 被强制淘汰、Redis 实例 OOM（内存溢出）。

### 大key如何解决？

1. **对大 key 进行拆分**
   - 将包含大量成员的一个大 key 拆成多个小 key；
   - 例如将一个大 hash 拆分为多个 hash，每个成员数控制在合理范围；
   - 在 Redis 集群架构中，拆分可提升数据分片间的内存均衡，防止数据倾斜。
2. **对大 key 进行清理**
   - 将不再使用或超出 Redis 承载能力的历史数据删除；
   - 可将数据迁移至其他存储（如数据库、对象存储）；
   - 删除过程需注意使用异步删除操作，避免阻塞 Redis 主线程。
3. **监控 Redis 的内存水位**
   - 配置监控系统，设置内存使用率告警阈值，例如超过 70% 报警；
   - 同时监控 Redis 一小时内内存增长率，若超出 20% 也应及时处理；
   - 便于提前识别大 key 造成的异常内存占用。
4. **定期清理过期数据**
   - 大量过期数据堆积会造成单 key 数据膨胀，形成大 key；
   - 例如 hash 中不断追加点赞、计数等数据；
   - 可通过定时任务或脚本定期清理失效字段，提升存储效率。

### 什么是热key？如何解决热key问题？

1. **什么是热 key**

   - 热 key 是指在短时间内被高频访问的 key，通常通过访问频率或资源占用比例来判断。

   - 常见判断标准如下：

     1.1 **QPS 集中在单 key 上**

     - Redis 实例总 QPS 为 10000，而其中一个 key 每秒访问量达到 7000，说明该 key 是热 key。

     1.2 **带宽使用集中在单 key 上**

     - 比如一个拥有上千个成员、体积达 1MB 的 hash key 每秒多次触发 `HGETALL` 请求，导致带宽异常消耗。

     1.3 **CPU 时间集中在单 key 上**

     - 比如一个 zset 类型的 key 每秒执行大量 `ZRANGE` 操作，造成 CPU 开销高度集中。

2. **如何解决热 key 问题**

   2.1 **在集群中进行热 key 拆分与复制**

   - 由于 Redis 集群中热 key 难以迁移或打散访问，可以对其进行多份复制；
   - 将热 key 拷贝出多个副本如 foo2、foo3、foo4，分配到不同数据分片，分担单点压力。

   2.2 **使用读写分离架构**

   - 如果热 key 来源于读操作，可以通过增加只读节点，将读请求分发到多个副本；
   - 可结合 LVS、Proxy 等组件做负载均衡，提升整体吞吐能力；
   - 但读写分离部署复杂，需注意数据一致性、节点数膨胀和故障转移处理等问题。

### 如何保证 Redis 和 MySQL 缓存一致性问题？

1. **问题背景**

   - Redis 作为缓存系统牺牲强一致性以提升读写性能，属于 CAP 理论中的 AP；
   - 与数据库（如 MySQL）组合使用时，更新时存在短暂数据不一致的风险；
   - 需要通过设计合理的更新策略确保最终一致性。

2. **读写策略选择**

   - 常用策略为：**读：Cache Aside（旁路缓存）策略**，**写：先更新数据库，再删除缓存**；
   - 读操作流程：先查 Redis，未命中则查数据库并写入缓存；
   - 写操作流程：先更新数据库，再删除 Redis 缓存，确保数据不脏读。

3. **删除缓存存在的问题**

   - 删除失败：若删除缓存失败会导致脏数据残留；
   - 延迟删除：缓存未及时失效，可能读到旧数据；
   - 延迟设置过长：浪费内存，影响更新时效；
   - 延迟设置过短：频繁击穿缓存，失去缓存意义。

4. **解决删除失败的两种方案**

   4.1 **引入消息队列，重试删除缓存**

   - 将删除缓存操作写入消息队列，由消费者读取后执行删除操作；
   - 若删除失败，可通过重试机制再次读取并删除缓存；
   - 确保最终一致性，但业务代码入侵性较高。

   4.2 **订阅 binlog，异步删除缓存**

   - 通过 Canal 订阅 MySQL 的 binlog 日志，获取数据库变更事件；
   - Canal 模拟从节点读取 binlog，转化为结构化消息供消费者使用；
   - 下游服务根据变更记录异步删除缓存，保证数据同步。

### 缓存雪崩、击穿、穿透是什么？怎么解决？

**1. 缓存雪崩**

**概念：**
 大量缓存数据在同一时间过期，或 Redis 故障不可用，导致大量请求同时落入数据库，形成访问风暴，引发系统雪崩。

**后果：**

- 数据库压力瞬间陡增，可能崩溃宕机
- 请求响应时间激增，用户体验下降
- 服务级联失败，系统整体不可用

**解决方案：**

1. **均匀设置过期时间：**
   - 给不同 key 设置不同过期时间，增加一定随机值，避免同一时间大量缓存过期。
2. **加互斥锁重建缓存（单线程构建）：**
   - 业务线程查询缓存不存在时，加锁保证同一时间内只有一个线程能构建缓存，其他线程等待。
3. **缓存“永不过期”+后台异步更新：**
   - 缓存设置为长期有效，业务线程不主动删除，定时任务或消息队列异步更新缓存内容。

**2. 缓存击穿**

**概念：**
 某个热点 key 在瞬间失效，导致大量请求同时穿透缓存访问数据库，造成数据库压力过大。

**后果：**

- 热点数据失效导致请求瞬间打爆数据库
- 单点压力聚集，影响服务稳定性

**解决方案：**

1. **互斥锁控制重建：**
   - 加锁保证只有一个线程能构建缓存，其他线程等待或返回默认值。
2. **热点 key 提前续期 / 预热：**
   - 提前更新热点缓存，避免同时过期。

**3. 缓存穿透**

**概念：**
 用户请求访问的数据，在缓存和数据库中都不存在，恶意攻击或无效请求穿透缓存层直接打数据库。

**后果：**

- 缓存无法命中，所有请求打到数据库
- 易被攻击利用，造成数据库资源耗尽

**解决方案：**

1. **非空值缓存（缓存空对象）：**
   - 对不存在的数据也缓存一个“空值”标识，下次直接返回默认值。
2. **布隆过滤器 / 白名单校验：**
   - 使用布隆过滤器提前拦截非法 key，减少无效访问。
3. **接口参数校验 / 限流防刷：**
   - API 层提前校验参数有效性、增加验证码、人机识别、限流保护等。 

### 布隆过滤器原理？

**布隆过滤器原理（进阶版）**

**1. 本质原理：空间换时间，位图 + 多哈希函数**

- 布隆过滤器由一个长度为 m 的位数组 `bit[]` 和 k 个独立哈希函数组成；
- 每个哈希函数都能将输入映射为 0~m-1 范围内的整数；
- 插入和查询均不存储原始数据，仅通过位图判断「可能存在」或「一定不存在」。

------

**2. 插入操作（以 key 为例）**

步骤如下：

1. 对 key 依次使用 k 个哈希函数，得到 k 个位置；
2. 将这些位置上的 bit 全部置为 1。

🔁 插入是幂等操作（重复插入无影响），但不同 key 的哈希冲突可能导致误判风险上升。

------

**3. 查询操作（仍以 key 为例）**

步骤如下：

1. 对 key 使用相同的 k 个哈希函数，得到 k 个位置；
2. 依次查看这 k 个位置的 bit 值：
   - 若存在某一位为 0 ⇒ key 一定不存在；
   - 若全部为 1 ⇒ key 可能存在（存在误判概率）。

------

**4. 误判率计算公式**

误判率 ≈ `(1 - e^(-kn/m))^k`

- `n` 是已插入元素个数
- `m` 是位图长度
- `k` 是哈希函数个数

✅ 给定 n 和 m，最佳的 k ≈ `ln2 × (m/n)`，此时误判率最低。

------

**5. 特点总结**

- ✅ 空间效率极高，适用于超大数据量场景（10 亿级布隆过滤器也只有几十 MB）；
- ✅ 插入 & 查询 O(k)，k 很小，性能极快；
- ❌ 无法删除元素（除非使用 Counting Bloom Filter）；
- ❌ 只能回答「一定不存在」或「可能存在」，存在误判。

------

**6. 应用场景**

- ✅ 黑名单判断（如垃圾邮件/非法请求）
- ✅ Redis 缓存穿透防护（判断请求 key 是否为非法 key）
- ✅ 分布式系统避免重复上传、数据去重（如 CDN 资源、短链接）
- ✅ 爬虫系统的 URL 去重、反爬机制前置过滤
- ✅ 大规模数据库分库前的预检索优化

------

**7. 与布谷鸟过滤器的区别（拓展点，面试爱问）**

| 特性         | 布隆过滤器           | 布谷鸟过滤器         |
| ------------ | -------------------- | -------------------- |
| 是否支持删除 | ❌ 不支持             | ✅ 支持删除           |
| 误判率       | 有，固定             | 更低，可调整         |
| 查询效率     | 快（O(k)）           | 更快（O(1)）         |
| 适用场景     | 一致性优先，写多读少 | 动态变更频繁，删改多 |

### 秒杀场景下的高并发与超卖问题的解决方案？

#### **一、问题是什么？**

秒杀场景具备以下典型特征：

- **高并发访问**：大量用户在极短时间内同时发起请求；
- **库存有限**：商品数量有限，易造成竞争；
- **数据一致性要求高**：不能出现**超卖**、**重复下单**等问题；
- **响应速度要求高**：用户秒杀体验需流畅迅速。

**主要问题**包括：

1. **库存超卖**：多个用户同时扣减库存，最终数据库库存可能小于 0；
2. **性能瓶颈**：大量请求直接打到数据库，数据库连接数耗尽、系统崩溃；
3. **请求排队 & 卡顿**：并发争抢资源，锁竞争激烈，响应变慢或失败；
4. **数据不一致**：分布式系统中并发更新库存，可能造成不一致。

#### **二、为什么要这么做？（做法背后的合理性）**

秒杀核心挑战是“**高并发下的并发控制 + 性能优化 + 数据一致性保障**”。

因此合理的设计思路应满足：

- **限流 + 排队**：避免瞬间压力击穿数据库；
- **并发安全**：防止并发操作导致库存被多扣或重复下单；
- **最终一致性**：确保最终数据库状态与缓存一致；
- **高性能处理**：尽量将请求挡在缓存/队列层，减轻数据库压力。

#### **三、常见解决方案有哪些？**

**1. 数据库层面加锁控制并发**

- **基本原理**：

  - 通过 `select ... for update` 行级锁串行化请求；
  - 在事务中执行库存扣减，避免并发冲突。

- **具体做法**：

  ```sql
  select * from goods where goods_id = ? for update;
  update goods set stock = stock - 1 where goods_id = ? and stock > 0;
  ```

- **合理性分析**：

  - 利用数据库原生事务和行锁机制，保证并发操作的安全性；
  - 能有效防止超卖。

- **缺点**：

  - 性能差：每次访问都要排队等待行锁；
  - 吞吐量低，数据库连接数易耗尽；
  - 无法支撑大规模并发请求。

**2. 分布式锁 + 分段缓存**

- **基本原理**：

  - 用 Redis 分布式锁控制并发；
  - 将库存拆分为多个段，每段独立加锁操作，提高并发度。

- **具体做法**：

  - 将库存 `100` 拆分为 `5` 个 key：`goods-01 ~ goods-05`，每个初始 `20`；
  - 用户下单时 `userId % 5` 选一个 key；
  - 获取锁，执行扣减操作；
  - 若库存不足，自动释放锁后尝试下一个 key。

- **示例**：

  ```sql
  key1=goods-01, value=20;
  key2=goods-02, value=20;
  ...
  ```

- **合理性分析**：

  - 分段加锁减少锁冲突，可支持多个用户同时购买同一商品；
  - 锁粒度更细，系统吞吐量更高。

- **缺点**：

  - 实现复杂：需要自动重试与段内库存协调；
  - 锁释放与切换逻辑繁琐；
  - 数据拆分需合理规划。

**3. Redis 原子指令 + 异步队列**

- **基本原理**：
  - 用 Redis 的 `decr` 实现原子扣减库存；
  - 请求成功后写入消息队列，异步落地数据库；
  - 避免高并发直接打数据库。
- **具体流程**：
  1. 系统启动时将库存同步到 Redis；
  2. 用户秒杀时先执行 `decr(goods-stock)`；
  3. 若库存 < 0，返回失败；
  4. 若成功，入队异步队列；
  5. 异步消费生成订单、扣减数据库库存；
  6. 用户轮询查询结果。
- **合理性分析**：
  - 预扣库存快速响应，Redis 单机支持超高 QPS；
  - 异步队列削峰填谷，提升系统整体稳定性；
  - 原子操作确保并发安全。
- **缺点**：
  - 系统架构复杂度上升，引入队列、轮询机制；
  - 存在数据不一致风险，需要保障消息投递成功；
  - 最终一致性需通过补偿或确认机制保障。
